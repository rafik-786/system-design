1. Card Accordion

Bug 1 Race Condition in Connection Pool

Incident: On 2024-02-15, the order service started returning HTTP 500 errors during peak traffic (12K req/s). PostgreSQL logs showed FATAL: too many connections for role "api_user".

The connection pool had max_pool_size = 20, but under concurrent load, two threads could both see 19 active connections, both open a 20th, and blow past the limit. The fix was double-checked locking with a semaphore:

private readonly SemaphoreSlim _semaphore = new(maxPoolSize);

public async Task<NpgsqlConnection> GetConnectionAsync(CancellationToken ct)
{
    await _semaphore.WaitAsync(ct);
    return await _pool.OpenConnectionAsync(ct);
}
Lesson: Always use a semaphore to guard pool access. if (count < max) is never thread-safe.

Bug 2 Cache Stampede on Expiry

When a popular cache key expired, 8,000 concurrent requests all hit the database simultaneously. The fix: stale-while-revalidate with a background refresh and a distributed lock.

Bug 3 N+1 Query in Order History

Fetching 50 orders triggered 50 individual SELECT queries for related items. The page took 3.2 seconds. Adding .Include(o => o.Items) reduced it to 42ms — one query with a JOIN.

2. Collapsible

Pitfall 1 Using SELECT * on Wide Tables

Fetching all 47 columns when the API only needs id, name, and email. Each row is 2.4 KB instead of 96 bytes — 25× more data over the wire. At 10K rows, that's 23 MB vs 940 KB.

Fix → SELECT id, name, email FROM users

Pitfall 2 Missing Index on Foreign Key

PostgreSQL doesn't auto-create indexes on foreign keys (unlike MySQL). A JOIN on orders.user_id does a sequential scan on 5M rows — 850ms per query.

Fix → CREATE INDEX idx_orders_user_id ON orders(user_id); drops it to 2ms.

7b. Code Window

public class UserService : IUserService, IDisposable
{
    private readonly IDbContext _db;
    private readonly ICache _cache;
    private readonly ILogger _logger;
    private readonly IConfiguration _config;

    public UserService(IDbContext db, ICache cache, ILogger logger, IConfiguration config, IHttpClientFactory httpFactory, IMemoryCache localCache)
    {
        _db = db;
        _cache = cache;
        _logger = logger;
        _config = config;
    }

    public async Task GetByIdAsync(int id, CancellationToken ct = default)
    {
        // Check cache first — O(1) lookup
        var cached = await _cache.GetAsync($"user:{id}", ct);
        if (cached != null) return cached;

        // Cache miss — hit database
        _logger.LogInformation("Cache miss for user:{UserId}, falling back to DB", id);
        var user = await _db.Users.Where(u => u.Id == id && u.IsActive).FirstOrDefaultAsync(ct);
        if (user != null)
            await _cache.SetAsync($"user:{id}", user, new DistributedCacheEntryOptions { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) }, ct);

        return user;
    }

    public async Task> SearchAsync(string query, int page, int pageSize, string sortBy, SortDirection direction, bool includeInactive = false)
    {
        var users = await _db.Users
            .Where(u => u.Name.Contains(query) || u.Email.Contains(query))
            .OrderByDynamic(sortBy, direction)
            .Skip((page - 1) * pageSize)
            .Take(pageSize)
            .ToListAsync();

        return new PagedResult(users, totalCount, page, pageSize);
    }

    public void Dispose() => _db?.Dispose();
}

8. Exercise Card

Medium Design a URL Shortener

Given 100M URLs/day, design a system that generates short URLs and redirects in under 10ms. Your system must handle 100:1 read:write ratio, guarantee uniqueness across distributed nodes, and support custom aliases. Consider: How do you generate 7-character unique keys at 1,150 writes/sec without collisions?

Hint

Base62 encoding of an auto-increment ID gives you 627 = 3.5 trillion unique keys. At 1,150/sec, that lasts 96,000 years. Pre-generate key ranges per node to avoid coordination.

33. Proof Block (#23)

Why Split-Brain Violates Consistency
Given: Two leaders accept writes simultaneously
∴ Both replicate to followers — conflicting state spreads
Therefore, split-brain violates consistency without a quorum mechanism. This is why Raft/Paxos require majority agreement.

36. What-If Card (#26)

What if the primary database goes down during Black Friday peak (25K writes/sec)?

All writes fail immediately — 25K errors/sec. Read replicas serve stale data (up to 4s behind). Shopping carts freeze. Payment confirmations lost. Estimated revenue loss: $47K/minute based on 2023 Black Friday average order of $113.

Recovery plan: (1) RDS Multi-AZ automated failover promotes standby replica in 60-120s. (2) Application retries with exponential backoff absorb the gap. (3) Idempotent payment endpoints prevent double charges on retry. (4) Post-recovery: replay failed writes from Kafka dead-letter queue. Total impact: 90-180s degraded, zero data loss.

38. Callout Trap (#28)

Common Trap
Caching everything sounds great until stale data causes more bugs than slow responses. Shopify cached product inventory for 30 seconds — during flash sales, customers would "buy" out-of-stock items because the cache showed old quantities. The fix wasn't removing the cache — it was reducing TTL to 2s for high-velocity items and using cache invalidation on write. Always ask: "What's the cost of showing stale data for this specific field?"

112. TL;DR Card

What: Define a family of algorithms, encapsulate each one, and make them interchangeable at runtime.

Why: So you can add new payment methods without touching existing checkout code. Adding Google Pay = one new class, zero changes to existing code.

Real example: Stripe's payment processing uses this pattern — credit card, PayPal, bank transfer, SEPA, and 20+ methods all implement the same IPaymentStrategy interface.

115. Concept Card

Polymorphism
The ability of different classes to respond to the same method call in their own way. The caller says "process this payment" — each strategy decides HOW.

116. Decision Compass

🧭 Decision: If your strategies need shared state or lifecycle management, use DI container registration. If they're pure functions, a simple dictionary lookup is cleaner.

118. Pros/Cons Split (L29)

Pros
  • Open for extension, closed for modification
  • Each strategy is independently testable
  • Easy to add new algorithms
Cons
  • More classes/files to manage
  • Client must know which strategy to pick
  • Overkill for 2-3 simple variations

119. Key Takeaway (L30)

Key Takeaway

The Strategy pattern isn't about removing if/else — it's about making your code open for extension. When a new payment method arrives, you write ONE new class. Zero existing code changes. Zero risk to working features.
