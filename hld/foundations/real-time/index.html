Four ways to make data show up the instant it happens — short polling, long polling, Server-Sent Events, and WebSockets. When to pick each, and how they scale.

Like sending a letter and waiting for a reply., and it powers every Google search, every page load, every form submission. Section 2

Why You Need This — The Scenario

You've just been asked to build a team chat application — think a simpler version of Slack. The core requirement sounds easy: when Alice sends a message, Bob should see it immediately. Not in 5 seconds. Not after refreshing the page. Instantly, like a text message on your phone.

You already know how normal websites work. The browser sends a request ("give me this page"), the server responds ("here you go"), and the connection closes. It's like ordering food at a counter — you ask, they hand it to you, transaction done. But chat is different. With chat, the server needs to push new messages to Bob the moment they arrive, even though Bob didn't ask for them. The server needs to tap Bob on the shoulder and say "hey, Alice just said something."

This is the fundamental tension: HTTP was designed for the client to initiate every conversation. The server can't just reach out to the browser unprompted. It's like a waiter who can only bring food when you order — they can't walk up and say "the chef just made something amazing, here you go." So how do we get around this limitation?

Section 3

First Attempt — Just Keep Asking

The simplest solution to the "how does Bob get new messages?" problem is embarrassingly obvious: Bob's browser asks the server "any new messages?" over and over and over again, on a timer. Maybe every second, maybe every 3 seconds. If the server says "nope, nothing new," the browser waits a bit and asks again. If the server says "yes, here's a new message from Alice," the browser shows it on screen.

This technique is called short pollingA technique where the client repeatedly sends HTTP requests at regular intervals (e.g., every 1-5 seconds) to check if new data is available. Simple to implement but wasteful because most requests return empty responses., and it's the most basic form of "real-time" communication. It works the same way a kid in the back seat keeps asking "are we there yet?" — annoying but effective, eventually.

Look at the diagram above. Alice sent her message at the 7-second mark, but Bob didn't find out about it until the 9-second mark, when his next poll happened. That's a 2-second delay. And 3 out of 4 requests returned absolutely nothing useful — they were wasted trips to the server.

short-polling.js 
// Short polling: ask for new messages every 3 seconds
function startPolling() {
  let lastMessageId = 0;

  setInterval(async () => {
    // Every 3 seconds, ask the server "anything new?"
    const response = await fetch(`/api/messages?since=${lastMessageId}`);
    const messages = await response.json();

    if (messages.length > 0) {
      // Got new messages! Show them on screen
      messages.forEach(msg => displayMessage(msg));
      lastMessageId = messages[messages.length - 1].id;
    }
    // If empty, do nothing — we'll ask again in 3 seconds
  }, 3000);
}

startPolling();

The code is dead simple — that's the appeal. A setInterval, a fetch, and you're done. It works. But if you zoom out and think about what's happening at scale, the problems become obvious fast.

The math gets ugly: If you have 10,000 users polling every 3 seconds, that's 3,333 HTTP requests per second hitting your server — and most of them return empty responses. You're paying for bandwidth, CPU cycles, and database queries that produce zero useful work.
Section 4

Where Short Polling Breaks — Death by a Thousand Requests

Short polling works fine on your laptop with 3 test users. But the moment you ship it to production, three brutal problems emerge — and they all get worse as your user count grows.

See the dilemma? You can have fast updates or cheap infrastructure, but not both. The faster you want messages to appear, the more requests you hammer the server with. And most of those requests come back empty-handed.

Think First

What if instead of Bob repeatedly asking "got anything?", he asked once and the server didn't answer until there actually was something? The server would just... hold the connection open, silently, waiting for new data. How would that change the math?

This one idea — "don't respond until you have something to say" — is the entire foundation of long polling.

That insight — "just don't respond until there's real data" — is the first major breakthrough in real-time communication. It reduces wasted requests to nearly zero, and it's been powering real-time features since the early 2000s. Let's see how.

Section 5

The Breakthrough — Persistent Connections

The fix for short polling's wastefulness is surprisingly elegant: instead of the server immediately responding "nothing new" to every request, the server holds the request open and waits. It doesn't send a response at all until there's actual data to deliver. The client sends one request, then just sits there, patiently, with the connection open. When new data finally arrives (maybe 30 seconds later, maybe 2 milliseconds later), the server sends its response. The client processes it, then immediately opens a new request, and the cycle repeats.

This technique is called long pollingA variation of polling where the server holds the client's request open until new data is available (or a timeout occurs). This dramatically reduces the number of empty responses compared to short polling. Used by early Facebook Messenger and old-school chat applications., and it was the backbone of real-time features for years before WebSockets existed. Facebook's original chat, early Google Talk, and many other systems used this approach.

The difference is dramatic. With short polling (3-second interval), Bob's browser would have fired 4 requests in those 12 seconds, 3 of which were useless. With long polling, Bob's browser fired just 1 request, and it got a response the instant Alice's message arrived. Zero wasted requests, near-zero latency.

long-polling.js 
// Long polling: one request at a time, server holds until data exists
async function longPoll(lastMessageId = 0) {
  try {
    // This request might hang for 30+ seconds — that's intentional!
    const response = await fetch(`/api/messages?since=${lastMessageId}&wait=true`);
    const messages = await response.json();

    if (messages.length > 0) {
      messages.forEach(msg => displayMessage(msg));
      lastMessageId = messages[messages.length - 1].id;
    }
  } catch (error) {
    // Network error? Wait a bit before retrying
    await new Promise(resolve => setTimeout(resolve, 3000));
  }

  // Immediately start the next long poll
  longPoll(lastMessageId);
}

longPoll();

But long polling isn't perfect. It still uses standard HTTP requests under the hood, which means each cycle involves full HTTP headersMetadata sent with every HTTP request and response — things like Content-Type, cookies, authentication tokens, caching directives. These headers typically add 200-800 bytes of overhead per request, which is wasteful when you're sending tiny messages like "user is typing..." (easily 500+ bytes of overhead per message). The connection is also one-directional per cycle: the server can only push data to the client when responding to an existing request. If the client needs to send something (like a chat message), that requires a separate HTTP request.

This brings us to an even better idea: what if the connection never closed at all? What if both sides could send data whenever they wanted, with zero overhead per message?

The evolution: Short polling → Long polling → SSE/WebSockets. Each step reduces overhead and latency. Long polling was a clever hack that worked within HTTP's limitations. The next solutions — SSE and WebSockets — were purpose-built for real-time from the ground up.
Section 6

How Each Technique Works — The Full Picture

Now that you understand the "why" — the journey from "just keep asking" to "keep the connection open" — let's look at each of the four techniques in detail. We'll see exactly what happens on the wire, what the code looks like, and where each one shines or stumbles.

1. Short Polling — The Brute-Force Approach

We covered this in Section 3, but let's add the complete picture. Short polling is the most basic form of "real-time" — the client sends regular HTTP requests on a timer, and the server responds immediately with whatever data is available (or an empty response).

When to use it: Prototypes, admin dashboards that refresh every 30 seconds, or any situation where "good enough" latency (seconds, not milliseconds) is acceptable and simplicity matters more than efficiency.

2. Long Polling — The Patient Waiter

Long polling flips the script: the client sends a request, and the server holds it open until there's data to send. No timer on the client side, no wasted empty responses. The server is the one who decides when to respond.

The key trade-off: long polling still uses one HTTP request per data delivery. Each response triggers a new request, which means new HTTP headers, a new TCP roundtrip (unless keep-aliveAn HTTP/1.1 feature that reuses the same TCP connection for multiple HTTP requests instead of opening a new connection each time. This saves the overhead of TCP handshakes but doesn't eliminate HTTP header overhead. is active), and new overhead. For applications with frequent updates (like fast-paced games), this overhead adds up.

When to use it: When you need near-instant delivery but WebSocket support is uncertain (old proxies, corporate firewalls), or when your framework makes it easy (e.g., SignalR uses long polling as a fallback). Facebook used this for their original chat in 2008.

3. Server-Sent Events (SSE) — The News Ticker

What if instead of closing the connection after each response, the server just... kept sending data on the same connection, over and over? That's exactly what Server-Sent EventsA browser API (EventSource) that opens a persistent HTTP connection to the server. The server can then push text-based events to the client at any time. It's one-directional (server → client only), auto-reconnects on failure, and works over standard HTTP. Perfect for live feeds, notifications, and dashboards. (SSE) does. The client opens a single HTTP connection, and the server streams events down that connection for as long as it wants — minutes, hours, even days.

Think of it like subscribing to a news ticker. You tune in once, and headlines keep scrolling across your screen without you having to refresh. But it's one-way — you can't talk back through the ticker. If you need to send data to the server (like posting a chat message), you'd use a separate regular HTTP request.

sse-client.js 
// SSE: the browser does most of the work for you
const source = new EventSource('/api/events');

// Listen for specific event types
source.addEventListener('message', (event) => {
  const data = JSON.parse(event.data);
  displayMessage(data);    // Show new chat message
});

source.addEventListener('typing', (event) => {
  showTypingIndicator(event.data);  // "Alice is typing..."
});

source.addEventListener('online', (event) => {
  updateUserList(event.data);  // Someone came online
});

// Auto-reconnects if connection drops!
source.onerror = () => {
  console.log('Connection lost, browser will auto-reconnect...');
};

Each event is a plain-text block separated by a blank line. The event: line names the type, the data: line carries the payload. Incredibly simple format — no binary protocol, no framing, just text.

SSE's killer feature is automatic reconnection. If the connection drops (Wi-Fi glitch, server restart), the browser automatically reconnects and sends a Last-Event-ID header so the server knows where to resume. You get this for free — no code needed. It also supports named event types, so you can have different listeners for "message", "typing", "online" events on the same connection.

When to use it: Live dashboards, stock tickers, notification feeds, news updates — anything where the server pushes data and the client just listens. SSE is simpler than WebSockets, works over standard HTTP (proxy-friendly), and auto-reconnects. If you don't need the client to send data back through the same channel, SSE is usually the right choice.

4. WebSockets — The Full Duplex Channel

WebSockets are the "gold standard" for real-time communication on the web. Unlike SSE (which is server-to-client only), a WebSocketA protocol (RFC 6455) that provides full-duplex (two-way) communication over a single TCP connection. Unlike HTTP, both the client and server can send messages at any time without waiting for a request. The connection starts as HTTP then "upgrades" to the WebSocket protocol. gives you a full two-way channel where both the client and server can send messages at any time, independently, with minimal overhead. It's like having an open phone call — both sides can talk whenever they want.

WebSockets start life as a normal HTTP request, then "upgrade" to a different protocol entirely. Once upgraded, the HTTP overhead disappears — messages are sent as tiny binary framesIn the WebSocket protocol, data is sent in small units called frames. Each frame has just 2-14 bytes of overhead (compared to hundreds of bytes for HTTP headers). This makes WebSockets extremely efficient for frequent, small messages like chat, gaming inputs, or real-time collaboration. with just 2-14 bytes of overhead (compared to 500+ bytes for HTTP headers).

websocket-client.js 
// WebSocket: full two-way communication
const ws = new WebSocket('wss://chat.example.com/ws');

// Connection opened
ws.onopen = () => {
  console.log('Connected! Both sides can talk now.');
};

// Server sends us something
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  if (data.type === 'message') displayMessage(data);
  if (data.type === 'typing')  showTypingIndicator(data.user);
  if (data.type === 'online')  updateUserList(data.user);
};

// We send something to the server (no HTTP request needed!)
function sendMessage(text) {
  ws.send(JSON.stringify({ type: 'message', text }));
}

// Handle disconnection
ws.onclose = () => {
  console.log('Disconnected. Reconnecting...');
  setTimeout(() => connectWebSocket(), 3000);
};

Notice the difference from SSE: the client can call ws.send() at any time to send data through the same connection — no separate HTTP request needed. This makes WebSockets perfect for truly interactive applications like chat, multiplayer games, and collaborative editors where data flows in both directions constantly.

When to use it: Chat applications, multiplayer games, collaborative editing (Google Docs-style), live auctions, trading platforms — any time both client and server need to send messages at any time. WebSockets are the most powerful option, but also the most complex to scale.

Section 7

Going Deeper — WebSocket Internals & Scaling

WebSockets are the most powerful real-time technique, but they're also the trickiest to run at scale. Let's peel back the layers and understand what's actually happening under the hood — the protocol upgrade, connection management, and the scaling challenges that hit you when you go from 100 users to 100,000.

The WebSocket Handshake (Protocol Upgrade)

Every WebSocket connection starts as a normal HTTP request. The client sends a special header saying "I'd like to upgrade this connection to the WebSocket protocol, please." If the server supports it, it responds with "101 Switching Protocols" and from that moment on, the connection speaks WebSocket instead of HTTP.

That 130x reduction in per-message overhead is why WebSockets dominate for high-frequency messaging. If your chat app sends 100 messages per second across all users, the bandwidth savings are enormous.

Scaling WebSockets with Redis Pub/Sub

Here's the challenge nobody tells you about until it's too late: WebSocket connections are statefulA stateful connection means the server must remember information about each connected client (who they are, what rooms they're in, what their last message was). This is the opposite of stateless HTTP, where each request is independent. Stateful connections make horizontal scaling much harder.. Each user has a persistent TCP connection to a specific server. If Alice is connected to Server 1 and Bob is connected to Server 2, and Alice sends a message — how does Bob get it? Server 1 has the message, but Bob isn't connected to Server 1.

The standard solution: use a message brokerA middleware system (like Redis, RabbitMQ, or Kafka) that sits between servers and routes messages. When Server 1 publishes a message, the broker ensures all other servers receive it so they can forward it to their connected clients. Think of it as a central post office that all servers check for new mail. like Redis to relay messages between servers.

This Redis Pub/Sub pattern is what powers most production WebSocket systems. Slack, Discord, and countless chat apps use exactly this architecture. The broker ensures that a message sent to any server reaches every connected client, no matter which server they're on.

Connection Limits & Heartbeats

Each WebSocket connection is a persistent TCP connection, which means it consumes resources on the server: a file descriptorAn operating system handle for an open connection (or file). Each TCP connection uses one file descriptor. Linux defaults to ~1,024 per process, but production servers raise this to 100K-1M. Running out of file descriptors means the server can't accept new connections., some memory for buffers, and a slot in the kernel's connection table. A single server can typically handle 10,000 to 100,000 concurrent WebSocket connections depending on hardware and how much memory each connection needs.

But connections can go stale. A user might close their laptop lid, walk through a dead Wi-Fi zone, or their ISP might silently drop idle connections. To detect these "zombie connections," servers send heartbeatA small, periodic message (often called a "ping") sent between client and server to verify the connection is still alive. If one side doesn't receive a heartbeat response within a timeout (e.g., 30 seconds), it assumes the connection is dead and closes it. WebSocket has built-in ping/pong frames for this. messages — tiny "are you still there?" pings.

Production tip: Set heartbeat intervals to 30-60 seconds. Too frequent wastes bandwidth; too infrequent means zombie connections linger and consume resources. The WebSocket protocol has built-in ping/pong frames specifically for this — use them.

Section 11

Common Pitfalls & Mistakes

Real-time systems have failure modes that catch experienced developers off-guard. Here are the mistakes that burn teams the most often.

Think First

You deploy a WebSocket-based chat app. Everything works perfectly in testing. Then you go live, and users report that messages "disappear" randomly — they send a message but the other person never sees it. What could cause this?

Think about what happens when a connection drops silently, without either side knowing.
Pitfall 1: Not handling reconnection

The mistake: Assuming WebSocket connections stay open forever. In reality, connections drop constantly — mobile users switch from Wi-Fi to cellular, corporate proxies timeout idle connections after 60 seconds, servers restart during deployments.

Why it's bad: Without reconnection logic, users silently stop receiving updates. They see a frozen UI and think the app is broken.

The fix: Always implement exponential backoff reconnection. Start with a 1-second retry, double it each attempt (2s, 4s, 8s...), cap at 30 seconds. Add jitter (random 0-1s) to prevent all clients reconnecting at once after a server restart (a thundering herdWhen a server restarts or recovers from an outage, all disconnected clients try to reconnect at the exact same moment, overwhelming the server before it can stabilize. Adding random jitter (small random delays) to reconnection timers prevents this stampede.).

Pitfall 2: Using WebSockets when SSE would suffice

The mistake: Defaulting to WebSockets for every real-time feature, even when data only flows from server to client.

Why it's bad: WebSockets are harder to scale (stateful connections, need Redis Pub/Sub), harder to debug (binary protocol, not visible in standard HTTP logs), and harder to deploy (some proxies and CDNs don't support them). If all you need is "server pushes updates to client," SSE gives you 90% of the benefit with 20% of the complexity.

The fix: Ask "does the client need to send data through this connection?" If no → SSE. If yes → WebSocket. Dashboard updates, live scores, notification feeds, stock tickers — all SSE candidates.

Pitfall 3: Forgetting about message ordering and delivery guarantees

The mistake: Assuming messages arrive in order and exactly once. With WebSockets, if the connection drops and reconnects, you might miss messages sent during the gap. Or if the server reprocesses a message, you might get duplicates.

Why it's bad: In a chat app, messages appear out of order. In a trading app, you might execute the same trade twice. In a collaborative editor, you get data corruption.

The fix: Assign each message a sequence numberA monotonically increasing integer assigned to each message. When the client reconnects, it tells the server "I last received message #42" and the server replays everything from #43 onward. This ensures no messages are lost during disconnection and duplicates can be detected and ignored. or timestamp. On reconnection, the client sends "I last saw message #42" and the server replays everything from #43. Deduplicate on the client side using message IDs.

Pitfall 4: No backpressure on fast producers

The mistake: A server producing events faster than clients can consume them. If 1,000 stock price updates arrive per second but the client can only render 30 frames per second, the client's message buffer grows without bound.

Why it's bad: Client memory explodes. Browser tabs crash. On mobile, the OS kills your app.

The fix: Implement server-side throttling (send at most 1 update per 100ms per client), or use backpressureA flow control mechanism where the consumer signals the producer to slow down when it can't keep up. In WebSockets, this might mean monitoring the send buffer size and dropping or batching updates when the client is falling behind. Without backpressure, fast producers overwhelm slow consumers. mechanisms where the server detects slow consumers and either batches updates or drops intermediate ones (sending only the latest state).

transport-negotiation.js 
async function connectRealTime() {
  // Try 1: WebSocket (best option)
  if ('WebSocket' in window) {
    try {
      const ws = new WebSocket('wss://api.example.com/ws');
      await waitForOpen(ws, 5000); // 5s timeout
      return { type: 'websocket', connection: ws };
    } catch (e) {
      console.log('WebSocket failed, trying SSE...');
    }
  }

  // Try 2: Server-Sent Events (good for one-way push)
  if ('EventSource' in window) {
    try {
      const sse = new EventSource('/api/events');
      await waitForOpen(sse, 5000);
      return { type: 'sse', connection: sse };
    } catch (e) {
      console.log('SSE failed, falling back to long polling...');
    }
  }

  // Try 3: Long Polling (works everywhere)
  return { type: 'longpoll', connection: startLongPoll() };
}

This is essentially what Socket.IO and SignalR do internally. The key insight: start with the best, gracefully degrade.

Exercise 4: Spot the Bug

This long-polling implementation has a critical bug that will eventually crash the browser. Can you find it?

buggy-long-poll.js 
async function longPoll() {
  try {
    const res = await fetch('/api/events?wait=true');
    const data = await res.json();
    handleData(data);
  } catch (error) {
    console.error('Poll failed:', error);
  }
  longPoll();  // Immediately retry
}
Solution

The bug: When the server is down, every fetch() fails immediately (no 30-second hold). The catch block runs and immediately calls longPoll() again, which fails instantly again. This creates an infinite loop of rapid-fire failed requests — thousands per second — which will crash the browser tab and DDoS your own server.

The fix: Add a delay in the catch block before retrying:

fixed-long-poll.js 
async function longPoll() {
  try {
    const res = await fetch('/api/events?wait=true');
    const data = await res.json();
    handleData(data);
    longPoll();  // Success: retry immediately
  } catch (error) {
    console.error('Poll failed:', error);
    await sleep(3000);  // Failure: wait 3s before retry!
    longPoll();
  }
}

Solution

Transport: SSE for browser delivery (notifications are server → client only; users don't need to send data back through the notification channel). WebSocket is overkill here.

Architecture: 2M connections at ~10KB each = ~20GB total. With 8GB/server at 50% utilization: ~5 servers for SSE connections. Use a connection registry (Redis hash: userId → serverId) so when a notification is generated, the system knows which server to route it to. Use Redis Pub/Sub to broadcast to the correct server.

Offline users: Store undelivered notifications in a database. When the user reconnects, they fetch missed notifications via a regular REST API call (GET /notifications?since=lastSeen).

Scaling: The notification generation service is separate from the notification delivery service. Generation writes to a queue (Kafka/RabbitMQ). Delivery workers consume from the queue, look up the user's server in Redis, and push through SSE.

Section 13

Cheat Sheet

Short Polling

Client asks on a timer. Simple, wasteful. Best for prototypes and slow-update dashboards.

Long Polling

Server holds request until data exists. Near-instant delivery. Fallback when WebSocket/SSE blocked.

SSE

One-way stream, server → client. Auto-reconnect. Perfect for feeds, tickers, notifications.

WebSocket

Full duplex, 2-byte overhead. Chat, gaming, collaboration. Hardest to scale (stateful).

Scaling WS

Redis Pub/Sub to relay messages between servers. Connection registry to route to right server.

Production Rule

Use a library (Socket.IO, SignalR) that auto-negotiates transport. Never hand-roll fallback logic.

Reconnection

Exponential backoff + jitter. Start 1s, double each retry, cap at 30s. Prevents thundering herd.

Message Ordering

Assign sequence numbers. On reconnect, send last-seen ID. Server replays missed messages.

Interview Tip

Always mention fallback chain and how you'd handle reconnection. Shows you think about reliability, not just the happy path.

Section 14

Interview Pitch — 30-Second Answer

When asked "How would you add real-time features to this system?"

"There are four main approaches to real-time on the web, each with different trade-offs. Short polling is simplest — the client asks on a timer — but it wastes bandwidth and adds latency. Long polling improves this by having the server hold the request until data exists, giving near-instant delivery. Server-Sent Events open a persistent one-way channel from server to client — great for feeds and notifications, with built-in auto-reconnect. WebSockets give full two-way communication with minimal overhead — the gold standard for chat, gaming, and collaboration."

"For this system, I'd choose [your pick] because [reason based on data direction, frequency, and scaling needs]. In production, I'd use a library like SignalR or Socket.IO that auto-negotiates the best transport and falls back gracefully. For scaling, I'd use Redis Pub/Sub to relay messages across multiple server instances, with heartbeats to detect and clean up dead connections."
