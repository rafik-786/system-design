DNS — The Internet's Phone Book

Every time you type a URL, a hidden lookup turns that human-friendly name into an IP address a computer can actually reach. That lookup is DNS — and if it breaks, the entire internet feels broken.

TL;DR — The One-Minute Version

Mental Model: DNS is the internet's phone book. You know your friend's name ("google.com"), but the internet only understands numbers ("142.250.80.46"). DNS translates names into numbers so you never have to memorize IP addresses. It's hierarchical (like a tree), heavily cached (so lookups are fast), and distributed across thousands of servers worldwide (so no single failure kills it).

Think about how you use contacts on your phone. You tap "Mom" — you don't dial 555-012-3456 from memory. DNS does the same thing for the internet. When you type www.google.com into your browser, DNS figures out which IP addressAn IP (Internet Protocol) address is a unique number that identifies every device on the internet — like a street address for computers. Example: 142.250.80.46 (IPv4) or 2607:f8b0:4004:800::200e (IPv6). that name points to, so your browser knows where to send its request.

The genius of DNS is that it's distributed — no single server holds all the answers. Instead, responsibility is split across a hierarchy: root servers at the top, then servers for .com, .org, .io, and so on, then servers for individual domains. And every step along the way, results get cachedCaching means storing a copy of data closer to where it's needed so you don't have to fetch it from the original source every time. DNS caches exist in your browser, your OS, your router, and your ISP. — so most lookups never travel the full chain.

One-line takeaway: DNS translates human-readable domain names into machine-readable IP addresses, using a distributed hierarchy of servers and aggressive caching to do it billions of times per day.

Section 5

The Breakthrough — Hierarchical Naming

In 1983, Paul Mockapetris published RFC 882 and RFC 883RFCs (Request for Comments) are the official documents that define how internet protocols work. RFC 882 and 883 defined the original DNS specification. They were later superseded by RFC 1034 and 1035, which are still the foundation of DNS today., and DNS was born. The breakthrough idea was beautifully simple: don't put everything in one place — organize names into a tree, and let each branch of the tree manage itself.

Think about how street addresses work. "123 Main Street, Springfield, Illinois, USA" isn't stored in one giant global address book. Instead:

  • The USA knows about its 50 states
  • Illinois knows about its cities
  • Springfield knows about its streets
  • Main Street knows about its house numbers

Nobody has to know everything. Each level only knows about the level directly below it. That's exactly how DNS works.

This solved every problem from the hosts file era:

How the Tree Fixes Everything
  • No single point of failure — thousands of servers share the work, each responsible for their slice of the tree
  • No name collisions — "mail.google.com" and "mail.yahoo.com" are totally different because they live in different branches
  • No bandwidth problem — you only ask for the one name you need, not the whole file
  • Distributed authority — Google controls everything under google.com. Wikipedia controls everything under wikipedia.org. No central gatekeeper needed

The name for each part of this tree has a specific term. The invisible dot at the very top is the root zoneThe root zone is the very top of the DNS tree. It's represented by a single dot (.) and managed by 13 root server clusters operated by organizations like ICANN, NASA, the US Army, and universities. Every DNS query ultimately traces back to root.. The next level down — .com, .org, .uk — are called Top-Level Domains (TLDs)TLDs are the highest level of domain names in the DNS hierarchy. They include generic TLDs (.com, .org, .net), country-code TLDs (.uk, .de, .jp), and newer ones (.io, .dev, .app). There are over 1,500 TLDs today.. Below that are the domains you recognize — google, github, wikipedia — each managed by whoever registered them. And below those are subdomainsA subdivision of a domain. In "mail.google.com," "mail" is a subdomain of "google.com." Domain owners can create unlimited subdomains — www, api, staging, dev — without asking anyone's permission. like www, mail, and api.

Section 6

How It Works — The Full Resolution Journey

When you type www.example.com into your browser, a chain of lookups happens behind the scenes — usually in under 50 milliseconds. Let's meet every player involved, then watch the full journey end-to-end.

The Recursive Resolver — Your DNS Agent

When your browser needs to resolve a domain name, it doesn't go hunting through the DNS tree itself. Instead, it asks a recursive resolverA DNS server (usually run by your ISP or a public provider like 8.8.8.8) that does all the legwork of querying the DNS tree on your behalf. You ask it one question; it chases down the answer through multiple servers and hands it back to you. — a middleman server whose entire job is to do the lookup on your behalf.

Think of it like calling a librarian. You say "I need the phone number for John Smith at 123 Main St, Springfield, IL." You don't walk through the library yourself — the librarian does the searching and comes back with the answer. That's the recursive resolver.

Your computer is pre-configured to know the IP address of at least one recursive resolver. Usually it's your ISP's resolver (set up automatically when you connect to the internet), but you can change it to a public resolver like Google's 8.8.8.8 or Cloudflare's 1.1.1.1.

Root Name Servers — The Starting Point

The recursive resolver's first stop is always a root name serverThe 13 root server clusters (named A through M) are the starting point for every DNS lookup. They don't know individual domain IPs — they only know which servers handle each TLD (.com, .org, .uk, etc.). There are actually hundreds of physical servers behind these 13 addresses, spread across the globe using anycast.. Root servers don't know the IP address of www.example.com — they only know one thing: "Who's in charge of .com?"

There are 13 root server clusters, labeled A through M. But "13" is misleading — behind those 13 addresses are over 1,700 physical servers spread across every continent (yes, even Antarctica research stations can reach them). They use a clever networking trick called anycastA routing technique where the same IP address is announced from multiple locations around the world. When you send a packet to that IP, the internet's routing system automatically sends it to the nearest physical server. This is how 13 root server IPs can serve the entire planet. to route you to the nearest one.

TLD Name Servers — The Category Index

Now the resolver knows which server handles .com. It goes there next and asks: "Who handles example.com?" The TLD serverTLD (Top-Level Domain) name servers are responsible for an entire domain extension like .com, .org, or .uk. Verisign operates the .com and .net TLD servers. Each country manages its own country-code TLD (e.g., .uk, .de, .jp). doesn't know the final IP address either — but it knows which servers are authoritative for that specific domain.

Think of it like a library's section index. The root server told you "fiction is on the 3rd floor." The 3rd floor index tells you "books by Author X are in aisle 7." You're getting closer, but you haven't found the exact book yet.

The .com TLD is managed by Verisign and handles lookups for over 160 million registered .com domains. It's one of the busiest parts of the DNS infrastructure.

Authoritative Name Servers — The Final Answer

This is the server that actually knows the answer. The authoritative name serverThe DNS server that holds the actual, official records for a domain. When you buy "example.com" and set up DNS records (A, CNAME, MX, etc.), those records live on your authoritative name server. It's the "source of truth" for that domain. for example.com holds the real records — the actual IP addresses, mail server addresses, and other data for that domain.

When you register a domain and set up DNS (through services like Cloudflare, Route 53, or GoDaddy), you're configuring records on an authoritative name server. This is the "source of truth" — if it says www.example.com points to 93.184.216.34, then that's the official answer.

The authoritative server responds with the IP address and a TTLTime-To-Live — a number (in seconds) that tells caches how long they can keep this answer before they must ask again. A TTL of 300 means "cache this for 5 minutes." Low TTL = fresher data but more DNS traffic. High TTL = less traffic but slower updates. (time-to-live) value that says "you can cache this answer for X seconds." That's the final piece of the puzzle.

DNS Caching — Why Most Lookups Are Instant

If every DNS lookup required traveling root → TLD → authoritative, the internet would be painfully slow. That's where caching saves the day. DNS responses are cached at multiple levels, so most lookups get answered instantly from a nearby copy.

The first time you visit google.com, the full chain runs. But for the next 5 minutes (or whatever the TTL says), every subsequent lookup is answered instantly from cache — no network round-trips at all. Since popular sites like Google and Facebook set TTLs of 60-300 seconds, and millions of people visit them constantly, the ISP resolver almost always has a cached copy ready.

Going Deeper — Record Types, TTL & GeoDNS

DNS doesn't just map names to IP addresses. A domain can have many different kinds of records, each serving a different purpose. When the authoritative server answers a query, it returns a specific record typeDNS records are key-value entries stored on authoritative name servers. Each record has a type (A, AAAA, CNAME, MX, etc.), a name (the domain or subdomain), a value (IP address, hostname, etc.), and a TTL (how long to cache it). depending on what was asked.

TTL (Time-To-Live)A value in seconds attached to every DNS record that tells caches "you can use this answer for this long, then you must ask again." Common values: 300 (5 min) for dynamic services, 86400 (24 hours) for stable records. is the number that controls how long a DNS answer stays cached. It's a trade-off between two competing goals:

GeoDNS — Location-Aware Resolution

Here's a problem: your users are spread across the globe, but your servers are in specific regions. A user in Tokyo shouldn't be routed to a server in Virginia — that's 200+ ms of unnecessary latency. GeoDNSA DNS technique where the authoritative server returns different IP addresses based on the geographic location of the user making the query. A user in Asia gets routed to an Asian server, a user in Europe gets a European server — same domain name, different IPs. solves this by returning different IP addresses depending on where the user is located.

This is how CDNsContent Delivery Networks (like Cloudflare, Akamai, CloudFront) use GeoDNS to route users to the nearest edge server. When you load an image from a CDN, DNS resolves to a server in your region — that's why CDN-served content feels instant. work under the hood. When Netflix serves you a movie, GeoDNS ensures the video streams from a server near you, not from a server on the other side of the planet.

DNS Propagation — Why Changes Take Time

You update your domain's IP address in your DNS provider dashboard. You hit save. Then you check — and nothing changed. You check again 10 minutes later. Some users see the new IP, others still see the old one. This maddening experience is called DNS propagationThe process of DNS changes spreading across the internet's cache hierarchy. Since every resolver, ISP, browser, and OS has cached the old answer with its own TTL countdown, they won't ask for the new answer until their cache expires. This is why DNS changes "propagate" gradually — different caches expire at different times..

It's not that DNS is "slow to propagate" — it's that thousands of caches around the world each have their own countdown timer, and they all started at different times. There's no "flush all caches" button for the internet. You just have to wait for each cache to expire naturally.

Section 8

Variations — Different Flavors of DNS

The basic DNS we've covered so far is "plain old DNS" — queries travel over UDPUser Datagram Protocol — a lightweight network protocol that sends packets without establishing a connection first. It's fast but unreliable (packets can be lost). DNS uses UDP because queries are tiny and speed matters more than guaranteed delivery. on port 53, completely unencrypted. But the DNS ecosystem has evolved to solve different problems. Here are the key variations you'll encounter.

Public DNS vs Private DNS

Public DNS is what your browser uses to resolve any domain on the internet. Google (8.8.8.8), Cloudflare (1.1.1.1), and your ISP all run public resolvers. Anyone can query them.

Private DNS is used inside companies to resolve internal names that don't exist on the public internet. For example, a company might have jenkins.internal.corp or db-primary.prod.internal — names that only make sense inside their network.

Most cloud providers offer private DNS zones. AWS has Route 53 Private Hosted Zones, GCP has Cloud DNS Private Zones, and Azure has Private DNS Zones. They let services find each other by name without exposing internal infrastructure to the public internet.

DNS-over-HTTPS (DoH) & DNS-over-TLS (DoT)

Traditional DNS has a big privacy problem: queries travel in plain text. Your ISP (and anyone snooping on the network) can see every domain you visit. Even if the website itself uses HTTPS, the DNS query to find that website's IP is completely visible.

DNS-over-HTTPS (DoH) wraps DNS queries inside regular HTTPS traffic on port 443. To anyone watching the network, it looks like normal web browsing. Firefox and Chrome support this natively.

DNS-over-TLS (DoT) encrypts DNS queries using TLSTransport Layer Security — the encryption protocol that makes HTTPS work. It creates an encrypted tunnel between two computers so that no one in between can read the data being exchanged. on port 853. It's the same encryption HTTPS uses, but on a dedicated port.

Split-Horizon DNS

Sometimes the same domain name should resolve to different IPs depending on whether the query comes from inside or outside the company network. This is called split-horizon (or split-brain) DNS.

For example, api.mycompany.com might resolve to a public IP (203.0.113.50) for external users, but to a private IP (10.0.1.50) for employees on the corporate network — skipping the load balancer and firewall for faster internal access.

This is commonly used in corporate environments, especially with tools like AWS Route 53Amazon's managed DNS service. It supports public hosted zones (internet-facing), private hosted zones (VPC-only), health checks, failover routing, GeoDNS, and weighted routing. It's one of the most popular DNS services for cloud infrastructure. private hosted zones or CoreDNS in Kubernetes clusters.

AWS Route 53 — DNS as Infrastructure

Amazon's Route 53 (named after DNS port 53) is more than a DNS service — it's a traffic management platform. Beyond basic resolution, it offers:

  • Health checks — Route 53 pings your servers. If one goes down, it automatically stops routing traffic there
  • Weighted routing — send 90% of traffic to the main server, 10% to a canary deployment
  • Latency-based routing — like GeoDNS but based on actual latency measurements, not just geography
  • Failover routing — primary/secondary setup where the secondary only gets traffic if the primary's health check fails

This turns DNS from a simple name lookup into a powerful tool for load balancingDistributing incoming traffic across multiple servers so no single server gets overwhelmed. DNS-based load balancing is the cheapest kind — it doesn't require a dedicated load balancer device — but it's also the crudest (no per-request intelligence, limited by TTL caching). and failover — all without the user noticing anything.

The Dyn DDoS Attack (2016) — When DNS Goes Down

On October 21, 2016, a massive DDoS attack targeted Dyn, a major DNS provider. The attack used a botnetA network of compromised devices (often IoT devices like cameras, routers, and DVRs) controlled by an attacker. The Mirai botnet that attacked Dyn consisted of ~100,000 infected devices, each flooding Dyn's DNS servers with fake queries. called Mirai — roughly 100,000 compromised IoT devices (security cameras, routers, DVRs) all flooding Dyn's DNS servers with fake queries simultaneously.

Section 10

The Anti-Lesson — When DNS Is the Wrong Tool

DNS is powerful, but people sometimes try to use it for things it was never designed to do. Here's when DNS is the wrong answer.

Think First

Your team wants to use DNS-based load balancing to distribute traffic evenly across 10 servers. Sounds reasonable — each DNS query returns a different server IP using round-robin. What could go wrong?

DNS Load Balancing Has No Brains

DNS round-robin returns server IPs in rotation: request 1 gets Server A, request 2 gets Server B, etc. But it has zero awareness of what's actually happening:

  • No health checks — DNS keeps returning a dead server's IP until someone manually removes it
  • No load awareness — one server might be drowning while another is idle, DNS doesn't care
  • Caching ruins distribution — ISP resolvers cache the answer, so millions of users behind one ISP all hit the same server
  • No session affinity — a user might hit Server A for login, then Server B for the next page (different session)

For real load balancing, use an actual load balancer (HAProxy, Nginx, AWS ALB). DNS can do coarse geographic routing (GeoDNS), but per-request balancing? That's not DNS's job.

DNS as a Service Discovery Hack

In microservice architectures, some teams try to use DNS for service discovery — Service A needs to find Service B, so they resolve service-b.internal via DNS. This works for simple cases, but breaks down when:

  • Services scale up/down frequently (DNS caches stale IPs)
  • You need metadata about the service (version, health status, capabilities)
  • You need sub-second failover (DNS TTL is too slow)

For dynamic microservices, use proper service discovery tools like Consul, etcd, or Kubernetes DNS (which has very low TTLs and tight integration with the container lifecycle).

Ultra-Low TTL = Pretending DNS Is Real-Time

Setting TTL to 1 second because you want "instant" failover is a trap. Many resolvers and operating systems enforce minimum TTLs (often 30-60 seconds) regardless of what you set. Java's default DNS cache is infinite (it caches forever until the JVM restarts). Browser DNS caches have their own minimum TTLs.

If you need sub-minute failover, DNS alone won't do it. Combine DNS with health-check-based load balancers, or use tools like Route 53 health checks which can failover within 30-60 seconds by using low TTLs and active health monitoring together.

The bottom line: DNS is a naming system, not a load balancer, not a service mesh, not a real-time health monitor. It's excellent at what it does — mapping names to addresses — but forcing it into other roles leads to fragile, unpredictable behavior.
Section 11

Common Mistakes — What Engineers Get Wrong

Mistake #1: "I changed the DNS record, why hasn't it updated?"

What people think: Changing a DNS record is like flipping a switch — instant everywhere.

The truth: Every cache in the chain (browser, OS, router, ISP resolver) holds the old answer until its TTL expires. If your old TTL was 86,400 seconds (24 hours), some users will see the old IP for up to 24 hours after your change.

The fix: Lower TTL to 60 seconds days before the change. Wait for the old high TTL to expire everywhere. Then make the change. Then raise TTL back when the migration is confirmed.

Mistake #2: Using CNAME at the zone apex

What people try: Setting a CNAME recordA CNAME (Canonical Name) record creates an alias — it says "this name is actually just another name for THAT name." The DNS resolver follows the chain until it reaches an A record with an actual IP. CNAMEs can't coexist with other record types at the same name. for the naked domain (example.com) pointing to a load balancer (lb.provider.com).

Why it breaks: The DNS spec (RFC 1034) says CNAME records cannot coexist with other record types at the same name. But the zone apex (example.com) MUST have NS and SOA records. So a CNAME there violates the spec, and most DNS servers reject it.

The fix: Use an ALIAS or ANAME record (a non-standard but widely supported extension), or use Cloudflare's "CNAME flattening." These behave like CNAMEs but resolve to A records at query time, avoiding the spec violation.

Mistake #3: Single DNS provider with no redundancy

What people do: Use one DNS provider because "DNS is boring, it just works."

What happens: That provider gets DDoS'd (Dyn 2016), has an outage (Cloudflare 2020), or makes a config error (Facebook BGP incident 2021 — took out their own DNS servers), and your entire online presence vanishes.

The fix: Configure at least two NS records pointing to different providers. Many registrars let you set NS records from multiple providers. Tools like NS1, Cloudflare, and Route 53 all support secondary/backup configurations.

Mistake #4: Forgetting that DNS uses UDP (mostly)

What people assume: DNS is reliable like HTTP.

The truth: DNS queries use UDP, which has no built-in retry or acknowledgment. If the packet is lost, it's just gone. Your OS has its own retry logic (typically 2-3 retries with increasing timeouts), but if all retries fail, you get ERR_NAME_NOT_RESOLVED.

Why it matters: In cloud environments with aggressive network policies, DNS UDP packets can be dropped by overloaded conntrack tablesConnection tracking tables in Linux firewalls that track every network connection. In high-traffic environments (especially Kubernetes), these tables can fill up, causing random packet drops — DNS queries being a common victim since they're small, frequent, and use UDP.. This causes intermittent, maddening DNS failures. If you're running Kubernetes, you've probably hit this.

Mistake #5: Not understanding negative caching

What people don't realize: DNS caches "this domain doesn't exist" responses too. This is called negative cachingWhen a DNS query returns NXDOMAIN (domain does not exist), resolvers cache that non-existence answer too, governed by the SOA record's minimum TTL. This prevents repeated lookups for domains that don't exist, but it also means a typo in your DNS config can haunt you even after you fix it..

The scenario: You try to access api.newservice.com before setting up the DNS record. The resolver caches "NXDOMAIN" (doesn't exist). You then add the record. But the resolver still returns "doesn't exist" for the next 5-60 minutes because it cached the negative result.

The fix: Always set up DNS records before you need them. If you're hit by negative caching, flush your local DNS cache (ipconfig /flushdns on Windows, sudo dscacheutil -flushcache on macOS) and wait for upstream caches to expire.

Interview Playbook — How DNS Shows Up

DNS almost never gets its own dedicated interview question. Instead, it comes up naturally when you're designing any system. Here's how to bring it up and how deep to go at each level.

What They Ask

"What happens when you type a URL into a browser?"

What They Want to Hear

  • Browser checks its DNS cache
  • If not cached, asks the OS, which asks a recursive resolver
  • Resolver walks the DNS tree: root → TLD → authoritative
  • Gets back an IP address, browser connects via TCP/HTTPS

Sample Answer

"First, the browser needs to turn the domain name into an IP address — that's DNS. It checks its local cache first. If the cache is empty, it asks the operating system, which asks a recursive resolver — usually your ISP's or a public one like 8.8.8.8. The resolver walks the DNS hierarchy: root servers point to .com servers, which point to example.com's authoritative servers, which return the actual IP. The resolver caches the result based on the TTL. Then the browser opens a TCP connection to that IP, does a TLS handshake for HTTPS, and sends the HTTP request."

What They Ask

"How would you handle DNS for a globally distributed application?"

What They Want to Hear

  • GeoDNS / latency-based routing to direct users to nearest region
  • Health checks + automatic failover (Route 53 style)
  • Low TTLs during migrations, higher TTLs for stability
  • Multiple DNS providers for redundancy

Sample Answer

"I'd use a managed DNS service like Route 53 or Cloudflare with latency-based routing. Users in Asia resolve to our Singapore servers, Europeans get Frankfurt, Americans get Virginia. Each record has health checks — if Singapore goes down, Route 53 automatically stops returning that IP and falls back to the next closest healthy region. I'd keep TTLs at 60 seconds so failover happens within a minute. And I'd configure a secondary DNS provider as backup, so even if our primary DNS gets DDoS'd, resolution still works."

What They Ask

"Our P99 latency spiked and we traced it to DNS resolution. How would you investigate and fix it?"

What They Want to Hear

  • Check if it's internal DNS (Kubernetes CoreDNS, etc.) or external
  • Investigate conntrack table overflows in Kubernetes (common cause)
  • Check for negative caching issues or stale records
  • Consider local DNS caching (nodelocaldns in k8s, nscd on Linux)
  • Measure: dig +stats, systemd-resolve --statistics, Prometheus metrics for CoreDNS

Sample Answer

"First, I'd isolate whether it's internal DNS (service-to-service within the cluster) or external. For internal, the usual culprit in Kubernetes is CoreDNS getting overwhelmed or conntrack table overflows dropping UDP packets. I'd check CoreDNS pod resource usage, look at conntrack entries with `conntrack -S`, and check for SERVFAIL or timeout rates in CoreDNS metrics. Quick wins: deploy NodeLocalDNS cache (DaemonSet that runs a cache on every node), increase CoreDNS replicas, or tune ndots in resolv.conf to reduce unnecessary search domain expansions. For external DNS, I'd check if we're hitting rate limits on our provider, consider adding a local unbound caching resolver, and verify TTLs aren't set absurdly low."

Interview tip: Whenever you draw a system design diagram, mention DNS at the very first step. "Users hit our domain, which goes through DNS — we use GeoDNS with Route 53 for regional routing and failover." It shows you think about the full request path, not just the backend.
Section 13

Exercises — Hands-On Practice

Exercise 1: Trace a Real DNS Lookup Beginner

Open your terminal and run dig +trace www.google.com (Linux/macOS) or nslookup -type=A www.google.com (Windows). Observe each step of the resolution chain. Identify the root server, TLD server, and authoritative server in the output.

Hint

With dig +trace, you'll see 3-4 groups of responses. The first group comes from root servers (look for lines ending in .), the second from .com TLD servers, and the final group from Google's authoritative servers (ns1.google.com, etc.). The last A record in the output is the final IP address.

Exercise 2: Explore Record Types Beginner

Use dig or nslookup to look up the A, MX, NS, and TXT records for google.com. For each record type, write one sentence explaining what it tells you about Google's infrastructure.

Hint

Try: dig google.com A, dig google.com MX, dig google.com NS, dig google.com TXT. The MX records show multiple mail servers with priority numbers — lower priority = preferred server. The TXT records reveal SPF email verification and various service verifications.

Exercise 3: TTL Experiment Intermediate

Pick 5 popular websites and look up their A record TTLs using dig. Compare the TTLs. Which sites use low TTLs? Which use high TTLs? Hypothesize why each site chose its TTL value based on what you know about their infrastructure.

Hint

The TTL value appears in the answer section of dig output (the number between the domain name and the record type). Sites behind CDNs (Cloudflare) often have low TTLs (60-300s) because they want flexibility to reroute traffic. Sites with stable infrastructure might have TTLs of 3600s or higher. Note: the TTL you see may be less than the original because your resolver has been counting down from when it cached the record.

Exercise 4: Design a DNS Failover Strategy Advanced

You're running a critical API at api.yourcompany.com with servers in US-East and EU-West. Design a DNS configuration that: (1) routes users to the nearest region, (2) fails over to the other region if one goes down, (3) propagates the failover within 2 minutes. Write out the DNS records, TTLs, and health check configuration you'd use.

Hint

Use Route 53 with latency-based routing + health checks. Create two A records for api.yourcompany.com, each pointing to a different region's IP, with routing policy set to "latency." Attach health checks that ping each region every 10 seconds. Set TTL to 60 seconds. When a health check fails, Route 53 stops returning that IP. With 60s TTL + 30s health check detection, failover happens within ~90 seconds. Consider also adding a failover policy as a backup strategy.

Exercise 5: Debug a DNS Outage Expert

Scenario: Your Kubernetes cluster is experiencing intermittent DNS failures. About 1 in 50 DNS queries from application pods are timing out. The application logs show SERVFAIL responses. CoreDNS pods show normal CPU/memory usage. The issue started after a traffic spike. Walk through your debugging process step by step, including the exact commands you'd run.

Hint

Classic conntrack table overflow. Steps: (1) Check conntrack -S for "insert_failed" counters. (2) Check dmesg | grep conntrack for "table full" messages. (3) Increase conntrack table size: sysctl -w net.netfilter.nf_conntrack_max=262144. (4) Long-term: deploy NodeLocalDNS (runs a DNS cache on every node as a DaemonSet, bypassing conntrack for local queries). (5) Also check resolv.conf ndots setting — if ndots is 5 (Kubernetes default), every query expands to 5-10 DNS queries (trying each search domain). Setting ndots: 2 can reduce DNS traffic dramatically.

Connected Topics — Where to Go Next

DNS touches almost every other topic in system design. Here are the most important connections.

Load Balancing

DNS is the first layer of load balancing — routing users to regions. Then hardware/software LBs take over for per-request distribution.

CDNs & Edge Computing

CDNs rely on GeoDNS to route users to the nearest edge server. Understanding DNS is key to understanding how CDN routing works.

Caching Strategies

DNS is one of the internet's most aggressive caching systems. The TTL-based cache hierarchy mirrors patterns used in application-level caching.

Networking Fundamentals

TCP/IP, UDP, anycast routing, BGP — the networking concepts that make DNS work under the hood. Essential for debugging DNS issues.

Service Discovery

Kubernetes DNS, Consul, etcd — how microservices find each other. DNS is the simplest form of service discovery.

Security & DDoS Protection

DNS amplification attacks, DNSSEC, DNS-based firewalls — the security landscape around the internet's most critical infrastructure.
