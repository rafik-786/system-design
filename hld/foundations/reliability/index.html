Core Principles

Reliability & Fault Tolerance

Things will break. Hardware fails, networks split, software crashes. The question isn't "will it happen?" — it's "will your system survive when it does?"

Section 1

TL;DR — The One-Minute Version

Mental Model: Reliability means your system keeps working correctly — even when things go wrong. A reliable system is like a car with a spare tire: when one tire blows, you don't abandon the car on the highway. You swap the tire and keep driving.

Here's the core idea. Every part of your system — servers, databases, networks, even the humans operating them — will eventually fail. Hard drives die — Google's landmark 2007 study of 100,000 drives found an annual failure rate of 1.7% for new drives, rising to over 8% after three years. Software has bugs. Someone will accidentally delete a database table at 2 AM — that's exactly what happened at GitLab in January 2017A GitLab engineer accidentally ran rm -rf on the production database directory instead of the replica. They lost 6 hours of data for 5,000 projects. The incident was so dramatic that GitLab live-streamed the recovery on YouTube., when an engineer ran rm -rf on the wrong database directory and wiped 300 GB of production data. That's not pessimism; it's just statistics. Run enough hardware for enough time, and failures are a mathematical certainty.

An unreliable system treats failure as a catastrophe. When one thing breaks, users see errors, data gets lost, and someone gets paged at 3 AM to manually fix things. A fault-tolerantA system's ability to keep operating correctly even when one or more of its components fail. The system "tolerates" the fault without the user noticing. system treats failure as routine. When one thing breaks, a backup takes over automatically, users don't even notice, and the engineer can fix the problem during business hours with a coffee in hand.

The difference between these levels isn't magic — it's engineering choices made before the failure happens. You add backup servers. You replicate your data. You build automatic detection so your system can heal itself. Netflix sits on the far right of that spectrum — they intentionally kill their own production servers every day using a tool called Chaos Monkey, and their customers never notice. Amazon's shopping cart is designed to survive entire datacenter outages. Every technique in this page is about moving your system to the right side of that spectrum.

One-line takeaway: Reliability isn't about preventing failures — it's about making failures boring. The system handles them automatically, and nobody loses sleep.

The First Attempt — "Just Restart It"

When something breaks in production, the universal instinct is: just restart it. Server crashed? Reboot it. Process hung? Kill and restart. Database unresponsive? Restart the service. And honestly, restarting works surprisingly often. Most software problems are transient — a memory leak, a deadlock, a corrupted connection pool — and a fresh start clears the slate.

So you build a simple playbook: when something breaks, an engineer remotely connects to the server (using a tool called SSHSecure Shell — a way to securely log into a remote server from your own computer and run commands on it, as if you were sitting right in front of it. Usage: ssh user@10.0.1.42), diagnoses the problem, and restarts whatever needs restarting. The first thing they type is probably systemctl status nginx to check if the web server is running, or journalctl -u mysql --since "1 hour ago" to see what went wrong with the database. Maybe dmesg | tail to check for hardware errors or uptime to see if the server recently rebooted on its own. Then: systemctl restart mysql. Problem solved, right?

Section 4

Where It Breaks — Death by Dominoes

The 3 AM scenario we just walked through was bad, but it was simple — one server died, one thing broke. Real-world outages are rarely that clean. The truly terrifying failures happen when one thing breaks, and that causes other things to break, which causes even more things to break, until your entire system collapses like a row of dominoes. Engineers call this a cascading failureWhen one component's failure overloads another component, causing it to fail too, which overloads the next, and so on — like dominoes falling. A single database timeout can bring down your entire system this way..

Here's how it happens. Your system has five services: an API gateway, a user service, an order service, a payment service, and a notification service. They all talk to each other. They all share a database. Everything works great — until one day, the payment service starts responding slowly because of a database lock.

Let's trace the dominoes. The payment service starts responding in 5 seconds instead of 200 milliseconds. The order service calls the payment service on every checkout, and each call now hangs for 5 seconds. The order service has a thread pool of 200 threads. At 40 checkouts per second, all 200 threads are now stuck waiting on payment responses. The order service has no threads left to handle any request — even non-checkout ones like order history.

Now the API gateway is calling the order service and getting timeouts. Its thread pool fills up. Now even the user service and notification service — which have absolutely nothing to do with payments — can't get a connection through the gateway. One slow database query in the payment service just took down your entire system.

This is why the concept of a single point of failureAny component that, if it fails, brings down the entire system. The goal of fault-tolerant design is to eliminate every single point of failure — so no single broken piece can take down the whole. is so dangerous. It's not just "one server dying." It's any place where a failure in one part can ripple outward and destroy everything. A shared database is a single point of failure. A shared thread pool is a single point of failure. A single network switch is a single point of failure. The list is longer than most teams realize.

The scariest part of cascading failures: the root cause is often tiny. A single slow SQL query. A misconfigured timeout. A memory leak that takes 3 days to surface. By the time the dominoes finish falling, the root cause is buried under a mountain of secondary failures, and your team spends hours debugging the wrong thing.
Real-world domino: The 2021 Facebook outage. On October 4, 2021, a routine BGPBorder Gateway Protocol — the "postal system" of the internet. It tells routers how to find paths to every network. When Facebook withdrew its BGP routes, the entire internet forgot how to reach facebook.com, instagram.com, and whatsapp.com. configuration change accidentally withdrew all of Facebook's network routes from the internet. Facebook, Instagram, and WhatsApp vanished for nearly 6 hours. But here's the cascading part: because Facebook's internal tools (including their own access badge system and internal communication) also depended on that same network, engineers couldn't get into the datacenter to fix it. They were literally locked out of their own building. The estimated revenue loss: $60 million. Root cause: one bad config update to their backbone routers.
Think First

In the Facebook outage, engineers couldn't fix the problem because their own access tools depended on the broken system. What principle does this violate, and how would you design around it?

Your recovery tools should NEVER depend on the system they're supposed to recover. If your monitoring, alerting, SSH access, and badge systems all run on the same infrastructure as your product, you've created a "recovery dependency loop." Keep your incident response path on a completely separate infrastructure — a different network, different DNS, different authentication.

Section 5

The Breakthrough — Design for Failure, Not Against It

For decades, engineers tried to build systems that don't fail. Better hardware. Redundant power supplies. Enterprise-grade storage with 5-year warranties. Banks ran on IBM mainframesMassive, ultra-reliable computers costing $500K-$5M+ that run mission-critical workloads for banks, airlines, and governments. A single IBM z16 mainframe can process 19 billion encrypted transactions per day. They achieve 99.999% uptime but at extraordinary cost. costing millions of dollars. Telecom companies used hardware from TandemTandem Computers (now part of HP) built "NonStop" servers in the 1970s-90s with redundant everything — dual CPUs, dual memory, dual I/O. They were designed so that no single hardware failure could cause downtime. Banks and stock exchanges loved them. that literally could not be shut down. The philosophy was: if we make each component reliable enough, the system will be reliable. Spend more money, get more reliability.

Then a company called Netflix blew that idea apart.

Netflix runs on hundreds of thousands of cloud servers. At that scale, they don't ask "will a server fail today?" — they ask "how many servers will fail today?" With that many machines, failures aren't rare events. They're a constant background noise. Waiting for failures and reacting to them isn't a strategy. So Netflix did something radical: they built a tool called Chaos MonkeyA Netflix tool that randomly terminates production servers during business hours to test whether the system can tolerate unexpected failures. If something breaks, the team fixes the weakness before a real outage does the same thing at 3 AM. that randomly kills their own production servers during business hours.

Read that again. They intentionally break their own system. Every day. On purpose. While real customers are watching movies.

Why? Because if your system can survive a random server dying at 2 PM on a Tuesday when your whole team is awake and watching dashboards, then it can definitely survive a server dying at 3 AM on a Saturday when nobody's around. The shift in thinking is profound:

This is the fundamental insight of modern reliability engineering: you cannot prevent all failures, so you must design systems that tolerate them. Instead of asking "how do we stop this from breaking?" you ask "when this breaks — and it will — what happens next?"

This idea goes by many names. Netflix calls it chaos engineeringThe practice of intentionally injecting failures into a production system to find weaknesses before real outages do. Netflix's Simian Army includes Chaos Monkey (kills instances), Latency Monkey (adds delays), and Chaos Kong (simulates entire region failures). Other tools: Gremlin, LitmusChaos, AWS Fault Injection Simulator.. Google calls it "designing for unreliable components" — their famous Site Reliability EngineeringSRE — Google's approach to operations, where software engineers build systems to run reliably at scale. Google's SRE book (free online) defines practices like error budgets, SLOs, and toil reduction that are now industry standard. (SRE) book is built on this idea. Amazon's CTO Werner Vogels famously said: "Everything fails, all the time." But the core principle is the same: build your system so that any single component can die at any moment, and the system keeps running.

The mindset shift in one sentence: A reliable system isn't one where nothing fails. It's one where anything can fail and the user never notices.
Think First

Netflix intentionally kills production servers every day. What would have to be true about their architecture for this to be safe? Think about what properties their system must have.

Every service must have multiple copies running. Data must be replicated so nothing is lost when a server dies. Traffic must be automatically rerouted to healthy servers. Health checks must detect failures in seconds. Essentially — every technique we're about to learn in the next section.
Section 6

How It Works — The Reliability Toolkit

Now that we know the philosophy — design for failure — let's talk about the actual tools that make it work. There are six core techniques that every fault-tolerant system uses in some combination. Think of them as your reliability toolbox: you don't use every tool on every job, but you need to know what each one does so you can pick the right one for each problem.

Tool 1: Redundancy — Have Spares Ready

The simplest way to survive a failure is to have a spare. If you have one server and it dies, you're dead. If you have two servers and one dies, you're fine. That's redundancyHaving more copies of a component than you strictly need, so that when one fails, others can take over. N+1 redundancy means one spare. N+2 means two spares. The extra copies sit ready to absorb the load if a primary fails. in a nutshell — having more copies of something than you strictly need, so that failure of any single copy doesn't take you down.

Engineers describe redundancy levels with a simple notation. N+1 means you need N servers to handle your load, and you add 1 spare. So if you need 3 servers for your traffic, you run 4. If one dies, the remaining 3 handle the load just fine. N+2 adds two spares — useful when you want to survive one failure AND still do maintenance on another server without risk.

Redundancy applies to everything, not just servers. You add redundant network paths so one switch dying doesn't cut off a rack (AWS regions have at least two Availability ZonesAn Availability Zone (AZ) is a physically separate datacenter within an AWS region, with independent power, cooling, and networking. US-East-1 has 6 AZs. Spreading your servers across 2-3 AZs means a single datacenter fire or power outage won't take you offline., each in a separate building). You add redundant power supplies so one PSU failing doesn't take a server offline — most datacenter servers have dual PSUs connected to separate power circuits. You add redundant data centers so one building losing power doesn't affect users — that's why Google has 35+ datacenters worldwide. The principle is always the same: no single thing should be so important that losing it breaks the system.

Real numbers: A single enterprise server has an MTBF of about 50,000-100,000 hours (roughly 6-11 years). Sounds great — until you run 1,000 servers, where you'd statistically expect a failure every 2-4 days. At Google's scale (millions of servers), they experience thousands of disk failures and dozens of machine deaths every single day. Redundancy isn't optional at scale — it's survival.

Tool 2: Replication — Keep Copies of Your Data

Redundancy for servers is straightforward — just run more copies of the same code. But what about your data? If your database server dies, it's not just the process that's lost — it's all the data stored on that disk. That's where replicationKeeping identical copies of your data on multiple machines. If one machine's disk dies, the data still exists on the other copies. There are two main approaches: synchronous (wait for all copies to confirm) and asynchronous (copy in the background, slightly behind). comes in — keeping identical copies of your data on multiple machines so that no single disk failure can cause data loss.

There are two main flavors of replication, and the choice between them is one of the most important trade-offs in distributed systems:

Synchronous replication waits until every copy has confirmed the write before telling the client "OK, your data is saved." This means zero data loss if the primary dies — the replica has everything. But it's slower, because every write has to wait for the network round-trip to the replica. And if the replica goes down, writes are blocked entirely.

Asynchronous replication writes to the primary and immediately tells the client "OK." It copies data to the replica in the background, typically a few milliseconds behind. This is much faster for the client, but there's a catch: if the primary dies before the replica catches up, those last few milliseconds of writes are gone forever. This gap is called replication lag.

The trade-off in one sentence: Sync replication trades speed for safety. Async replication trades safety for speed. Most real systems use a mix — sync for critical data (money, orders) and async for everything else (profiles, notifications).
Real-world examples: PostgreSQL supports both modes — you configure synchronous_commit = on for sync or off for async, and you can even set it per-transaction. MySQL uses async replication by default, with semi-sync available via a plugin. MongoDB replicates across a 3-node replica set — the primary gets writes, secondaries replicate asynchronously, and if the primary dies, the secondaries automatically elect a new leader within 10-12 seconds. Amazon RDS Multi-AZ does synchronous replication to a standby in a different datacenter — failover takes 60-120 seconds.

Tool 3: Failover — Automatic Switchover

Having redundant servers and replicated data is only half the battle. You also need a way to automatically switch from a failed component to a healthy one — without a human having to press a button. This automatic switching is called failoverThe automatic process of switching from a failed primary component to a standby backup. In active-passive mode, the backup sits idle until needed. In active-active mode, all copies handle traffic, so failure just means less capacity, not an outage..

There are two main patterns for failover, and they make very different trade-offs:

Active-passive is simpler. One server (the primary) handles all the traffic. The standby does nothing but replicate data and wait. When the primary fails, the standby takes over. The downside: you're paying for a server that sits idle 99% of the time. The upside: there's no risk of data conflicts because only one node is writing at a time.

Active-active is more efficient. Both servers handle traffic simultaneously, splitting the load between them. If one dies, the other absorbs all the traffic. No resources are wasted. But it's harder to build because both nodes are writing data at the same time, which can cause conflicts. (What happens when two users update the same record on different nodes at the same millisecond?)

Real-world failover tools: Patroni manages PostgreSQL failover — it uses a distributed consensus store (etcd or ZooKeeper) to elect a new primary within 10-30 seconds. Redis Sentinel monitors Redis instances and promotes a replica if the master dies. Nginx has built-in failover for upstream servers — if one backend returns errors, it automatically routes to the next one:
nginx.conf — upstream failover 
upstream backend {
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 backup;  # only used if both above fail
}

server {
    location / {
        proxy_pass http://backend;
        proxy_next_upstream error timeout http_502 http_503;
    }
}

Tool 4: Health Checks — Detecting Failures Fast

Redundancy and failover are useless if you don't know something is broken. You can't switch to a backup if nobody realizes the primary is dead. That's where health checks come in — a system for automatically and continuously checking whether each component is alive and healthy.

The simplest form is a heartbeatA periodic "I'm alive" signal sent between servers. If a server misses several consecutive heartbeats (e.g., 3 in a row), the system assumes it's dead and triggers failover. Typically sent every 1-5 seconds. — a tiny "I'm alive" message that every server sends every few seconds. Think of it like a doctor checking a patient's pulse. If the pulse stops, something is very wrong. In practice, a load balancer or orchestrator pings each server with a simple HTTP request (like GET /health) every 5 seconds. If a server misses 3 heartbeats in a row — 15 seconds of silence — the system declares it dead and reroutes traffic.

But a heartbeat only tells you "the server is up." A smarter health check also verifies that the server can actually do its job. Can it connect to the database? Is the disk not full? Is the CPU not at 100%? These deeper checks are called liveness probes (is it alive?) and readiness probes (is it ready to serve traffic?). Kubernetes uses both — a container can be alive but not ready, meaning it's still starting up or temporarily overloaded.

Here's what Kubernetes health checks look like in practice:

deployment.yaml — Kubernetes liveness & readiness probes 
containers:
  - name: api-server
    image: myapp:latest
    ports:
      - containerPort: 8080

    # "Are you alive?" — if this fails 3 times, K8s kills and restarts the pod
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 15    # wait 15s after startup before checking
      periodSeconds: 10          # check every 10 seconds
      failureThreshold: 3        # 3 failures = restart the container

    # "Can you handle traffic?" — if this fails, K8s stops sending requests
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      periodSeconds: 5           # check every 5 seconds
      failureThreshold: 2        # 2 failures = remove from load balancer

The distinction matters: a container might be alive (its process is running) but not ready (it's still loading a 2 GB machine learning model into memory). Without a readiness probe, Kubernetes would send traffic to it immediately and users would get timeouts. Tool 5: Circuit Breakers — Stop Calling a Dead Service

Remember the cascading failure from Section 4? The order service kept calling the slow payment service, which ate all its threads, which brought down the API gateway, which killed everything. A circuit breakerA pattern that monitors calls to a remote service and "trips" (stops making calls) when failures exceed a threshold. Like an electrical circuit breaker that cuts power to prevent a fire, it stops cascading failures by failing fast instead of waiting for timeouts. prevents exactly this scenario.

Think about the circuit breaker in your home's electrical panel. When too much current flows through a wire (a short circuit), the breaker trips — it cuts off the electricity immediately. It doesn't wait for the wire to melt and start a fire. It fails fast and safe. Software circuit breakers work the same way.

A circuit breaker has three states:

Closed (normal): calls flow through normally. The breaker counts failures. Open (tripped): when failures exceed a threshold (say, 5 failures in 10 seconds), the breaker trips. Now ALL calls to that service fail immediately — no waiting for a 30-second timeout, no consuming threads. The caller gets an instant "service unavailable" and can show a fallback response. Half-open (testing): after a cooldown period (say, 30 seconds), the breaker lets ONE test call through. If it succeeds, the breaker closes and traffic resumes. If it fails, the breaker stays open and waits another 30 seconds.

This is what prevents the domino effect. The order service has a circuit breaker around its calls to the payment service. When payment is slow, the breaker trips. Now the order service fails checkout requests instantly with a friendly "payment temporarily unavailable" message — but it can still serve order history, product pages, and everything else. The failure is contained.

Real libraries that implement this: In .NET, Polly is the go-to library — it supports circuit breakers, retries, timeouts, and bulkheads in a fluent API. In Java, Resilience4j (the successor to Netflix's Hystrix) is the standard. Envoy and Istio service meshes can add circuit breakers at the network layer without changing your code at all. Netflix's original Hystrix library pioneered this pattern — at its peak, Netflix ran millions of circuit breaker-protected calls per second.

Tool 6: Graceful Degradation — Bend, Don't Break

Here's a question most teams don't think about until it's too late: when something fails, what should the user see? The worst answer is a blank page or a cryptic "500 Internal Server Error." The best answer is: as much of the working experience as possible, with just the broken part gracefully hidden.

This is called graceful degradationWhen a system continues to operate with reduced functionality when a component fails, rather than failing completely. Like a car that still drives with a broken radio — you lose music, but you still get where you're going. — the idea that a system should bend under failure, not break. If your recommendation engine goes down, users should still see the product page — just with generic "popular items" instead of personalized suggestions. If your search service is overloaded, you can still show category pages and best-seller lists. The experience gets worse, but it doesn't disappear.

The key insight is that not all features are equally important. Checkout is critical — if users can't pay, you lose money. Recommendations are nice-to-have — showing "popular items" instead of personalized picks is a small quality drop, not a disaster. Graceful degradation means knowing the difference and building your system to shed non-essential features under stress, protecting the critical path at all costs.

Real-world graceful degradation: During peak traffic events (Prime Day, Black Friday), Amazon deliberately disables personalized recommendations and shows cached "popular items" instead — freeing up compute resources for checkout. Twitter used to show a "fail whale" error page during traffic spikes; after they redesigned for graceful degradation, they started showing a stripped-down read-only timeline instead. Netflix serves cached content and disables personalization when backend services are overloaded — you might not see the perfect recommendations, but you can still watch your show.

Going Deeper — The Math and Mechanics

You know the tools now — redundancy, replication, failover, circuit breakers. But how do you know if you're reliable enough? How do you put a number on it? This section is where reliability goes from "we added a backup server and hope for the best" to a measurable engineering target. And yes, this is the math that shows up in system design interviews.

MTBF & MTTR — The Two Numbers That Define Reliability

Reliability engineers live and die by two metrics. The first one answers the question: how often does stuff break? That's measured by something called MTBFMean Time Between Failures — the average time a system runs before something breaks. A higher MTBF means fewer failures. Example: MTBF of 720 hours means you can expect a failure roughly once a month. — Mean Time Between Failures. If your system fails once a week, your MTBF is 168 hours. If it fails once a month, your MTBF is 720 hours. Higher is better — it means more time between problems.

The second answers: how fast do you recover? That's MTTRMean Time To Recovery — the average time it takes to restore a system after a failure. Lower MTTR means less downtime. Automatic failover can reduce MTTR from 30+ minutes to under 5 seconds. — Mean Time To Recovery. If it takes you 30 minutes to get back online after a crash, your MTTR is 30 minutes. Lower is better — it means less downtime per incident.

Together, these two numbers determine your availability:

Availability Formula
Availability = MTBF / (MTBF + MTTR)
The percentage of time your system is up and running

Let's plug in some numbers to make this real.

Scenario A: Manual recovery
MTBF = 168 hours (1 failure/week), MTTR = 0.5 hours (30 min) Availability = 168 / (168 + 0.5) = 99.70%
That's 21.6 hours of downtime per year — not great.
Scenario B: Auto failover + same failure rate
MTBF = 168 hours (1 failure/week), MTTR = 0.001 hours (3.6 sec) Availability = 168 / (168 + 0.001) = 99.9994%
That's 3 minutes of downtime per year. Same failure rate, wildly different outcome.

The insight here is powerful: you don't always need to prevent more failures — you need to recover from them faster. Both scenarios have the same MTBF (same failure rate), but the one with automatic failover has 400× less downtime because its MTTR is 600× lower. In practice, reducing MTTR is often cheaper and faster than reducing MTBF.

The Nines of Availability — What 99.99% Actually Means

When engineers talk about reliability, they speak in "nines." Three nines (99.9%), four nines (99.99%), five nines (99.999%). Each additional nine sounds like a tiny improvement — just one more decimal place — but it actually represents a 10× reduction in allowed downtime. The jump from three nines to four nines isn't 0.09% better. It's 10 times harder to achieve.

Availability Name Downtime/Year Downtime/Month Typical Use Case
99% Two nines 3.65 days 7.3 hours Internal tools, batch jobs
99.9% Three nines 8.7 hours 43.8 min SaaS apps, content sites
99.95% Three and a half 4.4 hours 21.9 min E-commerce, APIs
99.99% Four nines 52.6 min 4.4 min Payment systems, databases
99.999% Five nines 5.3 min 26.3 sec Telecom, emergency systems
The cost curve is exponential. Going from 99% to 99.9% might cost you $10K/year (add a standby server). Going from 99.9% to 99.99% might cost $200K (multi-region, auto-failover). Going from 99.99% to 99.999% might cost $2M+ (redundant everything, chaos engineering team, custom tooling). Always ask: how much is downtime actually costing us? Don't buy five nines when three nines is enough.
Blast Radius & the Bulkhead Pattern

When something explodes, the blast radiusThe scope of impact when a failure occurs. A small blast radius means only one feature or one set of users is affected. A large blast radius means the whole system goes down. Good architecture minimizes blast radius. is how far the damage spreads. In systems, it's the same idea: when a component fails, how much of the system does it take down with it? A small blast radius means one feature breaks. A large blast radius means the whole system goes down.

The bulkheadA pattern borrowed from ship design: ships have watertight compartments (bulkheads) so a hull breach in one section doesn't flood the entire ship. In software, bulkheads isolate failures — e.g., separate thread pools for different services, so one slow service can't exhaust threads for everything else. pattern is borrowed directly from ship design. Ships are divided into watertight compartments. If the hull is breached and water floods one compartment, the bulkheads (walls) prevent the water from spreading to other compartments. The ship stays afloat even with one section flooded. Without bulkheads, one hole sinks the whole ship.

In software, bulkheads look like: separate thread pools for each downstream service (so a slow payment service can't eat threads meant for the user service), separate database connection pools, separate process groups, or even separate clusters. The cascading failure in Section 4 happened because everything shared resources. Bulkheads prevent that by putting walls between components.

Idempotency — Safe to Retry

Here's a sneaky problem with retries that catches a lot of teams off guard. You send a request, the server processes it successfully, but the response gets lost on the way back to you. From your side, it looks like the request failed. So you retry it. But the server already did the work — now it does it again. If that action was "charge the customer $50," you just double-charged them.

The fix is to make your operations safe to repeat. An operation that gives the same result whether you run it once or a hundred times is called idempotentAn operation that produces the same result whether you run it once or a hundred times. "Set balance to $100" is idempotent (running it twice still gives $100). "Add $50 to balance" is NOT idempotent (running it twice adds $100).. Think of it like pressing an elevator button. Pressing "3rd floor" once or mashing it ten times doesn't change where the elevator goes — you still end up on the 3rd floor. That's idempotency.

-- Add $50 to balance
-- If retried, customer gets charged TWICE!
UPDATE accounts
SET balance = balance + 50
WHERE user_id = 123;

-- Run once:  balance goes from $100 → $150  ✓
-- Run twice: balance goes from $150 → $200  ✗ WRONG!
-- Set balance to $150 (absolute value)
-- Retrying is safe — same result every time
UPDATE accounts
SET balance = 150
WHERE user_id = 123;

-- Run once:  balance = $150  ✓
-- Run twice: balance = $150  ✓ Same result!

-- OR: Use an idempotency key
INSERT INTO payments (idempotency_key, user_id, amount)
VALUES ('txn-abc-123', 123, 50)
ON CONFLICT (idempotency_key) DO NOTHING;

-- First call:  inserts the payment  ✓
-- Retry:       key already exists, skipped  ✓ Safe!

In reliable systems, retries are everywhere. Load balancers retry failed requests on a different server. Message queues redeliver messages when consumers crash. Clients retry when they get network timeouts. If your operations aren't idempotent, each of these retries is a potential bug. Making critical operations idempotent — usually with an idempotency key that deduplicates repeated requests — is a fundamental requirement for building fault-tolerant systems.

Section 8

Variations — The Five Types of Failure

Not all failures are created equal. A hard drive dying is very different from a developer accidentally deploying broken code, which is very different from a network cable getting unplugged. Each type of failure has different causes, different frequencies, and different defenses. Understanding the landscape helps you prioritize what to protect against first.

Type 1: Hardware Failures

What breaks: Hard drives, RAM sticks, CPUs, power supplies, fans, network cards. Physical components have a finite lifespan and will eventually fail. A typical hard drive has an annual failure rate of 1-4%. If you run 1,000 drives, expect 10-40 drive failures per year — that's almost one per week.

How it manifests: Sudden and total. The server is working one second and dead the next. There's no warning, no graceful shutdown. Data on the failed drive may be corrupted or unrecoverable.

Primary defenses: Redundancy (N+1 servers), replication (data on multiple drives), RAID arrays, hot-swappable components. Cloud providers handle most hardware failures transparently — that's one of the biggest reasons to use the cloud.

Real-world stat: Google's 2007 study found that 2-4% of hard drives fail per year, with failure rates increasing sharply after 3 years. At Google's scale (millions of drives), that's thousands of drive failures per month — and users never notice because everything is replicated.

Type 2: Software Bugs

What breaks: Application code, library dependencies, operating system updates, configuration changes. Unlike hardware failures (which are random and independent), software bugs are systematic — the same bug will hit every server running that code.

How it manifests: Sometimes sudden (a crash on every request), sometimes slow (a memory leak that takes 3 days to cause an OOM kill), sometimes conditional (only fails when a user's name contains an emoji). The sneaky ones are the slow leaks — everything looks fine for days until suddenly it's not.

Why it's different from hardware: Redundancy doesn't help here. If you have 10 servers running the same buggy code, all 10 will fail the same way. You can't survive a software bug by adding more copies of the broken software.

Primary defenses: Canary deployments (roll out to 1% of servers first, watch for errors before rolling out to 100%), feature flags, automated rollback, chaos testing to surface latent bugs, and thorough monitoring that catches memory leaks and error rate spikes early.

Type 3: Network Failures

What breaks: Network links between servers, between data centers, between your system and the internet. Cables get cut (a surprisingly common cause — one AWS outage was caused by a construction crew accidentally cutting a fiber cable). Switches fail. DNS servers go down. Routers get misconfigured.

How it manifests: This is the trickiest category because network failures are often partial. Server A can talk to Server B, and Server B can talk to Server C, but Server A can't talk to Server C. Or Server A can send messages to Server B, but not receive replies. This partial connectivity creates a nightmare scenario called a split-brainWhen a network partition causes two parts of a system to believe they're each the "primary." Both accept writes independently, leading to conflicting data. Resolving split-brain after the partition heals is one of the hardest problems in distributed systems. — where two parts of your system each think they're the primary and both start accepting writes, creating conflicting data.

Primary defenses: Redundant network paths, multiple internet providers, timeout policies, retry with backoff, and consensus protocols (like Raft or Paxos) that prevent split-brain by requiring a majority of nodes to agree before accepting writes.

The fundamental network truth: In a distributed system, the network is always unreliable. Packets get lost. Connections drop. Latency spikes happen. Any design that assumes the network is reliable will eventually fail in spectacular ways. This is so fundamental it's called the Fallacies of Distributed Computing, and the first fallacy is: "The network is reliable."

Type 4: Human Errors

What breaks: Everything. Humans are the most creative and unpredictable failure mode. Studies show that human errors cause 60-80% of outages — far more than hardware or software failures combined. A mistyped command, a bad config change, an accidental deletion, a deploy that wasn't tested, a "quick fix" that makes things worse.

How it manifests: "I thought I was running that command on the staging server." "I didn't know that config change would restart all the workers." "I accidentally dropped the production database instead of the test one." These aren't rare horror stories. They happen at every company, including the biggest and best-run ones.

Famous examples: In 2017, a single Amazon S3 engineer mistyped a command and took down a huge portion of the internet for 4 hours. GitLab lost 6 hours of production data in 2017 when an engineer accidentally deleted the wrong database during a maintenance operation. These weren't careless people — they were experienced engineers making mistakes under pressure.

Primary defenses: Make it hard to do the wrong thing. Use infrastructure-as-code (so changes go through code review). Require approval for destructive operations. Keep production access minimal. Build "undo" capabilities everywhere. Test disaster recovery procedures regularly so when humans do make mistakes, recovery is fast and well-practiced.

Type 5: Dependent Service Failures

What breaks: Third-party services your system depends on — payment processors (Stripe, PayPal), cloud services (AWS S3, Google Cloud), DNS providers (Cloudflare), email services (SendGrid), CDNs, and any external API your system calls.

How it manifests: You wrote perfect code. Your servers are healthy. Your database is fine. But Stripe is having an outage, and suddenly nobody can check out on your e-commerce site. You have zero control over the fix — all you can do is wait and hope they fix it soon. Your SLA is now at the mercy of every third-party service you depend on.

The math is brutal: If your system depends on 5 services, each with 99.9% availability, the combined availability is 99.9%⁵ = 99.5%. That's 44 hours of downtime per year from external dependencies alone. The more services you depend on, the worse it gets.

Combined availability of dependent services
5 services × 99.9% each = 0.999⁵ = 0.995 = 99.5% = 43.8 hours of downtime per year
Your system can't be more reliable than its least reliable dependency — unless you build fallbacks.

Primary defenses: Circuit breakers around every external call, fallback behavior (cache the last good response, show a degraded experience), timeout policies (don't let a slow third-party service hang your system for 30 seconds), and multi-provider strategies (use multiple DNS providers, multiple CDNs, or multiple payment processors so one provider's outage doesn't take you down).

Think First

Your e-commerce site depends on Stripe for payments, Cloudflare for CDN, SendGrid for email, and AWS S3 for image storage. Each has 99.9% availability. What's the probability that at least one of them is down at any given moment?

Probability all four are up: 0.999⁴ = 0.996. Probability at least one is down: 1 - 0.996 = 0.4%. That means in any given month, there's about a 12-hour window where at least one of your dependencies is unavailable. Plan for it.
Section 9

At Scale — Real Failure Stories

Theory is great, but nothing teaches reliability like watching a real system burn. These five incidents changed how the industry thinks about fault tolerance. Every one of them started with something small — a typo, a config change, a forgotten test — and snowballed into a disaster that affected millions.

AWS S3 Outage — A Typo Took Down Half the Internet (2017)

Impact: Thousands of websites and services — including Slack, Quora, Trello, and the SEC — went dark for nearly 5 hours because of a single mistyped command.

An engineer was debugging a billing system issue and ran a command to remove a small number of S3 servers. They accidentally typed a larger number than intended, which removed a critical chunk of the infrastructure that S3 itself depends on. The cascade was immediate: services that store images, host static files, or keep backups on S3 all broke at once.

The terrifying part? S3 was considered so reliable that many companies didn't even have a fallback plan. It was the invisible foundation under everything, and when it wobbled, the whole building shook.

Lessons Learned

  • Rate-limit dangerous operations — the tool should have refused to remove that many servers at once
  • Don't treat any service as "too reliable to fail" — always have a degradation plan
  • Human error is the #1 cause of outages — build guardrails, not blame

Facebook 6-Hour Outage — BGP Misconfiguration (2021)

Impact: Facebook, Instagram, WhatsApp, and Messenger were completely unreachable for over 6 hours — affecting 3.5 billion users worldwide.

During routine maintenance, a configuration change accidentally withdrew all BGPBorder Gateway Protocol — the system that tells the internet how to route traffic between networks. Think of it as the internet's GPS. routes for Facebook's servers. In plain English: Facebook told the internet "we don't exist anymore," and the internet believed it.

Because their internal tools (including the ones engineers use to fix problems) also ran on Facebook's network, engineers couldn't even log in remotely to fix it. They had to physically go to the data center and manually reconfigure the routers.

Lessons Learned

  • Out-of-band access is critical — your fix-it tools can't live on the same network they're supposed to fix
  • Canary your config changes — roll out network changes gradually, not all at once
  • Physical access plans matter — sometimes the only fix is walking into a data center

GitLab Database Deletion — Accidentally rm'd Production (2017)

Impact: 6 hours of user data permanently lost. 300 GB of live production data deleted by a tired engineer at 11 PM.

A GitLab engineer was troubleshooting database replication issues late at night. They needed to wipe a secondary database and re-sync it. But in their fatigue, they ran the delete command on the primary production database instead. By the time they realized the mistake, 300 GB of data was gone.

Then came the truly horrifying discovery: of their five backup methods, none of them actually worked. LVM snapshots were never configured, pg_dump hadn't run in months, and the Azure disk snapshots were empty. The only thing that saved them was a manual pg_dump someone had taken 6 hours earlier by coincidence.

Lessons Learned

  • Test your backups regularly — a backup you've never restored is not a backup, it's a hope
  • Tired humans make mistakes — don't do critical operations at 11 PM
  • Multiple backup methods only help if they actually run

Cloudflare Outage — A Bad Regex (2019)

Impact: Cloudflare's entire global network went down for 27 minutes, affecting millions of websites.

A developer deployed a new Web Application Firewall (WAF) rule that contained a poorly written regular expression. The regex caused catastrophic backtracking — meaning the CPU got stuck in an endless loop trying to evaluate it. Since the WAF runs on every request, every Cloudflare server worldwide pegged at 100% CPU simultaneously.

The irony? The WAF is meant to protect websites. Instead, a bad pattern in the protection layer became the thing that took everything down. One line of regex, global outage.

Lessons Learned

  • Regex needs review and testing — treat it like any other critical code path
  • Canary deployments for rules — roll to 1% of traffic first, not 100%
  • CPU circuit breakers — if a process spikes CPU, kill it automatically

Knight Capital — $440 Million Lost in 45 Minutes (2012)

Impact: A botched software deployment caused Knight Capital to lose $440 million in 45 minutes, bankrupting the firm.

Knight Capital was deploying new trading software to 8 servers. A technician forgot to update one of them. That one server still had old test code that was designed to aggressively buy and sell stocks as fast as possible. When the market opened, that server went rogue — executing millions of unintended trades at lightning speed.

By the time anyone realized what was happening, 45 minutes had passed. The company had accumulated $7 billion in unwanted positions and lost $440 million. Knight Capital went from a healthy company to bankrupt in under an hour.

Lessons Learned

  • Automated deployment or nothing — manual deploys to 8 servers will always miss one
  • Kill switches are mandatory — if spending spikes 1000x, auto-halt immediately
  • Dead code is dangerous code — remove old test logic, don't leave it dormant

Section 10

Anti-Lesson — What Not to Do

Not every reliability mistake is a missing backup or a forgotten health check. Some of the worst ones look like good engineering from the outside. Teams spend months building impressive-sounding systems that, when tested by a real outage, crumble like a sandcastle. These three anti-patterns are traps — easy to fall into, expensive to climb out of.

Chasing 99.999% When 99.9% Is Enough

Five nines (99.999%) means your system can only be down for about 5 minutes per year. That sounds impressive, but it comes at a staggering cost: redundant everything, multi-region failover, near-zero-downtime deployments, and a team of on-call engineers around the clock.

Knight Capital Group learned this the hard way in 2012 — but in reverse. They had invested heavily in trading speed but cut corners on deployment reliability. A botched software deployment (old code accidentally reactivated on one of 8 servers) caused their automated trading system to buy $7 billion in unwanted stock positions in 45 minutes. The firm lost $440 million and was acquired at a fire-sale price within days. The irony: the bug wasn't in the trading algorithm — it was a deployment process that lacked basic rollback safeguards.

The truth? Most applications are perfectly fine with three nines (99.9%), which allows about 8.7 hours of downtime per year. Your users probably won't even notice a few minutes of downtime at 3 AM. But the cost difference between 99.9% and 99.999% is often 10x or more. Set your target based on actual user impact, not engineering ego.

Rule of thumb: Set your SLOService Level Objective — the target level of reliability you commit to internally. Example: 'Our API will respond in under 200ms for 99.9% of requests.' based on what your users actually need, not what sounds impressive on a slide deck. If your internal dashboard goes down for 10 minutes and nobody notices, you don't need five nines.
Reliability Theater — Backups Never Tested

This is the GitLab story all over again, but it happens everywhere. Teams set up elaborate backup systems, tick the compliance checkbox, and never actually try restoring from those backups. When disaster strikes, they discover the backups are empty, corrupted, or pointed at the wrong database.

Reliability theater is any practice that looks like it improves reliability but has never been validated. Other examples: monitoring dashboards nobody checks, runbooks nobody reads, incident response plans that haven't been rehearsed.

The fix: Schedule regular "restore drills." Once a month, pick a random backup and actually restore it to a test environment. If it fails, you just found a critical bug before a real disaster. A backup you've never restored is just a file — not a safety net.
Ignoring the Human Factor

We build redundant servers, automatic failover, and self-healing clusters. But then we let a sleep-deprived engineer run a critical command at midnight with no guardrails and no peer review. The biggest single point of failure in most systems isn't a server — it's a tired human with root access.

Studies consistently show that 60-80% of outages are caused by human error — misconfigurations, bad deployments, wrong commands. Yet most reliability budgets go toward hardware and software redundancy, not toward making human operations safer.

Design for humans: Require two-person approval for production changes. Add confirmation prompts for destructive commands. Ban non-emergency changes after midnight. The best systems assume humans will make mistakes and catch them before they cause harm.
Section 11

Common Mistakes

If you've read the failure stories in Section 9, you might be thinking "that wouldn't happen to me." But these six mistakes are so common that even experienced teams fall into them — in real incidents, in design reviews, and in interviews. Memorize these, and you'll dodge landmines that trip up most engineers.

Never Testing Backups

The mistake: Setting up automated backups and assuming they work.

Why it happens: Backups run silently in the background. Nobody checks them until they're desperately needed. By then, you discover they've been failing for months.

The fix: Schedule monthly restore drills. Automate a "backup validation" job that restores to a scratch database and runs a row-count check. If it fails, page someone immediately.

Hidden Single Points of Failure

The mistake: Thinking you've eliminated all SPOFs when one is hiding in plain sight — a shared load balancer, a single DNS provider, one engineer who knows the system.

Why it happens: SPOFs hide in infrastructure layers people don't think about: the one network switch, the one config server, the one CI/CD pipeline.

The fix: Run a "what if this dies?" exercise. Walk through every component in your architecture and ask: "If this single thing disappears right now, what breaks?" Draw a dependency map and look for any node that, if removed, disconnects the graph.

Confusing Availability with Reliability

The mistake: Saying "our system is 99.9% reliable" when you actually mean "99.9% available." These are different things.

Why it matters: A system can be available (it responds to requests) but not reliable (it returns wrong data, drops transactions, or silently corrupts state). Availability is "is it up?" Reliability is "does it do the right thing?"

The fix: Track both metrics separately. Availability = uptime/total time. Reliability = correct responses/total responses. A system that's up but returning garbage is arguably worse than one that's honestly down.

No Runbooks

The mistake: When something breaks at 3 AM, the on-call engineer has no idea what to do and has to wake up the one person who built the system.

A runbookRunbook — a step-by-step guide for handling specific operational scenarios. Like a recipe book for incidents: 'If X happens, do Y, then Z.' is a simple document that says: "If this alert fires, here's exactly what to check and what to do." Without one, every incident becomes an improvisation exercise.

The fix: For every alert you create, write a matching runbook. Include: what the alert means, what to check first, how to mitigate, and who to escalate to. Keep it next to the alert definition.

Alert Fatigue

The mistake: Setting up so many alerts that the on-call engineer gets 200 notifications a day and starts ignoring all of them.

Why it's dangerous: When everything screams for attention, nothing gets attention. The real critical alert drowns in a sea of noise, and the engineer who should respond has already mentally tuned out.

The fix: Ruthlessly prune alerts. Every alert must be actionable — if the response is "do nothing, it'll fix itself," delete the alert. Aim for fewer than 5 pages per on-call shift. Categorize alerts into P1 (wake someone up) and P2 (check tomorrow morning).

No Chaos Testing

The mistake: Building fault tolerance in theory but never actually testing whether the system survives real failures.

Why it happens: Teams are afraid to intentionally break production. "What if the fallback doesn't work?" Well, that's exactly why you need to test it — better to discover the fallback is broken during a controlled test than during a real outage at peak traffic.

The fix: Start small. Kill one non-critical process in staging and see what happens. Gradually increase scope: kill a server, block a network path, introduce latency. Netflix's Chaos Monkey does this automatically in production — but you can start with a manual monthly drill.

Section 12

Interview Playbook

Reliability questions come up in almost every system design interview. The interviewer wants to know that you think about what happens when things go wrong, not just how things work when everything is fine. Here's how to structure your answers.

The magic phrase: "What happens when [X] fails?" — Say this proactively in your interview. Don't wait for the interviewer to ask about failures. Volunteer it: "Now let me think about what happens when the database goes down..." This immediately signals senior-level thinking.

What Beginners Should Cover

  • Redundancy: "I'd run at least 2 instances behind a load balancer so if one dies, the other handles traffic."
  • Backups: "I'd set up automated database backups and test restores monthly."
  • Health checks: "A /health endpoint that the load balancer pings to know if a server is alive."
  • Timeouts: "Every external call should have a timeout so we don't hang forever waiting."

At this level, showing you think about failure at all puts you ahead of most candidates.

What Intermediate Candidates Should Add

  • Circuit breakers: "If the payment service fails 5 times in a row, I'll stop calling it for 30 seconds and return a cached result or graceful error."
  • Retry with backoff: "Retries are fine, but with exponential backoff and jitter to avoid thundering herd."
  • RTORecovery Time Objective — the maximum acceptable time to restore service after an outage. RTO = 'how long can we be down?' and RPORecovery Point Objective — the maximum acceptable amount of data loss measured in time. RPO = 'how much data can we afford to lose?': "I'd set RTO at 15 minutes and RPO at 1 hour based on the business requirements."
  • Graceful degradation: "If recommendations are down, we show popular items instead of an error page."

What Senior/Staff Candidates Should Discuss

  • Error budgetsError Budget — the allowed amount of unreliability within your SLO. If your SLO is 99.9%, your error budget is 0.1% (about 43 min/month). Once the budget is spent, freeze feature releases and focus on reliability.: "With a 99.9% SLO, we have a 43-minute monthly error budget. If we burn through it, we freeze deploys and focus on reliability."
  • Chaos engineering: "We'd run regular game days — inject failures in staging, then graduate to production with kill switches."
  • Blast radius control: "We deploy to 1% canary first. If error rates spike, auto-rollback before it hits 100%."
  • PostmortemPostmortem — a blame-free document written after an incident. It covers what happened, why, what was the impact, and what we're changing to prevent recurrence. The goal is learning, not punishment. culture: "After every P1 incident, we write a blameless postmortem and track action items to completion."

Sample Interview Dialogue

Interviewer: "Design a payment processing system. How do you handle reliability?"

You: "Great question — let me walk through the failure modes first. The biggest risk is double-charging a customer, so I need idempotency keys on every payment request. If the payment gateway times out, I won't retry blindly — I'll check the status first. For infrastructure, I'd run the service across two availability zones with a load balancer doing health checks every 10 seconds. Database writes go to a primary with synchronous replication to a standby, giving us an RPO near zero. I'd set the RTO at 5 minutes using automated failover. And I'd add a circuit breaker around the gateway call — if it fails 3 times in 10 seconds, we trip the breaker and queue the payment for retry with exponential backoff."

Practice Exercises

Reading about reliability is one thing — designing for it is another. These exercises progress from beginner-friendly thought experiments to advanced hands-on challenges.

Exercise 1: SPOF Detective Beginner

You're given this architecture: a single web server, a single database, and a single load balancer (yes, even the load balancer is single). Users connect through one DNS provider.

Task: List every single point of failure. For each one, propose a fix that adds redundancy. Draw a before-and-after diagram.

Hint

There are at least 4 SPOFs in this architecture. Don't forget about the DNS provider and the network path itself.

Solution

SPOFs found:

  • Web server — Fix: Run 2+ instances behind a load balancer
  • Database — Fix: Primary-replica setup with automatic failover
  • Load balancer — Fix: Active-passive LB pair with a floating IP
  • DNS provider — Fix: Use two DNS providers (e.g., Route 53 + Cloudflare)

After fixing all four, there's no single component whose failure takes down the entire system.

Exercise 2: SLA Calculator Beginner

Your SLAService Level Agreement — a formal contract with customers specifying minimum uptime guarantees. Unlike SLOs (internal targets), SLAs carry financial penalties when breached. promises 99.95% uptime. Your system depends on three services in series: Service A (99.99%), Service B (99.97%), Service C (99.99%).

Task: Calculate the combined availability. Does it meet the SLA? What's the maximum downtime per month?

Solution

Combined availability = 0.9999 x 0.9997 x 0.9999 = 99.95%

This barely meets the SLA. Monthly downtime = 43,200 minutes x 0.0005 = 21.6 minutes.

In practice, you'd want your SLO to be stricter than your SLA (e.g., target 99.97% internally so you have a safety margin).

Exercise 3: Design a Health Check System Intermediate

Design a health check system for a microservices architecture with 12 services. The system should detect failures within 30 seconds and automatically remove unhealthy instances from the load balancer.

Considerations: How often do you poll? What counts as "unhealthy"? How do you avoid false positives? What about cascading failures where Service A is unhealthy only because Service B is down?

Hint

Think about shallow vs. deep health checks. A shallow check ("am I running?") vs. a deep check ("can I reach my database and all dependencies?"). What are the tradeoffs?

Solution

Design:

  • Poll interval: Every 10 seconds (3 checks within 30s window)
  • Unhealthy threshold: 3 consecutive failures before removal (avoids flapping)
  • Two-tier checks: Shallow /health (is process alive) every 10s + deep /health/ready (can I serve traffic) every 30s
  • Dependency isolation: If Service A reports unhealthy because B is down, mark A as "degraded" but don't remove it — it might still serve cached responses
  • Auto-recovery: Once a removed instance passes 2 consecutive checks, re-add it
Exercise 4: Chaos Engineering Game Day Advanced

You're planning a chaos engineering game day for a 3-tier e-commerce app (web servers, API servers, database cluster). Design a series of 5 failure injection experiments, ordered from least to most risky.

For each experiment: Define the hypothesis, the injection method, the expected behavior, the abort criteria, and the rollback plan.

Solution

Experiment progression:

  1. Kill one web server instance — Hypothesis: LB reroutes in <10s. Low risk.
  2. Inject 500ms latency on API-to-DB calls — Hypothesis: Timeouts fire, no cascading failure. Medium risk.
  3. Block network between API tier and cache — Hypothesis: API falls back to DB, response time increases but no errors. Medium risk.
  4. Trigger DB failover to replica — Hypothesis: Writes resume in <30s, no data loss. Higher risk.
  5. Simulate full AZ outage — Hypothesis: Traffic shifts to second AZ, all functionality intact. Highest risk.

Abort criteria for all: Error rate exceeds 5% OR p99 latency exceeds 5s OR any data inconsistency detected. Rollback: revert the failure injection immediately.

Exercise 5: Post-Incident Review Expert

An e-commerce site had a 2-hour outage on Black Friday. The root cause was a database connection pool exhaustion caused by a slow query that was introduced 3 weeks earlier. Monitoring didn't catch it because the slow query only triggers under high load.

Task: Write a full postmortem. Include: timeline, root cause analysis (use the "5 Whys" technique), impact assessment, what went well, what went poorly, and 5 concrete action items with owners and deadlines.

Hint

For the 5 Whys: Why did the connection pool exhaust? Because of a slow query. Why was there a slow query? Because it was missing an index. Why wasn't the missing index caught? ... Keep going until you reach the systemic cause.

Solution

5 Whys:

  1. Why did the site go down? — Connection pool exhausted.
  2. Why was the pool exhausted? — A slow query held connections for 30s each.
  3. Why was there a slow query? — A new feature added a full table scan (missing index).
  4. Why wasn't the missing index caught? — No query performance testing in CI/CD.
  5. Why no query perf testing? — Team didn't have load testing as part of the release checklist.

Action items:

  • Add connection pool usage alerts (threshold 80%) — SRE team, 1 week
  • Add slow query logging and alerting — DBA, 1 week
  • Integrate load testing into CI pipeline — Platform team, 2 weeks
  • Add query EXPLAIN analysis to PR review checklist — Engineering leads, 3 days
  • Create Black Friday capacity planning runbook — SRE + Product, 1 month
