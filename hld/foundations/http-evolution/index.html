HTTP/1.1, HTTP/2, HTTP/3 & QUIC

The web's delivery truck got faster three times — each upgrade solved a bottleneck the previous version couldn't fix. Here's the full story, from painfully slow page loads to blazing-fast zero-round-trip connections.

Section 1

TL;DR — The One-Minute Version

Mental Model: HTTP evolves like highway engineering. HTTP/1.0 was a single-lane dirt road (one car at a time). HTTP/1.1 paved it and let cars queue up. HTTP/2 turned it into a multi-lane highway (many cars at once). HTTP/3 said "forget roads entirely" and built a maglev train that doesn't get stuck when one rail breaks.

Every time you open a website, your browser talks to a server using HTTPHyperText Transfer Protocol — the language browsers and servers use to exchange web pages, images, APIs, and pretty much everything on the internet.. The problem? The original version was painfully slow for modern websites. Each new version of HTTP exists because the previous one had a specific bottleneck that made the web feel sluggish. Understanding these bottlenecks — and the clever solutions each version introduced — is essential for any engineer who builds things on the web.

Here's the pattern: HTTP/1.0 opened a brand-new connection for every single file — brutally wasteful. HTTP/1.1 fixed that by reusing connections, but requests still had to wait in line one after another. HTTP/2 let multiple requests fly over a single connection simultaneously — a massive leap. But it ran on TCPTransmission Control Protocol — a reliable transport layer that guarantees data arrives in order and without errors, at the cost of some overhead., and when one packet got lost, everything froze. HTTP/3 ditched TCP entirely, switching to a new transport called QUICQuick UDP Internet Connections — a transport protocol built on UDP that provides reliable delivery, encryption, and multiplexing without TCP's head-of-line blocking problem. that doesn't have this freezing problem.

One-line takeaway: HTTP/1.1 reuses connections. HTTP/2 multiplexes streams. HTTP/3 eliminates TCP's blocking with QUIC. Each version makes the web measurably faster.
Section 2

The Scenario — Why Does My Website Load So Slowly?

Imagine you've built a beautiful e-commerce website. The homepage has 1 HTML file, 3 CSS files, 5 JavaScript files, 40 product images, 2 font files, and a few API calls — roughly 55 individual resources. You deploy it, open it on your phone over a decent connection, and... it takes 8 seconds to fully load. Your competitor's site, with similar content, loads in 1.5 seconds. What's going on?

The bottleneck isn't your server speed. It isn't your internet bandwidth. The problem is how your browser fetches those 55 files. The protocol — the set of rules your browser and server use to talk to each other — is forcing your browser to be painfully polite, asking for one thing at a time and waiting patiently for each response before asking for the next.

The core pain: A modern web page needs 50-100 separate resources. If the protocol forces you to fetch them one by one, no amount of server optimization can save you. The protocol itself is the bottleneck.

The server can respond in 5 milliseconds. But the browser spends most of its time waiting — waiting for the connection to be established, waiting for the previous response to finish before sending the next request, waiting because the protocol says you must wait. The actual data transfer is lightning fast. The protocol overhead is what kills you.

🧠 Think First: If you had 55 files to download and each takes 5ms to serve, but creating a connection takes 100ms (a TCP handshake), how long would it take if you needed a brand-new connection for every single file? Do the math before reading on.

If you did the math: 55 files × (100ms handshake + 5ms transfer) = 5,775ms — nearly 6 seconds spent mostly on handshakes, not actual data. That's the world of HTTP/1.0, and it's where our story begins.

Section 3

The First Attempt — HTTP/1.0: One Request, One Connection

When the web was born in the early 1990s, web pages were simple — a single HTML file with maybe a handful of images. The protocol designed for this world, HTTP/1.0 (standardized in 1996), had a very straightforward rule: every time you need a file, you open a brand-new connection, download the file, and then close the connection. Done. Next file? Open a brand-new connection again.

Think of it like a phone call to a restaurant. You call to order a burger. They take your order, hang up. Now you want fries. You call again. They take your order, hang up. Want a drink? Call again. Every single item requires a completely new phone call — with all the dialing, ringing, and "hello, how can I help you?" overhead each time.

This was fine when a web page had 3 files. But by the mid-2000s, pages had 50-100 resources. Each one needed its own TCP handshakeA three-step process (SYN → SYN-ACK → ACK) that two computers use to establish a reliable connection. Takes one round trip — typically 50-150ms depending on distance. — a back-and-forth greeting that takes one full round trip. On a connection with 100ms latency (common for cross-continent requests), that's 100ms of pure overhead per file, before a single byte of actual data transfers.

Did HTTP/1.1 fix this? Partially. It introduced keep-alive — reusing the same connection for multiple requests. But it had its own problem, which we'll see in the next section. The connection stayed open, but requests still had to wait in line.
Section 4

Where It Breaks — Head-of-Line Blocking & Connection Limits

HTTP/1.1 (1997) made a crucial improvement: persistent connections. Instead of opening and closing a connection for every file, the browser keeps the connection open and sends multiple requests over it. Great! But here's the catch — requests must still go one after another on a single connection. You send request 1, wait for the response, then send request 2, wait for the response, and so on.

This is like being in a single-file checkout line at a grocery store. The person at the front is paying with a complicated coupon that takes 30 seconds to process. Everyone behind them — even people with a single item — has to wait. That slow request at the front blocks everything behind it. Engineers call this head-of-line blockingWhen the first item in a queue prevents all subsequent items from being processed, even if they could be handled immediately. The "head" (front) of the "line" (queue) blocks everyone behind it..

🧠 Think First: HTTP/1.1 allows persistent connections but processes requests sequentially. Browsers work around this by opening multiple connections in parallel (typically 6 per domain). If a page needs 55 resources and each connection handles requests one at a time, how many "batches" does the browser need with 6 connections? What if one resource is a slow 2-second API call?

Browsers worked around head-of-line blocking by opening up to 6 parallel connections per domain. This helped — instead of fetching 55 files sequentially, you fetch them 6 at a time. But it's a hack, not a solution:

  • Each connection has its own TCP handshake — 6 handshakes × 100ms = 600ms wasted before any data flows
  • Each connection has its own TLS handshakeTransport Layer Security handshake — an additional negotiation (1-2 round trips) on top of TCP where browser and server agree on encryption keys. Required for HTTPS. — for HTTPS sites, that's another 1-2 round trips per connection
  • 6 connections still isn't enough for 55+ resources — you need ~10 sequential batches
  • The server must maintain 6 open sockets per client — at scale, this means millions of open connections doing nothing most of the time

Web developers got creative with hacks: CSS spritesCombining many small images into one big image, then using CSS to show only the relevant portion. Reduces HTTP requests at the cost of complexity. (cramming 30 icons into one image), concatenating all JavaScript into a single massive file, inlining CSS directly into HTML, and even spreading assets across multiple domains (domain sharding) to bypass the 6-connection limit. These were clever workarounds for a broken protocol.

The fundamental problem: HTTP/1.1 is text-based, sequential, and has no way to interleave responses on a single connection. No matter how many connections you open, each individual connection processes one request at a time. The protocol needs a fundamentally different design — not more workarounds.
Section 5

The Breakthrough — Multiplexing: Many Requests, One Connection

What if instead of sending requests one after another (like a single-file checkout line), you could send them all at once and have the server respond with pieces of different files interleaved together? That's multiplexing — and it's the core idea behind HTTP/2.

Think of it like a restaurant kitchen with multiple chefs. You hand the waiter your entire order at once — burger, fries, drink, dessert. The kitchen doesn't finish the burger before starting the fries. All four items are being prepared simultaneously. As each one is ready, the waiter brings it to your table. The fries arrive first because they're quick. The burger takes longer. But nothing is waiting in line.

🧠 Think First: If multiplexing lets you send all 55 requests at once over a single connection, what's the theoretical minimum load time? Assume 100ms for the TCP handshake, and the largest file takes 200ms to transfer. Compare this to HTTP/1.1's sequential approach.

With HTTP/2 multiplexing, all 55 requests fly out immediately on a single connection. The server sends back pieces of different files interleaved — a chunk of CSS, then a chunk of JavaScript, then a chunk of an image — all mixed together on the same connection. The browser reassembles each file from its chunks. The total time is determined by the slowest individual file, not by the sum of all files.

This is the fundamental breakthrough of HTTP/2, and it came from Google's experimental SPDYAn experimental protocol created by Google in 2009 that introduced multiplexing, header compression, and server push. It was so successful that it became the basis for HTTP/2. protocol. Google tested it across Chrome and their own servers, saw 40-60% improvement in page load times, and it eventually became the HTTP/2 standard in 2015.

Key insight: HTTP/2 didn't change what HTTP does (request a resource, get a response). It changed how — by replacing the text-based, sequential wire format with a binary, multiplexed one. Your existing websites, APIs, and servers work with zero code changes.
Section 6

How It Works — Version by Version

Let's walk through each version's mechanics in detail. Each card below covers one HTTP version — what it introduced, how it works under the hood, and what problem it specifically solved.

HTTP/1.1 — Persistent Connections & Keep-Alive

HTTP/1.1 made one critical change: instead of closing the connection after each request, it defaults to keeping it open. This is the Connection: keep-alive header. The browser opens one TCP connection, sends a request, gets a response, and then sends the next request on the same connection — no new handshake needed.

HTTP/1.1 also introduced pipeliningA technique where the browser sends multiple requests without waiting for responses. Sounds great in theory, but responses must arrive in order, so a slow response still blocks everything. Most browsers disabled it. — sending multiple requests without waiting for responses. In theory, this could have solved head-of-line blocking. In practice, responses still had to arrive in order, so it didn't actually help. Most browsers disabled pipelining because it caused more problems than it solved.

Other features HTTP/1.1 brought: Host header (allowing multiple websites on one IP address), chunked transfer encodingA way to send a response in pieces without knowing the total size upfront. The server sends chunks as they're ready, each prefixed with its size, ending with a zero-length chunk. (streaming responses before the full size is known), and better caching with Cache-Control, ETag, and conditional requests.

HTTP/2 — Streams, Binary Frames & HPACK Compression

HTTP/2 (2015) reimagined the entire wire format. Instead of plain text, it uses a binary framing layer. Every message — request or response — is split into small framesThe smallest unit of communication in HTTP/2. Each frame has a fixed header (9 bytes) that says its length, type, stream ID, and flags. Types include HEADERS, DATA, SETTINGS, PUSH_PROMISE, and more., and each frame is tagged with a stream ID. Frames from different streams can be interleaved on a single connection — that's multiplexing.

The key concepts in HTTP/2:

  • Stream: A logical channel within the connection. Each request-response pair gets its own stream ID. Streams are independent — a slow response on stream 5 doesn't block stream 7.
  • Frame: The smallest unit of data. Each frame has a 9-byte header that tells the receiver its length, type, which stream it belongs to, and any flags. Think of frames as labeled envelopes on a shared conveyor belt.
  • HPACK: A compression algorithm specifically designed for HTTP headers. Both client and server maintain a synchronized table of previously-seen headers. When a header repeats (which happens constantly — cookies, user-agent, accept), it's sent as a tiny index number instead of the full text.

HTTP/2 also introduced server push — the server can proactively send resources the client hasn't asked for yet. If a client requests index.html, the server knows it will also need style.css, so it pushes the CSS immediately without waiting for the browser to discover and request it. (In practice, server push was tricky to get right, and Chrome eventually removed support for it in 2022.)

HTTP/3 — QUIC over UDP & Zero Round-Trip Connections

HTTP/2 solved the application-level head-of-line blocking problem — no more waiting for slow responses. But it still ran on TCP, and TCP has its own head-of-line blocking at the transport level. Here's the problem: TCP guarantees that all data arrives in order. If one packet gets lost, TCP stops delivering everything on that connection until the lost packet is retransmitted — even data for completely unrelated HTTP/2 streams.

🧠 Think First: HTTP/2 multiplexes 10 streams on one TCP connection. One packet carrying data for stream 3 gets lost. What happens to streams 1, 2, 4, 5, 6, 7, 8, 9, and 10? Think about TCP's ordering guarantee.

The answer: all 10 streams freeze. TCP doesn't know about HTTP/2 streams — it just sees one byte stream and insists on delivering it in order. A single lost packet belonging to stream 3 blocks the delivery of data for all other streams. This is TCP-level head-of-line blocking, and it's especially painful on lossy networks (mobile, Wi-Fi).

HTTP/3's solution was radical: replace TCP entirely. QUIC is a brand-new transport protocol built on top of UDPUser Datagram Protocol — a simple transport that sends packets without guarantees. No ordering, no retransmission, no handshakes. Fast and lightweight, but unreliable on its own. QUIC adds reliability on top of UDP.. But unlike raw UDP, QUIC provides everything TCP does — reliable delivery, congestion control, encryption — plus native multiplexing at the transport layer. Each QUIC streamAn independent, ordered sequence of data within a QUIC connection. Unlike TCP, streams are first-class citizens — loss on one stream doesn't affect others. is independent, so a lost packet on stream 3 only affects stream 3. Streams 1, 5, and 7 keep flowing.

Two more game-changers in HTTP/3:

  • 0-RTT connection resumption: If you've visited a server before, QUIC remembers the encryption keys. On your next visit, it sends data in the very first packet — zero round trips of handshaking. TCP + TLS needs 2-3 round trips before any data flows.
  • Built-in encryption: QUIC mandates TLS 1.3. There's no unencrypted HTTP/3. The TLS handshake is combined with the QUIC handshake, saving a round trip compared to TCP + TLS.
  • Connection migration: If your phone switches from Wi-Fi to cellular, the IP address changes. TCP connections break (they're identified by IP + port). QUIC connections survive because they're identified by a connection ID, not by IP. The same request continues seamlessly on the new network.

Going Deeper — Under the Hood

Now that you understand the big ideas, let's look at the internals that make each version tick. This is the stuff that separates "I know HTTP/2 uses multiplexing" from "I can explain exactly how multiplexing works at the byte level."

Binary Framing Layer (HTTP/2)

In HTTP/1.1, a request looks like this — plain ASCII text you could read with your eyes:

Stream Priorities & Dependencies

Not all resources are equally important. The CSS file that controls your page layout is more critical than a decorative footer image. HTTP/2 lets the client assign priorities to streams and create dependency trees. For example: "stream 1 (CSS) has weight 256, stream 3 (hero image) depends on stream 1 and has weight 220, stream 5 (footer image) has weight 16."

In practice, browser implementations of priority have been inconsistent. Chrome, Firefox, and Safari all used different prioritization schemes, making it hard for servers to optimize. HTTP/3 simplified this with the Extensible Priority Scheme (RFC 9218) — a cleaner urgency + incremental model.

Header Compression: HPACK vs QPACK

Both HTTP/2 and HTTP/3 compress headers, but they use different algorithms because of how their underlying transports work.

HPACK maintains a dynamic tableA key-value lookup table that both client and server maintain in sync. Previously sent headers get added to this table and can be referenced by index number in future requests — dramatically reducing header size. — a synchronized lookup table on both client and server. When you send User-Agent: Mozilla/5.0... for the first time, it gets added to the table at index 62. The next request refers to it as just the number 62 — a single byte instead of 80 bytes.

HPACK also uses Huffman encoding for header values that aren't in the table yet, squeezing them to about 70% of their original size. The static table includes 61 pre-defined entries for common headers like :method: GET, :status: 200, and content-type: text/html.

The catch: HPACK requires both sides to process headers in exact order. If header frame X arrives before header frame Y, they must be decoded in that order. This works fine on TCP (which guarantees order) but is a problem for QUIC (where packets can arrive out of order).

QPACK solves HPACK's ordering problem. It uses two separate QUIC streams for the dynamic table: an encoder stream (client → server, sending table updates) and a decoder stream (server → client, acknowledging updates). Request headers reference the table but can be decoded independently — even if they arrive out of order.

The trade-off: QPACK is slightly less efficient than HPACK at compression because it can't always reference the most recent table state (the update might not have been acknowledged yet). In practice, the difference is tiny — both achieve 85-95% header compression.

If the request is a GET, no harm done (same page is returned). If it's a POST that transfers money, the server might process the transfer twice.. That's why servers should only accept 0-RTT for idempotent requests (like GET) — requests that are safe to process more than once. Non-idempotent operations (POST, PUT, DELETE) should require a full handshake.

Variations — Protocols Built on Top of HTTP

HTTP isn't just for loading web pages. It's the foundation for many other protocols and patterns. Understanding how these build on HTTP versions helps you make better architectural choices.

gRPC — High-Performance RPCs over HTTP/2

gRPCGoogle Remote Procedure Call — a framework for calling functions on remote servers as if they were local. Uses Protocol Buffers for serialization and HTTP/2 for transport, enabling efficient bidirectional streaming. is Google's framework for service-to-service communication. It runs exclusively on HTTP/2 because it needs multiplexing for its four communication patterns:

Without HTTP/2's multiplexing, gRPC couldn't efficiently support streaming or handle hundreds of concurrent RPCs between microservices. This is why gRPC doesn't work on HTTP/1.1 (though gRPC-web provides a limited compatibility layer).

WebSocket — From HTTP to Full-Duplex

WebSocketA protocol that provides full-duplex (two-way) communication over a single TCP connection. Starts as an HTTP/1.1 request with an Upgrade header, then switches to a completely different protocol for real-time messaging. starts its life as a regular HTTP/1.1 request with a special Upgrade: websocket header. The server agrees, and the connection "upgrades" from HTTP to the WebSocket protocol. After that, HTTP is completely gone — it's a raw bidirectional data pipe.

With HTTP/2 and HTTP/3, WebSocket works differently. HTTP/2 introduced WebSocket over a single stream (RFC 8441) — the upgrade happens within one stream while other streams continue normal HTTP traffic. This is more efficient than HTTP/1.1 WebSocket, which took over the entire TCP connection.

When to use what: Need real-time bidirectional messaging (chat, gaming, live feeds)? WebSocket. Need efficient API calls between services? gRPC over HTTP/2. Need one-way server updates? Server-Sent EventsA simple standard for one-way real-time updates from server to client over a regular HTTP connection. Simpler than WebSocket but only server → client, not bidirectional. (SSE) over any HTTP version.

REST APIs — Same Semantics, Better Transport

The beautiful thing about HTTP/2 and HTTP/3 is that your existing REST APIs work without any changes. The HTTP methods (GET, POST, PUT, DELETE), status codes (200, 404, 500), and headers all stay exactly the same. The difference is entirely at the transport level.

This means upgrading from HTTP/1.1 to HTTP/2 is often as simple as updating your web server or load balancer configuration — your application code doesn't change at all. For HTTP/3, you need a QUIC-capable server (like Nginx with the QUIC module, Caddy, or Cloudflare's edge), and clients need to support it (all modern browsers do).

nginx.conf — Enable HTTP/2 
server {
    listen 443 ssl http2;       # ← just add "http2"
    server_name shop.example.com;
    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    # Everything else stays the same
    location /api/ {
        proxy_pass http://backend:3000;
    }
}
Caddyfile — HTTP/3 automatically enabled 
shop.example.com {
    # Caddy enables HTTP/3 by default!
    # Auto HTTPS, auto HTTP/2, auto HTTP/3
    reverse_proxy backend:3000
}

# To explicitly control:
shop.example.com {
    servers {
        protocol {
            experimental_http3  # if needed explicitly
        }
    }

Section 9

At Scale — How the Giants Adopted HTTP/2 & HTTP/3

The HTTP evolution wasn't just an academic exercise. Real companies saw massive improvements by upgrading their protocols. Here are the stories and numbers.

Google — From SPDY to HTTP/2 (The Origin Story)

In 2009, Google's Mike Belshe published research showing that on typical web pages, increasing bandwidth beyond 5 Mbps didn't improve load times. The bottleneck wasn't bandwidth — it was latencyThe time it takes for a single packet to travel from client to server and back (the round-trip time). Unlike bandwidth (how many packets per second), latency is limited by the speed of light and network hops.. All those TCP handshakes, sequential requests, and head-of-line blocking were the real problem.

Google built SPDY (pronounced "speedy") as an experiment. They deployed it between Chrome and Google's servers and measured: page load times dropped 40-60% on news sites, ~27-60% on complex web apps. The improvement was so dramatic that the IETF used SPDY as the starting point for HTTP/2.

Cloudflare — Early HTTP/3 Adoption at Massive Scale

Cloudflare enabled HTTP/3 for all their customers in 2019 — one toggle and suddenly millions of websites supported the newest protocol. Their measurements showed:

  • 15% faster Time to First Byte compared to HTTP/2 on mobile networks
  • Connection migration works — users switching from Wi-Fi to cellular in a subway didn't experience connection resets
  • 0-RTT resumption saved 100-300ms for returning visitors, depending on geographic distance to the nearest edge server

The biggest impact was on lossy networks — mobile connections, congested Wi-Fi, high-latency satellite links. On clean, fast connections, the difference between HTTP/2 and HTTP/3 was minimal. But on a 3% packet loss mobile connection, HTTP/3 was up to 3x faster than HTTP/2.

Section 10

The Anti-Lesson — When Upgrading HTTP Versions Hurts

Newer isn't always better. Here are real scenarios where upgrading the HTTP version caused problems or was completely pointless.

"We switched to HTTP/2 and our site got slower." — This actually happened to multiple companies. How? They had spent years optimizing for HTTP/1.1's limitations: concatenating files, using CSS sprites, domain sharding across 4 CDN domains. These HTTP/1.1 workarounds hurt HTTP/2 performance.

Why? Domain sharding forces HTTP/2 to open multiple connections (one per domain) instead of multiplexing everything on one. File concatenation means one huge JavaScript bundle instead of many small files — you can't cache individual components, and a one-line change invalidates the entire bundle. CSS sprites prevent HTTP/2 from prioritizing critical above-the-fold images.

Other anti-lessons:

  • "HTTP/3 will fix our slow API." If your API endpoint takes 2 seconds because of a slow database query, HTTP/3 won't help. Protocol improvements only help with transport overhead, not server-side processing time.
  • Corporate firewalls blocking QUIC. Many corporate networks block UDP traffic (since traditional web traffic only used TCP). HTTP/3 runs on UDP, so it gets blocked. Browsers fall back to HTTP/2, but the fallback detection itself adds latency. Some companies explicitly disable HTTP/3 to avoid this.
  • 0-RTT false security. Some developers see "0-RTT" and assume it's free speed with no downsides. But 0-RTT data can be replayed by attackers, so it must only be used for safe, idempotent requests.
The rule: Before upgrading HTTP versions, audit your existing optimizations. Remove HTTP/1.1 workarounds that conflict with HTTP/2. Test on your actual network conditions — the benefits of HTTP/3 are dramatic on lossy mobile networks but negligible on fast corporate LANs.
Section 11

Common Mistakes — What Engineers Get Wrong

These are the mistakes engineers and interview candidates make most often when discussing HTTP evolution.

Mistake #1: "HTTP/2 is always faster than HTTP/1.1"

Why it's wrong: On a page with very few resources (1-3 files), HTTP/2 can actually be slightly slower due to the overhead of the binary framing layer and connection setup negotiation. HTTP/2's multiplexing advantage only kicks in when there are many concurrent requests.

The truth: HTTP/2 is faster for typical web pages (20+ resources). For simple APIs returning one JSON response, the difference is negligible. The real win is at the margin — the 95th percentile load time, not the median.

Mistake #2: "HTTP/3 replaces TCP with UDP, so it's unreliable"

Why it's wrong: QUIC uses UDP as a transport, but builds reliability on top of it. QUIC has its own retransmission, congestion control, and ordering guarantees — just like TCP. It's not "raw UDP."

The truth: QUIC is essentially "TCP 2.0" implemented in userspace on top of UDP. It's just as reliable as TCP. The reason it uses UDP is practical: operating systems ship with TCP and UDP. Creating a brand-new transport protocol would require OS kernel changes that would take decades to deploy. UDP was the only available "blank canvas."

Mistake #3: "Server push makes websites faster automatically"

Why it's wrong: Server push sounds great in theory — send resources before the client asks! But in practice, the server doesn't know what's already in the browser's cache. If the browser already has style.css cached, pushing it again wastes bandwidth. Chrome removed server push support in 2022 because the cache invalidation problem made it counterproductive more often than helpful.

What replaced it: The 103 Early Hints status code. Instead of pushing the actual file, the server sends a hint: "you're going to need style.css soon." The browser can then decide whether to fetch it (if not cached) or ignore the hint (if cached). Same benefit, no wasted bandwidth.

Mistake #4: "HTTP/2 eliminates the need for CDNs"

Why it's wrong: HTTP/2 reduces the number of round trips, but doesn't reduce the speed of light. If your server is in Virginia and your user is in Mumbai, each round trip still takes ~200ms. HTTP/2 reduces the number of round trips, but you still want to serve from a location close to the user.

The truth: CDNs and HTTP/2 solve different problems. CDNs reduce latency by moving content closer to users. HTTP/2 reduces the number of round trips needed. Use both for the best results.

Mistake #5: "QUIC can't be inspected by firewalls, so it's a security risk"

The nuance: This is partially true and is a real concern in enterprise environments. QUIC encrypts almost everything — including the handshake metadata that firewalls traditionally inspect. Network middleboxes that do deep packet inspection on TCP/TLS traffic can't easily inspect QUIC.

The trade-off: This is by design — QUIC was built to prevent middlebox interference, which historically caused bugs when middleboxes corrupted TCP options or TLS extensions. But enterprises that need to inspect all traffic for compliance have a legitimate concern. The solution is usually: terminate QUIC at the edge (load balancer / reverse proxy) and use plain HTTP/2 or TCP internally.

  • HTTP/3 uses QUIC instead of TCP for better mobile performance

Sample answer when asked "Why is HTTP/2 faster?":

"HTTP/1.1 sends requests one after another on each connection. Even with 6 parallel connections, a page with 50 resources has to wait in batches. HTTP/2 sends all requests at once on a single connection — the server responds with pieces of different files interleaved together. So the total time equals the slowest single file, not the sum of all files."

What they expect on top of junior:

  • Binary framing layer and stream IDs
  • HPACK header compression and why it matters
  • TCP-level head-of-line blocking as HTTP/2's weakness
  • When to use gRPC (HTTP/2) vs REST vs WebSocket

How to bring it up naturally in a design:

"For service-to-service communication, I'd use gRPC over HTTP/2. It gives us multiplexing so hundreds of RPCs can share a single connection between each service pair, plus Protocol Buffers for compact serialization. For the client-facing API, standard REST over HTTP/2 — the frontend gets multiplexing for free, and we avoid the complexity of gRPC in the browser."

What sets senior answers apart:

  • Understanding trade-offs: 0-RTT replay risk, QUIC's UDP-blocking problem in enterprise networks
  • Knowing that HTTP/1.1 optimizations (domain sharding, sprites) become anti-patterns in HTTP/2
  • Connection migration for mobile-first architectures
  • When HTTP/3 matters (lossy networks) vs when it doesn't (datacenter)

Impressing the interviewer:

"For this mobile-first app, I'd serve through Cloudflare with HTTP/3 enabled. Our users are mostly on mobile — lossy networks where TCP-level HOL blocking kills HTTP/2 performance. QUIC's independent streams mean a lost packet on one image download doesn't freeze the API response. And connection migration means users walking through a building switching between Wi-Fi and cellular won't lose their session. We'd still need graceful fallback to HTTP/2 though — some corporate networks block UDP."

Frequently Asked Interview Questions

Q: "What's the difference between HTTP/1.1, HTTP/2, and HTTP/3?"

Framework answer: Each version solves the previous one's biggest bottleneck. HTTP/1.1 solved connection waste (keep-alive). HTTP/2 solved application-level head-of-line blocking (multiplexing). HTTP/3 solved transport-level head-of-line blocking (QUIC). Then add one concrete detail per version — binary framing for HTTP/2, 0-RTT for HTTP/3.

Q: "Why did HTTP/3 move from TCP to UDP?"

Key points: TCP's ordering guarantee causes all HTTP/2 streams to freeze when one packet is lost. Building a new transport protocol from scratch would require OS kernel changes — that takes decades. UDP was the only "blank slate" available that could be deployed immediately. QUIC adds reliability, congestion control, and encryption on top of UDP in userspace — it's essentially TCP 2.0 but deployable without OS upgrades.

Q: "Should our API use HTTP/2 or HTTP/3?"

Decision framework: For server-to-server (datacenter): HTTP/2 is fine — packet loss is minimal, and QUIC's benefits are negligible on low-latency LANs. For client-facing mobile-first: HTTP/3 if your CDN/load balancer supports it — the 0-RTT and connection migration benefits are significant. For internal microservices: gRPC over HTTP/2 — you get multiplexing, streaming, and compact serialization. Always have HTTP/2 as a fallback since not all clients/networks support QUIC.

Q: "What is head-of-line blocking and how does each version handle it?"

Three-layer answer: (1) HTTP/1.1 has HOL blocking at the application layer — one slow response blocks all requests behind it on the same connection. Workaround: open 6 connections. (2) HTTP/2 eliminates application-layer HOL blocking with multiplexing. But TCP still has HOL blocking — a lost packet blocks all streams. (3) HTTP/3 eliminates both — QUIC treats each stream independently, so loss on one stream doesn't affect others.

Hands-On Exercises

Theory is important, but nothing beats seeing these protocols in action. Here are exercises that go from "open your browser" to "analyze real traffic."

Exercise 1: Inspect HTTP Versions in Your Browser Beginner

Open Chrome DevTools (F12) → Network tab. Visit a popular website (e.g., google.com). Click the "Protocol" column header (right-click column headers to add it if missing). Notice which resources load over h2 (HTTP/2) vs h3 (HTTP/3). Try reloading — do more resources switch to h3 on the second load? Why?

Hint

On the first visit, the browser doesn't know the server supports HTTP/3. It connects via HTTP/2 and learns about HTTP/3 from the Alt-Svc response header. On the second visit, it uses HTTP/3 directly. This is called the "discovery" mechanism.

Exercise 2: Measure the Multiplexing Difference Intermediate

Set up a simple page that loads 50 small images. Serve it with Nginx configured for HTTP/1.1 only, then reconfigure for HTTP/2. Use Chrome DevTools' waterfall view to compare the loading patterns. Calculate the total load time difference and note how the waterfall shape changes (sequential bars vs parallel bars).

Hint

For HTTP/1.1, you'll see 6 parallel "lanes" of sequential downloads (the browser's 6-connection limit). For HTTP/2, you'll see all 50 images start at roughly the same time. The difference is most dramatic on high-latency connections — try throttling to "Slow 3G" in DevTools for a dramatic effect.

Exercise 3: Analyze Header Compression Intermediate

Using curl -v or Wireshark, capture the headers of 5 sequential requests to the same server. Calculate the total header bytes in HTTP/1.1 (plain text). Then estimate the HPACK-compressed size (most repeated headers reduce to 1-2 bytes after the first request). What's the compression ratio for your specific case?

Hint

Use curl -v --http1.1 https://example.com to see HTTP/1.1 headers, then curl -v --http2 https://example.com for HTTP/2. Count the bytes of common repeating headers (Cookie, User-Agent, Accept, Accept-Encoding). Typical compression: 85-95% for requests after the first one.

Exercise 4: Test 0-RTT with QUIC Advanced

Set up a Caddy server (it enables HTTP/3 by default). Visit it in Chrome, check the protocol in DevTools. Close the tab. Visit again and check if the connection time drops to near-zero (0-RTT). Use chrome://net-internals/#quic to inspect QUIC connection details including 0-RTT status.

Hint

The QUIC session ticket is cached for a limited time (usually a few hours). If you wait too long between visits, 0-RTT won't be available. Check chrome://net-internals/#quic for "zero_rtt_attempted" and "zero_rtt_accepted" fields in the session info.

Exercise 5: Simulate Packet Loss — HTTP/2 vs HTTP/3 Expert

Using a tool like tc (Linux traffic control) or clumsy (Windows), introduce 2-5% packet loss on your network. Load the same page over HTTP/2 and HTTP/3, measuring Time to First Byte and full page load time. Graph the results at 0%, 1%, 2%, 5%, and 10% loss rates. At what loss rate does HTTP/3 significantly outperform HTTP/2?

Hint

At 0-1% loss, the difference is minimal. At 2-3%, HTTP/3 starts to pull ahead significantly. At 5%+, HTTP/2 performance degrades dramatically (TCP retransmissions block everything) while HTTP/3 degrades gracefully (only affected streams slow down). This matches Cloudflare's real-world data.
