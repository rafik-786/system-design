Stripe processes billions in payments with PostgreSQL's SERIALIZABLE isolation — they'd rather be slow than lose a cent. Discord stores 12 billion messages per week in Cassandra — they'd rather be fast than perfectly synchronized. Two philosophies, both correct, for completely different reasons.

Section 2

The Scenario — The Disappearing $500

Let's make this painfully concrete. You open your banking app and transfer $500 from checking to savings. Behind the scenes, the database runs two SQL statements back to back:

bank-transfer.sql
-- This is a REAL PostgreSQL transaction. You can run this yourself.
BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;  -- debit checking
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;  -- credit savings
COMMIT;

Two statements. One subtracts. One adds. Seems simple enough. But what happens if the PostgreSQL server crashes after the first UPDATE (money leaves checking) but before the second UPDATE (money arrives in savings)?

Without ACID, your $500 is gone. Not in checking, not in savings — just... vanished. This is called a partial failureWhen a multi-step operation completes some steps but fails before finishing all of them, leaving the system in an inconsistent state. In the bank example: Account A lost $500 but Account B never gained it. The total money in the system decreased, which violates the most basic accounting rule. ACID prevents this by treating all steps as one atomic unit., and in banking, it's catastrophic. Not "bug report" catastrophic — "regulatory fine and front-page news" catastrophic.

Here's what actually happens when you wrap those two UPDATEs in a BEGIN; ... COMMIT; block. PostgreSQL writes every change to the WALWrite-Ahead Log — a sequential file on disk where PostgreSQL records every intended change BEFORE applying it. Think of it like writing "I'm about to move $500" in a notebook before actually doing it. If the server crashes, PostgreSQL reads the notebook on restart and undoes anything that wasn't fully completed. This is the "D" in ACID (Durability) and enables the "A" (Atomicity). Every ACID database has some version of this: MySQL calls it the redo log, SQL Server calls it the transaction log. (Write-Ahead Log) before touching the actual data. If the server crashes mid-transaction, PostgreSQL reads the WAL on restart, sees an incomplete transaction, and rolls back every change in it. Your checking account goes back to $1,000. The $500 never disappears.

This isn't theoretical. You can test it yourself. Start a transaction, kill the PostgreSQL process mid-way, restart it, and check — the data will be as if the transaction never started. The WAL is the safety net that makes ACID possible.

Now the Other Side: Discord Messages

Different scenario entirely. Someone sends "Hey everyone!" in a Discord server with 500,000 members. Discord writes that message to a CassandraA distributed NoSQL database built for massive write throughput. Discord chose Cassandra because it handles billions of message writes per week across multiple data centers with no single point of failure. The trade-off: when you write to one node, other nodes get the update a few milliseconds later (eventual consistency), not instantly. node in us-east-1. At that exact moment, your friend in London reads the channel from a node in eu-west-1 — and doesn't see the message yet. 100 milliseconds later, the EU node catches up and the message appears.

Was anyone harmed? Did data get corrupted? No. The message was just briefly invisible to some users. Discord processes 12 billion messages per week this way. They made a deliberate engineering choice: a 100ms delay in message visibility is a perfectly acceptable trade-off for the ability to handle that scale without any downtime.

Think First

If Discord used ACID for messages — every message must be confirmed written to ALL Cassandra nodes worldwide before showing it on anyone's screen — how long would each message send take? Their nodes span US, EU, and Asia.

Network round-trip US to Asia: ~150ms. US to EU: ~100ms. If a message must be confirmed on all nodes before appearing, every "Hey!" takes at least 150ms just for network physics. Multiply that by 12 billion messages per week. The system would collapse under the latency. By accepting eventual consistency (BASE), Discord writes locally in ~2ms and replicates in the background. The 100ms of staleness buys them 75x faster writes.

Two stories. Two very different stakes. When the server crashes during a bank transfer, someone loses real money. When a Discord message takes 100ms to propagate, nobody even notices. The entire ACID-vs-BASE debate comes down to one question: what happens when things go wrong? If the answer is "someone loses money," use ACID. If the answer is "someone sees a slightly stale number for a moment," use BASE.

First Attempt — "Just Make Everything ACID"

When you first learn about ACID, the reaction is obvious: "This is amazing! If ACID prevents data corruption, why would I ever use anything else?" Fair question. Let's try it.

Imagine you're building a social media platform. Every like, comment, feed refresh, and notification — all wrapped in strict SERIALIZABLE transactionsThe strongest isolation level in SQL databases. In PostgreSQL, you can enable it with: SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; This makes every transaction behave as if it ran completely alone, one at a time. It's the gold standard of correctness — but also the slowest, because the database must detect and abort conflicting concurrent transactions.. Maximum correctness for everything.

On a single PostgreSQL server, this actually works great. You can try it yourself:

-- Upgrade to the strictest level

-- Now every read and write in this transaction is bulletproof.
-- Open two psql sessions, try to UPDATE the same row simultaneously.
-- One will succeed. The other will get:
-- ERROR: could not serialize access due to concurrent update

All your data is in one place. TransactionsA group of SQL statements that the database treats as a single unit. Either ALL succeed (COMMIT) or ALL are undone (ROLLBACK). You can see the WAL location with: SHOW data_directory; and look inside the pg_wal/ folder. are fast because everything is local — no network calls, no waiting for other servers. A like takes 2ms. Life is good.

But now your app grows. A million users. Ten million. You need servers in multiple regions so users everywhere get fast responses. You spread your database across three data centers — Virginia, Ireland, Mumbai. And suddenly, every ACID transaction becomes agonizingly slow.

Why? Because ACID requires all replicas to agree before confirming a write. That means network round-trips. Virginia to Mumbai: ~150ms one way. Virginia to Ireland: ~85ms one way. A simple "like" that took 2ms locally now needs 300ms+ because it's waiting for acknowledgments from servers on other continents. That's not a bug — that's physics. Light travels through fiber at about 200,000 km/s, and Mumbai is 13,000 km from Virginia.

Let's do the math. Your ACID transaction needs confirmation from all 3 data centers. If the round-trips are sent in parallel (Virginia to Ireland = 85ms, Virginia to Mumbai = 150ms), what's the minimum latency for a single write? What if you're doing this for 100,000 likes per second?

In parallel: you wait for the slowest — 150ms (Mumbai round-trip). Add ~5ms local processing = 155ms per transaction. But here's the killer: at 155ms per transaction per row, a single row can only handle ~6 writes/second before transactions start queuing. For 100,000 likes/second spread across posts, you'd need locking coordination across continents on hot rows. The system would grind to a halt. This is why nobody runs ACID across global data centers for high-throughput operations.
The core problem: ACID on a single server is easy and fast (~2ms). ACID across a distributed system is painfully slow (~300ms) because of physics — light speed through fiber, plus the coordination overhead of two-phase commit2PC (Two-Phase Commit) is the standard protocol for distributed ACID transactions. When one data center goes down, ACID requires the entire system to stop (because you can't get agreement from a dead server). This is the exact problem that led engineers to invent BASE.
Section 4

Where It Breaks — 2PC Latency Math and the CAP Wall

Let's get specific about why distributed ACID is so painful. The standard protocol for making ACID work across servers is called Two-Phase Commit (2PC)A coordination protocol invented in the 1980s. Phase 1: the coordinator sends "PREPARE" to all nodes. Each node writes its changes to a local log and replies "READY." Phase 2: if all replied "READY," the coordinator sends "COMMIT" to all. If any replied "ABORT" (or didn't respond), coordinator sends "ROLLBACK" to all. The fatal flaw: if the coordinator dies between phases, all participants are locked, waiting for instructions that may never come.. Here's what it actually looks like in terms of time:

Let's break that down. For a single distributed write:

  • Phase 1 (Prepare): Coordinator sends PREPARE to Ireland (85ms) and Mumbai (150ms) in parallel. Waits for the slowest: 150ms. They process and reply: another 150ms back. Phase 1 total: ~300ms.
  • Phase 2 (Commit): Coordinator sends COMMIT. Same round-trips: ~300ms.
  • Grand total: ~600ms for a single write that took 2ms on one server. That's a 300x slowdown.

And that's the happy path. If Mumbai's network goes down mid-Phase-1, the coordinator is stuck. It sent PREPARE to Ireland (which locked its resources) but can't reach Mumbai. It can't commit (Mumbai didn't confirm). It can't rollback safely (maybe Mumbai got the message and is waiting). Everyone is blockedThis is the Achilles' heel of 2PC. When the coordinator fails between phases, participants who said "READY" are in limbo — they've locked their data but don't know whether to commit or rollback. They must wait for the coordinator to come back. This can block the entire system for minutes or hours. Modern alternatives like 3PC and Paxos reduce this problem but add even more latency., waiting for a node that might not come back for hours.

This is where the CAP theoremProven by Eric Brewer (UC Berkeley) in 2000 and formally proved by Seth Gilbert and Nancy Lynch (MIT) in 2002. It states: in a distributed system experiencing a network partition, you must choose between consistency (all nodes see the same data) and availability (all nodes respond to requests). You cannot have both during a partition. Since partitions are inevitable (cables break, routers fail), the practical choice is: do you favor consistency (ACID/CP) or availability (BASE/AP)? enters. When that cable between Virginia and Mumbai goes down (and it will — undersea cables get damaged by ship anchors about 100-200 times per year worldwide), you face an impossible choice:

This isn't a flaw in engineering — it's a mathematical impossibility, proven by Seth Gilbert and Nancy Lynch at MIT in 2002They formally proved Eric Brewer's 2000 conjecture: no distributed system can simultaneously guarantee Consistency, Availability, and Partition tolerance. The paper is "Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services" — one of the most cited papers in distributed systems.. No amount of clever code, hardware upgrades, or expensive consultants can give you consistency AND availability during a network partition. You must choose.

ACID systems choose consistency: they become unavailable during partitions. BASE systems choose availability: they accept temporary inconsistency during partitions. Neither is wrong — they're answering different questions about what's more important when things go bad.

Think First

The network between US and EU data centers goes down for 2 hours. During that time, a customer in Paris tries to transfer $10,000 between bank accounts. Meanwhile, a customer in Tokyo tries to send a Discord message. For each case: should the system reject the request (stay consistent) or accept it and sort it out later (stay available)?

Bank transfer: REJECT. If both US and EU independently process the transfer, the customer might end up with $10,000 deducted twice or credited twice when the partition heals. Wrong balances in banking means regulatory fines and lawsuits. Discord message: ACCEPT. Write the message to the local node. When the partition heals, both sides merge their messages. The worst case? A message shows up a couple of hours late. That's annoying but harmless. This is why Stripe uses PostgreSQL (CP) and Discord uses Cassandra (AP).
Section 5

The Breakthrough — Use Both (Amazon Invented This)

In 2007, Amazon published a paper that changed distributed systems forever. It was called Dynamo: Amazon's Highly Available Key-value Store, and it contained a sentence that became legendary in the engineering world:

Werner Vogels (Amazon CTO), 2007: "Customers should be able to view and add items to their shopping cart even if disks are failing, network routes are flapping, or data centers are being destroyed by tornados. Therefore, the service responsible for managing shopping cart state must always be available — it should be able to write to and read from its data store, under all circumstances."

This was the breakthrough: you don't pick one philosophy for your entire system. You pick the right philosophy for each piece of data. Money? ACID. Shopping cart? BASE. Same company, same application, different guarantees for different tables.

Amazon didn't invent this idea from theory — they learned it from pain. They discovered that making the shopping cart strictly consistent (ACID) meant that during network hiccups, customers couldn't add items. Every second of downtime on the "Add to Cart" button cost Amazon millions of dollars in lost sales. So they built DynamoDB — a BASE system — specifically for shopping cart state. If the cart briefly shows 2 items instead of 3 during a network partition, that's fine. When the partition heals, the items merge. But the cart was always available.

Here's the decision process in a single visual — the flowchart every senior engineer runs in their head:

You can see this pattern everywhere once you know to look for it. Every company worth their engineering salt uses both ACID and BASE:

  • Stripe: PostgreSQL with SERIALIZABLE isolation for every payment transaction. Every charge, refund, and transfer. They process billions of dollars this way. For user sessions and dashboard analytics? Redis and Kafka (BASE). If a dashboard chart is 30 seconds behind, nobody cares.
  • Discord: Cassandra for messages (BASE) — 12 billion messages per week, eventual consistency is fine. PostgreSQL for Nitro billing (ACID) — subscription charges must be exactly right.
  • Amazon (the Dynamo paper): DynamoDB for shopping cart (BASE) — "always available, even during failures." Aurora/RDS for order processing and payment settlement (ACID).

You can even see BASE tuning yourself. In DynamoDB, you choose consistency per read:

dynamodb-consistency-choice.sh
# Eventually consistent read (default) — fast, cheap, might be stale
aws dynamodb get-item \
  --table-name Users \
  --key '{"id":{"N":"1"}}'

# Strongly consistent read — slower, 2x cost, guaranteed fresh
aws dynamodb get-item \
  --table-name Users \
  --key '{"id":{"N":"1"}}' \
  --consistent-read

And in Cassandra, you tune consistency per query:

cassandra-consistency-levels.cql
-- ONE: fire-and-forget, fastest, least consistent
INSERT INTO messages (id, text) VALUES (uuid(), 'Hey!')
  USING CONSISTENCY ONE;

-- QUORUM: majority must acknowledge, good balance
INSERT INTO orders (id, total) VALUES (uuid(), 99.99)
  USING CONSISTENCY QUORUM;

-- ALL: every replica must acknowledge (basically ACID-like)
-- Don't use this unless you really need it — kills availability
INSERT INTO payments (id, amount) VALUES (uuid(), 500.00)
  USING CONSISTENCY ALL;

This is the real-world superpower: you don't just choose ACID or BASE at the system level — you choose it per table, per query, even per request. Cassandra's CONSISTENCY QUORUM is stricter than ONE but looser than ALL. DynamoDB's --consistent-read flag upgrades a single read from eventual to strong. The spectrum between "pure ACID" and "pure BASE" is continuous, and production systems live somewhere in the middle.

The one-question rule: For every piece of data in your system, ask: "What's the worst thing that happens if this data is wrong for 5 seconds?" If the answer involves money, health, or legal liability — ACID. If the answer is "a number is briefly off" or "a message appears a moment late" — BASE. Apply this per-table, not per-system.
Think First

You're designing an airline booking system. Classify each piece of data as ACID or BASE: (a) seat reservation, (b) loyalty miles balance, (c) "23 people viewing this flight right now", (d) baggage tracking location, (e) in-flight entertainment watch history.

(a) ACID — double-selling a seat means two people show up for the same spot; legal nightmare. (b) ACID — miles are currency; losing them triggers support tickets and lawsuits. (c) BASE — "23 people" vs "25 people" is just social proof; nobody cares if it's slightly off. (d) BASE — knowing the bag is "in Frankfurt" vs "arriving in Frankfurt" is close enough; exact real-time position can be eventually consistent. (e) BASE — if your watch history takes 30 seconds to sync between devices, that's totally fine.
Section 6

How It Works — The Properties Explained

Now that you understand why ACID and BASE exist, let's dig into the individual properties. ACID has four guarantees, BASE has three principles. Each one solves a specific problem.

The Four ACID Guarantees

Atomicity — All or Nothing

Imagine you're moving apartments. You hire a truck. Everything you own goes in the truck. The truck drives to the new apartment. Everything comes out. That's a successful move.

But what if the truck breaks down halfway? You don't want half your stuff at the old apartment and half on the highway. You'd rather the move "didn't happen" — everything stays at the old place, and you try again tomorrow.

That's atomicityThe "A" in ACID. A transaction is atomic if it's treated as a single, indivisible unit. Either every operation in the transaction succeeds (commit), or every operation is undone as if nothing happened (rollback). There's no in-between state.. A database transaction is treated as a single, indivisible unit. If any step fails, all steps are undone. The database rolls back to the state before the transaction started, as if nothing happened.

The key insight: atomicity means you don't have to write error-handling code for half-completed operations. The database handles the cleanup for you. If anything goes wrong, it's as if the entire transaction never started.

Consistency — Data Always Valid

Think of a game of chess. Every move must follow the rules — a bishop can only move diagonally, a pawn can't jump backwards. If someone tries an illegal move, the game rejects it. The board is always in a valid state.

ConsistencyThe "C" in ACID. It guarantees that a transaction can only move the database from one valid state to another valid state. All rules (constraints, foreign keys, triggers, check conditions) must be satisfied after the transaction. If any rule would be violated, the entire transaction is rejected. in ACID works the same way. You define rules for your data (a bank balance can't go negative, an email must be unique, a foreign key must point to a real record), and the database guarantees that no transaction can leave the data in a state that violates those rules.

If a transaction would result in a negative bank balance, the database rejects the entire transaction. Not just the offending step — the entire transaction. This works hand-in-hand with atomicity: if the transaction can't produce a valid end state, it's rolled back completely.

Important distinction: "Consistency" means something different in ACID vs BASE vs CAP. In ACID, it means "data follows all the rules." In CAP, it means "all nodes see the same data at the same time." Don't mix them up — interviewers love testing this.

Isolation — Transactions Don't Interfere

Imagine two people trying to buy the last concert ticket at the exact same time. Without isolation, both might read "1 ticket available," both proceed to purchase, and now you've sold 2 tickets when only 1 existed. That's a mess.

IsolationThe "I" in ACID. It ensures that concurrent transactions don't interfere with each other. Each transaction behaves as if it's the only one running, even when thousands are executing simultaneously. Isolation has levels (from Read Uncommitted to Serializable) that trade strictness for performance. makes each transaction behave as if it's the only one running, even when thousands are happening simultaneously. The database uses clever locking and versioning behind the scenes to prevent conflicts.

Isolation comes in different levels (we'll cover these in detail in Section 7). The strictest level, Serializable, makes transactions behave as if they ran one at a time. The most relaxed level, Read Uncommitted, lets you see other transactions' uncommitted changes. Most databases default to somewhere in the middle.

Durability — Committed Data Survives Crashes

You've just completed a bank transfer. The system says "Success!" Then the power goes out. When the server reboots, is your transfer still there?

With durabilityThe "D" in ACID. Once a transaction is committed (the database confirms success), the data is permanently saved — even if the server crashes, loses power, or reboots immediately after. This is typically achieved through write-ahead logs (WAL) written to disk before confirming the transaction., yes, always. Once the database says "committed," that data is written to permanent storage (disk). Even if the server crashes one millisecond later, when it reboots, your data is there. The database achieves this through a technique called a write-ahead log (WAL)A file where the database writes every change BEFORE applying it to the actual data files. If the server crashes mid-write, the WAL lets the database reconstruct and replay the changes on reboot. Think of it like a receipt — you write down what you're about to do before doing it. — it writes every change to a log file on disk before confirming the transaction. If the server crashes, it replays the log on reboot.

Common misconception: "I'm using an SSD, so my data is durable." Not quite. Durability requires the database to explicitly flush writes to disk (fsync). Without this, data might sit in an OS buffer and be lost on crash. The database must confirm the disk write, not just the buffer write.

The Three BASE Principles

If ACID is a set of guarantees ("the database WILL do this"), BASE is more of a philosophy ("the system is DESIGNED to work this way"). BASE systems don't enforce these properties through rigid rules — they embrace them as features of distributed architectures.

Basically Available — Always Responds

Think about Google Search. Have you ever gotten a "Service Unavailable" error from Google? Probably not. Even when parts of Google's infrastructure are having problems (and with tens of thousands of servers, something is always broken), the search page still loads and gives you results.

That's basically availableThe "BA" in BASE. The system guarantees that every request will receive a response — it won't hang or refuse connections. However, the response might be based on slightly stale data, or might come from a backup server. The focus is on always being responsive, even during failures.. The system always responds to requests, even during partial failures. The response might not be perfectly up-to-date — you might get cached data, or results from a backup server — but you always get something back. The system never goes fully dark.

Compare this to a strict ACID system during a failure: if it can't guarantee the data is correct, it might refuse to answer at all. BASE says "I'll give you the best answer I can right now" instead of "I can't answer, sorry."

Soft State — Data May Change Without Input

In a traditional ACID database, data only changes when someone explicitly writes to it. If nobody touches a row, it stays the same forever. That's "hard state" — solid, unchanging.

Soft stateThe "S" in BASE. The system's state may change over time even without new input, due to background processes like replication, cache expiration, or data propagation between nodes. The data "flows" through the system rather than being instantly "set" everywhere. means data can change even without any new input, because background processes are constantly syncing, replicating, and reconciling data across nodes. Your cached product price might update itself when the cache discovers the source has changed. A replica database might receive a new version of a row that it hadn't seen yet. The data is "soft" — fluid, flowing, always settling.

This sounds unsettling at first, but it's actually the normal state of most distributed systems. Think about DNS — when a website changes its IP address, that change takes hours to propagate across the world's DNS servers. During that time, different DNS servers return different answers. The state is soft, and that's by design.

Eventual Consistency — Given Time, All Nodes Converge

This is the most important BASE concept. When you write data to one node in a distributed system, it doesn't instantly appear on every other node. There's a delay — maybe milliseconds, maybe seconds. But the system guarantees that given enough time and no new updates, all nodes will eventually return the same value.

Think of it like a rumor spreading through a school. You tell one person. They tell two friends. Those friends tell two more. For a while, some people know the rumor and some don't. But given enough time, everyone knows it. That's eventual consistencyA consistency model where, after a write, not all nodes immediately see the update. However, given enough time (and assuming no further writes), all replicas will converge to the same value. The "inconsistency window" is the time between the write and full convergence — typically milliseconds to seconds..

The key question for any BASE system is: how long is "eventually"? For a like count on Instagram, 2 seconds is fine. For a product price on Amazon, maybe 5 seconds. For DNS records, hours are acceptable. You choose the inconsistency window based on what your users will tolerate.

Interview tip: Many candidates confuse ACID's "consistency" (data follows all rules/constraints) with CAP's "consistency" (all nodes see the same data at the same time). They're related but different concepts. ACID consistency is about validity. CAP consistency is about agreement. Be precise when using the word "consistency" in interviews.
Section 7

Going Deeper — The Mechanics Under the Hood

You now understand what ACID and BASE mean. This section digs into how they're implemented in real systems. Four topics that interviewers love to ask about — and that come up in every real-world system design.

Isolation Levels — From Chaos to Perfection

Remember how isolation means "transactions don't interfere with each other"? In practice, you get to choose how much isolation you want. More isolation = safer but slower. Less isolation = faster but riskier. The SQL standard defines four levels:

The key takeaway: you almost never use the extremes. Read Uncommitted is too dangerous (you're reading data that might be rolled back). Serializable is too slow (transactions queue up and wait for each other). The middle two levels — Read Committed and Repeatable Read — hit the sweet spot for most applications.

Two-Phase Commit (2PC) — Distributed ACID

ACID is straightforward on a single database. But what if your transaction spans multiple databases or services? For example, an e-commerce order that charges a payment (Payment DB) and reserves inventory (Inventory DB). Both must succeed or both must fail. How do you coordinate that?

The answer is the Two-Phase CommitA protocol for achieving ACID transactions across multiple databases or services. A coordinator asks all participants "Can you commit?" (Phase 1). If everyone says yes, the coordinator says "Commit!" (Phase 2). If anyone says no, the coordinator says "Abort!" and everyone rolls back. Simple in theory, problematic in practice because a coordinator failure can leave participants stuck waiting. (2PC) protocol. Think of it like a wedding ceremony:

2PC works, but it has a critical flaw: the coordinator is a single point of failure. If it crashes between Phase 1 and Phase 2, all participants are stuck holding locks, waiting for a decision that may never come. This is called the blocking problem, and it's why 2PC is increasingly replaced by patterns like the Saga (next collapsible).

Performance cost: 2PC requires at least 2 network round-trips per transaction (prepare + commit). In a system doing 10,000 transactions per second, that's 20,000 extra network calls. This is why distributed ACID is expensive — and why many systems choose BASE for cross-service operations instead.
Saga Pattern — ACID Alternative for Microservices

In a microservices world, each service owns its own database. You can't do a traditional ACID transaction across services because there's no single database engine managing everything. The Saga patternA design pattern for managing data consistency across microservices without distributed transactions. Instead of one big ACID transaction, a saga breaks the operation into a sequence of local transactions, each with a compensating action (undo) that runs if a later step fails. Think of it as "do, do, do... oops, undo, undo, undo." is the industry-standard solution.

Instead of one big all-or-nothing transaction, a saga breaks the operation into a sequence of small, local transactions. Each step has a compensating action (an undo) that runs if a later step fails.

The crucial difference: 2PC holds locks on all databases during the entire transaction. A saga never holds distributed locks — each step commits locally and immediately. If something fails later, compensating actions clean up. It's less strict than ACID (there's a brief window where data is inconsistent), but it's much faster and doesn't have the coordinator single-point-of-failure problem.

Think First

A customer orders a limited-edition sneaker. The saga creates the order (Step 1), reserves the last pair of sneakers (Step 2), then tries to charge the credit card — but the payment fails. The compensating action releases the sneaker. But during that brief window (between reservation and release), another customer tried to order the same sneaker and was told "out of stock." Is this a problem?

Yes, it's a real issue! The second customer was turned away even though the sneaker was about to become available again. This is the "inconsistency window" of sagas. Solutions include: retrying the second customer's order automatically, using a short reservation timeout, or accepting that this edge case is rare enough to tolerate. This is the trade-off you accept when choosing sagas over 2PC.
Eventual Consistency in Practice — How Long Is "Eventually"?

When people hear "eventual consistency," the first question is always: "How long do I have to wait?" The answer depends entirely on the system and how it's configured. Here are real-world examples:

A common technique is read-your-writes consistency: after you write data, your subsequent reads always see that data (because the system routes your reads to the node you wrote to). Other users might see stale data briefly, but you always see your own changes. This is often enough to feel consistent without the performance cost of full ACID.

Section 8

Variations — ACID vs BASE by Industry

Different industries have very different tolerances for inconsistency. A banking app and a gaming app have fundamentally different data requirements. Here's how four major industries approach the ACID vs BASE decision — and why.

Banking & Finance — Pure ACID, No Compromises

Banks are the textbook ACID use case. Money transfers, loan disbursements, interest calculations, regulatory reporting — every single one of these operations must be perfectly correct, all the time. There's no such thing as "eventually consistent" in a bank balance. If your account says $5,000, it must be exactly $5,000 right now, not "probably $5,000, give it a sec."

Banks typically use heavy-duty relational databases like Oracle, PostgreSQL, or SQL Server with the highest isolation levels. They also use two-phase commitA distributed transaction protocol that ensures all participants (databases, services) either commit or abort together. Banks use it for cross-system operations like wire transfers that touch multiple core banking systems. It's slow but guarantees correctness. for cross-system operations (like wire transfers between institutions).

The trade-off: banking systems are slower than social media or gaming backends. A wire transfer might take seconds. International transfers take hours or days. Banks accept this latency because the cost of a wrong answer ($500 disappearing) is infinitely worse than the cost of a slow answer ("your transfer is processing").

Fun fact: Even banks use a little bit of BASE. Your bank's mobile app might show a cached balance that's a few seconds behind the real-time ledger. The displayed balance is "eventually consistent" — but the actual ledger (the source of truth) is always ACID.

E-Commerce — Mixed Strategy (ACID for Orders, BASE for Catalog)

E-commerce is the poster child for the "use both" approach. Different parts of the system have very different needs:

  • ACID: Order placement (can't oversell), payment processing (can't double-charge), inventory management (must be accurate).
  • BASE: Product search (Elasticsearch index can lag a few seconds), product reviews (a new review can take a moment to appear), recommendation engine (yesterday's data is fine), product catalog browsing (cached data is acceptable).

Amazon's architecture is a famous example. Their shopping cart uses an eventually consistent system (Dynamo, the internal ancestor of DynamoDB). Why? Because a shopping cart being temporarily inconsistent across data centers is harmless — worst case, an item briefly appears or disappears, and the user adds it again. But the order and payment pipelines are strictly ACID.

Think First

During a flash sale, 10,000 people try to buy the last 100 units of a product. Should the inventory check be ACID or BASE?

ACID, absolutely. If you use BASE for inventory during a flash sale, two things happen: (1) you might sell 200 units when only 100 exist (overselling), and (2) you have to cancel half the orders, making customers furious. The few hundred milliseconds of extra latency from ACID is vastly preferable to the PR disaster of overselling.

Social Media — Mostly BASE (Feeds, Likes, Comments)

Social media platforms like Instagram, Twitter/X, and TikTok are almost entirely BASE. Feeds, likes, comments, stories, followers, DMs — all of these tolerate brief inconsistency just fine.

Think about what happens when a celebrity with 50 million followers posts a photo. That post needs to appear in 50 million feeds. With ACID, you'd need to update 50 million rows in a transaction, across dozens of data centers, atomically. That's physically impossible at any reasonable speed. With BASE, the post is written to one node and replicates outward. Users in different regions see it at slightly different times — some instantly, some 2-5 seconds later. Nobody notices or cares.

The only parts of a social media platform that typically use ACID are:

  • User account creation (email must be unique, no duplicate accounts)
  • Payment processing (ad purchases, subscriptions, in-app purchases)
  • Content moderation actions (when a post is removed, it must disappear everywhere immediately)

Everything else — the vast majority of the system — is BASE. This is why platforms like Facebook can handle billions of interactions per day: they don't pay the ACID tax on operations that don't need it.

Gaming — BASE for Game State, ACID for Purchases

Online gaming has a unique challenge: real-time state that changes 30-60 times per second. Player positions, health bars, ammunition counts, in-game events — this data is changing constantly and needs to be broadcast to all players with minimal latency.

Strict ACID for game state would be absurd. If every bullet fired required a distributed ACID transaction, the game would run at 2 FPS instead of 60. Game servers use heavily optimized BASE systems (or custom in-memory solutions) where:

  • Each game server is the authoritative source for its session
  • Player clients get "good enough" state updates at 20-60 Hz
  • Minor inconsistencies are smoothed out with client-side prediction and server reconciliation
  • If two players see slightly different things for 50ms, that's acceptable

But the moment money is involved — purchasing a skin, buying in-game currency, trading items with real-world value — the system switches to ACID. Nobody wants their $20 purchase to vanish because a game server hiccupped.

Pattern recognition: Notice the trend across all four industries? The split is always the same: ACID where money or critical data is at stake, BASE for everything else. The ratio changes (banks: 95/5, social media: 5/95), but the principle is universal.
Section 9

At Scale — How the Giants Use ACID & BASE

Theory is great, but what actually happens when billions of dollars and petabytes of data are on the line? Let's look at four real-world stories that show how companies pick — and sometimes combine — ACID and BASE.

Stripe — ACID for Every Cent

Stripe processes hundreds of billions of dollars in payments each year. A single misplaced decimal or a phantom readWhen a transaction re-runs a query and gets different rows because another transaction inserted data in between could mean a merchant gets paid twice — or not at all. That's why Stripe runs on PostgreSQL with strict serializable isolation.

Every charge, refund, and transfer lives inside an ACID transaction. If any step fails — charging the card, recording the ledger entry, notifying the merchant — everything rolls back. The user sees "payment failed," not "money disappeared." Stripe even built custom tooling on top of Postgres to shard their data while keeping transactions ACID-compliant within each shard.

Key takeaway: When money is involved, ACID isn't optional — it's a legal and business requirement. The performance cost of serializable isolationThe strictest isolation level — transactions behave as if they ran one at a time, preventing all concurrency anomalies is worth it when the alternative is financial inconsistency.

Cassandra — BASE for Write-Heavy Workloads

Apache Cassandra was designed at Facebook to handle the inbox search problem: billions of messages written across global data centers. Waiting for every node to agree before confirming a write would make the system unusably slow. So Cassandra went all-in on BASE.

When you write a message, Cassandra confirms the write as soon as a quorumA majority of replicas — e.g., 2 out of 3 nodes must acknowledge the write before it's considered successful of replicas acknowledge it. The remaining replicas catch up in the background through a process called anti-entropy repair. For a brief window, different replicas might return different versions of the data — that's eventual consistencyGiven enough time with no new writes, all replicas will converge to the same value — but reads during the convergence window might be stale in action.

Companies like Netflix, Discord, and Instagram use Cassandra for timelines, activity feeds, and messaging — workloads where writing fast matters more than reading the absolute latest value.

Amazon — Hybrid: BASE for Cart, ACID for Orders

Amazon famously uses both models in the same shopping flow. Here's how:

Shopping cart → DynamoDB (BASE). Your cart is stored in DynamoDB with eventual consistency. If you add an item on your phone and it takes 200 milliseconds to show on your laptop, nobody panics. Availability matters — Amazon never wants "add to cart" to fail, even during peak traffic like Prime Day.

Order placement → RDS/Aurora (ACID). The moment you click "Place Order," everything switches to a relational database with full ACID guarantees. Inventory gets decremented, payment gets charged, and the order record gets created — all atomically. If payment fails, inventory doesn't change.

The lesson: You don't have to pick one model for your entire system. Use ACID where correctness is critical (payments, orders, inventory) and BASE where speed and availability matter more (carts, recommendations, browsing history).

Google Spanner — Global ACID (Having It All)

For decades, the industry believed you had to choose: ACID or global scale. Google Spanner challenged that assumption. It provides full ACID transactions across data centers on different continents.

The secret sauce? TrueTimeGoogle's globally synchronized clock system using GPS receivers and atomic clocks in every data center — gives time uncertainty bounds of ~7 milliseconds — a custom clock system using GPS receivers and atomic clocks in every data center. By knowing the exact time (within a few milliseconds) everywhere, Spanner can order transactions globally without the expensive round-trips that traditional two-phase commitA protocol where a coordinator asks all participants to prepare, then tells them all to commit — guarantees atomicity but is slow and blocking requires.

The trade-off? Latency. A Spanner transaction takes 10-15 ms (compared to sub-millisecond on a local Postgres). But for Google's advertising and financial systems, that latency is acceptable because correctness at global scale is non-negotiable.

Section 10

Anti-Lesson — Myths That Refuse to Die

These three myths show up in blog posts, tech talks, and even interviews. They sound reasonable — but they're wrong, and believing them leads to bad architecture decisions.

Myth: "NoSQL Means No Transactions"

Why people believe it: Early NoSQL databases (Cassandra, Riak, early MongoDB) genuinely did not support multi-document transactions. The "NoSQL movement" of 2009-2012 marketed itself as a departure from relational databases — and transactions were part of what got left behind.

Why it's wrong now: Modern NoSQL databases have caught up. MongoDB added multi-document ACID transactions in version 4.0 (2018). Azure Cosmos DB supports transactions within a partition. FaunaDB was built from the ground up with distributed ACID. Even DynamoDB added TransactWriteItemsA DynamoDB API that lets you group up to 100 write actions into a single all-or-nothing transaction across multiple items and tables for cross-item atomic operations.

The reality: "NoSQL" describes the data model (document, key-value, graph), not the consistency model. You can absolutely have a NoSQL database with ACID guarantees — and many production systems rely on exactly that.

Myth: "Eventual Consistency = Data Loss"

Why people believe it: The word "eventual" sounds scary — like your data might never arrive. If a system is "eventually consistent," surely some writes get lost along the way?

Why it's wrong: Eventual consistency has nothing to do with durabilityThe guarantee that once a write is acknowledged, it won't be lost — even if the server crashes immediately after. When Cassandra acknowledges a write, that data is persisted to disk on multiple nodes. It's durable. The "eventual" part only means that not all replicas have the latest version yet. A reader might temporarily see an older value — but the new value is safely stored and will propagate.

Analogy: Imagine you update your profile photo on social media. Your friend in another country might see the old photo for a few seconds. That doesn't mean the new photo was lost — it just hasn't reached their nearest server yet. That's eventual consistency: the data is safe, the view is temporarily stale.

Myth: "ACID Is Always the Better Choice"

Why people believe it: ACID sounds objectively better — "I want my data consistent, isolated, and durable. Who wouldn't?" It feels like choosing safety.

Why it's wrong: ACID guarantees come with a performance cost. Strong isolation means locking rows or using multi-version snapshots. Distributed ACID means coordinating across nodes with protocols like two-phase commit — which means higher latency and lower throughput. At massive scale (millions of writes per second across global data centers), those coordination costs become the bottleneck.

Real example: Discord handles billions of messages daily. If they used strict ACID for every message insert, they'd need expensive cross-datacenter coordination for each "lol" sent in a chat. Instead, they use Cassandra (BASE) — because a chat message arriving 100ms late on one server is far better than the whole messaging system slowing to a crawl.

The right framing: It's not "ACID vs. BASE — which is better?" It's "For this specific piece of data, what are the consequences of temporary inconsistency?" If the answer is "financial loss or safety risk," use ACID. If the answer is "a user sees a stale like-count for a second," BASE is fine.

Section 11

Common Mistakes

These are the mistakes engineers actually make in production — not theoretical edge cases, but real bugs that cause real outages.

Mistake 1: Treating eventual consistency as "close enough to consistent"

What happens: A developer builds a checkout flow on an eventually consistent store. User clicks "buy," the order is written, then the code immediately reads back the order to show a confirmation page — but the read hits a replica that hasn't received the write yet. The user sees "no orders found" and clicks buy again, creating a duplicate order.

Fix: Use read-your-own-writes consistency. After a write, route subsequent reads for that user to the same node or use a session token that guarantees the read reflects the latest write. Most managed databases (DynamoDB, Cosmos DB) support this as a configurable consistency level.

Mistake 2: Using long-running transactions

What happens: A transaction opens, calls an external API (payment gateway, email service), waits for the response, then commits. The transaction holds locks for 2-5 seconds. Under load, other transactions queue up waiting for those locks, creating a cascading slowdown that brings down the database.

Fix: Keep transactions short. Do external calls outside the transaction. Use the pattern: (1) gather all data, (2) call external services, (3) open transaction, write results, commit. If you need atomicity across the external call, use the Saga patternA sequence of local transactions where each step has a compensating action — if step 3 fails, you undo steps 1 and 2 instead of rolling back a single giant transaction or outbox patternWrite the event to an 'outbox' table in the same transaction as the data change, then a separate process reads the outbox and sends the actual message instead.

Mistake 3: Ignoring conflict resolution in BASE systems

What happens: Two users update the same record on different replicas at the same time. The system uses last-write-winsA conflict resolution strategy where the write with the latest timestamp overwrites all others — simple but can silently discard valid updates (LWW) by default, silently discarding one user's changes. Nobody notices until a customer complains their profile update vanished.

Fix: Choose a conflict resolution strategy deliberately. Options include: application-level merge logic, CRDTs (conflict-free replicated data types) for counters and sets, or vector clocks to detect conflicts and surface them to the user (like Google Docs does).

Mistake 4: Not handling idempotency in retry logic

What happens: A BASE system times out on a write. The client retries. But the original write actually succeeded — now the same record is inserted twice, or a counter is incremented twice.

Fix: Make all writes idempotentAn operation that produces the same result whether you execute it once or multiple times — safe to retry without side effects. Use unique request IDs (idempotency keys). Before processing, check if this request ID was already handled. Payment systems like Stripe require an idempotency key on every API call for exactly this reason.

Mistake 5: Choosing consistency model per-database instead of per-operation

What happens: A team picks DynamoDB (BASE) for their entire service because "we need scale." But some operations — like updating account balances — genuinely need strong consistency. They end up building fragile workarounds on top of an eventually consistent store.

Fix: Many databases let you choose consistency per-query. DynamoDB supports ConsistentRead: true. Cosmos DB has five consistency levels selectable per request. Use strong consistency for critical reads and eventual consistency for everything else — on the same database.

Mistake 6: Assuming distributed transactions are free

What happens: A team introduces cross-service ACID transactions using two-phase commit. During normal operation, everything is fine. During a network partition, the coordinator can't reach a participant — and the entire transaction blocks indefinitely, taking down both services.

Fix: Avoid distributed transactions across service boundaries. Use the Saga pattern with compensating actions instead. If you must use 2PC, add aggressive timeouts and a "presumed abort" policy so blocked participants don't hold locks forever.

Section 12

Interview Playbook

ACID vs BASE is a favourite system design interview topic because it reveals whether you think in trade-offs or just memorize definitions. Here's what to expect at each level.

Typical question: "Explain ACID and BASE. When would you use each?"

What they're testing: Can you explain the four ACID properties and the three BASE properties in your own words — not just recite acronyms?

How to answer:

  1. Start with a one-sentence summary: "ACID prioritizes correctness; BASE prioritizes availability."
  2. Walk through each ACID property with a bank transfer example.
  3. Contrast with BASE using a social media feed example — a like count being briefly stale is fine.
  4. End with: "The choice depends on the cost of inconsistency for that specific data."

Typical question: "You're designing an e-commerce platform. Which parts use ACID and which use BASE?"

What they're testing: Can you apply the theory to a real system? Do you think in terms of trade-offs per component, not one-size-fits-all?

How to answer:

  1. Identify the data domains: payments, orders, inventory, cart, product catalog, reviews, recommendations.
  2. Classify each: "Payments and inventory need ACID — a sold-out item showing as available causes real harm."
  3. "Product catalog reads can be eventually consistent — a price update propagating in 2 seconds is fine."
  4. Mention the CAP theorem: "In a partition, our payment service chooses consistency (CP), our catalog chooses availability (AP)."

Sample dialogue:

Interviewer: "What if inventory needs to scale to millions of reads per second?"
You: "I'd keep writes ACID on the source of truth, but replicate to a read-optimized store with eventual consistency. Reads hit the replica — slightly stale but fast. The checkout flow does a strong-consistency read to get the real count before decrementing."

Typical question: "How would you implement cross-region consistency for a financial ledger?"

What they're testing: Deep understanding of distributed transaction protocols, clock synchronization, and the real-world costs of each approach.

How to answer:

  1. Frame the problem: "Cross-region ACID requires consensus — the question is how to minimize the latency cost."
  2. Discuss options: two-phase commit (simple but blocking), Paxos/Raft (fault-tolerant consensus), Spanner-style TrueTime (timestamp-based ordering).
  3. Mention Sagas as an alternative: "If we can tolerate compensating transactions, Sagas avoid the coordination overhead entirely."
  4. Discuss CRDTs for data that can be merged without coordination (counters, sets).

Bonus point: Mention that Google Spanner waits out the clock uncertainty window (typically ~7ms) before committing — "They literally sleep() to guarantee ordering. That's the cost of global ACID."

Practice Exercises

Exercise 1: Classify the Data Easy

You're building a ride-sharing app (like Uber). For each of the following data types, decide whether it needs ACID or BASE consistency, and explain why:

  1. Driver location updates (GPS pings every 3 seconds)
  2. Payment processing (charging the rider after the trip)
  3. Ride request matching (connecting rider to nearest driver)
  4. Trip history (past rides shown in the app)
  5. Surge pricing calculations
Hint

Ask yourself: "If this data is wrong for 5 seconds, what's the worst that happens?" If the answer involves money or safety, lean ACID. If the answer is "mild inconvenience," BASE is fine.

Solution
  1. Driver locations → BASE. GPS pings are high-frequency, write-heavy, and a 3-second-old location is perfectly usable.
  2. Payments → ACID. Charging the wrong amount or double-charging is unacceptable.
  3. Ride matching → BASE (with caveats). Matching uses approximate driver locations. A slightly stale position is fine; the driver confirms anyway.
  4. Trip history → BASE. Showing a trip 10 seconds late in the history tab causes no harm.
  5. Surge pricing → BASE. Pricing is recalculated frequently. A brief lag in surge multiplier is acceptable; the locked-in price at booking time uses a consistent snapshot.
Exercise 2: Spot the Bug Medium

A developer wrote this pseudocode for a ticket booking system using an eventually consistent database:

available = read_ticket_count(event_id)    // eventual read
if available > 0:
    decrement_ticket_count(event_id)
    create_booking(user_id, event_id)
    charge_payment(user_id, amount)
    return "Booking confirmed!"

Identify at least three problems with this approach.

Solution
  1. Race condition: Two users read available = 1 at the same time. Both proceed, both book — but only one ticket exists. The eventual read doesn't reflect concurrent decrements.
  2. No atomicity: If charge_payment fails after decrement_ticket_count, the ticket count is wrong but no payment was made. These operations need to be atomic (all succeed or all roll back).
  3. No idempotency: If the response times out and the client retries, the user could be charged twice and get two bookings.

Fix: Use a strongly consistent read + atomic transaction for the critical path (check → decrement → book → charge). Add an idempotency key to prevent duplicate bookings on retry.

Exercise 3: Design the Hybrid Medium

You're architecting a social media platform with these features: user profiles, posts/feed, direct messages, and creator monetization (tipping/subscriptions). Design a data architecture that uses both ACID and BASE appropriately. For each feature, specify:

  1. Which consistency model and why
  2. What database technology you'd consider
  3. What happens during a network partition
Hint

Think about what users would complain about most. "My $50 tip disappeared" is a support nightmare. "My post took 2 extra seconds to appear in my friend's feed" — nobody notices.

Exercise 4: Eventual Consistency Timeline Easy

A system has 3 replicas (A, B, C). A user writes score = 100 to Replica A at time T=0. Draw a timeline showing:

  1. When each replica has the updated value (assume replication takes ~200ms per hop)
  2. What a reader would see if they read from Replica C at T=50ms, T=150ms, and T=500ms
  3. At what point the system becomes "consistent"
Solution

Timeline: T=0 → A has 100. T=200ms → B gets 100. T=400ms → C gets 100 (assuming chain replication A→B→C).

Reads from C: T=50ms → old value (stale). T=150ms → old value (still stale). T=500ms → 100 (consistent).

Fully consistent: At T=400ms when all three replicas have the same value. The "eventual" window was 400ms.

Exercise 5: Saga vs. Two-Phase Commit Hard

A travel booking system needs to atomically book a flight, a hotel, and a rental car across three independent services. Design both approaches:

  1. Two-Phase Commit: Draw the message flow. What happens if the hotel service crashes after voting "yes" but before the commit message arrives?
  2. Saga with compensating transactions: Define the saga steps and the compensating action for each. What happens if the car rental fails after flight and hotel are booked?
  3. Compare: latency, failure handling, complexity, and which you'd recommend for this scenario.
Hint

For the Saga approach, the compensating actions are: cancel flight booking, cancel hotel reservation. The key insight is that compensating actions must be idempotent — you might need to cancel the same booking twice if a retry happens.
