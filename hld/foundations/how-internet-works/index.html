Section 1

TL;DR — The One-Minute Version

Mental Model: The internet is a postal system for digital data. Your message gets split into small envelopes (packets), each stamped with an address (IP), routed through a chain of post offices (routers), and reassembled at the destination. DNS is the phone book that converts names to addresses.

When you type google.com in your browser, here's what happens in about 200 milliseconds. Your computer asks a DNS serverDomain Name System — translates human-readable names like "google.com" into IP addresses like 142.250.80.46 that computers use to find each other. "Hey, what's the IP address for google.com?" It gets back a number like 142.250.80.46. Your computer then opens a TCP connectionTransmission Control Protocol — a reliable, ordered way to send data between two computers. Like a phone call where both sides confirm each message was received. to that address, sends an HTTP request saying "Give me the homepage," and the server sends back HTML, CSS, and images that your browser turns into the page you see.

The entire internet runs on layers — each layer does one job and trusts the layer below it to handle the rest. Your browser speaks HTTP, which rides on TCP, which rides on IP, which rides on electrical signals in a cable. It's like mailing a letter: you write the letter (HTTP), put it in an envelope with an address (IP), and the postal service (physical network) figures out how to actually get it there. You don't need to know the truck routes.

One-line takeaway: The internet is layers of trust — each layer solves one problem and hands off to the next, so nobody has to understand the whole thing at once.
Section 2

The Scenario — You Type google.com

It's Friday afternoon. You open your laptop, click on the browser, and type google.com. You press Enter. In less than a blink, Google's search page appears. You've done this ten thousand times. But let's stop and think about what actually just happened.

Your laptop is sitting in your bedroom in, say, Mumbai. Google's nearest server might be in a data center in Chennai — 1,000 km away. Or it could be in Singapore. Or Iowa. How did your laptop know where to send that request? How did the response find its way back? And how did it all happen faster than you can blink?

Here's the beautiful thing: the internet solves this exact problem billions of times per second, for 5 billion people, across 200+ countries. And it does it using a surprisingly elegant system of layers, addresses, and agreements between machines. Let's see how.

Section 3

First Attempt — The "Magic Box" Mental Model

If you're like most people, your mental model of the internet looks something like this: you type a URL, "the internet" does something magical, and a webpage appears. It's a black box. Stuff goes in, stuff comes out. You don't think about the middle.

And honestly? That works fine for daily life. You don't need to know how the postal system works internally to mail a letter. But if you're a software engineer designing systems that serve millions of users — you do need to understand the postal system. Because when something goes wrong (and it will), you need to know WHERE it went wrong.

Let's try a slightly better model. What if the internet is just a really long cable connecting your computer to Google's computer? You send a message down the cable, Google sends one back. Simple!

This model is closer to reality — there ARE physical cables involved. But it falls apart immediately when you start asking questions:

  • How does your computer know which cable leads to Google? There are billions of devices on the internet. You can't have a direct cable to each one.
  • What if the cable breaks? If there's only one path, you're stuck.
  • How do you share the cable? You're not the only person trying to reach Google right now. Millions of people are making requests at the same time.
  • How does Google's response find its way back to YOU? Among billions of devices, how does the response know exactly where you are?
Think First

Imagine you want to send a physical letter to someone, but you only know their name — not their address. And the postal system has no roads — just a giant mesh of paths. How would you design a system to get that letter delivered reliably?

Hint: You'd probably need (1) a way to look up addresses, (2) a way to pick a route, and (3) a way to confirm it arrived.

The real internet solves all of these problems, and it does it through layers. Instead of one magic box, there are multiple layers, each responsible for exactly one piece of the puzzle. That's the breakthrough we'll explore next.

Section 4

Where It Breaks — Why One Layer Isn't Enough

Let's say we tried to build the internet as one single system — no layers, just one big protocol that handles everything: finding addresses, routing data, ensuring delivery, and formatting messages. What would go wrong?

This is the exact same problem that hits any monolithic system. Remember from the scalability page — a single giant application that does everything eventually becomes impossible to change, test, and scale. The internet's architects in the 1970s and 80s figured this out early. Their solution? Split the problem into layers.

Think of it like building a house. You don't want the plumber to also be the electrician, the roofer, and the painter. Each trades-person has one job. The plumber doesn't care how the electricity works — they just know water comes in through the pipes and goes out through the drain. Similarly, each internet layer does one thing and trusts the layer below it.

Key insight: Layers allow independent evolution. We went from dial-up to fiber optics to 5G — and HTTP didn't change one bit. That's because physical transport is a lower layer that the application layer doesn't see. Change one layer without touching the others.
Section 5

The Breakthrough — Layers of Trust

The internet is built on a simple, brilliant idea: split the problem into layers, and let each layer only worry about its own job. This is called the TCP/IP modelThe 4-layer architecture the internet actually uses: Application, Transport, Internet (Network), and Link. Named after its two most important protocols — TCP and IP., and it has four layers. (There's also a 7-layer model called OSIOpen Systems Interconnection — a theoretical 7-layer model used for teaching. The real internet uses the simpler 4-layer TCP/IP model, but OSI terms like "Layer 7" are still widely used. used in textbooks, but the real internet uses TCP/IP's four.)

Here's the analogy that makes it click. Imagine you're sending a physical package to a friend in another country:

The key insight: each layer wraps the data from the layer above it. When you send an HTTP request, the Transport layer wraps it in a TCP segment. The Network layer wraps that in an IP packet. The Link layer wraps that in a frame. It's like putting a letter in an envelope, then putting that envelope in a bigger envelope with the street address, then putting that in a mailbag for the truck. At the destination, each layer peels off its envelope and passes the contents up.

This wrapping-and-unwrapping process is called encapsulationEach layer adds its own header (and sometimes trailer) around the data from the layer above. At the destination, each layer removes its header and passes the payload up. Like nested envelopes.. It's one of the most elegant ideas in computer science — and it's why the internet has survived 50 years of radical change without a redesign.

Why this matters for system design: When an interviewer asks "what happens when you type a URL," they want to see that you understand layers — not just memorized steps. Each layer has different failure modes, different scaling strategies, and different optimization opportunities. Load balancers work at Layer 4 or Layer 7. CDNs work at the Application layer. Firewalls can work at Layer 3 or 4. Understanding layers lets you reason about WHERE to put each tool.
Section 6

How It Works — The Five Layers, One by One

Let's walk through each layer from bottom to top — from the physical cables all the way up to the HTTP request your browser sends. We'll use five cards, one per layer. (We're using five layers here instead of four because it helps to separate the Physical and Data Link layers — the TCP/IP model lumps them together, but understanding them individually makes things clearer.)

Think First

Why does the internet use multiple layers instead of one protocol that does everything? Think about real-world systems — why do cars have separate systems for engine, brakes, and steering instead of one unified mechanism?

Hint: What happens when you want to upgrade just ONE part without replacing the whole thing?

Layer 1 — Physical: Bits on a Wire

At the very bottom, data is just electrical signals, light pulses, or radio waves. This layer doesn't know about addresses, websites, or even what a "message" is. It just moves 1s and 0s from point A to point B.

There are three main ways to physically move data:

  • Copper cables (Ethernet) — electrical signals travel through copper wires. Cheap, works for short distances (up to ~100 meters for Cat 6).
  • Fiber optic cables — light pulses travel through glass fibers. Much faster, works for thousands of kilometers. This is what submarine cablesOver 500 cables laid on the ocean floor connecting continents. They carry 99% of intercontinental internet traffic — not satellites, as many people assume. use to connect continents.
  • Radio waves (Wi-Fi, 4G, 5G) — electromagnetic waves through air. Convenient but slower and less reliable than wired connections.

Fun fact: 99% of international internet traffic travels through submarine fiber optic cables on the ocean floor — not satellites. There are over 500 of these cables crisscrossing the world's oceans, some spanning over 20,000 km. When one gets damaged (by anchors, earthquakes, or even sharks), entire regions can lose connectivity.

Layer 2 — Data Link: Talking to Your Neighbors

The Physical layer can move raw bits, but it has no idea who those bits are for. That's where the Data Link layer comes in. It handles communication between devices on the same local network — your laptop to your router, for example.

Every network device has a MAC addressMedia Access Control address — a 48-bit unique hardware identifier burned into every network card at the factory. Example: A4:83:E7:2B:9C:01. Unlike IP addresses, MAC addresses don't change. — a unique hardware ID baked into the device at the factory (like a serial number). The Data Link layer uses these to deliver data within a local network.

The Data Link layer wraps IP packets into frames, adds the source and destination MAC addresses, and sends them across the local wire (or Wi-Fi). Ethernet and Wi-Fi are both Data Link protocols — they just use different physical media.

Important limitation: MAC addresses only work on a local network. They can't route across the internet. That's why we need the next layer — the Network layer — with its globally-routable IP addresses.

Layer 3 — Network: Addressing and Routing

This is the layer that makes the inter-net possible — connecting different networks together. The Network layer's job is simple: get a packet from source to destination, even if they're on completely different networks in different countries.

It does this using IP addressesInternet Protocol address — a numerical label assigned to each device on a network. IPv4 uses 32 bits (e.g., 192.168.1.1), giving ~4.3 billion addresses. IPv6 uses 128 bits for practically unlimited addresses.. Unlike MAC addresses (which are hardware-based and only work locally), IP addresses are logical — they can be assigned, changed, and routed across the entire internet.

Routers are the heroes of the Network layer. Each router has a routing tableA table stored in every router that maps destination IP ranges to the next hop (the next router to forward to). Like a signpost at each intersection saying "for addresses in 142.x.x.x, go this way." — a set of rules that says "for packets going to this range of IP addresses, forward them to that next router." No single router knows the complete path from source to destination. Each one just knows the next step. It's like asking for directions at every intersection.

Layer 4 — Transport: Reliable Delivery

IP can get packets from A to B, but it makes no promises. Packets can arrive out of order, get duplicated, or disappear entirely. The Network layer's attitude is "best effort" — I'll try, but no guarantees.

That's where the Transport layer steps in. It adds reliability on top of IP. There are two main protocols here:

TCP (Transmission Control Protocol) guarantees that every byte you send arrives, in order, without corruption. It does this by:

  • Numbering every byte so they can be reassembled in order
  • Requiring the receiver to acknowledge every segment
  • Retransmitting anything that doesn't get acknowledged
  • Controlling the sending speed to avoid overwhelming the receiver (flow controlTCP's mechanism to prevent a fast sender from overwhelming a slow receiver. The receiver advertises a "window size" — how much data it can handle right now. The sender respects this limit.)

The trade-off? It's slower. All that acknowledgement and retransmission takes time. But for web pages, emails, file downloads — you need every byte to be correct. Missing a pixel in an image or a word in an email is not acceptable.

UDP (User Datagram Protocol) is the opposite philosophy: send it and forget it. No acknowledgements, no ordering, no retransmission. It's like shouting a message across a room — fast, but you can't be sure the other person heard every word.

Why would anyone use this? Because for some applications, speed matters more than completeness:

  • Video calls — if a frame arrives late, it's useless. Better to skip it and show the next one.
  • Online games — a player's position from 200ms ago is irrelevant. Send the latest one.
  • DNS lookups — a tiny request/response pair. The overhead of TCP's handshake would double the time.
  • Live streaming — a few dropped frames are invisible. A 2-second delay is annoying.

Layer 5 — Application: What the User Sees

This is the layer you interact with every day. HTTPHyperText Transfer Protocol — the language browsers and web servers speak. You send a request (GET /page), the server sends a response (200 OK + HTML). Simple, text-based, and stateless. for web browsing, SMTP for email, FTP for file transfer, DNSDomain Name System — the internet's phone book. Converts human-readable names (google.com) to IP addresses (142.250.80.46). Without DNS, you'd have to memorize numbers for every website. for name resolution. These protocols define the actual meaning of the data being sent.

When your browser sends a request, it's written in HTTP — a simple text-based protocol:

HTTP Request 
GET / HTTP/1.1
Host: www.google.com
User-Agent: Chrome/120.0
Accept: text/html
Connection: keep-alive

And Google's server responds with:

HTTP Response 
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 14823

<!DOCTYPE html>
<html>
  <head><title>Google</title></head>
  ...
</html>

That's it. HTTP is just text. A request says "I want this resource" and a response says "Here it is" (or "Not found" or "You need to log in"). The entire web is built on this simple conversation.

Your browser then takes the HTML, fetches any CSS and JavaScript files referenced in it (each one is a separate HTTP request), and renders the page — turning code into the visual layout you see on screen.

The beautiful part: when engineers upgraded the world from HTTP/1.1 to HTTP/2 (and then HTTP/3), they didn't have to change TCP, IP, Ethernet, or the physical cables. They only changed the Application layer. When the world upgraded from Ethernet to Wi-Fi, they only changed the Link layer. Each layer evolves independently. That's the power of layered design.
Section 7

Going Deeper — DNS, TCP Handshake, NAT, and Routing

Now that you understand the layers, let's zoom into the four mechanisms that power the everyday internet experience. These come up constantly in system design interviews and debugging real-world issues.

DNS Resolution — The Internet's Phone Book

You type google.com, but computers only understand numbers (IP addresses). So before anything else, your computer needs to look up the IP address for that domain name. This process is called DNS resolutionThe process of converting a domain name (like google.com) into an IP address (like 142.250.80.46). It involves multiple servers in a hierarchy: local cache → recursive resolver → root → TLD → authoritative nameserver., and it works like a chain of referrals.

Think First

There are hundreds of millions of domain names. Would you store all of them in one giant database? What problems would that cause? How would you design a lookup system that's fast AND doesn't have a single point of failure?

Hint: Think about how phone books work — you don't call a single phone book operator for the entire world.

The clever part: results are cached at every level. Your browser caches DNS results. Your OS caches them. Your ISP's resolver caches them. So most DNS lookups resolve instantly from cache without making any network requests at all. A fresh lookup (no cache) takes about 20-120ms. A cached one takes under 1ms.

TCP Three-Way Handshake — Establishing Trust

Once you have the IP address, your browser needs to set up a TCP connectionA logical link between two endpoints (IP:port pairs) that provides reliable, ordered data transfer. Must be established with a handshake before any data flows.. But TCP doesn't just start blasting data — it first does a "handshake" to make sure both sides are ready. Think of it like a phone call:

This handshake takes one round tripThe time for a message to travel from A to B and a response to come back. Also called RTT (Round-Trip Time). For a server in the same country, typically 10-50ms. Cross-continent: 100-200ms. (your message goes to the server and a response comes back). For a server in the same country, that's about 20-50 ms. For a server across the ocean, it could be 100-200 ms. This is why TCP has some latency overhead — and why protocols like QUICA modern transport protocol (used by HTTP/3) that combines connection setup and encryption in a single round trip, reducing initial latency. Built on UDP instead of TCP. (used in HTTP/3) try to reduce handshake time.

NAT — How Millions Share One Address

Here's a puzzle: there are only about 4.3 billion IPv4 addresses, but there are over 15 billion internet-connected devices. How does that math work?

The answer is NATNetwork Address Translation — a router technique where all devices on your local network share one public IP address. The router keeps a table mapping internal IP:port pairs to the single external IP, swapping addresses on packets as they flow through. (Network Address Translation). Your home router has one public IP address given by your ISP. All the devices in your home — laptop, phone, smart TV, thermostat — share that single public IP. The router keeps a translation table that tracks which internal device made which request.

When your laptop sends a request to Google, the router replaces your private IP (192.168.1.10) with its public IP (203.0.113.42) and notes which internal device made the request. When Google's response comes back, the router checks its table and forwards it to the right device. Your laptop never knows the difference.

IP Routing — How Packets Find Their Way

IP routing is one of the most impressive engineering feats of the internet. Your packet might travel through 10-20 routers to reach its destination, and no single router knows the full path. Each router just knows: "for this destination, forward to THAT next router." It's like navigating a city by asking for directions at every intersection.

Routers exchange information about their knowledge of the network using routing protocols like BGP (Border Gateway Protocol). BGP is how the internet's backbone routers know how to reach every network in the world — it's been called "the protocol that holds the internet together." When BGP misconfigures (which happens more often than you'd think), entire regions can go offline.

TLS — Encrypting the Connection

After the TCP handshake completes, if the URL starts with https:// (which is almost every website today), there's an additional step: the TLS handshakeTransport Layer Security — a cryptographic protocol that encrypts data between browser and server. The handshake negotiates encryption keys without sending them over the wire (using Diffie-Hellman key exchange). TLS 1.3 does this in just 1 round trip.. This establishes an encrypted tunnel so nobody between you and the server (ISPs, Wi-Fi hackers, governments) can read your data.

TLS 1.3 (the latest version) completes in just one round trip — a big improvement over TLS 1.2's two round trips. It also supports 0-RTT resumption, where a returning visitor can start sending encrypted data immediately without any handshake, using keys from a previous session. This is how modern HTTPS websites feel nearly as fast as unencrypted HTTP.

The complete connection setup: DNS (~20ms) + TCP handshake (1 RTT, ~30ms) + TLS handshake (1 RTT, ~30ms) = about 80ms before the first byte of your actual request even leaves. This is why performance engineers obsess over reducing round trips — HTTP/3 (QUIC) combines the TCP and TLS handshakes into a single round trip.

Variations — IPv4 vs IPv6, Ports, and Address Types

Now that you understand the core mechanisms, let's look at the important variations and details that come up in practice and interviews.

IPv4 vs IPv6 — Running Out of Addresses

When the internet was designed in the 1980s, 4.3 billion addresses seemed like more than enough. After all, there were only a few hundred computers on the network. Fast forward to today: we have smart watches, fridges, doorbells, and cars all wanting IP addresses. We ran out of new IPv4 addresses in 2011.

IPv4 uses 32-bit addresses written as four numbers separated by dots:

IPv4 Address
192.168.1.42

  │     │  └──── Subnet

  └───────────── Network

Total: 2^32 = ~4.3 billion addresses
Problem: We ran out in 2011!

IPv6 also removes the need for NAT (every device can have its own public address), has built-in security features, and simplifies routing. Adoption is around 40% globally as of 2025 — it's happening, just slowly.

Ports — Multiple Conversations on One Address

An IP address gets you to the right machine, but a single server runs many services — a web server, a database, an email server, an SSH daemon. How does the machine know which service a packet is for? That's where portsA 16-bit number (0-65535) that identifies a specific service on a machine. Think of IP as the street address and port as the apartment number. Web servers default to port 80 (HTTP) or 443 (HTTPS). come in.

A port is just a number from 0 to 65535. Think of IP addresses as street addresses and ports as apartment numbers. The combination of IP + port uniquely identifies a specific service on a specific machine.

Section 9

At Scale — How the Internet Actually Runs

The internet isn't run by one company or one government. It's a network of networks — thousands of independent organizations that agree to connect their networks together. Here's how it works at scale.

Submarine Cables — The Internet's Backbone

Forget satellites. Over 99% of intercontinental internet traffic travels through submarine fiber optic cables laid on the ocean floor. There are more than 500 of these cables, some over 20,000 km long. They carry data at the speed of light (well, about 2/3 the speed of light in glass) and are each thinner than a garden hose.

CDNs — Bringing Content Closer

If Google's servers are in Iowa but you're in Mumbai, every request travels 15,000 km — adding 150+ ms of latency. The solution? CDNsContent Delivery Networks — a distributed network of servers placed in cities worldwide. They cache static content (images, CSS, JS) close to users, so most requests travel 10 km instead of 10,000 km. (Content Delivery Networks) like CloudFlare, Akamai, and Fastly place copies of popular content in servers around the world. When you request a cat video, it comes from a server in your city — not from the original server across the ocean.

Companies like Google, Netflix, and Facebook operate their own CDNs with thousands of edge servers worldwide. CloudFlare alone has servers in 310+ cities. The result: most of the internet's static content is served from a server within 50 km of you — even though the origin server might be on another continent.

Internet Exchange Points (IXPs)

When your ISP needs to send data to another ISP, they could route it across the entire internet — or they could meet at a local IXPInternet Exchange Point — a physical building where ISPs and networks connect directly to exchange traffic. Like a meeting point where postal services hand packages to each other instead of routing through a central hub.. These are physical buildings where networks plug in and exchange traffic directly. The largest IXP in the world — DE-CIX in Frankfurt — handles over 14 Tbps of peak traffic.

BGP — The Protocol That Holds the Internet Together

How do routers across different companies and countries know how to reach each other? Through BGPBorder Gateway Protocol — the routing protocol used between autonomous systems (networks run by different organizations). BGP is how your ISP knows how to reach Google's network, and vice versa. It's been called "the most important protocol you've never heard of." (Border Gateway Protocol). Every ISP, cloud provider, and large company operates an Autonomous System (AS) — essentially their own mini-internet. BGP is how these autonomous systems announce their routes to each other.

Real-world BGP incidents are surprisingly common. In 2021, a Facebook BGP misconfiguration took down Facebook, Instagram, and WhatsApp for 6 hours — the BGP withdrawal made Facebook's servers literally unreachable from the entire internet. In 2022, a Rogers (Canadian ISP) BGP issue knocked out internet for 12 million Canadians. BGP is critical infrastructure with very little built-in security.

Why this matters: in system design interviews, when you say "put a CDN in front of static assets" or "deploy to multiple regions," you're leveraging this infrastructure. Understanding HOW the internet routes traffic helps you make better placement decisions. Putting your servers in US-East might work for American users, but your Indian users will experience 200ms+ extra latency unless you deploy to a closer region or use a CDN.
Section 10

The Anti-Lesson — What People Get Wrong

Understanding how the internet works can lead to some dangerous over-simplifications. Here are the most common myths that even experienced engineers fall for:

Think First

Before reading below, which of these statements do you think is TRUE? (a) Most internet traffic travels through satellites. (b) HTTPS makes all communication completely secure. (c) More bandwidth always means faster browsing. (d) None of the above.

Hint: All three are common misconceptions. The answer is (d).

"The Internet Runs on Satellites"

This is one of the biggest myths. Satellites provide internet to remote areas (ships, planes, rural villages), but they carry less than 1% of global traffic. The latency alone makes them impractical for most uses — geostationary satellites add 500-600ms of round-trip delay. Even SpaceX's Starlink (low-Earth orbit) has 20-40ms latency, which is worse than a local fiber connection's 1-5ms.

Reality: Submarine fiber optic cables carry 99% of intercontinental traffic. Satellites are a last resort, not the backbone.

"HTTPS Makes Everything Secure"

HTTPS encrypts the data in transit between your browser and the server. It does NOT protect against:

  • DNS spoofing — someone can redirect your DNS query to a malicious IP before HTTPS even kicks in
  • Compromised servers — HTTPS doesn't help if the server itself is hacked
  • Metadata leaks — your ISP can still see WHICH domains you visit (via DNS queries and TLS SNI), even if they can't see the content
  • Man-in-the-middle with a trusted cert — corporate proxies often install their own certificates on employee machines

Reality: HTTPS is essential and solves transit encryption. But it's one layer of defense, not a silver bullet.

"More Bandwidth = Faster Internet"

Bandwidth is how much data you can send per second (like the width of a pipe). LatencyThe time it takes for a single packet to travel from source to destination. Measured in milliseconds. Determined by physical distance, routing hops, and processing delays. No amount of bandwidth can reduce latency below the speed of light. is how long it takes for data to arrive (like the length of the pipe). Increasing bandwidth from 100 Mbps to 1 Gbps won't help if your problem is a 200ms round-trip to a distant server.

Reality: For large file downloads, bandwidth matters. For web browsing, latency matters more — most webpages make 50-100 small requests, each waiting for the previous one. Lower latency helps every single request.

Section 11

Common Mistakes — What Engineers Get Wrong

Mistake 1: Confusing IP addresses with MAC addresses

What people think: "MAC and IP are interchangeable — both identify a device."

Why it's wrong: MAC addresses are hardware-level and only work on a local network segment. They never leave your router. IP addresses are logical and can be routed across the entire internet. Your laptop's MAC address is the same whether you're at home or at a coffee shop, but your IP address changes based on which network you're on.

The truth: MAC = who you are (permanent hardware ID). IP = where you are (changes based on network).

Mistake 2: Thinking TCP is always better than UDP

What people think: "TCP is reliable, so always use TCP."

Why it's wrong: Reliability comes at a cost — higher latency, more overhead. For video calls, gaming, and live streaming, a dropped packet from 200ms ago is useless. Retransmitting it would be worse than just skipping it. UDP's "fire and forget" model is exactly what these applications need.

The truth: Use TCP when correctness matters (web, email, files). Use UDP when timeliness matters (video, gaming, DNS).

Mistake 3: Ignoring DNS in system design

What people think: "DNS just works. I don't need to think about it."

Why it's wrong: DNS is a critical piece of infrastructure that affects latency, availability, and even security. DNS outages have taken down major sites (the 2016 Dyn attack knocked out Twitter, Netflix, GitHub, and Reddit for hours). DNS TTLs affect how quickly you can fail over to a backup server. DNS-based load balancing (GeoDNS) is how many companies route users to the nearest data center.

The truth: In system design, DNS is a first-class concern — mention it when discussing domain routing, failover strategy, and global distribution.

Mistake 4: Assuming the network is reliable

What people think: "If I send a message, it will arrive."

Why it's wrong: This is literally the first of the 8 Fallacies of Distributed ComputingClassic assumptions that new distributed systems developers make: (1) The network is reliable, (2) Latency is zero, (3) Bandwidth is infinite, (4) The network is secure, (5) Topology doesn't change, (6) There is one administrator, (7) Transport cost is zero, (8) The network is homogeneous.. Packets get dropped, connections time out, cables get cut. Any system that doesn't handle network failures will eventually fail in spectacular ways.

The truth: Design for failure. Implement retries with backoff, circuit breakers, and timeouts. Assume every network call CAN fail, and plan what happens when it does.

Mistake 5: Not understanding latency numbers

What people think: "All network requests take about the same time."

Why it's wrong: The difference between local network (0.5ms), same datacenter (0.5ms), same continent (50ms), and cross-ocean (150ms) is enormous. An API that makes 10 sequential calls to a cross-continent service adds 1.5 seconds of latency — before any processing even starts.

The truth: Always know where your data lives relative to your users. Keep frequently-accessed data close (caching, CDNs, regional deployment). Avoid chatty cross-region calls.

Section 12

Interview Playbook — "What Happens When You Type a URL?"

This is one of the most common interview questions in the tech industry — possibly THE most popular warm-up question in backend and infrastructure interviews. It can be asked in a 2-minute version (phone screen) or a 30-minute deep dive (onsite). The depth of your answer signals your seniority. Here's how to answer it at three levels.

Think First

Before reading the answer below, try to list all the steps that happen when you type "https://www.google.com" and press Enter. Time yourself — can you hit 10+ steps in under 2 minutes?

Hint: DNS → TCP → TLS → HTTP → Server → Response → Render. But there's more at each step.

The 2-minute answer — hit the key steps clearly:

  1. Browser checks its cache for the URL. If cached and fresh, display it.
  2. DNS lookup to resolve the domain name to an IP address.
  3. TCP three-way handshake to establish a connection.
  4. TLS handshake (for HTTPS) to encrypt the connection.
  5. Browser sends an HTTP request (GET /).
  6. Server processes the request and sends back an HTTP response with HTML.
  7. Browser parses HTML, discovers CSS/JS/image resources, fetches them in parallel.
  8. Browser renders the page (DOM construction, CSS layout, paint).
Interviewer impression: "They understand the basic flow. Good foundation."

The 10-minute answer — add technical depth to each step:

  1. Browser cache check: check memory cache → disk cache → check Cache-Control and Expires headers
  2. DNS resolution: browser cache → OS cache → local DNS resolver → recursive resolver → root nameservers → TLD nameservers → authoritative nameserver. Mention TTL and caching.
  3. TCP handshake: SYN → SYN-ACK → ACK. Mention that this costs one RTT (round trip time). For HTTPS, add TLS 1.3 handshake (one additional RTT, or 0-RTT with session resumption).
  4. HTTP request: method, headers (Host, User-Agent, Accept, cookies), potentially HTTP/2 multiplexing.
  5. Load balancer: request likely hits a load balancer first (L4 or L7), which routes to a backend server.
  6. Server processing: web server (nginx) → application server → possibly database query → response generation.
  7. Response: status code, content-type, cache headers, possibly gzip compressed.
  8. Rendering: parse HTML → build DOM → parse CSS → build CSSOM → combine into render tree → layout → paint → composite.
Interviewer impression: "They understand the details. They've actually thought about this."

The 30-minute deep dive — mention everything above, plus:

  • HSTS check: browser may have a preloaded list forcing HTTPS — never sends HTTP in the first place
  • DNS over HTTPS (DoH): modern browsers encrypt DNS queries to prevent ISP snooping
  • Anycast: the DNS resolver IP (like 8.8.8.8) is anycast — routes to the nearest Google DNS server
  • TCP congestion control: slow start, congestion avoidance, fast retransmit. Connection starts slow and ramps up.
  • CDN involvement: the response might come from a CDN edge server, not the origin. Mention cache-hit vs cache-miss.
  • HTTP/2 features: multiplexing (many requests on one TCP connection), header compression (HPACK), server push
  • HTTP/3 / QUIC: uses UDP instead of TCP. 0-RTT connection establishment. Survives network changes (Wi-Fi → cellular).
  • Connection pooling: browser keeps TCP connections open (keep-alive) to reuse for subsequent requests
  • Critical rendering path: render-blocking CSS, async/defer JS loading, preload hints, font loading strategies
  • Service workers: may intercept the request for offline support or custom caching strategies
Interviewer impression: "They have deep understanding. They can reason about trade-offs at every layer."
Section 13

Hands-On Exercises

Exercise 1: Trace a Real DNS Lookup Beginner

Open your terminal and run nslookup google.com (Windows) or dig google.com (Mac/Linux). Then try the same for three other websites you use daily. Answer these questions:

  1. What IP address did google.com resolve to? Is it the same every time you run it?
  2. What's the TTL (time-to-live) of the DNS record?
  3. Try nslookup google.com 8.8.8.8 — how does the answer differ from your ISP's resolver?
Hint

Google uses multiple IP addresses and load balances across data centers. Running the lookup multiple times may give different IPs — this is DNS-based load balancing. The TTL tells you how long to cache the result before asking again.

Exercise 2: Trace the Route Beginner

Run tracert google.com (Windows) or traceroute google.com (Mac/Linux) and observe:

  1. How many hops does the packet take?
  2. At which hop does latency significantly increase? (That's likely where the packet leaves your city/country.)
  3. Are there any hops that time out (show * * *)? What might cause that?
Hint

Timeouts usually mean a router is configured to drop ICMP packets (for security). It doesn't mean the route is broken. Big latency jumps between hops indicate long-distance links (possibly submarine cables).

Exercise 3: TCP vs UDP in Action Intermediate

Open two browser tabs. In one, start a YouTube video. In the other, download a large file. Now:

  1. Briefly disconnect your Wi-Fi (2-3 seconds) and reconnect. What happens to the video vs. the download?
  2. The download pauses and resumes exactly where it left off (TCP retransmission). The video might skip or buffer (UDP tolerates loss).
  3. Now use netstat -an to see active connections. How many TCP connections does your browser have open?
Explanation

YouTube actually uses QUIC (UDP-based) for video delivery in Chrome. The file download uses TCP. This is exactly the TCP vs UDP trade-off in practice — video prioritizes low latency, file downloads prioritize completeness.

Exercise 4: Inspect Real HTTP Traffic Intermediate

Open Chrome DevTools (F12) → Network tab. Load google.com and answer:

  1. How many HTTP requests are made to fully load the page?
  2. What's the largest resource? What's the total transfer size?
  3. Find a request that came from cache (look for "(disk cache)" or "(memory cache)"). What's its response time vs. a network request?
  4. Click on a request and look at the Timing tab. Identify DNS lookup, TCP connection, TLS, and server response times.
Exercise 5: Design a Simplified Internet Advanced

On paper or whiteboard, design a simplified network for 4 offices in different cities. Each office has 50 computers. Specify:

  1. IP address scheme for each office (use private IP ranges)
  2. How offices connect to each other (draw the physical topology)
  3. How a computer in Office A sends a message to a computer in Office D
  4. What happens if the direct link between Office A and Office D breaks?
Hint

Use 10.1.0.0/24 for Office A, 10.2.0.0/24 for B, etc. Connect offices in a mesh or ring topology. If a direct link breaks, routing protocols automatically find an alternate path through other offices — this is exactly how the real internet works.

The Layers
Application  → HTTP, DNS, FTP
Transport    → TCP (reliable), UDP (fast)
Network      → IP (addressing + routing)
Data Link    → Ethernet, Wi-Fi (local)
Physical     → Cables, fiber, radio waves
Key Numbers
DNS cached lookup   : <1ms
DNS fresh lookup    : 20-120ms
TCP handshake       : 1 RTT
TLS handshake       : 1-2 RTT
Same datacenter RTT : 0.5ms
Same continent RTT  : 20-50ms
Cross-ocean RTT     : 100-200ms
TCP vs UDP
TCP: reliable, ordered, slower
  → Web, email, file transfer, APIs
  → Connection-oriented (handshake)

UDP: unreliable, unordered, faster
  → Video calls, gaming, DNS, streaming
  → Connectionless (fire-and-forget)
IPv4 vs IPv6
IPv4: 32-bit, 4.3B addresses
  → 192.168.1.1 format
  → Ran out in 2011, needs NAT

IPv6: 128-bit, 340 undecillion
  → 2001:db8::1 format
  → No NAT needed, ~40% adoption
