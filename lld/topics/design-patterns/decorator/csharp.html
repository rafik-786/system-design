Section 1

TL;DR — The Gift Wrapping Pattern

  • What Decorator is and why inheritance isn't always the answer
  • How to add behavior to objects at runtime without modifying them
  • The wrapping concept — each decorator wraps the previous one
  • How .NET uses Decorator everywhere (Stream, middleware, logging)

Add new behavior to an object by wrapping it in another object — like gift wrapping. Each layer adds something new, but the gift inside stays untouched.

Think about ordering coffee. You start with a basic espresso. Want milk? Wrap it — now it's a latte. Want whipped cream? Wrap it again — latte with whip. Want caramel drizzle? One more wrap. Each addition is a "decorator" — it adds something on top without changing the original espresso. And you can combine them in any order, any quantity. THAT's Decorator — layer behavior on top of existing behavior, endlessly, without modifying the original.

Imagine you have a perfectly working email notification service. It sends emails. That's its job, and it does it well. Now your boss says: "We also need SMS notifications." Easy — add an SMS class. Then: "We need Slack notifications too." Fine — another class. But then: "We need email + SMS together. And email + Slack. And SMS + Slack. And all three. Oh, and sometimes just email with logging. And sometimes SMS with retry logic."

If you try to solve this with inheritanceWhen a class extends another class to reuse its behavior. "EmailWithSmsNotifier extends EmailNotifier." The problem is you need a new subclass for every combination, leading to an explosion of classes., you're looking at a class for every single combination: EmailSmsNotifier, EmailSlackNotifier, SmsSlackNotifier, EmailSmsSlackNotifier, EmailWithLoggingNotifier, SmsWithRetryNotifier... That's called class explosionWhen the number of subclasses grows exponentially because you need one for every possible combination of features. With N features, you'd need 2^N subclasses. Decorator solves this by letting you mix and match features at runtime., and it's a nightmare. With 5 optional features, you'd need 25 = 32 subclasses.

Decorator says: "Don't create a new class for every combination. Instead, create one small wrapperA class that holds a reference to another object and delegates work to it, optionally adding behavior before or after. The key is that the wrapper and the wrapped object share the same interface, so they're interchangeable. for each feature. Each wrapper takes in the thing it's wrapping, does its own work, then passes the call along." Want email + SMS + logging? Just nest them: LoggingDecorator(SmsDecorator(EmailNotifier())). Three small classes instead of thirty-two. And you can assemble any combination at runtime — no new classes needed.

The wrapped object doesn't know it's been wrapped, and the outside world doesn't know it's dealing with a wrapper.

// The shared interface — both the real object and decorators implement this public interface INotifier { void Send(string message); } // Core component — the "gift" inside all the wrapping public class EmailNotifier : INotifier { public void Send(string message) => Console.WriteLine($"[Email] {message}"); } // Decorator 1 — wraps ANY INotifier and adds SMS public class SmsDecorator(INotifier inner) : INotifier { public void Send(string message) { inner.Send(message); // delegate to inner Console.WriteLine($"[SMS] {message}"); // add own behavior } } // Decorator 2 — wraps ANY INotifier and adds Slack public class SlackDecorator(INotifier inner) : INotifier { public void Send(string message) { inner.Send(message); // delegate to inner Console.WriteLine($"[Slack] {message}"); // add own behavior } } // Compose at runtime — nest like Russian dolls: INotifier notifier = new SlackDecorator( new SmsDecorator( new EmailNotifier())); notifier.Send("Server is on fire!"); // Output: // [Email] Server is on fire! // [SMS] Server is on fire! // [Slack] Server is on fire!

How Decorator Works — Layers of Wrapping

Think of Decorator as concentric layers. A request comes in from the outside, passes through each wrapper layer (each adding its own behavior), reaches the core component, then the response travels back out through those same layers. Each layer is transparent — from the outside, the whole stack looks like a single INotifier.

Decorator-Wrapping-Layers

Notice the key insight: every layer implements the same interface. The Logging Decorator, Caching Decorator, Auth Decorator, and the Core Component all look identical from the outside — they're all INotifier (or IRepository, or Stream, or whatever your interface is). That's why wrapping is transparent. The client code doesn't know and doesn't care how many layers are wrapped around the core. It just calls the interface method, and the layers handle the rest.

Decorator wraps objects layer by layer to add behavior dynamically. Each wrapper and the core component share the same interface, making the wrapping transparent. Instead of 2^N subclasses for N features, you create N small wrappers and compose them at runtime.

Prerequisites

Decorator is one of the more intuitive structural patterns, but these four concepts will make the C# code examples click much faster. You don't need to be an expert — just comfortable enough that nothing in the code surprises you.

  • Interfaces in C# — Decorator depends on a crucial idea: both the real object and every decorator implement the same interfaceA contract that says "any class implementing me must have these methods." For Decorator, both the core component and all wrappers implement the same interface, making them interchangeable from the outside.. That's what makes wrapping transparent — the outside world can't tell whether it's talking to the real thing or a wrapped version. If "programming to an interface" is new to you, review that concept first — it's the foundation of the entire pattern.

  • Composition — Decorator is the poster child for composition over inheritance"Has-a" instead of "is-a." Rather than a CachingEmailNotifier that inherits from EmailNotifier, you create a CachingDecorator that HAS a reference to any INotifier. Composition is more flexible because you can swap the inner object at runtime.. Instead of saying "CachingNotifier is-a EmailNotifier" (inheritance), you say "CachingDecorator has-a INotifier reference" (composition). The decorator wraps the inner object and delegates to it. This "has-a" relationship is what makes Decorator so flexible — the inner object can be anything that implements the interface, including other decorators.

  • Constructor Injection — Every decorator receives the object it wraps through its constructorThe method called when creating a new instance of a class. In Decorator, the constructor takes the inner component as a parameter: new CachingDecorator(innerNotifier). This is how you "nest" the layers.. You'll see patterns like new LoggingDecorator(new CachingDecorator(new SqlRepository())). In production, the DI container handles this wiring for you, but understanding how constructor injection works helps you see how decorators chain together.

  • Single Responsibility Principle — Decorator is SRP in action. Instead of one fat class that logs, caches, retries, AND does the real work, each decorator handles exactly one concern. LoggingDecorator logs. CachingDecorator caches. RetryDecorator retries. The core component does the actual business logic. Each class has one reason to change. That's clean code.

Analogies

Walk into any coffee shop and watch how they build your drink. You start with a base — a plain espresso. That's your core component. Now, each topping or modification you add is a decorator. Want milk? That wraps the espresso into a latte. Want whipped cream on top of that? Another layer. Caramel drizzle? One more layer.

Here's the thing that makes this analogy perfect for Decorator: the barista doesn't need a pre-built recipe for "espresso + milk + whip + caramel." They just layer the toppings one at a time. They don't even need to know what's already in the cup to add whipped cream — they just add it on top of whatever's there. That "whatever's there" is the inner object, and "adding it on top" is the wrapping.

And notice: each topping adds to the price. The espresso is $3. Milk adds $0.50. Whip adds $0.75. Caramel adds $0.60. The total price is calculated by each layer asking the layer below "what's your price?" and then adding its own. That's exactly how decorators delegate — call the inner object's method, then add your own behavior on top of the result.

Section 4

Core Concept Diagram

Before diving into the class diagram, let's meet the four participants in the Decorator pattern. Each one has a clear, simple job:

1. Component (IComponent) — The interface that both the real object and all decorators implement. This is the key to the entire pattern. Because decorators and the real object share the same interface, they're interchangeable. The outside world doesn't know (or care) whether it's talking to the real thing or a wrapped version. Think of it as the contract that says "I can do X" — whether "I" am the real deal or a wrapper.

2. ConcreteComponent — The real object being decorated. This is the "espresso" in our coffee analogy — the thing that actually does the core work. A FileStream that reads bytes from disk, an EmailNotifier that sends emails, a SqlRepository that queries the database. It implements the IComponent interface and contains the actual business logic.

3. BaseDecorator — An abstract class that implements IComponent and holds a reference to another IComponent (the thing it's wrapping). By default, it just delegates every method call to the inner component — doing nothing extra. This exists so that concrete decorators only need to override the methods they care about, not every single interface method.

4. ConcreteDecorators — The actual wrappers that add specific behavior. LoggingDecorator logs before and after delegating. CachingDecorator checks a cache before delegating. RetryDecorator catches exceptions and retries. Each one extends BaseDecorator, overrides the methods it cares about, and adds its behavior before or after calling base.Method() (which delegates to the inner component).

GoF Definition: "Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality."

Also notice the composition arrow from BaseDecorator back to IComponent. The decorator doesn't just implement the interface — it also holds a reference to another IComponent. That reference could be a ConcreteComponent, or it could be another decorator. This self-referential structure is what lets decorators stack infinitely: Decorator A wraps Decorator B wraps Decorator C wraps the real thing. Each layer doesn't know (or care) what it's wrapping.

That's the Decorator secret: every wrapper is just "before + delegate + after." The composition arrow back to IComponent is the key — it's what lets decorators wrap decorators indefinitely.

Code Implementations

Let's build Decorator three different ways. Each implementation solves a different real-world problem, but they all follow the same fundamental structure: shared interface → core component → decorators that wrap and delegate. By seeing the pattern repeated across different domains, the structure will become second nature.

This is the textbook Decorator example, perfect for learning. We have an INotifier interface, a core EmailNotifier, and decorators that add SMS and Slack capabilities. The key learning: every decorator takes an INotifier in its constructor and is an INotifier itself — that's what makes nesting work.

Notice that we skip the abstract BaseDecorator here because the interface has only one method. When your interface has many methods, a base decorator avoids repeating boilerplate delegation in every concrete decorator. For a one-method interface, concrete decorators can delegate directly.

// The Component interface — both real notifiers and decorators implement this. // From the outside, everything is just an INotifier. public interface INotifier { void Send(string message); } // ConcreteComponent — the real thing. Does actual email sending. // This is the "espresso" — the core object everything wraps around. public class EmailNotifier : INotifier { public void Send(string message) { Console.WriteLine($"[Email] Sending: {message}"); // In production: SmtpClient, SendGrid, etc. } } // ConcreteDecorator #1 — adds SMS notification on top of whatever it wraps. // It takes an INotifier (could be EmailNotifier, could be another decorator) // and IS an INotifier itself — transparent wrapping. public class SmsDecorator : INotifier { private readonly INotifier _inner; public SmsDecorator(INotifier inner) { _inner = inner; // store reference to the wrapped object } public void Send(string message) { _inner.Send(message); // delegate to inner first Console.WriteLine($"[SMS] Sending: {message}"); // then add own behavior } } // ConcreteDecorator #2 — adds Slack notification. // Same pattern: wrap inner, delegate, add behavior. public class SlackDecorator : INotifier { private readonly INotifier _inner; public SlackDecorator(INotifier inner) { _inner = inner; } public void Send(string message) { _inner.Send(message); // delegate to whatever we're wrapping Console.WriteLine($"[Slack] Posting to #alerts: {message}"); } } // Compose decorators at runtime — nest like Russian dolls // Just email: INotifier email = new EmailNotifier(); email.Send("Server backup complete"); // Output: [Email] Sending: Server backup complete // Email + SMS: INotifier emailPlusSms = new SmsDecorator(new EmailNotifier()); emailPlusSms.Send("Deployment failed"); // Output: // [Email] Sending: Deployment failed // [SMS] Sending: Deployment failed // Email + SMS + Slack (full stack): INotifier fullStack = new SlackDecorator( new SmsDecorator( new EmailNotifier())); fullStack.Send("Database is down!"); // Output: // [Email] Sending: Database is down! // [SMS] Sending: Database is down! // [Slack] Posting to #alerts: Database is down! // The variable type is always INotifier — the client code // doesn't know or care how many layers are wrapped inside.

What just happened: We created every possible notification combination without a single subclass for each combo. EmailNotifier, SmsDecorator, and SlackDecorator — three classes, not seven (which is what you'd need for all combos with inheritance). Adding a fourth channel, like Teams? One new decorator class. The existing code doesn't change at all.

This is where Decorator really shines in production .NET code. You have a repositoryA class that abstracts data access. Instead of writing SQL directly in your service, you call repository.GetById(id). This makes it easy to swap databases, add caching, or test with fake data. that talks to a database. You want to add caching (so you don't hit the DB every time) and logging (so you can track what's happening). Instead of polluting your repository with caching and logging code, you wrap it.

// The shared interface — core repo and decorators all implement this public interface IRepository<T> where T : class { Task<T?> GetByIdAsync(int id); Task<IReadOnlyList<T>> GetAllAsync(); Task AddAsync(T entity); } // ConcreteComponent — does real database work public class SqlProductRepository : IRepository<Product> { private readonly AppDbContext _db; public SqlProductRepository(AppDbContext db) => _db = db; public async Task<Product?> GetByIdAsync(int id) => await _db.Products.FindAsync(id); public async Task<IReadOnlyList<Product>> GetAllAsync() => await _db.Products.ToListAsync(); public async Task AddAsync(Product entity) { _db.Products.Add(entity); await _db.SaveChangesAsync(); } } // Decorator #1: adds caching behavior // Checks the cache BEFORE hitting the database. // If the data is cached, returns it immediately (no DB call). // If not, delegates to the inner repository and caches the result. public class CachingRepositoryDecorator<T> : IRepository<T> where T : class { private readonly IRepository<T> _inner; private readonly IMemoryCache _cache; private readonly TimeSpan _expiration = TimeSpan.FromMinutes(5); public CachingRepositoryDecorator( IRepository<T> inner, IMemoryCache cache) { _inner = inner; _cache = cache; } public async Task<T?> GetByIdAsync(int id) { string key = $"{typeof(T).Name}:{id}"; // Try cache first — avoid DB call if possible if (_cache.TryGetValue(key, out T? cached)) return cached; // Cache miss — delegate to the real repository var result = await _inner.GetByIdAsync(id); if (result is not null) _cache.Set(key, result, _expiration); return result; } public async Task<IReadOnlyList<T>> GetAllAsync() { // For collections, cache with a different key string key = $"{typeof(T).Name}:all"; if (_cache.TryGetValue(key, out IReadOnlyList<T>? cached)) return cached!; var result = await _inner.GetAllAsync(); _cache.Set(key, result, _expiration); return result; } public async Task AddAsync(T entity) { await _inner.AddAsync(entity); // Invalidate the "all" cache after adding _cache.Remove($"{typeof(T).Name}:all"); } } // Decorator #2: adds logging behavior // Logs every call with timing information. // Doesn't know or care about caching — single responsibility. public class LoggingRepositoryDecorator<T> : IRepository<T> where T : class { private readonly IRepository<T> _inner; private readonly ILogger _logger; public LoggingRepositoryDecorator( IRepository<T> inner, ILogger<LoggingRepositoryDecorator<T>> logger) { _inner = inner; _logger = logger; } public async Task<T?> GetByIdAsync(int id) { _logger.LogInformation("Getting {Type} by id {Id}", typeof(T).Name, id); var sw = Stopwatch.StartNew(); var result = await _inner.GetByIdAsync(id); _logger.LogInformation( "Got {Type} by id {Id} in {Ms}ms — {Status}", typeof(T).Name, id, sw.ElapsedMilliseconds, result is not null ? "found" : "not found"); return result; } public async Task<IReadOnlyList<T>> GetAllAsync() { _logger.LogInformation("Getting all {Type}", typeof(T).Name); var sw = Stopwatch.StartNew(); var result = await _inner.GetAllAsync(); _logger.LogInformation( "Got {Count} {Type} records in {Ms}ms", result.Count, typeof(T).Name, sw.ElapsedMilliseconds); return result; } public async Task AddAsync(T entity) { _logger.LogInformation("Adding {Type}: {@Entity}", typeof(T).Name, entity); await _inner.AddAsync(entity); _logger.LogInformation("Added {Type} successfully", typeof(T).Name); } } // Wire decorators through DI — order matters! // The OUTERMOST decorator is what gets injected. // Logging wraps Caching wraps SqlRepository: // Logging → Caching → SQL // This means logging sees ALL calls (including cache hits), // and caching intercepts BEFORE hitting SQL. var builder = WebApplication.CreateBuilder(args); builder.Services.AddDbContext<AppDbContext>(); builder.Services.AddMemoryCache(); // Manual decorator wiring: builder.Services.AddScoped<IRepository<Product>>(sp => new LoggingRepositoryDecorator<Product>( new CachingRepositoryDecorator<Product>( new SqlProductRepository( sp.GetRequiredService<AppDbContext>()), sp.GetRequiredService<IMemoryCache>()), sp.GetRequiredService<ILogger< LoggingRepositoryDecorator<Product>>>())); // Or with Scrutor (much cleaner — see S8): // builder.Services.AddScoped<IRepository<Product>, SqlProductRepository>(); // builder.Services.Decorate<IRepository<Product>, // CachingRepositoryDecorator<Product>>(); // builder.Services.Decorate<IRepository<Product>, // LoggingRepositoryDecorator<Product>>();

Why this approach wins: The SqlProductRepository does ONE thing: talk to the database. It has no caching logic, no logging logic. The CachingDecorator does ONE thing: check cache, delegate on miss, store result. The LoggingDecorator does ONE thing: log timing. Each class is small, focused, and independently testable. Want to disable caching? Remove one wrapper. Want to add retry logic? Create a RetryDecorator and slip it between the layers.

Here's the best part: you've probably already used Decorator without knowing it. .NET's entire System.IO.Stream hierarchy is the Decorator pattern. FileStream reads bytes from disk. BufferedStream wraps any stream and adds buffering. GZipStream wraps any stream and adds compression. CryptoStream wraps any stream and adds encryption. Each takes a Stream in its constructor and is a Stream itself. Stack them however you like.

using System.IO; using System.IO.Compression; using System.Security.Cryptography; // .NET's Stream is the IComponent interface. // Every stream decorator takes a Stream and IS a Stream. // Layer 1: FileStream — reads raw bytes from disk (ConcreteComponent) using FileStream fileStream = File.OpenRead("data.txt"); // Layer 2: BufferedStream — wraps FileStream, adds buffering // WHY? Reduces the number of actual disk reads by reading ahead // into an in-memory buffer. Fewer disk reads = faster I/O. using BufferedStream buffered = new BufferedStream(fileStream, bufferSize: 4096); // Layer 3: GZipStream — wraps BufferedStream, adds decompression // WHY? The file is gzipped. GZipStream transparently decompresses // the bytes as you read. The code above doesn't know or care. using GZipStream decompressed = new GZipStream(buffered, CompressionMode.Decompress); // Each layer wraps the previous one. Reading from 'decompressed' // triggers: decompress → unbuffer → read disk. // The decoration chain: GZip(Buffered(File)) using StreamReader reader = new StreamReader(decompressed); string content = await reader.ReadToEndAsync(); // 'content' is the decompressed text — all the wrapping is invisible.

The beauty: StreamReader doesn't know it's reading from a compressed, buffered file. It just sees a Stream. You could swap FileStream for MemoryStream or NetworkStream and the decoration chain would still work perfectly. That's the power of transparent wrapping — every layer speaks the same interface.

Section 6

Junior vs Senior

Problem Statement

Build a data pipeline that reads from a database, caches results, logs every access, and validates the response. The ops team says they'll want to add rate limiting and circuit breaking later. How do you structure this?

How a Junior Thinks

"I'll put everything in one class. It reads from the DB, checks the cache, logs the call, and validates the result. It's all in one place, so it's easy to follow. I'll add rate limiting when they ask for it — just another if-block in the method."

// The "god class" approach — everything in one place public class DataService { private readonly AppDbContext _db; private readonly IMemoryCache _cache; private readonly ILogger _logger; public DataService(AppDbContext db, IMemoryCache cache, ILogger<DataService> logger) { _db = db; _cache = cache; _logger = logger; } public async Task<Product?> GetProductAsync(int id) { // Logging concern _logger.LogInformation("Getting product {Id}", id); var sw = Stopwatch.StartNew(); // Caching concern string cacheKey = $"product:{id}"; if (_cache.TryGetValue(cacheKey, out Product? cached)) { _logger.LogInformation("Cache hit for product {Id}", id); return cached; } // Database concern var product = await _db.Products.FindAsync(id); // Validation concern if (product is not null && string.IsNullOrEmpty(product.Name)) { _logger.LogWarning("Product {Id} has no name!", id); throw new ValidationException("Product name is required"); } // Caching concern (store) if (product is not null) _cache.Set(cacheKey, product, TimeSpan.FromMinutes(5)); // Logging concern (end) _logger.LogInformation("Got product {Id} in {Ms}ms", id, sw.ElapsedMilliseconds); return product; // Next quarter: add rate limiting here? Circuit breaking? // This method is already 40 lines and mixes 4 concerns. // Adding 2 more will make it 60+ lines of interleaved logic. } }

Problems

Single Responsibility Violation

One method handles logging, caching, database access, AND validation. That's four reasons to change in one place. A caching bug means editing the same method as a logging change. They're tangled together — you can't modify one without risking the others.

Can't Mix and Match

What if one endpoint needs caching + logging but NOT validation? Or validation + logging but NOT caching? With everything baked into one class, you get all-or-nothing. You can't compose different feature combinations for different use cases without messy boolean flags.

Testing Is Painful

To test just the caching logic, you need to set up a database context, a logger, AND trigger the right code path. You can't test caching in isolation because it's interleaved with logging and validation code. Every test needs all 3 dependencies mocked.

How a Senior Thinks

"Each concern gets its own class. The database service reads from the DB — that's it. Caching is a decorator that wraps it. Logging is another decorator. Validation is another. I compose them through DI. Adding rate limiting next quarter means one new decorator class and one line of DI registration. Nothing else changes."

Design Decisions

Each Decorator = One Concern

CachingDataDecorator only caches. LoggingDataDecorator only logs. ValidationDataDecorator only validates. Each is 30-40 lines, easy to understand, easy to test. The junior's 60-line monster is split into four clean, focused classes.

Composable Feature Combinations

Need caching + logging but no validation for one endpoint? Just compose differently: Logging(Caching(Sql)). Need validation but no caching for another? Validation(Sql). Each combination is just a different nesting order — no code changes, no boolean flags.

Isolated Testing

Testing caching? Mock IDataService (the inner), inject a real IMemoryCache, test that cache hits skip the inner call. No logger needed. No validator needed. Each decorator is tested in complete isolation from the others.

Fixing SRP Violation

The senior pulls each concern into its own decorator class. SqlDataService handles only database access. CachingDecorator handles only caching. LoggingDecorator handles only logging. Each class has exactly one reason to change. A caching bug means editing only the caching decorator — logging and validation are untouched.

Enabling Mix and Match

Because each concern is a separate wrapper, you compose exactly the features you need. For endpoint A: Logging(Caching(Sql)). For endpoint B: Validation(Sql). For endpoint C: Logging(Validation(Caching(Sql))). Different DI registrations, same decorator classes, zero code duplication.

Making Testing Easy

Each decorator is tested independently. To test caching: create a CachingDecorator with a mock IDataService inner. Call GetProductAsync twice. Verify the inner was called only once (cache hit on second call). No database, no logger, no validator — just the caching logic in isolation.

Juniors mix all concerns into one god class (logging + caching + validation + DB in one method). Seniors split each concern into its own decorator, compose them through DI, and add new features by creating new decorators — never modifying existing code.

Evolution & History

Decorator is one of the original GoF patterns from 1994, and it's aged beautifully. While some patterns have faded into obscurity, Decorator has grown more relevant with every generation of .NET. The reason is simple: as applications get more complex, the need to add cross-cutting behavior (logging, caching, auth, retry) without modifying existing code only increases.

Decorator-Evolution-Timeline

The Gang of Four catalogued Decorator as one of 23 foundational patterns. Their canonical example was Java's java.io package: BufferedReader wraps InputStreamReader wraps FileInputStream. Each layer adds behavior (buffering, character decoding, file access) without modifying the layers beneath. The GoF explicitly positioned Decorator as the answer to subclass explosionThe combinatorial problem that happens when you try to extend functionality through inheritance. With N optional features, you need 2^N subclasses. Decorator eliminates this by making features composable wrappers instead. — when inheritance creates too many classes, composition via wrapping is the escape hatch.

// GoF structure (adapted to C#-like pseudocode) interface IComponent { void Operation(); } class ConcreteComponent : IComponent { public void Operation() { /* real work */ } } abstract class Decorator : IComponent { protected IComponent _inner; public Decorator(IComponent inner) => _inner = inner; public virtual void Operation() => _inner.Operation(); // delegate } class ConcreteDecoratorA : Decorator { public ConcreteDecoratorA(IComponent inner) : base(inner) { } public override void Operation() { AddedBehavior(); // before base.Operation(); // delegate to inner } }

When .NET 1.0 shipped, its entire I/O system was built on Decorator. The Stream abstract class is the Component. FileStream, MemoryStream, and NetworkStream are ConcreteComponents. BufferedStream, CryptoStream, and GZipStream are Decorators — each takes a Stream in its constructor, adds behavior (buffering, encryption, compression), and is itself a Stream. The design was so successful that it hasn't changed in 20+ years. Why? Because Decorator is inherently extensible — Microsoft could add new Stream decorators without modifying any existing stream class.

Comparisons

Decorator gets mixed up with several other patterns because they all involve "wrapping" or "chaining" in some form. The confusion is understandable because they look similar in code. The difference is always about intent: what is the wrapping trying to achieve? Let's compare the three patterns that cause the most confusion.

Decorator vs Proxy

Both patterns wrap an object behind the same interface. Structurally, they look almost identical in a class diagram. The difference is why they wrap. A Decorator adds new behavior (logging, caching, validation) to make the object do more. A ProxyProxy pattern provides a surrogate that controls access to the real object. Common types: virtual proxy (lazy loading), protection proxy (access control), remote proxy (network calls), and caching proxy. The proxy decides IF and WHEN the real object is used. controls access to the object (lazy loading, security checks, network indirection). Decorator says "let me add to what you do." Proxy says "let me decide if you get to do it at all."

When to pick which: If you're adding functionality that wasn't there before (logging every method call, caching results, retrying on failure), that's Decorator. If you're gatekeeping the real object (checking permissions before allowing access, lazily creating an expensive object only when first needed, routing a call over the network), that's Proxy. A common confusion is caching proxyA caching proxy controls access by returning cached data instead of hitting the real service. It sits at the boundary between access control and behavior addition. Most teams call it a CachingDecorator if it always delegates on cache miss, or a CachingProxy if it sometimes prevents access entirely (e.g., circuit breaker with cache fallback). vs caching decorator. The practical answer: if it always delegates to the real object on a cache miss, call it a Decorator. If it sometimes prevents access entirely, call it a Proxy.

Decorator vs Strategy

This one trips people up because both patterns "change behavior." But they change behavior in completely different ways. Decorator wraps the outside of an object — it's like putting on a coat. The object doesn't know it's been decorated. StrategyStrategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. The host object holds a reference to a strategy and delegates to it. You swap the INTERNAL algorithm without changing the object itself. swaps the inside of an object — it replaces the algorithm the object uses internally. The object actively collaborates with its strategy. Decorator changes the skin; Strategy changes the guts.

When to pick which: If you need to add behavior around an existing operation (log before, cache after, retry on failure), Decorator is your tool. If you need to change the core algorithm the object uses (different sorting algorithms, different pricing calculations, different compression formats), Strategy is the fit. In practice, you might use both together: a Strategy to select the core algorithm, then Decorators to add cross-cutting concerns around whichever strategy is selected.

Decorator vs Chain of Responsibility

Both patterns chain handlers together. In fact, they look very similar in code: each handler holds a reference to the "next" one and calls it. The key difference is in whether the chain must reach the end. Decorator always delegates to the next layer — every decorator runs, and eventually the core object runs too. Chain of ResponsibilityChain of Responsibility passes a request along a chain of handlers. Each handler can either process the request and stop, or pass it to the next handler. Unlike Decorator, the chain CAN stop early. Used in middleware, exception handlers, and approval workflows. may stop at any handler — if one handler processes the request, it can choose not to pass it along.

When to pick which: If every layer must always run and there's always a real object at the center, that's Decorator. If handlers should be able to "claim" a request and stop it from propagating further (like exception handlers, approval workflows, or middleware that short-circuits with a 401 response), that's Chain of Responsibility. ASP.NET Core middleware is actually a blend: it's structurally like Decorator (wraps the next), but functionally like CoR (authentication middleware can short-circuit the chain).

Decorator adds behavior (wraps the outside). Strategy swaps algorithms (changes the inside). Proxy controls access (gates the object). Chain of Responsibility chains handlers that may stop early, while Decorator always delegates through every layer.

Decorator is one of the most SOLID-aligned patterns in the GoF catalog. It was practically designed to make the Open/Closed PrincipleOCP: classes should be open for extension but closed for modification. Decorator is the textbook way to achieve OCP for cross-cutting concerns. a reality for cross-cutting concerns. Let's walk through each principle and see how Decorator connects.

In .NET Core, the built-in DI container (IServiceProvider) manages object lifetimes and can wire decorator chains automatically using libraries like Scrutor. — the DI container is the natural place to compose decorator chains.

Closing insight: Decorator scores 4/5 "Supports" on SOLID, with ISP as the one conditional. That's not a coincidence — Decorator was literally designed to let you extend objects without modifying them (OCP), through a common abstraction (DIP), with single-purpose wrappers (SRP), that are substitutable for the original (LSP). The pattern is SOLID incarnate.

Pitfalls & Anti-Patterns

Decorator is elegant when used correctly, but there are common traps that turn it from a clean solution into a maintenance headache. Each pitfall below includes an SVG diagram and bad/good code comparison. Click to expand.

Mistake: Applying Decorator to an interface with 15+ methods. Every decorator must implement and delegate all 15 methods, even if it only cares about 2 of them. The other 13 are pure boilerplate pass-throughs.

Why This Happens: The original interface was designed without Decorator in mind. It grew organically as features were added. Nobody stopped to ask "is this still one responsibility?" Now you have IRepository<T> with GetById, GetAll, GetPaged, Add, Update, Delete, BulkInsert, Count, Exists, Query, and 5 more. Decorating it means implementing all of them in every decorator.

Imagine being told to gift-wrap a present, but the "present" is a 15-piece furniture set. You have to wrap each piece individually, even though you only wanted to put a bow on the table. That's what decorating a bloated interface feels like.

// ❌ BAD: 15-method interface forces massive decorators public interface IRepository<T> { Task<T> GetByIdAsync(int id); Task<IEnumerable<T>> GetAllAsync(); Task<PagedResult<T>> GetPagedAsync(int page, int size); Task AddAsync(T entity); Task UpdateAsync(T entity); Task DeleteAsync(int id); Task BulkInsertAsync(IEnumerable<T> entities); Task<int> CountAsync(); Task<bool> ExistsAsync(int id); // ... 6 more methods } // CachingDecorator must implement ALL 15 methods. // Only GetByIdAsync and GetAllAsync need caching. // The other 13 are pure pass-throughs = wasted code. // ✅ GOOD: Split into focused interfaces (ISP) public interface IReader<T> { Task<T> GetByIdAsync(int id); Task<IEnumerable<T>> GetAllAsync(); } public interface IWriter<T> { Task AddAsync(T entity); Task UpdateAsync(T entity); Task DeleteAsync(int id); } // Now CachingDecorator only wraps IReader<T> — 2 methods, zero boilerplate public class CachingReaderDecorator<T> : IReader<T> { private readonly IReader<T> _inner; private readonly IMemoryCache _cache; // Only 2 methods to implement! }

Fix: Apply the Interface Segregation PrincipleISP (Interface Segregation Principle): Split fat interfaces into smaller, focused ones. Clients should only depend on the methods they actually use. This makes decoration practical — you only wrap the slice you care about. before decorating. Split the fat interface into smaller, focused interfaces. Your caching decorator only needs to wrap the read methods? Create IReader<T> and decorate that. Write methods don't need caching? They stay on IWriter<T> undecorated.

Mistake: Registering decorators in the wrong order and getting unexpected behavior. Logging inside caching means cache hits are invisible to logs. Retry outside logging means the same request gets logged multiple times. The decoration order determines the behavior, but it's rarely documented.

Why This Happens: Decoration order in DI configuration reads like a flat list of registrations, but it actually creates a nested structure. Developers add decorators one at a time without thinking about the full chain. The order matters more than most people realize, and the consequences of getting it wrong are subtle — things appear to work but behave incorrectly.

Think of getting dressed: coat over sweater over shirt makes sense. Shirt over coat over sweater does not. The order of layers matters, and getting it wrong doesn't cause an error — it just looks weird.

// ❌ BAD: Cache wraps Logging — cache hits invisible services.Decorate<IRepository, LoggingDecorator>(); // Inner services.Decorate<IRepository, CachingDecorator>(); // Outer // Result: Cache(Log(SQL)) — logging only fires on cache misses // ✅ GOOD: Logging wraps Cache — sees everything // ORDER MATTERS! Read bottom-up: outermost = last registered services.Decorate<IRepository, ValidationDecorator>(); // Layer 1 (innermost) services.Decorate<IRepository, CachingDecorator>(); // Layer 2 services.Decorate<IRepository, LoggingDecorator>(); // Layer 3 services.Decorate<IRepository, RetryDecorator>(); // Layer 4 (outermost)

Fix: Always document the intended decoration order in comments. Write an integration test that resolves the service and checks service.GetType().Name matches the expected outermost decorator. Use the recommended order: Retry > Logging > Caching > Validation > Core.

Mistake: A decorator silently modifies the data passing through it — trimming strings, converting formats, normalizing values — without the caller or inner component knowing. This turns a "transparent wrapper" into an invisible data transformer.

Why This Happens: It seems like a convenient place to add data cleanup. "While we're logging, let's also trim whitespace on inputs." But decorators should add concerns (logging, caching, retry), not silently transform data. If a decorator changes the input, the inner component receives different data than the caller sent. If it changes the output, the caller receives different data than the inner component produced.

Imagine a postal service worker who opens your packages, replaces your handwritten letter with a typed version, reseals the package, and delivers it. The recipient gets a "better" letter, but it's not what you sent. That's not the postal service's job.

// ❌ BAD: Decorator silently modifies data public class "CleanupDecorator" : IRepository { public async Task AddAsync(Product product) { product.Name = product.Name?.Trim(); // ❌ Changes input! product.Description = product.Description?.Trim(); await _inner.AddAsync(product); } } // ✅ GOOD: Keep decoration transparent, add separate validation public class ValidationDecorator : IRepository { public async Task AddAsync(Product product) { if (string.IsNullOrWhiteSpace(product.Name)) throw new ValidationException("Product name is required"); await _inner.AddAsync(product); // ✅ Pass through unchanged } } // If you need data cleanup, make it an explicit step: public class ProductService { public async Task CreateAsync(ProductDto dto) { var product = SanitizeInput(dto); // ✅ Explicit, visible await _repository.AddAsync(product); } }

Fix: Decorators should observe, log, cache, retry, or reject — but never silently transform the data. If you need data transformation, make it an explicit step (a separate validator, sanitizer, or mapper), not a hidden decorator behavior. The caller should be able to predict exactly what data reaches the inner component.

Mistake: Decorator wraps IShape and overrides Draw(), but forgets to delegate the Color property. Downstream code that reads shape.Color gets null or a default value instead of the actual color from the inner shape.

Why This Happens: Developers focus on the methods they're decorating and forget about properties. In C#, properties are just syntactic sugar over getter/setter methods, but they feel different mentally. If the interface has 3 methods and 2 properties, the decorator needs to handle all 5 — not just the methods.

It's like wrapping a gift box but accidentally covering the "this side up" label. The wrapping looks great, but critical information is hidden.

// ❌ BAD: Forgot to delegate Color property public class BorderDecorator : IShape { private readonly IShape _inner; public string Color { get; set; } // ❌ Own property, ignores inner! public void Draw() { DrawBorder(); _inner.Draw(); } } // ✅ GOOD: Forward all properties to inner public class BorderDecorator : IShape { private readonly IShape _inner; public string Color { get => _inner.Color; // ✅ Delegate getter set => _inner.Color = value; // ✅ Delegate setter } public void Draw() { DrawBorder(); _inner.Draw(); } }

Fix: When implementing a decorator, go through every member of the interface (methods AND properties) and ensure each one delegates to _inner. A base decorator class that delegates everything by default can help — concrete decorators only override the members they need to modify.

Mistake: Using Decorator to add a single, fixed enhancement to one class. The team creates an interface, a base decorator, and a concrete decorator — all to add one behavior that will never be composed with anything else.

Why This Happens: Pattern enthusiasm. Someone learned about Decorator and now everything looks like a nail. But patterns exist to solve specific problems. If you have one class that needs one fixed enhancement that will never change, plain inheritance is simpler, more readable, and more honest about what's happening.

You don't need a Swiss Army knife to open a letter. If the job is simple, use a simple tool.

// ❌ Over-engineered: Decorator for ONE fixed behavior public interface ILogger { void Log(string msg); } public class BaseLoggerDecorator : ILogger { /* all pass-through */ } public class TimestampDecorator : BaseLoggerDecorator { /* adds timestamp */ } // 3 files, 1 interface, 1 base — for ONE behavior that never changes // ✅ Just inherit — simple, clear, honest public class TimestampLogger : FileLogger { public override void Log(string msg) => base.Log($"[{DateTime.UtcNow:O}] {msg}"); } // Done. One class. If you never compose, don't over-architect.

Fix: Before reaching for Decorator, ask: "Will I ever need to compose this behavior with other behaviors? Will different objects need different combinations?" If both answers are "no," inheritance is the right tool. Save Decorator for when you actually need composable, mix-and-match behavior.

Mistake: Code depends on CachingDecorator instead of IRepository. This defeats the entire purpose of transparent wrapping — callers should never know they're talking to a decorator.

Why This Happens: A developer needs to access the decorator's specific functionality (like clearing the cache). Instead of creating a proper way to manage the cache, they cast or inject the concrete decorator type. Now the calling code is tightly coupled to the decoration structure.

If you peek inside the gift wrapping to check what brand of paper was used before opening the gift, you're missing the point of the wrapping. The whole idea is that it's transparent.

// ❌ BAD: Depends on concrete decorator public class ProductService { private readonly CachingDecorator _repo; // ❌ Tied to decorator type public void ClearCache() => _repo.ClearCache(); // ❌ Knows about caching } // ✅ GOOD: Depends on interface, separate cache management public class ProductService { private readonly IRepository _repo; // ✅ Depends on abstraction } // Expose cache management through a separate interface public interface ICacheManager { void ClearCache(string key); } public class CachingDecorator : IRepository, ICacheManager { /* ... */ } // Inject ICacheManager separately where cache management is needed public class CacheController(ICacheManager cache) { [HttpPost("clear")] public IActionResult Clear() { cache.ClearCache("products"); return Ok(); } }

Fix: Never inject or depend on concrete decorator types. If you need decorator-specific functionality (like cache clearing), expose it through a separate interface. The calling code depends on IRepository for data access and ICacheManager for cache management — never on CachingDecorator directly.

The most important tests: mock the inner component and verify the decorator adds its behavior correctly. This is fast, deterministic, and catches bugs in the decorator logic itself. You should have the most of these tests.

Let's be direct: Decorator adds trivial overhead per layer. The real performance story is that a well-placed CachingDecorator can make your application orders of magnitude faster by eliminating redundant work. Here's the honest breakdown of costs and benefits.

Virtual dispatchWhen calling a method through an interface (IRepository.GetByIdAsync), the runtime must look up the actual method implementation via a vtable. This costs ~1-2 nanoseconds per call. Easy (Q1 – Q10)

Q1: What is the Decorator pattern and why does it exist?
Easy
Think First You have a perfectly working object. You want to add behavior to it — but WITHOUT modifying the original class and WITHOUT creating a separate subclass for every combination. How would you do that?

The Decorator pattern lets you wrap an object inside another object that adds new behavior — while keeping the same interface. Think of it like putting a phone case on your phone. The phone still works exactly the same (same buttons, same screen), but now it has extra protection. You can stack cases if you want — a bumper case inside a wallet case — and the phone doesn't care.

Why does this exist? Because the alternative is inheritance, and inheritance doesn't scale. If you have a Notifier class and you want to add SMS, Email, and Slack capabilities, inheritance forces you to create SmsNotifier, EmailNotifier, SlackNotifier, SmsEmailNotifier, SmsSlackNotifier, EmailSlackNotifier, SmsEmailSlackNotifier — that's 7 classes for 3 features. With Decorator, you create 3 wrapper classes and stack them in any order at runtime. Done.

The key principle is composition over inheritance: instead of "is-a" relationships (SmsNotifier IS A Notifier), you use "has-a" (SmsDecorator HAS A notifier inside it and adds SMS behavior on top).

Great Answer Bonus "Decorator solves the combinatorial explosion problem. With N features, inheritance needs 2^N classes. Decorator needs N wrapper classes that you stack at runtime — same interface, transparent to the caller."
Q2: How is Decorator different from simply subclassing?
Easy
Think First Suppose you have a Coffee class with 5 optional add-ons (milk, sugar, whip, caramel, chocolate). How many subclasses would you need to cover every combination? Now think about what happens when you add a 6th option.

With inheritance, every combination of features needs its own class. Five add-ons means 2^5 = 32 subclasses. Add a 6th and it doubles to 64. This is called the class explosion problem — the number of subclasses grows exponentially with each new feature. It's a maintenance nightmare.

Decorator avoids this entirely. Each add-on is a single wrapper class. Five add-ons = 5 decorator classes. Adding a 6th? Just write one more class — no existing code changes. You compose them at runtime in any order: new WhipDecorator(new CaramelDecorator(new Espresso())).

There's another subtle difference: timing. Inheritance is decided at compile time — you write new MilkSugarCoffee() and that's fixed forever. Decorator is decided at runtime — you can add or remove layers based on user input, configuration, or conditions. This makes Decorator far more flexible for anything that needs to be configurable.

Great Answer Bonus "Inheritance is compile-time, static, and exponential. Decorator is runtime, dynamic, and linear. Whenever you see combinatorial subclass growth, Decorator is the fix."
Q3: Give a real-world .NET example of the Decorator pattern
Easy
Think First You've used new BufferedStream(new FileStream(...)) in .NET. What pattern is at play? Why does BufferedStream wrap FileStream instead of FileStream having a bool enableBuffering flag?

The most classic example in .NET is the Stream class hierarchy. Every stream in .NET inherits from the abstract Stream base class. But some streams don't do I/O themselves — they wrap another stream and add behavior on top.

FileStream reads raw bytes from disk. BufferedStream wraps any stream and adds an in-memory buffer. GZipStream wraps any stream and compresses bytes. CryptoStream wraps any stream and encrypts bytes.

The beauty: you can stack them. new CryptoStream(new GZipStream(new BufferedStream(new FileStream(...)))) gives you encrypted, compressed, buffered file I/O — and the code at the top just calls stream.Read() without knowing about any layers.

Great Answer Bonus "The .NET Stream hierarchy is the textbook Decorator implementation. Every decorator stream wraps another Stream, adds one behavior, and passes everything else through."
Q4: What makes a good candidate for decoration?
Easy
Think First Think about the things you add to MANY services — logging, caching, retry logic, validation. What do they all have in common?

A behavior is a good decoration candidate when it ticks these boxes:

1. Cross-cutting concern. The behavior is needed in many places — logging, caching, authentication, retry logic. These aren't part of any one service's core job; they cut across the entire application.

2. Same interface in and out. The decorator must look identical to the original from the outside. If wrapping changes the interface, it's not Decorator — it's Adapter.

3. Independent and stackable. Each decorator adds exactly one behavior. Logging shouldn't also do caching. This independence means you can stack them in any order.

4. Runtime flexibility needed. If the behavior is always needed and never changes, a simple base class might be simpler. Decorators shine when you need to add/remove behavior based on configuration or environment.

Great Answer Bonus "The sweet spot is cross-cutting concerns that need the same interface: logging, caching, retry, validation. If you're copy-pasting the same try/catch/log into every service method, that's a decorator waiting to be extracted."
Q5: What is "transparent wrapping" and why does it matter?
Easy
Think First If you wrap a service with a LoggingDecorator, should the code that calls the service need to change? What if you add a CachingDecorator on top later?

Transparent wrapping means the caller has absolutely no idea it's talking to a decorated object. From the outside, the decorated object looks and behaves exactly like the original — same interface, same method signatures, same return types. The caller says IRepository repo and doesn't care whether it's the real repository, a cached one, or one wrapped in five layers of decorators.

This matters for two reasons. First, you can add decorators without touching any caller code. Want to add caching to your repository? Just change the DI wiring — every controller, service, and handler that injects IRepository automatically gets the cached version. Zero code changes across the app.

Second, you can swap decorators in different environments. Development might skip the caching decorator for easier debugging. Production stacks logging + caching + retry. Same callers, different wiring, different behavior.

Great Answer Bonus "Transparent wrapping is what makes Decorator composable. Because every layer shares the same interface, you can add, remove, or reorder layers without touching a single line of calling code."
Q6: Can you decorate a sealed class?
Easy
Think First A sealed class can't be inherited from. But Decorator uses composition (has-a), not inheritance (is-a). Does that change anything?

Yes — as long as the sealed class implements an interface. Decorator doesn't inherit from the object it wraps; it wraps it via composition. The decorator implements the same interface, holds a reference to the inner object, and delegates calls to it. The inner object's class being sealed is irrelevant because the decorator never extends it.

The catch: if the sealed class doesn't implement an interface, you have nothing to wrap against. You'd need to create an interface first (extract it from the class's public methods), or use the Adapter patternAdapter wraps an object behind a NEW interface. Unlike Decorator (which preserves the interface), Adapter changes the interface shape to make incompatible classes work together. to create a wrappable abstraction around it.

Q7: What is the role of the base decorator abstract class?
Easy
Think First If every decorator needs to delegate all methods to the inner object, and you have 5 methods on the interface, that's a lot of repetitive code. How do you avoid writing the same delegation 5 times in every decorator?

The base decorator is an optional convenience class that provides default pass-through delegation for every method in the interface. Concrete decorators inherit from it and override only the methods where they add behavior. Everything else automatically delegates to the inner object.

Without a base decorator, every concrete decorator must implement every method — even ones it doesn't enhance. That's tedious and bug-prone (if the interface adds a new method, every decorator that forgot to delegate it silently breaks). The base decorator centralizes the delegation so concrete decorators stay small and focused.

That said, in modern C# with primary constructorsA C# 12 feature that lets you declare constructor parameters directly in the class declaration: class Foo(int x). The parameter is available throughout the class body. and small interfaces (2-3 methods), many teams skip the base class entirely. Each decorator is so short that the delegation is barely a burden. It's a pragmatic choice: use a base decorator for interfaces with 4+ methods, skip it for tiny interfaces.

// Base decorator — delegates everything by default public abstract class RepositoryDecorator(IProductRepository inner) : IProductRepository { public virtual Task<Product?> GetByIdAsync(int id) => inner.GetByIdAsync(id); public virtual Task<IReadOnlyList<Product>> GetAllAsync() => inner.GetAllAsync(); public virtual Task AddAsync(Product product) => inner.AddAsync(product); public virtual Task DeleteAsync(int id) => inner.DeleteAsync(id); } // Concrete decorator — only overrides what it enhances public class CachingDecorator(IProductRepository inner, IMemoryCache cache) : RepositoryDecorator(inner) { public override async Task<Product?> GetByIdAsync(int id) { if (cache.TryGetValue(id, out Product cached)) return cached; var product = await base.GetByIdAsync(id); // delegates to inner if (product is not null) cache.Set(id, product); return product; } // GetAllAsync, AddAsync, DeleteAsync — auto-delegated by base class! }
Great Answer Bonus "The base decorator is a delegation convenience — it implements pass-through for every method so concrete decorators only override what they enhance. Skip it for tiny interfaces, use it for 4+ methods."
Q8: How many decorators can you stack? Is there a limit?
Easy
Think First Each decorator is just an object wrapping another object. Is there a technical limit on how deep you can nest objects?

There's no hard limit — you can stack as many decorators as you want. Each decorator is just an object that holds a reference to another object. Stacking 3, 5, or 10 decorators works fine technically.

That said, there are practical limits. Each layer adds one virtual method call (~1-2 nanoseconds) and one object allocation. For 3-5 layers, this is completely negligible. For 20+ layers, you might notice the overhead in profiling — but you'd almost never need that many. If you're stacking more than 6-7 decorators, it's a design smell: either your interface is doing too many things (violating ISPInterface Segregation Principle — keep interfaces small so that decorators only need to implement the methods they care about.) or you should consider a pipeline/middleware approach instead.

The real limit is human comprehension. Debugging a chain of 10 decorators means tracing through 10 layers of delegation. Keep your chains at 3-5 layers and your teammates will thank you.

Great Answer Bonus "No technical limit — but a practical limit of 3-5 layers. Beyond that, debugging becomes painful and the overhead starts to show. If you need more, consider a pipeline pattern with explicit ordering."
Q9: Does decoration order matter?
Easy
Think First Imagine putting on clothes. Shirt first, then jacket — or jacket first, then shirt? The order matters because each layer goes OVER the previous one. Same thing with decorators.

Yes, order matters enormously. The outermost decorator runs first (receives the call), then delegates inward. So if you wrap Logging(Caching(Repository)), logging runs first, then caching checks the cache, then (on a miss) the repository does the real query. But Caching(Logging(Repository)) means caching runs first — on a cache hit, the request never reaches the logging layer, so cache hits are invisible in your logs.

The rule of thumb: put the concern you want to see ALL calls first as the outermost layer. Logging usually goes outermost so it captures everything. Caching goes closer to the core to short-circuit expensive operations. Validation should be outermost or near-outermost so invalid requests are rejected before any work happens.

Great Answer Bonus "Order matters because the outermost decorator runs first. I always put logging outermost for full visibility, caching closer to the core to short-circuit DB calls, and document the order in Program.cs with comments."
Q10: What is "composition over inheritance" and how does Decorator use it?
Easy
Think First "Is-a" means "this thing IS that thing" (a Dog IS an Animal). "Has-a" means "this thing CONTAINS that thing" (a Car HAS an Engine). Which relationship does Decorator use?

Composition over inheritance is a design principle that says: instead of building behavior by extending classes (inheritance / "is-a"), build it by combining objects (composition / "has-a"). Prefer holding a reference to an object and delegating work to it, rather than inheriting from it.

Decorator is the poster child for this principle. A LoggingDecorator doesn't inherit from ProductRepository. Instead, it holds a reference to an IProductRepository and delegates every call to it, adding logging before and after. This means:

1. Flexibility: You can wrap ANY IProductRepository — not just one specific class. The inner object can be the real repo, a fake for testing, or another decorator.

2. Runtime changes: You can swap the inner object at runtime. Inheritance locks you in at compile time.

3. Multiple behaviors: You can stack multiple decorators. Inheritance only gives you one parent class in C# (no multiple inheritance).

Great Answer Bonus "Decorator IS composition over inheritance. Instead of CachedLoggingRepository extends Repository, you write new LoggingDecorator(new CachingDecorator(repository)). Same behavior, but composable, testable, and runtime-configurable."

Q11: How do you wire decorators with .NET Dependency Injection?

Q15: How do you handle async decorators in C#?
Medium
Think First If the interface method returns Task<T>, the decorator's method must also be async. What pitfalls emerge when you mix async/await with decoration?

Async decorators work exactly like sync ones — just with async/await. The interface returns Task<T>, the decorator awaits the inner call and wraps behavior around it. But there are three pitfalls:

1. Never use .Result or .Wait(). These block the thread and cause deadlocks in ASP.NET. Always await the inner call.

2. Pass CancellationToken through. If the interface accepts a cancellation token, every decorator must forward it to the inner call. Dropping the token means the operation can't be cancelled from the outermost layer.

3. Handle OperationCanceledException. When a token is cancelled mid-chain, the inner call throws. Decide whether your decorator should catch and handle it, or let it propagate. Usually, let it propagate — the outermost layer decides how to respond to cancellation.

public class CachingDecorator(IProductRepository inner, IMemoryCache cache) : IProductRepository { public async Task<Product?> GetByIdAsync(int id, CancellationToken ct = default) { if (cache.TryGetValue(id, out Product? cached)) return cached; // ALWAYS await — never .Result or .Wait() // ALWAYS forward the CancellationToken var product = await inner.GetByIdAsync(id, ct); if (product is not null) cache.Set(id, product, TimeSpan.FromMinutes(5)); return product; } }
Great Answer Bonus "Async decorators must always await, never .Result, and always forward CancellationToken. These two rules prevent deadlocks and ensure the chain is fully cancellable."

Q17: How do you prevent circular decoration?
Medium
Think First What happens if decorator A wraps B and B wraps A? Or if a decorator accidentally resolves itself from the DI container instead of the inner service?

Circular decoration (A wraps B which wraps A) creates an infinite loop that ends in a StackOverflowException. This usually happens through DI misconfiguration, not intentional code. Two common causes:

1. DI resolving the decorator as its own inner. If a CachingDecorator asks for IRepo from the container, and the container is already configured to return CachingDecorator for IRepo, you get a circle. Fix: use Scrutor (which handles this automatically) or register the concrete class separately from the interface.

2. Two decorators that reference each other. Extremely rare in practice, but possible if someone manually wires A to wrap B and B to wrap A. Fix: use a linear chain where each layer only knows about "inner" — never about a specific decorator type.

Prevention strategies:

• Always use Scrutor's Decorate() — it builds a linear chain by design.

• Constructor should accept the interface, never a specific decorator type.

• Add a startup health check that logs the resolved chain and verifies it's acyclic.

Great Answer Bonus "Circular decoration = StackOverflow at runtime. Prevent it by using Scrutor for linear chains, accepting interfaces (not concrete types) in constructors, and adding a startup health check that logs the decoration chain."

Q19: How do you debug issues in deeply nested decorator chains?

Q24: Compare Decorator to AOP (Aspect-Oriented Programming)

Q27: How do you handle decorator state and thread safety?
Hard
Think First A CachingDecorator stores data in a dictionary. A CircuitBreakerDecorator tracks failure counts. Both are state. What happens when multiple threads call the service concurrently?

Most decorators are stateless — they don't hold data between calls. A LoggingDecorator writes to a logger and forgets. These are inherently thread-safe.

But stateful decorators (caching, circuit breaker, rate limiter) need thread-safety measures:

1. Use concurrent collections. Replace Dictionary<K,V> with ConcurrentDictionary<K,V>. Use IMemoryCache (already thread-safe) instead of rolling your own cache.

3. Scope via DI lifetime. Register stateful decorators as Scoped (one per request) instead of Singleton. This gives each request its own decorator state, eliminating cross-request race conditions. Exception: circuit breakers genuinely need shared state across requests — register as Singleton but use thread-safe types.

4. Avoid locks in async code. Never use lock around await. Use SemaphoreSlim if you need async-compatible mutual exclusion.

    Deep Dive

    ScrutorA popular .NET library that extends Microsoft's built-in DI container with assembly scanning and decoration support. Its Decorate method makes it trivial to wrap registered services with decorators. is a library that adds decoration support to Microsoft's built-in DI container. Instead of writing factory lambdas to wire decorator chains manually, you call services.Decorate<IService, LoggingDecorator>() and Scrutor handles the rest.

    Under the hood, Scrutor replaces the existing service registration. When you register IRepo → ProductRepo and then call Decorate<IRepo, CachingDecorator>, Scrutor changes the registration to: "Create a CachingDecorator and inject the original ProductRepo into it." Stack multiple Decorate calls — each wraps the previous. The last call becomes the outermost layer.

    builder.Services.AddScoped<IProductRepository, ProductRepository>(); builder.Services.Decorate<IProductRepository, ValidationDecorator>(); builder.Services.Decorate<IProductRepository, CachingDecorator>(); builder.Services.Decorate<IProductRepository, LoggingDecorator>(); // Resolution: Caller → Logging → Caching → Validation → ProductRepository // Works with open generics too builder.Services.AddScoped(typeof(IRepository<>), typeof(Repository<>)); builder.Services.Decorate(typeof(IRepository<>), typeof(LoggingRepository<>));
    Common Scrutor Pitfall

    If you call Decorate before the base registration, Scrutor silently does nothing. Always register the concrete class first, then decorate. And remember: the last Decorate call is the outermost layer, not the first.

    Both Decorator and AOPAspect-Oriented Programming — a paradigm for separating cross-cutting concerns from business logic. Instead of writing wrapper classes, you define "aspects" that are woven into your code automatically at compile time (PostSharp) or runtime (Castle DynamicProxy). add cross-cutting behavior without modifying the original code. But they do it in very different ways — and the trade-offs matter.

    Decorator is explicit. You write a wrapper class, implement every method, manually delegate. Verbose but transparent — you can step through everything in a debugger.

    AOP is implicit. Libraries like Castle DynamicProxy generate proxy objects at runtime. PostSharp rewrites compiled code at build time. You define one interceptor and it applies to all methods automatically.

    AspectDecoratorAOP (Castle / PostSharp)
    VisibilityFully explicitImplicit — behavior added behind the scenes
    BoilerplateHigh — must delegate every methodLow — one interceptor for all methods
    DebuggingEasy — step through wrapper codeHarder — generated proxy, noisy stack traces
    PerformanceMinimal (one virtual call per layer)DynamicProxy: reflection; PostSharp: zero
    Best forSmall interfaces, explicit wiringLarge interfaces, uniform cross-cutting
    Rule of Thumb

    Use Decorator when the interface is small (3-5 methods) and you want explicit, testable wiring. Use AOP when the interface is large (10+ methods) and you want the same behavior applied uniformly without boilerplate.

    If you look at Decorator from a distance, it creates an onion shapeA layered architecture where each layer wraps the next. Requests flow inward through the layers, hit the core, and responses flow back outward. — layers wrapped around a center. ASP.NET Core's middleware pipeline, clean architecture, and even network protocol stacks all follow this same pattern.

    When a request arrives, it enters the outermost layer first. That layer does some work (authenticate, log), then passes inward. This continues until it reaches the core (business logic). The core produces a response, which travels back outward through the same layers in reverse. Each layer can add behavior on the way out (stop the timer, add headers).

    app.Use(async (context, next) => { Console.WriteLine($"[1-IN] Request: {context.Request.Path}"); var sw = Stopwatch.StartNew(); await next(context); // Pass inward sw.Stop(); Console.WriteLine($"[1-OUT] Response: {context.Response.StatusCode} in {sw.ElapsedMilliseconds}ms"); }); app.Use(async (context, next) => { Console.WriteLine("[2-IN] Checking authentication..."); await next(context); Console.WriteLine("[2-OUT] Auth header added"); }); app.MapGet("/api/products", () => Results.Ok(new[] { "Widget", "Gadget" })); // Output: [1-IN] → [2-IN] → (core) → [2-OUT] → [1-OUT]
    The Onion Is Everywhere

    TCP/IP network layers (Application → Transport → Network → Physical → back up), HTTP middleware, database connection pooling (acquire → execute → release), and even try/catch blocks. Decorator is the OOP expression of this universal pattern.

    Mini-Project: Extensible Logger

    Let's build a logging system where processing steps can be added and removed dynamically. We'll evolve from a monolithic mess to a clean, configurable decorator pipeline.

    Why this project? Logging is the most common cross-cutting concern. Every production system needs it, and requirements always grow: "Add timestamps." "Add thread IDs." "Format as JSON." "Write to a file AND console." Decorator handles this gracefully.

    One Logger class with boolean flags for every feature. With 5 features, you have 2^5 = 32 possible configurations crammed into one class. Every new feature adds more if statements.

    public class MonolithicLogger { public bool AddTimestamp { get; set; } public bool AddThreadId { get; set; } public bool FormatAsJson { get; set; } public bool WriteToFile { get; set; } public bool WriteToConsole { get; set; } = true; private readonly string? _filePath; public MonolithicLogger(string? filePath = null) => _filePath = filePath; public void Log(string message) { var output = message; if (AddTimestamp) output = $"[{DateTime.Now:yyyy-MM-dd HH:mm:ss.fff}] {output}"; if (AddThreadId) output = $"[Thread-{Environment.CurrentManagedThreadId}] {output}"; if (FormatAsJson) output = JsonSerializer.Serialize(new { timestamp = DateTime.Now, message }); if (WriteToConsole) Console.WriteLine(output); if (WriteToFile && _filePath is not null) File.AppendAllText(_filePath, output + Environment.NewLine); } } // Problems: FormatAsJson ignores flags, 32 paths to test, SRP violated

    Separate each processing step into its own decorator. Stack any combination at runtime.

    public interface IAppLogger { void Log(string message); } public class ConsoleLogger : IAppLogger { public void Log(string message) => Console.WriteLine(message); } public class TimestampDecorator(IAppLogger inner) : IAppLogger { public void Log(string message) => inner.Log($"[{DateTime.Now:yyyy-MM-dd HH:mm:ss.fff}] {message}"); } public class ThreadIdDecorator(IAppLogger inner) : IAppLogger { public void Log(string message) => inner.Log($"[Thread-{Environment.CurrentManagedThreadId}] {message}"); } public class JsonFormatDecorator(IAppLogger inner) : IAppLogger { public void Log(string message) => inner.Log(JsonSerializer.Serialize(new { timestamp = DateTime.Now, message })); } public class FileWriterDecorator(IAppLogger inner, string filePath) : IAppLogger { public void Log(string message) { File.AppendAllText(filePath, message + Environment.NewLine); inner.Log(message); } } // Stack any combination: IAppLogger logger = new ConsoleLogger(); logger = new TimestampDecorator(logger); logger = new ThreadIdDecorator(logger); logger.Log("User logged in"); // Output: [Thread-1] [2026-04-07 10:30:00.123] User logged in

    The builder provides a fluent API that hides the nesting complexity. The user says what features they want; the builder wires the nesting automatically.

    public class LoggerBuilder { private readonly List<Func<IAppLogger, IAppLogger>> _decorators = new(); private IAppLogger _core = new ConsoleLogger(); public LoggerBuilder UseConsole() { _core = new ConsoleLogger(); return this; } public LoggerBuilder UseFile(string path) { _decorators.Add(i => new FileWriterDecorator(i, path)); return this; } public LoggerBuilder AddTimestamp() { _decorators.Add(i => new TimestampDecorator(i)); return this; } public LoggerBuilder AddThreadId() { _decorators.Add(i => new ThreadIdDecorator(i)); return this; } public LoggerBuilder FormatAsJson() { _decorators.Add(i => new JsonFormatDecorator(i)); return this; } public IAppLogger Build() { IAppLogger logger = _core; foreach (var decorate in _decorators) logger = decorate(logger); return logger; } } var logger = new LoggerBuilder() .UseConsole() .AddTimestamp() .AddThreadId() .Build(); logger.Log("Application started"); // Output: [Thread-1] [2026-04-07 10:30:00.123] Application started
    Why Builder + Decorator?

    Decorator handles runtime composition. Builder handles configuration. Together: configurable, testable, extensible, and easy to read.

    Define an IService interface with the core methods. The existing God class implements it. Lowest-risk step: adding an interface, not changing logic.

    Only adding an interface. Existing code works unchanged.
