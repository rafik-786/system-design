Factory Method

Instead of creating objects yourself, ask a factory to make them for you — and the factory decides which exact type to give back.

Section 1

TL;DR

Quick note: If you only have ONE type to create, you don't need a factory — just register it in DI directly. Factory Method shines when the choice of what to create depends on something you only know at runtime — like user input, a config setting, or which environment you're in. This page teaches both the classic pattern and the modern .NET approach.

What: Normally when you need an object, you write new SomeClass() directly. But what if you don't know which class to create until the program is running? Maybe the user picked "Email" notification, or the config says "use PostgreSQL." Factory Method solves this: instead of deciding yourself, you ask a factory — and the factory figures out which specific class to create and hand back.

When: Use it whenever the type of object you need depends on something that can change — user choice, configuration, environment, or business rules.

In C# / .NET: The framework already has built-in factories like IHttpClientFactory (creates HTTP clients) and IDbContextFactory<T> (creates database contexts). For your own code, you can build a factory class or register a factory function in DI.

Quick Code:

FactoryMethod.cs 
public interface INotification { void Send(string message); }

public interface INotificationFactory
{
    INotification Create(NotificationType type);
}

public class NotificationFactory(IServiceProvider sp) : INotificationFactory
{
    public INotification Create(NotificationType type) => type switch
    {
        NotificationType.Email => sp.GetRequiredService<EmailNotification>(),
        NotificationType.Sms   => sp.GetRequiredService<SmsNotification>(),
        NotificationType.Push  => sp.GetRequiredService<PushNotification>(),
        _ => throw new ArgumentOutOfRangeException(nameof(type))
    };
}

Prerequisites

Before diving in:
Interfaces & Polymorphism — The factory returns objects through an interface, so your code works with any type the factory gives back. If "programming to an interface" is unfamiliar, review that first. Dependency Injection — Modern factories in .NET are wired through DI. ClassicFactory.cs 
// Product interface
public interface IDocumentFormatter
{
    byte[] Format(ReportData data);
    string ContentType { get; }
}

// Concrete products
public class PdfFormatter : IDocumentFormatter
{
    public byte[] Format(ReportData data) => PdfEngine.Render(data);
    public string ContentType => "application/pdf";
}

public class ExcelFormatter : IDocumentFormatter
{
    public byte[] Format(ReportData data) => ExcelEngine.Render(data);
    public string ContentType => "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet";
}

// Creator (abstract class with factory method)
public abstract class DocumentExporter
{
    // Factory Method — subclasses decide which formatter to create
    protected abstract IDocumentFormatter CreateFormatter();

    // Template method that uses the factory method
    public byte[] Export(ReportData data)
    {
        var formatter = CreateFormatter();  // ← polymorphic call
        var bytes = formatter.Format(data);
        return AddWatermark(bytes);
    }

    private byte[] AddWatermark(byte[] doc) => doc; // simplified
}

// Concrete creators
public class PdfExporter : DocumentExporter
{
    protected override IDocumentFormatter CreateFormatter() => new PdfFormatter();
}

public class ExcelExporter : DocumentExporter
{
    protected override IDocumentFormatter CreateFormatter() => new ExcelFormatter();
}
InterfaceFactory.cs 
// Factory interface — no inheritance needed
public interface IExporterFactory
{
    IDocumentFormatter Create(ExportFormat format);
    bool Supports(ExportFormat format);
}

// Each format gets its own factory
public class PdfExporterFactory : IExporterFactory
{
    public IDocumentFormatter Create(ExportFormat format) => new PdfFormatter();
    public bool Supports(ExportFormat format) => format == ExportFormat.Pdf;
}

public class ExcelExporterFactory : IExporterFactory
{
    public IDocumentFormatter Create(ExportFormat format) => new ExcelFormatter();
    public bool Supports(ExportFormat format) => format == ExportFormat.Excel;
}

// Composite factory — finds the right factory at runtime
public class CompositeExporterFactory(IEnumerable<IExporterFactory> factories)
    : IExporterFactory
{
    public IDocumentFormatter Create(ExportFormat format) =>
        factories.FirstOrDefault(f => f.Supports(format))
            ?.Create(format)
        ?? throw new NotSupportedException($"No factory for {format}");

    public bool Supports(ExportFormat format) =>
        factories.Any(f => f.Supports(format));
}

// Registration
builder.Services.AddTransient<IExporterFactory, PdfExporterFactory>();
builder.Services.AddTransient<IExporterFactory, ExcelExporterFactory>();
builder.Services.AddTransient<CompositeExporterFactory>();

Section 6

Junior vs Senior Implementation

Problem Statement

Build a payment processing system that supports Stripe, PayPal, and RazorPay. Each provider has its own SDK and configuration. New providers get added every quarter.

How a Junior Thinks

"I'll just add a switch statement in the controller. Simple! One file, everything in one place."

PaymentController.cs
// Junior: everything in the controller
[HttpPost("process")]
public async Task<IActionResult> ProcessPayment(PaymentRequest req)
{
    switch (req.Provider)
    {
        case "stripe":
            var stripeClient = new StripeClient("sk_live_abc123");  // ❌ hardcoded key
            var intent = await stripeClient.PaymentIntents.CreateAsync(
                new() { Amount = (long)(req.Amount * 100), Currency = req.Currency });
            return Ok(new { id = intent.Id, success = intent.Status == "succeeded" });

        case "paypal":
            var ppClient = new PayPalHttpClient(
                new SandboxEnvironment("client_id", "secret"));  // ❌ hardcoded
            var order = await ppClient.Execute(new OrdersCreateRequest()
                .RequestBody(BuildPayPalOrder(req)));
            return Ok(new { id = order.Result<Order>().Id, success = true });

        case "razorpay":
            var rzp = new RazorpayClient("key", "secret");  // ❌ hardcoded
            var payment = rzp.Payment.Create(new Dictionary<string, object>
            {
                { "amount", req.Amount * 100 }, { "currency", req.Currency }
            });
            return Ok(new { id = payment["id"], success = true });

        default:
            return BadRequest($"Unknown provider: {req.Provider}");
    }
}

Problems

Hardcoded new + API Keys

Junior writes new StripeClient("sk_live_abc123") directly in the controller. Keys are in source control, can't differ per environment (dev/staging/prod).

Giant Switch in Controller

switch(provider) with 5 cases in the action method. Controller owns creation logic + business logic + error handling. Adding PayPal means modifying the controller.

Untestable

Can't mock the payment providers — they're created inline with new. Unit testing requires real Stripe/PayPal SDKs and API keys. No interface = no mock.

How a Senior Thinks

"Each provider is a strategy behind an interface. A factory resolves the right one at runtime. DI wires everything. Adding a new provider = new class + one line of registration."

IPaymentProcessor.cs
public interface IPaymentProcessor
{
    PaymentProvider Provider { get; }
    Task<PaymentResult> ChargeAsync(decimal amount, string currency, CancellationToken ct);
    Task<PaymentResult> RefundAsync(string transactionId, CancellationToken ct);
}

public enum PaymentProvider { Stripe, PayPal, RazorPay }

public class StripeProcessor(IOptions<StripeSettings> settings) : IPaymentProcessor
{
    public PaymentProvider Provider => PaymentProvider.Stripe;

    public async Task<PaymentResult> ChargeAsync(decimal amount, string currency, CancellationToken ct)
    {
        var client = new Stripe.PaymentIntentService();
        var intent = await client.CreateAsync(
            new() { Amount = (long)(amount * 100), Currency = currency },
            cancellationToken: ct);
        return new PaymentResult(intent.Id, intent.Status == "succeeded");
    }

    public Task<PaymentResult> RefundAsync(string transactionId, CancellationToken ct) =>
        /* Stripe refund logic */ Task.FromResult(new PaymentResult(transactionId, true));
}
PaymentProcessorFactory.cs
public interface IPaymentProcessorFactory
{
    IPaymentProcessor Create(PaymentProvider provider);
}

public class PaymentProcessorFactory : IPaymentProcessorFactory
{
    private readonly Dictionary<PaymentProvider, IPaymentProcessor> _processors;

    public PaymentProcessorFactory(IEnumerable<IPaymentProcessor> processors)
    {
        _processors = processors.ToDictionary(p => p.Provider);
    }

    public IPaymentProcessor Create(PaymentProvider provider) =>
        _processors.TryGetValue(provider, out var processor)
            ? processor
            : throw new NotSupportedException(
                $"Payment provider '{provider}' is not registered.");
}

// Adding a new provider? Create the class + register it. ZERO changes here.
Program.cs
// Each processor reads its own config section via IOptions<T>
builder.Services.Configure<StripeSettings>(builder.Configuration.GetSection("Stripe"));
builder.Services.Configure<PayPalSettings>(builder.Configuration.GetSection("PayPal"));
builder.Services.Configure<RazorPaySettings>(builder.Configuration.GetSection("RazorPay"));

// Register all processors
builder.Services.AddTransient<IPaymentProcessor, StripeProcessor>();
builder.Services.AddTransient<IPaymentProcessor, PayPalProcessor>();
builder.Services.AddTransient<IPaymentProcessor, RazorPayProcessor>();

// Factory collects all implementations automatically
builder.Services.AddTransient<IPaymentProcessorFactory, PaymentProcessorFactory>();

// Usage — clean and testable
app.MapPost("/api/payments", async (
    PaymentRequest req,
    IPaymentProcessorFactory factory,
    CancellationToken ct) =>
{
    var processor = factory.Create(req.Provider);
    var result = await processor.ChargeAsync(req.Amount, req.Currency, ct);
    return result.Success ? Results.Ok(result) : Results.BadRequest(result);
});

Design Decisions

Dictionary-Based O(1) Lookup

Factory builds a Dictionary<PaymentProvider, IPaymentProcessor> from IEnumerable<T>. O(1) resolution instead of iterating with .First().

Open/Closed Principle

Adding SquareProcessor = new class + one DI line. Zero changes to factory, controller, or existing processors.

Config via IOptions<T>

Each processor reads its own config section. No hardcoded keys. Environment-specific settings via appsettings.{Environment}.json.

Section 7

Evolution of Factory Method in .NET

Factory Method has evolved from manual switch/case blocks to DI-native patterns. Understanding this history helps you recognize legacy code and choose the right modern approach.

.NET 1.0–2.0 (2002–2005): Manual Switch/Case

No generics, no DI containersFrameworks that automatically resolve and inject dependencies at runtime based on registered type mappings.. Factories were static methods with big switch blocks.

Factory_v1.cs
// .NET 1.x — static factory, no generics, no interfaces
public class ReportFactory
{
    public static object CreateReport(string type)
    {
        switch (type)
        {
            case "pdf":  return new PdfReport();
            case "excel": return new ExcelReport();
            default: throw new ArgumentException("Unknown type");
        }
    }
}

// Caller must cast — no type safety
PdfReport report = (PdfReport)ReportFactory.CreateReport("pdf");
Problems

Returns object — callers cast everywhere. Adding a new type means modifying the factory (violates OCPOpen/Closed Principle — classes should be open for extension but closed for modification). Static method — untestable, can't mock.

.NET 2.0–3.5 (2005–2008): Generics + Activator

Generics enabled typed factories. Activator.CreateInstance<T>() provided reflectionThe ability to inspect and manipulate types, methods, and properties at runtime — powerful but slower than compile-time approaches.-based creation.

Factory_v2.cs
// .NET 2.0+ — generic factory interface
public interface IFactory<T> where T : new()
{
    T Create();
}

public class ReportFactory<T> : IFactory<T> where T : IReport, new()
{
    public T Create() => new T();
    // Or reflection: Activator.CreateInstance<T>()
}
Improvement

Type-safe returns, no casting. But new() constraint limits dependency injection. Activator.CreateInstance is 50-100x slower than direct new.

.NET 4.0 (2010): MEF Plugin Factories

Managed Extensibility Framework (MEFManaged Extensibility Framework — a .NET library for building plugin-based applications using attribute-based discovery.) introduced [Export]/[Import] for plugin-style factories.

Factory_MEF.cs
// MEF — attribute-based discovery
[Export(typeof(IReportGenerator))]
[ExportMetadata("Format", "pdf")]
public class PdfReportGenerator : IReportGenerator { }

// Consumer — MEF resolves the right export
public class ReportService
{
    [ImportMany]
    public IEnumerable<Lazy<IReportGenerator, IFormatMetadata>> Generators { get; set; }

    public IReportGenerator GetGenerator(string format) =>
        Generators.First(g => g.Metadata.Format == format).Value;
}
Improvement

Auto-discovers implementations via attributes. ExportFactory<T> enables on-demand creation. But MEF was complex and mostly used in desktop apps (Visual Studio plugins).

ASP.NET Core (2016+): DI-Native Factories

The built-in DI container made factory patterns first-class citizens. Framework ships with real factory interfaces.

Factory_DI.cs
// Built-in factory pattern — IHttpClientFactory
builder.Services.AddHttpClient("github", c =>
{
    c.BaseAddress = new Uri("https://api.github.com");
    c.DefaultRequestHeaders.Add("User-Agent", "MyApp");
});

// Usage — factory creates configured, pooled HttpClient
public class GitHubService(IHttpClientFactory factory)
{
    public async Task<string> GetReposAsync() =>
        await factory.CreateClient("github").GetStringAsync("/repos");
}

.NET 8+ (2023+): Keyed Services + Source Generators

Keyed servicesA .NET 8+ DI feature allowing multiple implementations of the same interface, resolved by a string or enum key. eliminate many custom factory classes. Source generatorsA C# compiler feature that generates additional source code at compile time, enabling zero-reflection patterns. remove reflection overhead.

Factory_NET8.cs
// .NET 8 — keyed services replace custom factory classes
builder.Services.AddKeyedTransient<IExporter, PdfExporter>("pdf");
builder.Services.AddKeyedTransient<IExporter, ExcelExporter>("excel");
builder.Services.AddKeyedTransient<IExporter, CsvExporter>("csv");

// Controller — resolve by key, no custom factory needed
public class ExportController(
    [FromKeyedServices("pdf")] IExporter pdfExporter)
{
    [HttpGet("export/pdf")]
    public IActionResult ExportPdf() => File(pdfExporter.Export(data), "application/pdf");
}

// When you still need a factory (runtime key):
app.MapGet("/export/{format}", (string format, IServiceProvider sp) =>
{
    var exporter = sp.GetRequiredKeyedService<IExporter>(format);
    return Results.File(exporter.Export(data));
});
Modern Best Practice

Use keyed services for simple key→type mapping. Keep custom factories for complex creation logic (parameterized, async, cross-cutting concernsFunctionality needed across multiple parts of an app (logging, auth, validation) that doesn't belong to any single module.). Source generators can produce factory code at compile time — zero reflection, zero runtime cost.

.NET 9+ (2024+): HybridCache Factory + Auto-Keyed Discovery

HybridCache (preview in .NET 9) uses factory delegates internally — the GetOrCreateAsync method takes a Func<TState, CancellationToken, Task<T>> factory to produce values on cache miss. Your factory becomes the cache's creation strategy.

Factory_NET9.cs
// .NET 9 — HybridCache uses factory delegate for creation
builder.Services.AddHybridCache();

app.MapGet("/report/{format}/{id}", async (
    ReportFormat format, int id,
    HybridCache cache,
    IReportGeneratorFactory factory) =>
{
    // Factory delegate runs ONLY on cache miss
    var bytes = await cache.GetOrCreateAsync(
        $"report:{format}:{id}",
        async (ct) =>
        {
            await using var generator = factory.Create(format);
            return await generator.GenerateAsync(
                await GetReportData(id, ct), ct);
        },
        new HybridCacheEntryOptions
        {
            Expiration = TimeSpan.FromMinutes(15),
            LocalCacheExpiration = TimeSpan.FromMinutes(5)
        });

    return Results.File(bytes, "application/octet-stream");
});

// Auto-keyed discovery — register all IExporter implementations
// by scanning assemblies (community pattern emerging in .NET 9+)
builder.Services.Scan(scan => scan
    .FromAssemblyOf<PdfExporter>()
    .AddClasses(c => c.AssignableTo<IExporter>())
    .AsImplementedInterfaces()
    .WithTransientLifetime());
Where It's Heading

HybridCache blurs the line between factory and cache — your factory delegate is the creation strategy, the cache handles L1/L2 layering and stampede protection. Assembly scanningAutomatically discovering and registering types from compiled assemblies at startup based on naming conventions or interfaces. with keyed auto-registration is reducing boilerplate further. The factory pattern stays, but the ceremony keeps shrinking.

Factory Method is baked into the .NET Core framework. You use these factories daily — here's how they're organized:

IHttpClientFactory — Pooled HTTP Clients

The poster child of Factory Method in .NET Core. Creates HttpClient instances with proper handler poolingReusing HttpMessageHandler instances across HttpClient calls to prevent socket exhaustion and improve performance., DNS rotationPeriodically refreshing DNS lookups so connections switch to new server IPs — important for cloud deployments., and named configurations.

HttpClientFactory.cs
// Registration — configure named clients
builder.Services.AddHttpClient("stripe", c =>
{
    c.BaseAddress = new Uri("https://api.stripe.com/v1/");
    c.DefaultRequestHeaders.Authorization =
        new AuthenticationHeaderValue("Bearer", config["Stripe:ApiKey"]);
}).AddPolicyHandler(GetRetryPolicy());  // Polly retry

// Usage — factory handles handler pooling & DNS rotation
public class StripeService(IHttpClientFactory factory)
{
    public async Task<Charge> CreateChargeAsync(ChargeRequest req)
    {
        var client = factory.CreateClient("stripe");  // Factory Method!
        var response = await client.PostAsJsonAsync("charges", req);
        return await response.Content.ReadFromJsonAsync<Charge>();
    }
    // client is NOT disposed — the factory manages the handler lifetime
}

IServiceScopeFactory — Manual DI Scopes

When a Singleton needs to use Scoped services (like DbContext), it creates a manual scope.
The factory pattern prevents the captive dependencyA scoped or transient service accidentally captured by a singleton, causing it to live far longer than intended. bug.

ScopeFactory.cs
// Background service (Singleton) needs DbContext (Scoped)
public class OrderCleanupService(
    IServiceScopeFactory scopeFactory) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            using var scope = scopeFactory.CreateScope();  // Factory Method!
            var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            await db.Orders
                .Where(o => o.Status == "abandoned" && o.CreatedAt < DateTime.UtcNow.AddDays(-30))
                .ExecuteDeleteAsync(ct);

            await Task.Delay(TimeSpan.FromHours(1), ct);
        }
    }
}

Every time you use IOptions<T>, the framework's IOptionsFactory<T> creates and validates your settings object.
It runs IConfigureOptions<T>, IPostConfigureOptions<T>, and IValidateOptions<T> in order.

    .ValidateOnStart();          // fail at startup, not first request

public class SmtpSettings
{

// 1. new SmtpSettings()

Bug 1: Service Locator Disguised as Factory

The Incident

2023, e-commerce platform, .NET 7. A mid-sized e-commerce company had a PaymentFactory that looked perfectly clean on the surface. It implemented IPaymentProcessorFactory, had a proper Create() method, and was neatly registered in the DI container. Code reviews passed without a single comment.

But under the hood, there was a problem hiding in plain sight. Instead of injecting specific dependencies, the factory injected the entire DI container — IServiceProvider — and called GetRequiredService<T>() to resolve payment processors. This is what's called a Service LocatorAn anti-pattern where a class resolves its own dependencies from a global container instead of receiving them via constructor injection. — an anti-pattern where a class reaches into the container to grab whatever it needs, instead of declaring its dependencies up front.

For months, everything worked fine. Then a developer registered StripeProcessor as Scoped — it needed per-request tenant context to know which merchant account to charge. The factory, however, was registered as Singleton. This created a captive dependencyWhen a longer-lived service (e.g., Singleton) captures a shorter-lived service (e.g., Scoped), keeping it alive beyond its intended lifetime.: the Scoped processor was resolved once from the root scope and then held forever inside the Singleton factory.

The result was devastating. Request 1 from Tenant A created a StripeProcessor configured with Tenant A's merchant account. Request 2 from Tenant B got the same processor instance — still configured for Tenant A. One customer was charged using another customer's payment method. The team discovered the bug when a customer called support saying "I don't have a Stripe account, why was I charged through Stripe?"

Time to Diagnose

3 hours. The bug only appeared when two tenants used different payment providers within the same app restart. The factory's IServiceProvider injection made it look like a proper factory, but it was actually a Service Locator wearing a factory costume.

PaymentFactory.cs
// ❌ Service Locator anti-pattern disguised as a factory
// Registered as: services.AddSingleton<IPaymentProcessorFactory, PaymentFactory>();
public class PaymentFactory(IServiceProvider sp) : IPaymentProcessorFactory
{
    // ❌ Injecting the entire container — hides real dependencies
    // ❌ Factory is Singleton, but processors might be Scoped!
    public IPaymentProcessor Create(string provider) => provider switch
    {
        "stripe"   => sp.GetRequiredService<StripeProcessor>(),   // ← Scoped!
        "paypal"   => sp.GetRequiredService<PayPalProcessor>(),   // ← Scoped!
        "razorpay" => sp.GetRequiredService<RazorPayProcessor>(), // ← Scoped!
        _ => throw new ArgumentException($"Unknown: {provider}")
    };
    // The IServiceProvider was captured at Singleton construction time.
    // It resolves from the ROOT scope — Scoped services live FOREVER.
    // Request 1: StripeProcessor created with Tenant A's context
    // Request 2: Same StripeProcessor returned — still has Tenant A's context!
    // Result: Tenant B gets charged with Tenant A's payment method
}

Walking through the buggy code: The factory takes IServiceProvider in its constructor. Since the factory is a Singleton, it gets the root service provider — not the per-request scoped one. When it calls GetRequiredService<StripeProcessor>(), it resolves from the root scope. Scoped services resolved from the root scope are never disposed and never refreshed. They live as long as the application lives. So the first tenant's processor gets cached and every subsequent tenant gets that same stale instance.

PaymentFactory.cs
// ✅ Inject IServiceScopeFactory — create a fresh scope per product
public class PaymentFactory(IServiceScopeFactory sf) : IPaymentProcessorFactory
{
    public IPaymentProcessor Create(string provider)
    {
        // Each call creates a new scope — Scoped services get fresh instances
        var scope = sf.CreateScope();
        return scope.ServiceProvider
            .GetRequiredKeyedService<IPaymentProcessor>(provider);
        // Caller is responsible for disposing the scope when done
    }
}

// Program.cs — enable scope validation to catch this at startup
builder.Host.UseDefaultServiceProvider(opts =>
{
    opts.ValidateScopes = true;  // throws if Singleton captures Scoped
    opts.ValidateOnBuild = true;
});

Why the fix works: Instead of injecting the raw container, the factory now injects IServiceScopeFactory. Every time Create() is called, it creates a brand new scope. The processor resolved inside that scope gets a fresh set of Scoped dependencies — including the correct tenant context for the current request. The scope validation flags in Program.cs add a safety net: if anyone accidentally resolves a Scoped service from a Singleton, the app will throw at startup rather than silently serving stale data.

Lesson Learned

If your factory injects IServiceProvider, it's a Service Locator in disguise. Inject specific factory interfaces or IServiceScopeFactory instead. Enable ValidateScopes = trueA DI container option that detects when scoped services are resolved from the root scope — catches captive dependency bugs. in development to catch this at startup.

How to Spot This in Your Code

Search your codebase for any factory class that takes IServiceProvider as a constructor parameter. If you find one, check its DI lifetime — if it's Singleton and it resolves Scoped services, you have this bug. Run grep -rn "IServiceProvider" --include="*Factory*.cs" as a quick sanity check. Better yet, enable ValidateScopes in your development environment and let the framework catch it for you.

Bug 2: Factory Returning Scoped from Singleton

The Incident

2024, SaaS platform, .NET 8. A team building a financial reporting service registered their ReportFactory as a Singleton for performance — they wanted one factory instance shared across all requests, which sounds reasonable.

The factory created ReportGenerator objects that needed a database connection. In .NET, database connections are typically managed through DbContext, which is registered as Scoped — meaning each HTTP request gets its own fresh connection. But because the factory was a Singleton, the DbContext it received during construction was the only one it would ever see.

The first user to generate a report got perfect results. The second user hit an ObjectDisposedException — the DbContext from the first request had already been cleaned up when that request ended, but the Singleton factory was still trying to use it. Even worse, in some cases the DbContext survived (no explicit dispose), and its internal cache — called the change trackerAn EF Core mechanism that keeps track of all entities loaded in a DbContext, including their original and current values. — still held data from the previous user. User B could see User A's financial reports.

The team spent hours staring at database logs. The database was returning correct data. The bug was in the application layer — the factory was feeding every report generator a dead or stale database connection.

Time to Diagnose

2 hours. The ObjectDisposedException was the easy clue. The "seeing other users' data" part took longer — it only happened when DbContext's change tracker had cached entities from the previous request.

ReportFactory.cs
// ❌ Singleton factory directly injects Scoped dependency
// Registered as: services.AddSingleton<IReportFactory, ReportFactory>();
public class ReportFactory(AppDbContext db) : IReportFactory  // ← Singleton!
{
    // ❌ AppDbContext is Scoped — captured once at Singleton construction
    // The SAME DbContext instance is reused across ALL requests
    public IReportGenerator Create(ReportFormat format) => format switch
    {
        ReportFormat.Pdf   => new PdfReportGenerator(db),   // stale DbContext!
        ReportFormat.Excel => new ExcelReportGenerator(db), // stale DbContext!
        _ => throw new ArgumentOutOfRangeException(nameof(format))
    };
    // Request 1: db is alive, PdfReportGenerator works perfectly
    // Request 1 ends: db.Dispose() called by DI scope cleanup
    // Request 2: db is DEAD — ObjectDisposedException
    // Request 3: If db survived (no dispose), it returns cached entities
    //            from Request 1's change tracker — User B sees User A's data
}

Walking through the buggy code: The factory's constructor takes AppDbContext directly. Since the factory is a Singleton, the DI container creates it once at app startup and passes it the DbContext from the root scope. That DbContext is now trapped inside the factory forever. When Request 1 ends, the framework tries to dispose all Scoped services — including that DbContext. But the Singleton factory still holds a reference to it. Request 2 comes in, the factory passes the disposed DbContext to a new generator, and the generator crashes.

ReportFactory.cs
// ✅ Factory creates a fresh scope per product — each gets its own DbContext
public class ReportFactory(IServiceScopeFactory scopeFactory) : IReportFactory
{
    public IReportGenerator Create(ReportFormat format)
    {
        var scope = scopeFactory.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();

        return format switch
        {
            ReportFormat.Pdf   => new PdfReportGenerator(db, scope),
            ReportFormat.Excel => new ExcelReportGenerator(db, scope),
            _ => throw new ArgumentOutOfRangeException(nameof(format))
        };
        // Product implements IDisposable and disposes the scope + DbContext
    }
}

// Alternative: Use IDbContextFactory<T> (built for exactly this)
public class ReportFactory(IDbContextFactory<AppDbContext> dbFactory) : IReportFactory
{
    public IReportGenerator Create(ReportFormat format)
    {
        var db = dbFactory.CreateDbContext(); // fresh DbContext every time
        return new PdfReportGenerator(db);   // caller disposes db
    }
}

Why the fix works: The first approach injects IServiceScopeFactory instead of DbContext directly. Every Create() call builds a new scope with its own fresh DbContext. The product takes ownership of the scope and disposes it when done. The second approach is even cleaner — IDbContextFactory<T> was built by Microsoft specifically for this scenario. It creates independent DbContext instances that aren't tied to any scope, so they work perfectly inside Singleton factories.

How to Spot This in Your Code

Look for any Singleton-registered class that takes DbContext (or any Scoped service) in its constructor. The compiler won't warn you — the DI container silently satisfies the request from the root scope. Enable ValidateScopes = true during development, and it will throw on startup instead of failing silently in production.

Bug 3: Factory Caching Stale Instances

The Incident

2024, multi-tenant SaaS, .NET 8. A developer on the reporting team had a clever idea: cache report generator instances by tenant ID in a ConcurrentDictionary. Creating a report generator was expensive — it loaded templates, initialized PDF rendering engines, and established database connections. Caching seemed like an obvious win.

And it worked beautifully — until a tenant updated their configuration. Specifically, the tenant "acme" migrated from one database server to another. The admin updated the connection string in the config file. But the factory didn't care. It already had a cached generator for "acme" that was built with the old connection string, and GetOrAdd means "get the existing one if it's there — never create a new one."

For 6 hours, the factory happily returned the cached generator pointing at the old database. The old database still existed (it was in the process of being decommissioned), but it had last month's data. Customers started calling support: "My reports look outdated. The numbers don't match what I see in the dashboard."

The team initially blamed the database migration. They checked the database server, verified the new connection string, and everything looked correct. It wasn't until someone examined the factory's in-memory cache and compared its stored connection strings with the current config that they found the real culprit: the factory's cache had no invalidation mechanism whatsoever.

Time to Diagnose

4 hours. The team initially blamed the database. It wasn't until they compared the factory's cached connection string with the current config that they found the mismatch.

ReportGeneratorFactory.cs
// ❌ Factory caches products forever — ignores config changes
public class ReportGeneratorFactory(IOptions<TenantSettings> options)
    : IReportGeneratorFactory
{
    // ❌ ConcurrentDictionary acts as a permanent cache — no eviction
    private readonly ConcurrentDictionary<string, IReportGenerator> _cache = new();

    public IReportGenerator Create(string tenantId)
    {
        // ❌ GetOrAdd = create once, cache forever
        return _cache.GetOrAdd(tenantId, id =>
        {
            var settings = options.Value.Tenants[id];
            var connString = settings.ConnectionString;  // read ONCE
            return new ReportGenerator(connString);
        });

        // 10:00 AM: Tenant "acme" created with connString "Server=db1"
        // 11:00 AM: Admin changes acme's DB to "Server=db2" in config
        // 11:01 AM: Factory returns CACHED generator with "Server=db1"
        // 5:00 PM: Customer calls — "my reports show last month's data"
        // IOptions<T> reads config once at startup. Even if config changes,
        // the cached generator still uses the old connection string
    }
}

Walking through the buggy code: The factory uses IOptions<T>, which reads configuration once at startup and never updates. Even if someone edits appsettings.json, the options object keeps the old values. Combine that with GetOrAdd — which only creates a new entry if the key doesn't exist — and you get a cache that will never refresh. The generator for "acme" was built at 10 AM with the old connection string, and every call after that returns the same stale instance.

ReportGeneratorFactory.cs
// ✅ Use IOptionsMonitor to detect config changes and invalidate cache
public class ReportGeneratorFactory : IReportGeneratorFactory, IDisposable
{
    private readonly ConcurrentDictionary<string, IReportGenerator> _cache = new();
    private readonly IOptionsMonitor<TenantSettings> _monitor;
    private readonly IDisposable? _changeListener;

    public ReportGeneratorFactory(IOptionsMonitor<TenantSettings> monitor)
    {
        _monitor = monitor;
        // ✅ Listen for config changes — evict stale entries
        _changeListener = monitor.OnChange((settings, tenantId) =>
        {
            if (tenantId != null) _cache.TryRemove(tenantId, out _);
        });
    }

    public IReportGenerator Create(string tenantId)
    {
        return _cache.GetOrAdd(tenantId, id =>
        {
            // ✅ IOptionsMonitor.CurrentValue always returns latest config
            var settings = _monitor.CurrentValue.Tenants[id];
            return new ReportGenerator(settings.ConnectionString);
        });
    }

    public void Dispose() => _changeListener?.Dispose();
}

Why the fix works: Two changes make this safe. First, IOptionsMonitor<T> replaces IOptions<T> — it actively watches the config file and always returns the latest values through .CurrentValue. Second, the OnChange callback fires whenever the config changes and evicts the affected tenant from the cache. The next time Create() is called for that tenant, GetOrAdd sees no entry and builds a fresh generator with the updated connection string.

Lesson Learned

Don't cache factory products that depend on mutable configuration. If you must cache, use IOptionsMonitor<T> change callbacks to invalidate. Or better — don't cache at all and let DI handle lifetimes.

How to Spot This in Your Code

Search for ConcurrentDictionary or Dictionary fields in your factory classes. If they're used for caching and the values depend on configuration or external state, ask: "What happens when the config changes?" If the answer is "nothing — the cache never invalidates," you have this bug. Switch from IOptions<T> to IOptionsMonitor<T> and wire up OnChange callbacks.

Bug 4: Factory-Created Disposables Never Disposed

The Incident

2024, document service, .NET 8. A document export service handled about 200,000 exports per day. The IExporterFactory.Create() method returned PdfExporter instances, each of which allocated a MemoryStream and a PdfDocument internally to generate the PDF file.

Here's the ownership problem: the factory created the exporter, the controller used it, and then... nothing. Nobody called Dispose(). The developers assumed the garbage collector would clean everything up eventually. And technically it can — but "eventually" is the key word. MemoryStream allocates a byte array, and large byte arrays (over 85KB) end up on the Large Object HeapA special memory area in .NET for objects larger than 85KB. The LOH is collected less frequently and can cause memory fragmentation., which the garbage collector collects much less frequently.

Memory grew at 500MB per hour. The Kubernetes pod running the service hit its memory limit every 4 hours and got OOMKilled (killed for using too much memory). Kubernetes restarted it automatically, and the cycle repeated. The team's first instinct? Increase the memory limit. They doubled it from 2GB to 4GB. The pod lasted 8 hours instead of 4. They doubled it again. Eventually someone asked the right question: "Why is memory growing at all?"

A dotMemoryA .NET memory profiling tool by JetBrains that shows heap snapshots, object allocations, and retention paths. snapshot revealed 47,000 undisposed MemoryStream objects — each holding onto its byte array, none being cleaned up in a timely manner. The fix took 10 minutes: add IDisposable to the interface and using to the caller.

Time to Diagnose

2 days. The team kept increasing pod memory limits. A dotMemory snapshot finally showed 47,000 undisposed MemoryStream objects.

ExportController.cs
// ❌ IExporter does NOT extend IDisposable — no compile-time reminder
public interface IExporter
{
    byte[] Export(ReportData data);
    // Missing: IDisposable — callers don't know they need to dispose
}

public class PdfExporter : IExporter
{
    // ❌ MemoryStream allocated on construction — never freed
    private readonly MemoryStream _stream = new();
    private readonly PdfDocument _doc = new();

    public byte[] Export(ReportData data)
    {
        _doc.AddPage().DrawString(data.Title);
        _doc.Save(_stream);
        return _stream.ToArray();
        // _stream and _doc are NEVER disposed
        // GC will eventually collect them, but MemoryStream pins
        // a byte[] on the Large Object Heap — GC pressure builds up
    }
}

// ❌ Controller never disposes the exporter
[HttpGet("export/{format}")]
public IActionResult Export(string format, int reportId)
{
    var exporter = _factory.Create(format); // created
    var data = _reportService.GetData(reportId);
    var bytes = exporter.Export(data);       // used
    return File(bytes, "application/pdf");
    // exporter goes out of scope — Dispose() never called
    // 200K times/day x ~2.5KB MemoryStream = 500MB/hour leak
}

Walking through the buggy code: The IExporter interface doesn't extend IDisposable, so there's no signal to callers that they need to clean up. The controller calls Create(), uses the exporter, returns a response, and lets the exporter fall out of scope. The garbage collector will eventually reclaim the memory, but "eventually" means "when it gets around to it" — and for large objects on the LOH, that could be minutes or hours later. With 200K calls per day, the undisposed objects pile up faster than the GC can clean them.

ExportController.cs
// ✅ IExporter extends IDisposable — compiler reminds callers
public interface IExporter : IDisposable
{
    byte[] Export(ReportData data);
}

public class PdfExporter : IExporter
{
    private readonly MemoryStream _stream = new();
    private readonly PdfDocument _doc = new();

    public byte[] Export(ReportData data)
    {
        _doc.AddPage().DrawString(data.Title);
        _doc.Save(_stream);
        return _stream.ToArray();
    }

    public void Dispose()
    {
        _doc.Dispose();
        _stream.Dispose();  // frees the byte[] from LOH
    }
}

// ✅ Controller disposes via 'using' — ownership is explicit

{

    var data = _reportService.GetData(reportId);
    var bytes = exporter.Export(data);
    return File(bytes, "application/pdf");
    // exporter.Dispose() called here — MemoryStream freed
}

Why the fix works: Making IExporter extend IDisposable does two things. First, it's documentation — anyone looking at the interface immediately knows "this thing holds resources I need to clean up." Second, the using keyword tells the compiler to generate a try/finally block that calls Dispose() automatically when the variable goes out of scope, even if an exception is thrown. The MemoryStream and PdfDocument get freed immediately after use, not "whenever the GC feels like it."

Lesson Learned

Document ownership: if a factory creates it, someone must dispose it. Make IDisposable part of the product interface so the compiler reminds callers. Rule: "Factory creates, caller disposes."

How to Spot This in Your Code

Enable Roslyn analyzer rule CA2000 — it flags variables that implement IDisposable but aren't disposed before going out of scope. Also search for factory Create() calls that aren't wrapped in using. If the product holds streams, connections, or handles, it almost certainly needs disposal.

Bug 5: Circular Factory Dependency

The Incident

2023, microservice, .NET 7. A team was doing a big refactor: they converted three services into factory-based designs in a single pull request. The OrderFactory needed an IValidator to validate orders during creation. The ValidatorFactory needed an IOrderParser to parse validation rules. And the OrderParserFactory needed an IOrderFactory to create template orders for testing the parser.

Each factory innocently depended on the product of another factory. But together, they formed a circle: A depends on B, B depends on C, C depends on A. When the DI container tried to create OrderFactory, it needed IValidator, which needed ValidatorFactory, which needed IOrderParser, which needed OrderParserFactory, which needed IOrderFactory — which is where we started. Infinite loop.

The DI container detected this at startup and threw a clear error: InvalidOperationException: A circular dependency was detected for the service 'IOrderFactory'. The app wouldn't even start. The error message told them what was wrong, but not how to fix it — untangling three intertwined factories took 3 hours of redesign.

This is a common pitfall when teams "factory-ify" everything in one go. Each factory individually makes sense, but the dependency graph as a whole forms a cycle. The root cause is usually that the interfaces are too broad — each one does too many things, so it pulls in more dependencies than it really needs.

Time to Diagnose

20 minutes — the error message was clear. But 3 hours to untangle the circular design.

CircularFactories.cs
// ❌ Three factories that form a circular dependency chain
// OrderFactory → IValidator → ValidatorFactory
//   → IOrderParser → OrderParserFactory → IOrderFactory → BOOM

public class OrderFactory(IValidator validator) : IOrderFactory
{
    public Order Create(OrderRequest req)
    {
        var order = new Order(req);
        validator.Validate(order);  // needs IValidator
        return order;
    }
}

public class ValidatorFactory(IOrderParser parser) : IValidatorFactory
{
    public IValidator Create(string ruleSet)
    {
        var rules = parser.ParseRules(ruleSet);  // needs IOrderParser
        return new Validator(rules);
    }
}

public class OrderParserFactory(IOrderFactory orderFactory) : IOrderParserFactory
{
    public IOrderParser Create(string format)
    {
        // Uses IOrderFactory to create template orders for validation
        var template = orderFactory.Create(OrderRequest.Template);  // needs IOrderFactory

// Program.cs
builder.Services.AddSingleton<IOrderFactory, OrderFactory>();
builder.Services.AddSingleton<IValidatorFactory, ValidatorFactory>();
builder.Services.AddSingleton<IOrderParserFactory, OrderParserFactory>();
// BOOM at startup:

// for the service 'IOrderFactory'.

Walking through the buggy code: Follow the chain. To build OrderFactory, the DI container needs IValidator. To build Validator, it needs a ValidatorFactory, which needs IOrderParser. To build a parser, it needs OrderParserFactory, which needs IOrderFactory. And we're back where we started. The DI container can't figure out which one to create first, because every one of them needs another one to already exist.

// ✅ Option 1: Break cycle with Lazy<T> (quick fix)
public class OrderFactory(Lazy<IValidator> validator) : IOrderFactory
{
    public Order Create(OrderRequest req)
    {
        var order = new Order(req);
        validator.Value.Validate(order);  // resolved on first use, not construction
        return order;
    }
}

// Register Lazy<T> wrapper in DI
builder.Services.AddSingleton<IOrderFactory, OrderFactory>();
builder.Services.AddSingleton(sp =>
    new Lazy<IValidator>(() => sp.GetRequiredService<IValidator>()));

// ✅ Option 2 (better): Extract focused interfaces to break the cycle
// Before: OrderFactory → IValidator (heavy — has parsing + validation)
// After:  OrderFactory → IValidationRule (just validation, no parsing)
//         ValidatorFactory → IParsingRule (just parsing, no order creation)
// Each depends on a focused interface — no circular reference

public interface IValidationRule  // extracted from IValidator
{
    bool Validate(Order order);
}

public class OrderFactory(IEnumerable<IValidationRule> rules) : IOrderFactory
{
    public Order Create(OrderRequest req)
    {
        var order = new Order(req);
        foreach (var rule in rules) rule.Validate(order);
        return order;
    }
}

Why the fix works: Option 1 uses Lazy<T> to defer resolution — the validator isn't actually resolved until someone calls .Value, which breaks the construction-time cycle. It's a quick patch. Option 2 is the real fix: instead of each factory depending on a big, do-everything interface, you extract small focused interfaces. OrderFactory only needs validation rules, not the full validator. ValidatorFactory only needs parsing rules, not the full order parser. By splitting the responsibilities, you break the cycle naturally without any tricks.

Lesson Learned

Circular factory dependencies usually mean your abstractions are too coarse. Extract smaller, focused interfaces. Lazy<T> is a band-aid — fixing the design is the real solution.

How to Spot This in Your Code

The DI container will tell you immediately with a clear error message at startup. The harder part is preventing it. Before refactoring multiple services into factories, sketch the dependency graph on a whiteboard. If you can draw a circle through the dependencies, you need to extract smaller interfaces. A good rule of thumb: if a factory depends on more than 3-4 services, it's probably doing too much.

Bug 6: Thread-Unsafe Factory Dictionary Initialization

The Incident

2024, high-traffic API, .NET 8. A notification service had a NotificationFactory that lazily built its internal lookup dictionary. The developer's reasoning seemed sound: "Why build the dictionary at startup if nobody has called Create() yet? Let's build it on first use." They used a simple null-check pattern: if the dictionary is null, create it and add entries.

This worked perfectly in development because there's only one developer sending requests. In production, the API handled hundreds of requests per second. Under a traffic spike, 40 concurrent requests hit the factory simultaneously — all before the dictionary had been populated. Multiple threads saw _factories is null at the same time, so multiple threads entered the initialization block at the same time.

Here's the insidious part: Dictionary<K,V> is not thread-safe. When two threads call .Add() simultaneously, the dictionary's internal bucket array can get corrupted. The dictionary doesn't throw an error — it just silently loses entries. After the corruption, "sms" was in the dictionary but unreachable. TryGetValue("sms") returned false. The factory threw KeyNotFoundException for a channel that was definitely registered.

The bug was maddeningly sporadic. It only happened when the first batch of requests arrived simultaneously. Once the dictionary was populated (even with corruption), subsequent requests either worked or failed consistently. The team couldn't reproduce it locally and spent 4 hours adding logging, load testing, and finally inspecting the dictionary's internal state with a debugger attached to a staging pod.

Time to Diagnose

4 hours — sporadic KeyNotFoundException only under load. Dictionary corruption doesn't throw on write — it silently loses entries.

UnsafeFactory.cs
// ❌ Lazy-init dictionary in a Singleton factory — not thread-safe
public class NotificationFactory : INotificationFactory
{
    private readonly IServiceProvider _sp;
    private Dictionary<string, Func<INotification>>? _factories; // ❌ nullable, lazy-init

    public NotificationFactory(IServiceProvider sp) => _sp = sp;

    public INotification Create(string channel)
    {
        // ❌ Race condition: 40 threads enter here simultaneously
        // before _factories is assigned
        if (_factories is null)
        {
            _factories = new Dictionary<string, Func<INotification>>();
            // ❌ Dictionary.Add is NOT thread-safe
            // Two threads adding "email" key simultaneously → bucket corruption
            _factories.Add("email", () => _sp.GetRequiredService<EmailNotification>());
            _factories.Add("sms", () => _sp.GetRequiredService<SmsNotification>());
            _factories.Add("push", () => _sp.GetRequiredService<PushNotification>());
        }

        // ❌ After corruption: _factories.ContainsKey("sms") returns false
        // even though "sms" was added — internal hash buckets are mangled
        return _factories.TryGetValue(channel, out var factory)
            ? factory()
            : throw new KeyNotFoundException($"No handler for '{channel}'");
        // ^ Sporadic KeyNotFoundException for valid channels under load
    }
}

Walking through the buggy code: The factory defers dictionary creation to the first Create() call. Under load, many threads reach the if (_factories is null) check before any thread finishes populating the dictionary. They all enter the block. Thread 1 creates a new Dictionary and starts adding entries. Thread 2 also creates a new Dictionary (overwriting Thread 1's reference) and also starts adding. Meanwhile Thread 1 is still adding to the old reference. The result is a corrupted internal state where hash buckets point to the wrong entries — keys are "there" but unreachable through the normal lookup path.

SafeFactory.cs
// ✅ Build dictionary eagerly in constructor — runs once, before any Create() call
public class NotificationFactory : INotificationFactory
{
    // ✅ readonly + initialized in constructor = immutable after construction
    // No race condition possible — DI guarantees single-threaded construction
    private readonly FrozenDictionary<string, Func<INotification>> _factories;

    public NotificationFactory(IServiceProvider sp)
    {
        // ✅ Built once in constructor — no lazy-init, no locking needed
        _factories = new Dictionary<string, Func<INotification>>
        {
            ["email"] = () => sp.GetRequiredService<EmailNotification>(),
            ["sms"] = () => sp.GetRequiredService<SmsNotification>(),
            ["push"] = () => sp.GetRequiredService<PushNotification>(),
        }.ToFrozenDictionary(); // ✅ FrozenDictionary (.NET 8+) — optimized for read-heavy
    }

    public INotification Create(string channel) =>
        _factories.TryGetValue(channel, out var factory)
            ? factory()
            : throw new ArgumentOutOfRangeException(nameof(channel),
                $"No handler for '{channel}'. Available: {string.Join(", ", _factories.Keys)}");
}

// ✅ Alternative: use IEnumerable injection for auto-discovery
public class NotificationFactory(IEnumerable<INotification> notifications) : INotificationFactory
{
    private readonly FrozenDictionary<string, INotification> _lookup =
        notifications.ToFrozenDictionary(n => n.Channel); // ✅ built in constructor

    public INotification Create(string channel) => _lookup[channel];
}

Why the fix works: Moving dictionary creation to the constructor eliminates the race condition entirely. The DI container constructs Singleton services on a single thread — no concurrent access is possible during construction. Marking the field readonly means it can't be reassigned after construction. Using FrozenDictionary goes a step further: it creates an immutable, read-optimized structure that's mathematically incapable of corruption because it can't be modified after creation. All concurrent reads are safe by design.

Lesson Learned

Never lazy-init mutable state in a Singleton. Build your factory's lookup in the constructor — DI guarantees it runs once. Use FrozenDictionaryA .NET 8+ read-only dictionary optimized for fast lookups after initial construction. (.NET 8+) for read-optimized, immutableAn object whose state cannot be changed after creation — inherently thread-safe lookups. If you must lazy-init, use Lazy<T> or ConcurrentDictionary.

How to Spot This in Your Code

Search for nullable Dictionary fields in Singleton classes: Dictionary<...>? with a null check in a method body. If you see if (_dict is null) { _dict = new ...; _dict.Add(...); }, that's a race condition waiting for traffic. Move the initialization to the constructor, or use Lazy<T> which guarantees single-threaded initialization.

Section 13

Pitfalls & Anti-Patterns

1. Factory for a single type

Mistake: Creating INotificationFactory when there's only one INotification implementation.

Why This Happens: A developer reads about Factory Method and thinks "every object creation should go through a factory." They create INotificationFactory even though the only notification type is EmailNotification. It feels like "good architecture" — until you realize the factory always returns the same type. There's no decision point, no runtime choice, no polymorphism. It's a wrapper that does nothing useful.

The second reason is "future-proofing." Developers think "we might add SMS later." That's a valid thought, but you're paying the complexity cost now for a feature that might never arrive. When you actually need a second type, creating a factory takes 5 minutes. There's no need to pre-build the abstraction.

Unnecessary.cs — ❌ Over-engineering
// ❌ Factory for a single type — pointless abstraction
public interface INotificationFactory
{
    INotification Create(); // always returns EmailNotification
}

public class NotificationFactory : INotificationFactory
{
    public INotification Create() => new EmailNotification(); // only one type!
}

// ❌ Caller has to go through a factory that adds zero value
var notification = _factory.Create(); // why not just inject INotification?
Simple.cs — ✅ Direct injection
// ✅ Just register it directly — no factory needed
services.AddTransient<INotification, EmailNotification>();

// ✅ Caller injects the interface directly
public class OrderService(INotification notification)
{
    public void PlaceOrder(Order order) => notification.Send(order.Confirmation);
}

// When you need a second type later, THEN add a factory. Not before.

The connection is simple: a factory exists to make a choice. If there's no choice to make, skip the factory and use direct DI injection.

2. God Factory

Mistake: One factory that creates 15 different unrelated types — notifications, reports, validators, parsers.

Why This Happens: It starts innocently. Someone creates a ServiceFactory to build notifications. Then another developer needs a factory for reports and thinks "we already have a factory — I'll just add a method." A third developer adds validator creation. Before long, one class is responsible for creating everything in the system.

This violates the Single Responsibility PrincipleA class should have only one reason to change. If a factory creates 15 types, it has 15 reasons to change.. Every time any team member adds a new type, they modify this one class. It becomes a merge conflict magnet, a testing nightmare, and a 500-line file that nobody wants to touch.

GodFactory.cs — ❌ Dumping ground
// ❌ One factory for everything — SRP violation
public class ServiceFactory : IServiceFactory
{
    public INotification CreateNotification(string type) => ...;
    public IReport CreateReport(ReportFormat fmt) => ...;
    public IValidator CreateValidator(string ruleSet) => ...;
    public IParser CreateParser(string format) => ...;
    public IExporter CreateExporter(ExportFormat fmt) => ...;
    // 15 more methods... every new type = change this class
}
FocusedFactories.cs — ✅ Each factory has one job
// ✅ Split into focused factories — each has one responsibility
public class NotificationFactory : INotificationFactory { ... }
public class ReportFactory : IReportFactory { ... }
public class ValidatorFactory : IValidatorFactory { ... }

// Adding a new notification type? Only NotificationFactory changes.
// Adding a new report format? Only ReportFactory changes.
// No merge conflicts, no giant class, easy to test individually.

Think of it like a restaurant kitchen: you want a pizza station, a pasta station, and a grill station — not one cook doing everything. Each factory should be focused on one family of related types.

3. Returning object or dynamic

Mistake: Factory returns object and callers cast: var report = (PdfReport)factory.Create("pdf");

Why This Happens: A developer doesn't want to create an interface for the products. It feels like "extra work" to define IReport, so they return object instead. Or they use dynamic to avoid the cast. Either way, the compiler can't help you — all type checking happens at runtime, meaning bugs that could be caught at compile time only show up in production.

The irony is that the whole point of Factory Method is to decouple the caller from concrete types. But if the caller casts the result back to a concrete type, you've defeated the purpose entirely. The caller now knows exactly what type it's getting, and you've just added a factory layer that provides no benefit.

Untyped.cs — ❌ No compile-time safety
// ❌ Returns object — callers must cast blindly
public object Create(string type) => type switch
{
    "pdf"   => new PdfReport(),
    "excel" => new ExcelReport(),
    _ => throw new ArgumentException()
};

// ❌ Caller must know the concrete type and cast
var report = (PdfReport)factory.Create("pdf"); // runtime crash if wrong
Typed.cs — ✅ Type-safe interface
// ✅ Returns typed interface — compiler catches misuse
public IReport Create(ReportFormat format) => format switch
{
    ReportFormat.Pdf   => new PdfReport(),
    ReportFormat.Excel => new ExcelReport(),
    _ => throw new ArgumentOutOfRangeException()
};

// ✅ Caller works with the interface — no casting needed
IReport report = factory.Create(ReportFormat.Pdf);
byte[] bytes = report.Generate(data); // type-safe!

If you find yourself casting the factory's return value, it's a signal: either the factory's return type is too broad (object), or the interface is missing methods the caller needs.

4. Static factory methods in business logic

Mistake: PaymentProcessor.Create("stripe") as a static method.

Why This Happens: Static methods feel convenient. No need to inject anything, no interface, no DI registration. Just call ClassName.Method() from anywhere. It's the path of least resistance, and for simple utilities like int.Parse() or DateTime.Now, it works fine.

But for business logic factories — things that create payment processors, report generators, or notification services — static methods are a trap. You can't mock a static method in unit tests. You can't inject dependencies into a static context. Every class that calls PaymentProcessor.Create() is now tightly coupledWhen components depend directly on concrete implementations rather than abstractions, making changes ripple through the codebase to that concrete class. Swapping out the implementation requires changing every call site.

Static.cs — ❌ Untestable
// ❌ Static factory — can't mock, can't inject dependencies
public static class PaymentProcessor
{
    public static IPaymentProcessor Create(string provider) => provider switch
    {
        "stripe" => new StripeProcessor(/* hardcoded API key? */),
        _ => throw new ArgumentException()
    };
}

// ❌ Every test that touches OrderService hits the real factory
public class OrderService
{
    public void Charge(Order order)
    {
        var processor = PaymentProcessor.Create(order.Provider); // can't mock this!
        processor.Charge(order.Amount);
    }
}
Injectable.cs — ✅ Instance-based, testable
// ✅ Instance-based factory — mockable, injectable
public class PaymentProcessorFactory(IServiceProvider sp) : IPaymentProcessorFactory
{
    public IPaymentProcessor Create(string provider) => ...;
}

// ✅ OrderService receives factory via DI — easily testable
public class OrderService(IPaymentProcessorFactory factory)
{
    public void Charge(Order order)
    {
        var processor = factory.Create(order.Provider); // can mock this!
        processor.Charge(order.Amount);
    }
}

The rule: static methods are fine for pure utilities with no dependencies (parsing, formatting). For business logic that needs configuration, database access, or API keys — always use instance-based factories with DI.

5. Leaking concrete types

Mistake: Factory returns INotification but callers downcastCasting a base type reference to a derived type — a code smell that often indicates a leaky abstraction to EmailNotification to access .AttachmentPath.

Why This Happens: The interface was designed before all the requirements were known. Later, someone needs a feature that's specific to one concrete type (like adding an attachment to emails). Instead of updating the interface, they cast the returned object to the concrete type. It's quick, it works, and it quietly undermines the entire factory pattern.

The problem snowballs. Once one developer downcasts, others copy the pattern. Soon you have 20 places in the codebase doing (EmailNotification)factory.Create("email"). Now you can't change the concrete type without breaking all those cast sites. The factory's abstraction has leaked — callers know exactly what's behind the interface.

Leaky.cs — ❌ Downcasting defeats the factory
// ❌ Caller downcasts — knows the concrete type
INotification notification = _factory.Create("email");
var email = (EmailNotification)notification; // ❌ cast = leaked abstraction
email.AttachmentPath = "/reports/q4.pdf";    // this property isn't on INotification
Complete.cs — ✅ Interface covers all needs
// ✅ Option 1: Add the capability to the interface
public interface INotification
{
    void Send(string message);
    void AddAttachment(string path); // throws NotSupportedException for SMS
}

// ✅ Option 2: Create a focused interface for the capability
public interface IAttachableNotification : INotification
{
    string AttachmentPath { get; set; }
}

// Caller checks capability without downcasting to concrete type
if (notification is IAttachableNotification attachable)
    attachable.AttachmentPath = "/reports/q4.pdf";

If callers need features beyond what the interface provides, the interface is incomplete. Enrich the interface or create a focused sub-interface — don't reach past the abstraction.

6. Factory with side effects in creation

Mistake: Factory constructor calls database, sends HTTP requests, or pre-loads data during Create().

Why This Happens: A developer wants the factory product to be "ready to use" the moment it's created. So they put database calls, HTTP requests, or file I/O inside the factory's Create() method. The intent is good — nobody wants to call product.Initialize() separately. But the cost is paid at creation time, which blocks the thread and slows down DI resolution.

This is especially dangerous when the factory is resolved during startup. If Create() makes a database call and the database is slow (or down), the entire application startup blocks. In Kubernetes, the health check fails, the pod gets killed, Kubernetes restarts it, the database is still slow, and you're in a restart loop.

HeavyCreation.cs — ❌ Slow factory
// ❌ Factory does heavy work during Create()
public IReportGenerator Create(ReportFormat format)
{
    var templates = _db.Templates.ToList();        // ❌ database call!
    var fonts = _httpClient.GetAsync("/fonts");     // ❌ HTTP call!
    var config = File.ReadAllText("rules.json");    // ❌ file I/O!
    return new PdfGenerator(templates, fonts, config); // 2 seconds to create!
}
LazyInit.cs — ✅ Cheap creation, lazy init
// ✅ Factory creates cheaply — product initializes lazily
public IReportGenerator Create(ReportFormat format)
{
    return new PdfGenerator(_db, _httpClient); // instant — no I/O
}

// Product loads resources on first use, not at creation
public class PdfGenerator(AppDbContext db, HttpClient http) : IReportGenerator
{
    private List<Template>? _templates;

    public async Task<byte[]> GenerateAsync(ReportData data, CancellationToken ct)
    {
        _templates ??= await db.Templates.ToListAsync(ct); // lazy load
        // ... generate report using _templates
    }
}

The principle: factory creation should be cheap and fast — just wire up dependencies. Heavy work (I/O, network, database) belongs in the product's own methods, not in the factory.

7. Forgetting to dispose factory products

Mistake: Factory creates it, caller uses it, nobody disposes it. 200K exports/day with undisposed MemoryStreams.

Why This Happens: When you resolve a service from the DI container normally, the container tracks it and disposes it at the end of the scope. Developers get used to this — "DI handles disposal." But factory-created objects are different. The factory builds them manually with new, so the DI container doesn't know they exist. Nobody tracks them, nobody disposes them.

The confusion is understandable: DI-resolved objects are managed by the container, but factory-created objects are unmanaged. It's the caller's responsibility to dispose them, but nothing in the code makes that obvious unless the product interface extends IDisposable.

Leaked.cs — ❌ No disposal
// ❌ Factory creates, caller uses, nobody disposes
var exporter = _factory.Create(format);
var bytes = exporter.Export(data);
return File(bytes, "application/pdf");
// exporter with MemoryStream goes out of scope — leak!
Disposed.cs — ✅ Using statement
// ✅ Product extends IDisposable — compiler reminds you
public interface IExporter : IDisposable { ... }

// ✅ Caller wraps in 'using' — auto-dispose at end of scope
using var exporter = _factory.Create(format);
var bytes = exporter.Export(data);
return File(bytes, "application/pdf");
// Dispose() called automatically — MemoryStream freed!

The golden rule: "Factory creates, caller disposes." Make IDisposable part of the product interface, and enable RoslynThe .NET compiler platform providing C# and VB.NET compilers as services, enabling code analysis and source generators. analyzer CA2000 to flag missed disposals automatically.

8. Non-exhaustive switch/dictionary

Mistake: Adding a new enum value without updating the factory. The switch falls through to default which returns null.

Why This Happens: A developer adds ExportFormat.Markdown to the enum, creates a MarkdownExporter class, registers it in DI — and forgets to add the case to the factory's switch statement. The switch falls through to the default case, which returns null (or worse, silently does nothing). The bug doesn't show up until someone actually requests a Markdown export in production.

C# switch expressions can catch this at compile time, but only if you enable the right analyzer rule. Without it, adding a new enum value is a silent change that can break the factory.

Incomplete.cs — ❌ Silent null return
// ❌ default returns null — new enum values silently get nothing
public IExporter Create(ExportFormat format) => format switch
{
    ExportFormat.Pdf   => new PdfExporter(),
    ExportFormat.Excel => new ExcelExporter(),
    // Someone adds ExportFormat.Markdown — not handled here!
    _ => null  // ❌ silent null — caller gets NullReferenceException later
};
Exhaustive.cs — ✅ Fail fast + compile-time check
// ✅ default throws — fail fast with clear message
public IExporter Create(ExportFormat format) => format switch
{
    ExportFormat.Pdf   => new PdfExporter(),
    ExportFormat.Excel => new ExcelExporter(),
    _ => throw new ArgumentOutOfRangeException(nameof(format),
        $"No exporter for {format}. Did you forget to add a case?")
};

// ✅ .editorconfig — make the compiler catch unhandled enum values
// dotnet_diagnostic.CS8509.severity = error

Always use throw in the default case, and enable CS8509 at error severity in your .editorconfig. The compiler will refuse to build if a new enum value isn't handled.

9. Factory that swallows exceptions

Mistake: Factory catches creation exceptions and returns null or a no-op fallback instead of failing fast.

Why This Happens: A developer doesn't want the factory to crash the application. They wrap the creation logic in a try/catch and return a "safe" fallback — a no-op implementation that does nothing. The intent is resilience: "if Stripe is down, at least the app doesn't crash." But the side effect is far worse than a crash.

Imagine a payment factory that catches all exceptions and returns a NoOpProcessor. When Stripe's API is temporarily unreachable, every order goes through without being charged. The customer gets the product for free. Nobody notices until end-of-day reconciliation shows $0 revenue. A 500 error would have been infinitely better — the customer would retry, or customer support would intervene. Silent data loss is always worse than a loud failure.

Swallowed.cs — ❌ Silent failures
// ❌ DON'T — swallowing exceptions
public IPaymentProcessor Create(string provider)
{
    try { return ResolveProcessor(provider); }
    catch { return new NoOpProcessor(); }  // orders go through for FREE
}
FailFast.cs — ✅ Loud failures
// ✅ Fail fast — let the exception propagate
public IPaymentProcessor Create(string provider)
{
    return ResolveProcessor(provider);
    // If this throws, the caller gets a 500 error.
    // The customer retries. Support gets alerted.
    // $0 revenue at end of day? Never happens.
}

// If you need resilience, handle it at a higher level:
// retry policies (Polly), circuit breakers, fallback queues
// — NOT a silent no-op in the factory

Factories should fail fast and fail loud. If creation fails, let the error propagate to the caller. Resilience belongs in dedicated middleware (retry policies, circuit breakers) — not in a catch-all inside the factory.

10. Factory coupled to configuration shape

Mistake: Factory reads raw IConfiguration and knows the exact JSON structure — key names, nesting, types.

Why This Happens: Injecting IConfiguration is easy — it's always available in the DI container, and you can read any key with a string path like config["Exporters:Pdf:StorageConnection"]. No need to create a settings class, no need to set up options binding. It works on the first try.

But now your factory is tightly coupled to the exact shape of your JSON config file. Rename a key? The factory silently gets null. Move a section to a different nesting level? Same thing. The factory has no way to validate the config at startup — it just reads strings and hopes for the best. This is the exact problem the Options patternA .NET pattern for accessing configuration using strongly-typed classes injected via IOptions<T>. was designed to solve.

MagicStrings.cs — ❌ Coupled to JSON shape
// ❌ Factory knows the exact JSON shape
public class ExporterFactory(IConfiguration config) : IExporterFactory
{
    public IExporter Create(string format)
    {
        var connStr = config["Exporters:Pdf:StorageConnection"];  // ❌ magic string
        var maxSize = config.GetValue<int>("Exporters:Pdf:MaxFileSizeMb"); // ❌ coupled
        return new PdfExporter(connStr, maxSize);
        // If someone renames the key in appsettings.json, connStr is silently null
    }
}
TypedOptions.cs — ✅ Strongly-typed settings
// ✅ Settings class — owns the config shape
public class ExporterSettings
{
    public string StorageConnection { get; init; } = "";
    public int MaxFileSizeMb { get; init; } = 100;
}

// ✅ Register with validation

    .BindConfiguration("Exporters:Pdf")
    .ValidateDataAnnotations()
    .ValidateOnStart(); // ← fail at startup if config is wrong

{

        new PdfExporter(opts.Value.StorageConnection, opts.Value.MaxFileSizeMb);

Let the Options layer own the config shape. The factory gets a strongly-typed POCOPlain Old CLR Object — a simple class with properties and no framework dependencies. with validated properties. If the config is wrong, the app fails at startup with a clear error — not silently in production with null values.

Performance Considerations

Most factory overhead is negligible — but in hot pathsCode that executes extremely frequently (e.g., per-request in a web server) — performance optimizations matter most here. (100K+ calls/second), the creation mechanism matters. Here's what to know:

Creation MethodApprox. TimeUse Case
new ConcreteType() ~1 ns When you know the type at compile time
Func<T> delegate ~3 ns Lightweight factory — no vtable lookupThe runtime mechanism for virtual method dispatch — the CLR follows a pointer to find the correct implementation.
IFactory<T> interface ~5 ns Standard factory pattern — slight virtual dispatch cost
Activator.CreateInstance ~200 ns Reflection — avoid in hot paths

Q1: What is the Factory Method pattern?
Easy
Think First What problem does Factory Method solve that plain new doesn't?

Think of a restaurant where you order "a drink" from the menu. You don't walk into the kitchen and grab a specific bottle yourself — you tell the waiter what you want, and the kitchen decides how to make it. That's Factory Method in a nutshell: you ask for a product through an interface, and a separate component decides which concrete type to create and how to configure it.

More formally, Factory Method is a GoF creational pattern that:

  1. Defines an interface for creating objects
  2. Lets subclasses or implementations decide which concrete class to instantiate
  3. Decouples the creator from the product — the caller doesn't know which concrete type it gets

A real .NET example: IHttpClientFactory.CreateClient("stripe") — the caller gets a configured HttpClient but doesn't know about handler pooling, DNS rotation, or retry policies. The factory hides all that complexity. If Microsoft changes how HttpClient pooling works internally, your code doesn't change at all.

Great Answer Bonus "The key insight is that it's not just about wrapping new — it's about deferring the type decision to a separate component. In modern .NET, the DI container often plays the role of the ConcreteCreator."
Q2: Why not just use new directly?
Easy
Think First What happens when you need to support a new payment provider?

Imagine you're building a house, and instead of hiring a plumber, you install every pipe yourself. If you need to change the plumbing later, you have to do the work again. That's what new StripeProcessor(apiKey, webhookUrl) does in code — you've hardwired the specific type and its configuration directly into the class that uses it.

Using new directly creates tight couplingWhen a class depends directly on a concrete implementation rather than an abstraction, making it hard to change or test.:

  • Can't swap implementations without changing the caller
  • Can't test — mocking requires an interface, new creates a concrete
  • Violates OCP — adding a new type means modifying existing code
  • Hides dependencies — constructor args like API keys end up hardcoded

That said, new is perfectly fine for simple value objects: new Point(x, y), new List<int>(). The decision rule is: if the choice of type matters at runtime (e.g., Stripe vs PayPal based on user preference), use a factory. If there's only ever one type and no decisions to make, new is simpler and better.

Q3: What's the difference between Factory Method and Simple Factory?
Easy
Think First Is Simple Factory even a real design pattern?

This is a classic "they sound the same but work differently" interview trap. Think of it like ordering food: a Simple Factory is like a single cook who takes all orders and uses a big if/else to decide what to make. If you want to add sushi to the menu, the cook has to learn a new recipe — you modify the existing cook (class).

Simple Factory: A single class with a static method that uses switch/if to create objects. Not a GoF pattern — it's a coding idiom. Adding a new type means modifying the factory class (violates OCP).

Factory Method: Uses an interface or virtual method. Each product type can get its own factory implementation. It's like having specialized chefs — a sushi chef, a pizza chef — each knowing their domain. Adding sushi means hiring a sushi chef (new class). The pizza chef doesn't change at all. Zero changes to existing code (OCP satisfied).

In practice, many .NET factories use a dictionary-based approach that sits somewhere in between. The factory itself doesn't change, but new products register themselves via DI. This gives you OCP benefits without needing a separate factory class per product type.

Great Answer Bonus "Simple Factory is a stepping stone — it centralizes creation logic but doesn't support extension. Factory Method takes it further with polymorphism. In practice, I'd use a dictionary-based factory registered in DI, which gives OCP benefits without needing separate factory classes per product."

Q5: When should you NOT use Factory Method?
Easy

Knowing when NOT to use a pattern is just as important as knowing when to use it. Factory Method is powerful, but it adds a layer of indirection. If that layer doesn't earn its keep, it's just complexity for complexity's sake. Here's when to skip the factory:

  • Only one implementation — just register it directly in DI
  • No runtime decision — the type is known at compile time
  • Simple value objects — new Point(x, y) doesn't need a factory
  • Func<T> is sufficient — don't create a class just to wrap a lambda
  • Over-abstracting CRUD — new Order() doesn't need an OrderFactory

Think of it this way: a factory exists to make a choice at runtime. If there's no choice to make (only one type, or the type is known at compile time), the factory has no job. DI is already a factory in disguise — it creates instances, manages lifetimes, and injects dependencies. Don't put a factory on top of a factory unless the top one adds real decision logic.

Great Answer Bonus "The decision rule is simple: if no runtime value determines the type, you don't need a factory. DI is already a factory — don't add another layer on top of it without a reason."
Q6: How does IHttpClientFactory implement Factory Method?
Medium
Think First Why can't you just new HttpClient()? What goes wrong?

This is one of the best real-world examples of Factory Method because most .NET developers have used it, and the "why" is immediately tangible. Let's start with the problem it solves.

If you just new HttpClient() every time you make an HTTP call, each instance opens its own TCP connection. Under load, you burn through all available sockets — that's called socket exhaustionRunning out of available TCP sockets because HttpClient instances weren't reused — sockets stay in TIME_WAIT state for ~4 minutes after closing.. On the flip side, if you reuse a single HttpClient forever, it caches DNS results and doesn't pick up DNS changes (like when a server IP changes during a deployment).

IHttpClientFactory solves both problems. It maps perfectly to the Factory Method roles:

  1. Creator: IHttpClientFactory (the interface)
  2. ConcreteCreator: DefaultHttpClientFactory (internal framework class)
  3. Product: HttpClient
  4. Factory Method: CreateClient(string name)

Internally, the factory manages an HttpMessageHandler pool with a 2-minute handler lifetime. This prevents both socket exhaustion (too many connections) and DNS staleness (cached DNS entries). The caller just gets a ready-to-use HttpClient — all the pooling is hidden behind the factory. If Microsoft changes how pooling works in .NET 10, your code doesn't change at all.

Q9: Factory Method vs Service Locator — what's the difference?
Medium
Think First Why is injecting IServiceProvider considered an anti-pattern?

This is a common interview question because the two look deceptively similar — both "create" objects from some container. But they have fundamentally different philosophies about how a class should declare its needs.

Service Locator: Injects the DI container (IServiceProvider) and calls GetService<T>() directly. Dependencies are hidden — you can't tell what a class needs by looking at its constructor. It's like giving someone a master key to every room in the building. They can open any door, but you have no idea which ones they actually use.

Factory Method: Injects a focused factory interface (IExporterFactory). Dependencies are explicit — you can see from the constructor that this class creates exporters, nothing else. It's like giving someone a key to the supply room only. Clear, bounded, predictable.

Key difference: with Service Locator, the class can resolve anything from the container — its real dependencies are invisible. With Factory Method, the class can only create what the factory interface allows. The constructor signature is honest: "I need an exporter factory" is much clearer than "I need access to the entire DI container."

Testing impact: To mock a Service Locator, you have to set up a fake IServiceProvider and pre-register every service the class might request (which you can't know without reading the implementation). To mock a factory, you set up one method: Create(). That's it.

Q11: How does IDbContextFactory<T> solve the Blazor Server problem?
Medium
Think First What's different about Blazor Server's lifetime compared to traditional HTTP requests?

This question tests whether you understand DI scoping beyond the basic "one request = one scope" model. In traditional ASP.NET Core, each HTTP request creates one DI scope, which creates one DbContext. The request ends, the scope is disposed, the DbContext is cleaned up. Clean and simple.

Blazor Server changes the game. A circuit (SignalR connection) represents a user's browser session. // db disposed immediately — no stale tracking, no memory leak

This is Factory Method at its finest: the component doesn't manage DbContext lifetimes. It asks the factory for a fresh one, uses it, disposes it. The factory handles all the configuration and connection pooling behind the scenes.

Q14: How does Factory Method enable the Open/Closed Principle? Walk through adding a new type.
Hard

Q15: Design an async factory — Task<IProduct> CreateAsync(). When and why?
Hard

Most factory methods are synchronous because object creation is cheap — just allocating memory and wiring up dependencies. But sometimes creation itself involves I/O: loading configuration from a database, downloading a license file, establishing a connection, or warming up a cache. In those cases, a synchronous factory would block the calling thread, which is unacceptable in async web applications.

The solution is an async factory: Task<IProduct> CreateAsync(CancellationToken ct). The key design decisions are: always accept a CancellationToken (the caller might cancel during a long creation), and always use ConfigureAwait(false) in library code to avoid deadlocks.

AsyncFactory.cs
public interface IReportGeneratorFactory
{
    Task<IReportGenerator> CreateAsync(
        ReportFormat format, CancellationToken ct = default);
}

public class ReportGeneratorFactory(
    IDbContextFactory<AppDbContext> dbFactory) : IReportGeneratorFactory
{
    public async Task<IReportGenerator> CreateAsync(
        ReportFormat format, CancellationToken ct)
    {
        await using var db = await dbFactory.CreateDbContextAsync(ct);
        var template = await db.Templates
            .FirstAsync(t => t.Format == format, ct);

        return format switch
        {
            ReportFormat.Pdf => new PdfGenerator(template),
            ReportFormat.Excel => new ExcelGenerator(template),
            _ => throw new ArgumentOutOfRangeException(nameof(format))
        };
    }
}

Important: Always accept CancellationToken — async factories might be called during HTTP requests that can be cancelled.

// Retry decorator — wraps any IExporter
public class RetryExporter(IExporter inner, int maxRetries = 3) : IExporter
{
    public byte[] Export(ReportData data)
    {
        for (int i = 0; i < maxRetries; i++)

}

{

Q20: How would you build a plugin architecture using Factory Method + AssemblyLoadContext?

public class PluginExporterFactory : IExporterFactory
{
    private readonly Dictionary<string, Type> _exporterTypes = new();

    public void LoadPlugins(string pluginDir)
    {
        foreach (var dll in Directory.GetFiles(pluginDir, "*.dll"))
        {
            var ctx = new AssemblyLoadContext(dll, isCollectible: true);
            var assembly = ctx.LoadFromAssemblyPath(Path.GetFullPath(dll));

            foreach (var type in assembly.GetTypes()
                .Where(t => typeof(IExporter).IsAssignableFrom(t) && !t.IsAbstract))

                if (attr != null) _exporterTypes[attr.FormatName] = type;
            }

            ? (IExporter)Activator.CreateInstance(type)!
            : throw new ArgumentException($"No plugin for format: {format}");

Q22: Open generic registration — how does typeof(IRepository<>) → typeof(Repository<>) work?
Hard

This is one of the most elegant factory patterns in .NET — the DI container itself becomes a generic factory. Instead of registering IRepository<Order>, IRepository<Product>, and IRepository<Customer> separately, you register the open generic once and the container figures out the rest.

Here's the step-by-step process when you request IRepository<Order>:

  1. Container sees request for IRepository<Order>
  2. Matches against open generic registration IRepository<>
  3. Substitutes type argument: Repository<> → Repository<Order>
  4. Creates instance of Repository<Order> via reflection (compiled expression for speed)

When it fails: generic constraints mismatch (e.g., where T : class but you request IRepository<int>), missing nested generic arguments, or the implementation has constructor parameters the container can't resolve. These errors appear at resolution time, not at registration time — use ValidateOnBuild = true to catch them at startup.

Q23: How does IOptionsFactory<T> work internally?

Q24: How does MediatR resolve the right handler? Is IMediator.Send() a Factory Method?
Hard

Yes, conceptually — and this is a great example of how Factory Method thinking appears in libraries you use every day, even when it's not called "factory." Think about what mediator.Send() does: you give it a request object, and it figures out which handler to create and invoke. That's fundamentally a factory operation — "here's what I need, give me the right object to handle it."

When you call mediator.Send(new CreateOrderCommand()):

  1. MediatR looks at the type of the request: CreateOrderCommand
  2. It resolves IRequestHandler<CreateOrderCommand, OrderResult> from the DI container
  3. The DI container acts as the factory — it knows the mapping from request type to handler type
  4. The handler is created (or retrieved from the current scope) and its Handle() method is invoked

The clever part is what serves as the "key." In a traditional factory, you pass a string or enum: factory.Create("pdf"). In MediatR, the type parameter itself is the key. The C# type system ensures at compile time that a handler exists for each request type. If you create CreateOrderCommand but forget to implement its handler, the DI container will throw at runtime — and with ValidateOnBuild = true, it'll throw at startup instead.

This is a type-safe factory where the generic type argument replaces the traditional factory key. No strings to misspell, no enums to forget updating. The compiler does the work.

Q26: How do you handle factory product versioning when multiple API versions coexist?
Hard
Think First How do you route API v1 requests to one factory product and v2 to another? What if v1 and v2 share 90% of the logic?

Use a composite key — the factory resolves by both type and version:

  1. Keyed services with composite key: AddKeyedTransient<IExporter>("pdf-v1", ...) and AddKeyedTransient<IExporter>("pdf-v2", ...)
  2. Version-aware factory: Create(ExportFormat format, ApiVersion version) — factory holds a Dictionary<(ExportFormat, int), Func<IExporter>>
  3. Shared logic via inheritance or composition: PdfExporterV2 wraps PdfExporterV1 and overrides only the changed behavior (Decorator pattern)
  4. Fallback strategy: If no v2-specific product exists, fall back to v1 — the factory handles this transparently
Great Answer Bonus "I'd avoid version-specific factory classes. Instead, the factory accepts a version parameter and falls back gracefully. This keeps the number of factory classes linear, not products × versions."
Q27: How would you implement a factory that creates products with different lifetimes (transient, scoped, singleton)?
Hard
Think First Can a Singleton factory create Scoped products? Who owns the scope? What about disposal?

This is a senior-level question that combines Factory Method with DI lifetime management. The core challenge: a Singleton factory lives forever, but some products need shorter lifetimes. How do you reconcile that?

This is the captive dependency problem applied to factories. The factory must respect product lifetimes:

  1. Transient products: Factory creates a new instance every call. Caller owns disposal
  2. Scoped products: Factory must create a scope via IServiceScopeFactory, resolve the product within that scope, and return both the product and a dispose handle
  3. Singleton products: Factory returns the same instance every time — effectively a cache. Use Lazy<T> or ConcurrentDictionary for thread-safe init

The key pattern: return a wrapper that implements IAsyncDisposable and disposes the scope when the product is disposed:

ScopeAwareFactory.cs
public IExporter Create(ExportFormat format)
{

    var exporter = scope.ServiceProvider
        .GetRequiredKeyedService<IExporter>(format);
    return new ScopedExporterWrapper(exporter, scope);
    // wrapper.DisposeAsync() disposes both exporter AND scope
}
Great Answer Bonus "The caller shouldn't know whether the product is scoped or transient — the factory abstracts that. IDbContextFactory<T> does exactly this: it creates a scope internally so DbContext works outside the request scope."
Q28: How does FrozenDictionary (.NET 8+) improve factory lookup performance, and when would you use it?
Hard
Think First How is FrozenDictionary different from Dictionary or ConcurrentDictionary? What does "frozen" buy you?

FrozenDictionary<TKey, TValue> (System.Collections.Frozen) is an immutable, read-optimized dictionary:

  1. Construction: Expensive — analyzes key distribution, picks optimal hash strategy (length-based for strings, ordinal comparison, etc.)
  2. Lookup: 2-5x faster than Dictionary<K,V> for string keys — no hash collision chains, branch-free lookup for small dictionaries
  3. Thread safety: Immutable after construction — zero locking needed, no ConcurrentDictionary overhead

Perfect for factories because: factory lookup tables are built once (constructor) and read millions of times (every Create() call). The one-time construction cost is amortized over the application's lifetime.

Great Answer Bonus "I'd build the factory's Dictionary in the constructor, then call .ToFrozenDictionary(). For a factory with 5-10 product types, the frozen dictionary might use a perfect hash — zero collisions, essentially an array index lookup."
Q29: How would you design a factory that supports hot-reload — adding new product types without restarting the application?
Hard
Think First How do you add new types to a running application? What about assembly loading? Config-driven vs code-driven?

This is an advanced architecture question that tests your understanding of dynamic extensibility. There are two very different levels of "hot reload" — one is straightforward, the other is deeply complex. A great answer distinguishes between them.

Two approaches depending on the degree of dynamism:

1. Config-driven hot-reload (simpler): Types are already compiled, config determines which are active.

  1. Register all possible products at startup
  2. Factory reads IOptionsMonitor<FactoryConfig> to decide which products are enabled
  3. IOptionsMonitor.OnChange() callback rebuilds the internal lookup dictionary
  4. Use Interlocked.Exchange to swap the dictionary atomically — no locking on reads

2. Plugin-based hot-reload (advanced): New DLLs loaded at runtime via AssemblyLoadContext.

  1. File watcher monitors a /plugins directory for new DLLs
  2. Load assembly into a collectible AssemblyLoadContext
  3. Scan for types implementing IExporter, register in a secondary dictionary
  4. To unload: remove from dictionary, unload the AssemblyLoadContext, GC collects the types
Great Answer Bonus "Config-driven covers 90% of cases. Plugin-based is for true extensibility — think VS Code extensions. The key challenge with plugin unloading is making sure no references to the loaded types survive, otherwise the AssemblyLoadContext can't be collected."

PluginExporterFactory.cs 
public class PluginExporterFactory : IExporterFactory, IDisposable
{
    private readonly ConcurrentDictionary<string, Type> _exporters = new();
    private readonly FileSystemWatcher _watcher;
    private AssemblyLoadContext? _context;

    public PluginExporterFactory(string pluginDir)
    {
        LoadPlugins(pluginDir);

        _watcher = new FileSystemWatcher(pluginDir, "*.dll");
        _watcher.Created += (_, e) => ReloadPlugins(pluginDir);
        _watcher.EnableRaisingEvents = true;
    }

    private void LoadPlugins(string dir)
    {
        _context?.Unload();
        _context = new AssemblyLoadContext("Plugins", isCollectible: true);

        foreach (var dll in Directory.GetFiles(dir, "*.dll"))
        {
            var asm = _context.LoadFromAssemblyPath(Path.GetFullPath(dll));
            foreach (var type in asm.GetTypes()
                .Where(t => typeof(IExporter).IsAssignableFrom(t)
                          && !t.IsAbstract))
            {
                var instance = (IExporter)Activator.CreateInstance(type)!;
                _exporters[instance.Format.ToString()] = type;
            }
        }
    }

    private void ReloadPlugins(string dir)
    {
        _exporters.Clear();
        LoadPlugins(dir);
    }

    public IExporter Create(string format) =>
        _exporters.TryGetValue(format, out var type)
            ? (IExporter)Activator.CreateInstance(type)!
            : throw new ArgumentException($"No plugin for: {format}");

    public void Dispose() { _watcher.Dispose(); _context?.Unload(); }
}

Deep Dive: Open Generic Registration & Factory Internals

This section goes beyond "create a factory" and explains how the .NET DI container itself acts as a factory — including open generics, covariance, and Func<T> as lightweight factory.

Open Generic Registration Internals

When you register services.AddTransient(typeof(IRepository<>), typeof(Repository<>)), the DI container becomes a generic factory:

OpenGenerics.cs
// Registration — open generic (no type argument specified)
builder.Services.AddTransient(typeof(IRepository<>), typeof(Repository<>));

// What happens when you resolve IRepository<Order>:
// 1. Container receives request: IRepository<Order>
// 2. Matches against open generic: IRepository<> ✅
// 3. Extracts type argument: T = Order
// 4. Constructs closed generic: Repository<Order>
// 5. Resolves constructor deps of Repository<Order>
// 6. Creates instance via compiled expression (fast)

public interface IRepository<T> where T : class
{
    Task<T?> GetByIdAsync(int id);
    Task<IReadOnlyList<T>> GetAllAsync();
    Task AddAsync(T entity);
}

public class Repository<T>(AppDbContext db) : IRepository<T> where T : class
{
    public async Task<T?> GetByIdAsync(int id) => await db.Set<T>().FindAsync(id);
    public async Task<IReadOnlyList<T>> GetAllAsync() => await db.Set<T>().ToListAsync();
    public async Task AddAsync(T entity) { await db.Set<T>().AddAsync(entity); await db.SaveChangesAsync(); }
}

// Now inject IRepository<Order>, IRepository<Product>, IRepository<Customer>
// — all resolved automatically, no individual registration needed!
When It Fails

Constraint mismatch: where T : class but you request IRepository<int> → runtime error.
Missing nested generics: IRepository<Wrapper<T>> doesn't match IRepository<>.
Constructor not resolvable: Repository<T> has a dependency the container can't fulfill for that specific T.

Func<T> as Lightweight Factory

Register a Func<T> in DI to create a factory without a separate class.
The container doesn't natively support Func<T> — you register it explicitly:

FuncFactory.cs
// Register Func<T> as factory delegate
builder.Services.AddTransient<EmailNotification>();
builder.Services.AddTransient<SmsNotification>();
builder.Services.AddTransient<PushNotification>();

builder.Services.AddSingleton<Func<NotificationType, INotification>>(sp =>
    type => type switch
    {
        NotificationType.Email => sp.GetRequiredService<EmailNotification>(),
        NotificationType.Sms   => sp.GetRequiredService<SmsNotification>(),
        NotificationType.Push  => sp.GetRequiredService<PushNotification>(),
        _ => throw new ArgumentOutOfRangeException(nameof(type))
    });

// Inject the delegate — no IFactory interface needed
public class NotificationService(Func<NotificationType, INotification> factory)
{
    public async Task SendAsync(NotificationType type, string message)
    {
        var notification = factory(type);
        await notification.SendAsync(message);
    }
}

When Func<T> is enough: simple key→type mapping, no complex creation logic, no need to mock in tests.
When you need IFactory<T>: complex creation, parameterized construction, cross-cutting concerns, or testability requirements.

// Health check — validates every registered generator can produce output
public class ReportGeneratorHealthCheck(
    IReportGeneratorFactory factory,
    ILogger<ReportGeneratorHealthCheck> logger) : IHealthCheck
{
    // Dummy data for smoke-test generation
    private static readonly ReportData _smokeData = new()
    {
        Title = "HealthCheck", Rows = [new Row("test", 1)]
    };

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken ct = default)
    {
        var failures = new Dictionary<string, object>();

        foreach (var format in factory.SupportedFormats)
        {
            try
            {
                await using var generator = factory.Create(format);
                var bytes = await generator.GenerateAsync(_smokeData, ct);

                if (bytes.Length == 0)
                    failures[format.ToString()] = "Empty output";
            }
            catch (Exception ex)
            {
                logger.LogError(ex, "Health check failed for {Format}", format);
                failures[format.ToString()] = ex.Message;
            }
        }

        return failures.Count == 0
            ? HealthCheckResult.Healthy(
                $"All {factory.SupportedFormats.Count} generators OK")
            : HealthCheckResult.Unhealthy(
                $"{failures.Count} generator(s) failed",
                data: failures);
    }
}

public class LoggingReportDecorator(
    IReportGenerator inner,
    ILogger logger,
    IMeterFactory meterFactory) : IReportGenerator
{
    private static readonly Meter _meter = meterFactory.Create("Reports");
    private static readonly Counter<long> _generated =
        _meter.CreateCounter<long>("reports.generated");
    private static readonly Histogram<double> _duration =
        _meter.CreateHistogram<double>("reports.duration_ms");

    public ReportFormat Format => inner.Format;
    public string ContentType => inner.ContentType;

    public async Task<byte[]> GenerateAsync(
        ReportData data, CancellationToken ct = default)
    {
        var sw = Stopwatch.StartNew();
        try
        {
            var result = await inner.GenerateAsync(data, ct);

            _generated.Add(1, new("format", Format.ToString()));
            _duration.Record(sw.Elapsed.TotalMilliseconds,
                new("format", Format.ToString()));

            logger.LogInformation(
                "Generated {Format} report ({Size:N0} bytes) in {Ms:F1}ms",
                Format, result.Length, sw.Elapsed.TotalMilliseconds);

            return result;
        }
        catch (Exception ex)
        {
            logger.LogError(ex,
                "Failed to generate {Format} report after {Ms:F1}ms",
                Format, sw.Elapsed.TotalMilliseconds);
            throw;
        }
    }

    public ValueTask DisposeAsync() => inner.DisposeAsync();
}

Migration Guide: Switch/Case → Factory Method

You've inherited a codebase with a 200-line switch statement in a controller that creates different report types. Here's how to migrate safely, one step at a time.

Step 1: Extract Creation Logic into a Static Method

Step1_Extract.cs
// BEFORE: switch in the controller action
[HttpGet("report/{format}")]
public IActionResult Generate(string format) { switch(format) { ... } }

// STEP 1: Extract to a static method — same file, minimal change
private static IReportGenerator CreateGenerator(string format) => format switch
{
    "pdf" => new PdfReportGenerator(),
    "excel" => new ExcelReportGenerator(),
    _ => throw new ArgumentException($"Unknown: {format}")
};

[HttpGet("report/{format}")]
public IActionResult Generate(string format)
{
    var generator = CreateGenerator(format);
    var bytes = generator.Generate(data);
    return File(bytes, generator.ContentType);
}

Risk: Zero. Same behavior, just moved code. The controller action is now 3 lines instead of 200.

Step 2: Extract Interface + Concrete Factory Class

Step2_Interface.cs
// New file: IReportGeneratorFactory.cs
public interface IReportGeneratorFactory
{
    IReportGenerator Create(string format);
}

// New file: ReportGeneratorFactory.cs
public class ReportGeneratorFactory : IReportGeneratorFactory
{
    public IReportGenerator Create(string format) => format switch
    {
        "pdf" => new PdfReportGenerator(),
        "excel" => new ExcelReportGenerator(),
        _ => throw new ArgumentException($"Unknown: {format}")
    };
}

// Controller — still uses new (we'll fix in Step 3)
private readonly IReportGeneratorFactory _factory = new ReportGeneratorFactory();

Risk: Minimal. New files added, controller now depends on interface. Tests can be written against IReportGeneratorFactory.

Step 3: Register in DI, Inject into Controller

Step3_DI.cs
// Program.cs — register factory and products
builder.Services.AddTransient<PdfReportGenerator>();
builder.Services.AddTransient<ExcelReportGenerator>();
builder.Services.AddSingleton<IReportGeneratorFactory, ReportGeneratorFactory>();

// Controller — inject via constructor
public class ReportController(IReportGeneratorFactory factory) : ControllerBase
{
    [HttpGet("report/{format}")]
    public IActionResult Generate(string format)
    {
        var generator = factory.Create(format);
        var bytes = generator.Generate(data);
        return File(bytes, generator.ContentType);
    }
}

// Tests — mock the factory
var mockFactory = new Mock<IReportGeneratorFactory>();
var controller = new ReportController(mockFactory.Object);

Win: Controller is testable. Factory is mockable. Products get their dependencies from DI.

Step 4: Upgrade to Keyed Services (.NET 8+)

Step4_Keyed.cs
// .NET 8+ — keyed services can replace the custom factory class
builder.Services.AddKeyedTransient<IReportGenerator, PdfReportGenerator>("pdf");
builder.Services.AddKeyedTransient<IReportGenerator, ExcelReportGenerator>("excel");

// Option A: Inject specific key at compile time
public class PdfController([FromKeyedServices("pdf")] IReportGenerator generator) { }

// Option B: Resolve by runtime key (still need some factory logic)
app.MapGet("/report/{format}", (string format, IServiceProvider sp) =>
{
    var generator = sp.GetRequiredKeyedService<IReportGenerator>(format);
    return Results.File(generator.Generate(data), generator.ContentType);
});
Where Keyed Services DON'T Work

Complex creation logic (parameterized factories), async initialization, cross-cutting decorator chains, or when you need to validate/transform the key before resolution. In these cases, keep your custom IReportGeneratorFactory.

Code Review Checklist

Use this checklist when reviewing Factory Method code in PRs. Print it, bookmark it, tattoo it on your forearm.

#CheckWhy It MattersRed Flag
1 Does factory return an interface, not a concrete type? Callers should depend on abstraction — enables swapping and testing public PdfExporter Create() — returns concrete
2 Is the factory registered in DI (not instantiated manually)? Manual new Factory() hides dependencies and can't be mocked _factory = new ReportFactory() in controller
3 Is product disposal responsibility documented? Undisposed factory products leak memory (see Bug Study 4) Products implement IDisposable but no using in caller
4 Is the switch/dictionary exhaustive for all enum values? Missing case = runtime exception in production No default / _ throw, or CS8509 suppressed
5 Does the factory avoid side effects in creation? DB calls or HTTP in factory constructor blocks DI resolution Factory constructor with await HttpClient.GetAsync()
6 Is the factory testable (interface-based, mockable)? Static factories can't be mocked in unit tests public static IExporter Create(string format)
7 Does it avoid Service Locator anti-pattern? Injecting raw IServiceProvider hides dependencies Factory constructor takes IServiceProvider and calls GetService<T>
8 Are factory products validated after creation? Misconfigured product causes downstream failures Factory returns product without checking required properties are set
9 Is the factory focused (SRP)? God factory creates unrelated types — violates SRP One factory creates exporters, validators, AND notification services
10 Does it handle null/invalid input gracefully? Null key = NullReferenceException deep in dictionary lookup No null check on factory key parameter
11 Are Roslyn analyzers enabled for factory patterns? Catch issues at compile time, not production CS8509 (non-exhaustive switch) and CA2000 (disposal) not enabled
12 Does the factory support CancellationToken for async creation? Long-running factory creation should be cancellable Task<IProduct> CreateAsync() without CancellationToken parameter
Automate it: Enable these Roslyn analyzers to catch factory issues at compile time:
  • CS8509 — Non-exhaustive switch expression (missing enum cases)
  • CA2000 — Dispose objects before losing scope
  • CA1822 — Mark members as static (catches factory methods that should be instance methods)
  • CA1063 — Implement IDisposable correctly (for disposable factory products)
  • Microsoft.Extensions.DependencyInjection.Analyzers — detects captive dependencies and lifetime mismatches (.NET 8+)
