Section 1

TL;DR

Observer in one line: "Hey, something changed! Everyone who cares — you're notified. Everyone who doesn't care — totally fine, zero involvement." The Subject screams into a list of listeners; the listeners react however they want. The Subject doesn't care who's listening or what they do with the news.

What: Observer defines a one-to-many dependencyOne Subject (publisher) notifies many Observers (subscribers). The key word is "dependency" — observers DEPEND on the subject for their data/events, but the subject is blissfully unaware of who those observers are. Add 100 observers, the subject code doesn't change by a single line. between objects. One Subject holds state; when that state changes, it automatically notifies a collection of ObserversObjects that have registered interest in a Subject's state changes. In code: any class implementing IObserver<T> or subscribing to an event. Each observer receives the notification and decides independently how to react — log it, update a UI, send an email, trigger another workflow.. The magic: the Subject only knows about the IObserver abstraction, never about concrete observer types.

When: Use it when a change in one object needs to trigger updates in others — and you don't want those objects to be tightly coupled. Classic scenarios: UI data binding, event-driven systems, pub/sub messaging, audit logs, real-time dashboards.

Modern .NET: You have THREE built-in mechanisms. event / delegateC#'s native Observer mechanism. Declare 'public event EventHandler<StockChangedEventArgs> StockChanged;' and any number of subscribers can attach with '+='. The compiler generates multicast delegate infrastructure — thread-safe add/remove, automatic multicasting. This is the most common Observer implementation in .NET codebases. (the everyday workhorse), IObservable<T>/IObserver<T>Introduced in .NET 4.0, formalized in .NET 4.5. The "reactive" push model. IObservable<T> is the Subject; IObserver<T> is the Observer with three methods: OnNext(T), OnError(Exception), OnCompleted(). This is the foundation of Rx.NET (System.Reactive) and represents a sequence of push-based notifications rather than a single event. (reactive/streaming), and Channel<T>System.Threading.Channels — introduced in .NET Core 3.0. A high-performance, async-first pipe for producer/consumer scenarios. The producer writes to ChannelWriter<T>, the consumer reads from ChannelReader<T>. Think of it as a thread-safe async queue. Not strictly Observer, but solves similar decoupling problems with backpressure support. + IAsyncEnumerable<T> (async streaming). Choose based on whether you need synchronous push, reactive composition, or async streaming.

Observer-at-a-glance.cs 
// .NET's built-in IObserver<T> / IObservable<T> interfaces
// (both are in System namespace — no packages needed)

// The Subject: produces stock price updates
public sealed class StockTicker : IObservable<decimal>
{
    private readonly List<IObserver<decimal>> _observers = new();

    public IDisposable Subscribe(IObserver<decimal> observer)
    {
        _observers.Add(observer);
        return new Unsubscriber(_observers, observer);   // caller holds this to unsubscribe
    }

    public void UpdatePrice(decimal newPrice)
    {
        foreach (var observer in _observers.ToList())    // ToList = safe iteration
            observer.OnNext(newPrice);
    }

    private sealed class Unsubscriber(List<IObserver<decimal>> observers, IObserver<decimal> observer)
        : IDisposable
    {
        public void Dispose() => observers.Remove(observer);
    }
}

// An Observer: reacts to price changes
public sealed class PriceAlertObserver(string stockSymbol, decimal threshold)
    : IObserver<decimal>
{
    public void OnNext(decimal price)
    {
        if (price > threshold)
            Console.WriteLine($"ALERT: {stockSymbol} crossed {threshold:C} — now {price:C}!");
    }
    public void OnError(Exception error) => Console.WriteLine($"Feed error: {error.Message}");
    public void OnCompleted() => Console.WriteLine("Market closed.");
}

Before reading this page, you should be comfortable with: Observer is one of those patterns where the code looks simple but the failure modes are nasty. The prerequisites below are the exact things that will bite you if you skip them.
Interfaces & Polymorphism — Observer is interface-driven. The Subject holds a List<IObserver<T>> and calls OnNext() through the interface. This encapsulation is crucial for Observer — you don't want some random class clearing everyone's subscriptions by doing 'ticker.StockChanged = null'. is literally Observer baked into the language. Generics — IObservable<T>, IObserver<T>, EventHandler<TEventArgs>, List<T> — everything in modern Observer is generic. Be comfortable reading and writing generic type signatures. Core UML

Key Insight — The arrows point INWARD: The Subject depends on the IObserver<T> abstractionThe Subject's _observers list is typed as List<IObserver<T>> — not List<PriceDisplay> or List<EmailNotifier>. This means you can add brand new observer types years later and the Subject code never changes. The Subject is closed for modification but open for extension — that's OCP in action., not on any concrete Observer type. ConcreteSubject has an aggregation to IObserver<T>, not to PriceDisplay or Logger. This is the Dependency Inversion PrincipleDIP: High-level modules (Subject) should not depend on low-level modules (concrete observers). Both should depend on abstractions (IObserver<T>). In Observer, the Subject is the high-level policy — "notify everyone when state changes." The concrete observers are low-level details. The Subject never imports them, never instantiates them, never knows they exist. at work — the most important structural insight in this pattern.

GoF Roles Explained

  • ISubject (Observable): Defines Attach, Detach, Notify. In .NET this is IObservable<T> with a single Subscribe() method.
  • ConcreteSubject: Holds actual state. Calls Notify() when state changes. Manages the observers list.
  • IObserver: Defines Update(). In .NET: IObserver<T> with OnNext/OnError/OnCompleted.
  • ConcreteObserver: Implements reaction logic. Can hold a reference back to the Subject to pull more data if needed (pull model).

Push vs Pull Model

  • Push (preferred): Subject packages the changed data and sends it directly to OnNext(T value). Observer gets everything it needs in the call. Used by IObservable<T> and typed events.
  • Pull: Subject just calls Update(this), passing a reference to itself. Observer calls back on the Subject to fetch what it needs. More flexible but creates back-reference coupling.
  • GoF originally described pullThe original GoF pattern had Update(Subject subject) — observers would call subject.GetState() themselves. Modern .NET strongly prefers push via IObservable<T> or EventArgs, because it makes the data flow explicit and avoids the observer needing to know the subject's API. — modern .NET strongly prefers push.

ClassicObserver.cs 
// file-scoped namespace — C# 10+
namespace Observer.Classic;

// ── Interfaces (the GoF contract) ──────────────────────────────────────────

public interface IStockObserver
{
    void Update(string symbol, decimal price);
}

public interface IStockSubject
{
    void Attach(IStockObserver observer);
    void Detach(IStockObserver observer);
    void Notify();
}

// ── Concrete Subject ───────────────────────────────────────────────────────

public sealed class StockMarket : IStockSubject
{
    private readonly List<IStockObserver> _observers = new();
    private decimal _price;
    private string _symbol;

    public StockMarket(string symbol) => _symbol = symbol;

    // State property — calling setter triggers notification
    public decimal Price
    {
        get => _price;
        set
        {
            _price = value;
            Notify();   // automatic notification on state change
        }
    }

    public void Attach(IStockObserver observer)
    {
        if (!_observers.Contains(observer))
            _observers.Add(observer);
    }

    public void Detach(IStockObserver observer) => _observers.Remove(observer);

    public void Notify()
    {
        // ToList() snapshot prevents issues if an observer detaches itself
        // during notification (a real production concern)
        foreach (var observer in _observers.ToList())
            observer.Update(_symbol, _price);
    }
}

// ── Concrete Observers ─────────────────────────────────────────────────────

// Primary constructor (C# 12) — clean, no boilerplate
public sealed class PriceDisplay(string name) : IStockObserver
{
    public void Update(string symbol, decimal price) =>
        Console.WriteLine($"[{name}] {symbol}: {price:C}");
}

public sealed class TradingBot(decimal buyThreshold, decimal sellThreshold) : IStockObserver
{
    public void Update(string symbol, decimal price)
    {
        if (price <= buyThreshold)
            Console.WriteLine($"[Bot] BUY {symbol} at {price:C}");
        else if (price >= sellThreshold)
            Console.WriteLine($"[Bot] SELL {symbol} at {price:C}");
        else
            Console.WriteLine($"[Bot] HOLD {symbol} at {price:C}");
    }
}

public sealed class AuditLogger : IStockObserver
{
    private readonly List<string> _log = new();

    public void Update(string symbol, decimal price)
    {
        var entry = $"{DateTime.UtcNow:HH:mm:ss} | {symbol} | {price:C}";
        _log.Add(entry);
        Console.WriteLine($"[Audit] Logged: {entry}");
    }

    public IReadOnlyList<string> GetLog() => _log.AsReadOnly();
}

namespace Observer.Events;

// ── EventArgs DTO — carries notification data ──────────────────────────────

// record = immutable DTO, perfect for event args (C# 9+)
public sealed record StockChangedEventArgs(
    string Symbol,
    decimal OldPrice,
    decimal NewPrice,
    DateTimeOffset Timestamp) : EventArgs;

// ── Publisher (Subject) ────────────────────────────────────────────────────

public sealed class StockTicker
{
    private decimal _price;

    // The event IS the Observer registration mechanism
    // EventHandler<T> = void(object? sender, T e) delegate signature
    public event EventHandler<StockChangedEventArgs>? StockChanged;

    public string Symbol { get; }
    public decimal Price
    {
        get => _price;
        set
        {
            var old = _price;
            _price = value;
            OnStockChanged(old, value);
        }
    }

    public StockTicker(string symbol, decimal initialPrice)
    {
        Symbol = symbol;
        _price = initialPrice;
    }

    // Protected virtual — allows subclassing if needed
    protected virtual void OnStockChanged(decimal oldPrice, decimal newPrice)
    {
        // '?.Invoke' is null-safe: only calls if there are subscribers
        StockChanged?.Invoke(this, new StockChangedEventArgs(
            Symbol, oldPrice, newPrice, DateTimeOffset.UtcNow));
    }
}

// ── Subscribers (Observers) — just methods, no interface needed! ───────────

public sealed class Dashboard
{
    public void OnStockChanged(object? sender, StockChangedEventArgs e)
    {
        var arrow = e.NewPrice > e.OldPrice ? "↑" : "↓";
        Console.WriteLine($"[Dashboard] {e.Symbol} {arrow} {e.OldPrice:C} → {e.NewPrice:C}");
    }
}

public sealed class RiskManager
{
    private const decimal MaxDropPercent = 0.05m; // 5% drop triggers alert

    public void OnStockChanged(object? sender, StockChangedEventArgs e)
    {
        if (e.OldPrice > 0)
        {
            var dropPercent = (e.OldPrice - e.NewPrice) / e.OldPrice;
            if (dropPercent >= MaxDropPercent)
                Console.WriteLine($"[RISK] ALERT: {e.Symbol} dropped {dropPercent:P1}!");
        }
    }
}

IObservable<T>System.IObservable<T> is the Subject in .NET's reactive pattern. It has one method: IDisposable Subscribe(IObserver<T> observer). The returned IDisposable is the unsubscription token. IObserver<T> has three methods: OnNext (normal data), OnError (error in stream), OnCompleted (stream ended). This pair is the foundation of Rx.NET. — .NET's built-in reactive interfaces. No Rx.NET package required for basic usage.

ReactiveObserver.cs 
namespace Observer.Reactive;

// ── Data type ──────────────────────────────────────────────────────────────

public sealed record TemperatureReading(string Sensor, double Celsius, DateTimeOffset At);

// ── Subject implements IObservable<T> ─────────────────────────────────────

public sealed class TemperatureSensor : IObservable<TemperatureReading>
{
    // Thread-safe snapshot for concurrent subscribe/unsubscribe
    private readonly object _lock = new();
    private readonly List<IObserver<TemperatureReading>> _observers = new();

    public string Name { get; }
    public TemperatureSensor(string name) => Name = name;

    public IDisposable Subscribe(IObserver<TemperatureReading> observer)
    {
        lock (_lock)
        {
            if (!_observers.Contains(observer))
                _observers.Add(observer);
        }
        return new Subscription(this, observer);
    }

    private void Remove(IObserver<TemperatureReading> observer)
    {
        lock (_lock) { _observers.Remove(observer); }
    }

    public void Publish(double celsius)
    {
        var reading = new TemperatureReading(Name, celsius, DateTimeOffset.UtcNow);
        List<IObserver<TemperatureReading>> snapshot;
        lock (_lock) { snapshot = _observers.ToList(); }

        foreach (var observer in snapshot)
            observer.OnNext(reading);
    }

    public void Fail(string errorMessage)
    {
        List<IObserver<TemperatureReading>> snapshot;
        lock (_lock) { snapshot = _observers.ToList(); }
        var ex = new InvalidOperationException(errorMessage);
        foreach (var observer in snapshot)
            observer.OnError(ex);
    }

    public void Shutdown()
    {
        List<IObserver<TemperatureReading>> snapshot;
        lock (_lock) { snapshot = _observers.ToList(); _observers.Clear(); }
        foreach (var observer in snapshot)
            observer.OnCompleted();
    }

    // Nested subscription token — clean unsubscription via IDisposable
    private sealed class Subscription(TemperatureSensor sensor, IObserver<TemperatureReading> observer)
        : IDisposable
    {
        private bool _disposed;
        public void Dispose()
        {
            if (!_disposed) { sensor.Remove(observer); _disposed = true; }
        }
    }
}

// ── Observers implement IObserver<T> ──────────────────────────────────────

public sealed class ThermostatController(double setPoint) : IObserver<TemperatureReading>
{
    public void OnNext(TemperatureReading r)
    {
        if (r.Celsius > setPoint + 2)
            Console.WriteLine($"[Thermostat] Too hot ({r.Celsius:F1}°C) — cooling ON");
        else if (r.Celsius < setPoint - 2)
            Console.WriteLine($"[Thermostat] Too cold ({r.Celsius:F1}°C) — heating ON");
        else
            Console.WriteLine($"[Thermostat] OK ({r.Celsius:F1}°C)");
    }
    public void OnError(Exception e) => Console.WriteLine($"[Thermostat] Sensor error: {e.Message}");
    public void OnCompleted() => Console.WriteLine("[Thermostat] Sensor offline.");
}

// ── Usage ──────────────────────────────────────────────────────────────────

Think of it as a thread-safe async pipe: one or more producers write to ChannelWriter<T>, one or more consumers read from ChannelReader<T> via await foreach. Supports backpressure (bounded channels block/drop when full). + IAsyncEnumerable<T> — the modern choice for async event streams in .NET 8.

AsyncObserver.cs 
using System.Threading.Channels;
namespace Observer.Async;

public sealed record OrderEvent(string OrderId, string Status, DateTimeOffset At);

// ── Async Subject — broadcasts to multiple channel consumers ───────────────

public sealed class OrderEventBus : IAsyncDisposable
{
    // Bounded channel: if consumer falls behind, writer will await or drop
    private readonly List<Channel<OrderEvent>> _channels = new();
    private readonly object _lock = new();

    public ChannelReader<OrderEvent> Subscribe()
    {
        // Bounded(100): at most 100 buffered events per consumer
        var channel = Channel.CreateBounded<OrderEvent>(
            new BoundedChannelOptions(100)
            {
                FullMode = BoundedChannelFullMode.DropOldest,  // drop old events if consumer slow
                SingleWriter = false,
                SingleReader = true
            });

        lock (_lock) { _channels.Add(channel); }
        return channel.Reader;
    }

    public async Task PublishAsync(OrderEvent evt, CancellationToken ct = default)
    {
        List<Channel<OrderEvent>> snapshot;
        lock (_lock) { snapshot = _channels.ToList(); }

        // Broadcast to all consumers concurrently
        var tasks = snapshot.Select(ch =>
            ch.Writer.WriteAsync(evt, ct).AsTask());
        await Task.WhenAll(tasks);
    }

    public async ValueTask DisposeAsync()
    {
        List<Channel<OrderEvent>> snapshot;
        lock (_lock) { snapshot = _channels.ToList(); _channels.Clear(); }
        foreach (var ch in snapshot)
            ch.Writer.Complete();   // signals OnCompleted to all readers
    }
}

// ── Async Observers — consume via await foreach ────────────────────────────

public static class OrderProcessors
{
    public static async Task RunEmailNotifier(
        ChannelReader<OrderEvent> reader, CancellationToken ct)
    {
        await foreach (var evt in reader.ReadAllAsync(ct))
        {
            Console.WriteLine($"[Email] Order {evt.OrderId} status: {evt.Status}");
            await Task.Delay(10, ct);   // simulate async email send

    public static async Task RunAnalytics(
        ChannelReader<OrderEvent> reader, CancellationToken ct)
    {
        await foreach (var evt in reader.ReadAllAsync(ct))

}

Problem 1-3: Coupling & Safety

  • Tight coupling: List<PriceDisplay> means only PriceDisplay observers are supported. Adding MobileDisplay requires modifying PriceTracker.
  • No abstraction: Should be List<IObserver<decimal>> or a custom interface — anything that decouples via abstraction.
  • Concurrent modification: If an observer unsubscribes itself during Refresh(), the foreach throws InvalidOperationException. Fix: iterate over displays.ToList().

Problem 4-5: Exceptions & Leaks

  • Exception propagation: If d2.Refresh() throws, d3, d4, etc. never receive the notification. Wrap each call in try/catch and log — don't let one bad observer break the chain.
  • Memory leakThe PriceTracker holds a strong reference to every PriceDisplay ever added. Even if the display is no longer shown on screen and your UI code drops all references to it, the PriceTracker keeps it alive because it's in _observers. The GC cannot collect it. Over time (especially in long-running services), this creates a memory leak that grows until OutOfMemoryException.: No RemoveDisplay() method means every added display is referenced forever. In long-running processes this is a slow OOM crash.

namespace StockTracker;

public sealed class StockTicker
{
    // ✅ Lock for thread-safe subscribe/unsubscribe
    private readonly Lock _lock = new();
    private readonly List<IStockObserver> _observers = new();
    private readonly ILogger<StockTicker> _logger;

    private decimal _price;
    public string Symbol { get; }

    public StockTicker(string symbol, ILogger<StockTicker> logger)
    {
        Symbol = symbol;
        _logger = logger;
    }

    // ✅ Returns IDisposable — subscribe and get a cleanup token
    public IDisposable Subscribe(IStockObserver observer)
    {
        ArgumentNullException.ThrowIfNull(observer);
        lock (_lock)
        {
            if (!_observers.Contains(observer))
                _observers.Add(observer);
        }
        return new Subscription(this, observer);
    }

    private void Unsubscribe(IStockObserver observer)
    {
        lock (_lock) { _observers.Remove(observer); }
    }

    public decimal Price
    {
        get => _price;
        set
        {
            var old = _price;
            if (old == value) return;   // ✅ No spurious notifications
            _price = value;
            NotifyAll(old, value);
        }
    }

    private void NotifyAll(decimal oldPrice, decimal newPrice)
    {
        List<IStockObserver> snapshot;
        lock (_lock) { snapshot = _observers.ToList(); }   // ✅ Safe snapshot

        foreach (var observer in snapshot)
        {
            try
            {
                observer.OnPriceChanged(Symbol, newPrice, oldPrice);
            }
            catch (Exception ex)
            {
                // ✅ Isolate failures — one bad observer can't break others
                _logger.LogError(ex, "Observer {Type} threw during notification",
                    observer.GetType().Name);
            }
        }
    }

    // ✅ Inner Subscription — auto-removes when disposed
    private sealed class Subscription(StockTicker ticker, IStockObserver observer)
        : IDisposable
    {
        private int _disposed;

        public void Dispose()
        {
            // ✅ Idempotent dispose — safe to call multiple times
            if (Interlocked.Exchange(ref _disposed, 1) == 0)
                ticker.Unsubscribe(observer);
        }
    }
}

Why events over manual list?

The event keyword gives you thread-safe add/remove, multicast delegate infrastructure, and a clear subscription API — all for free. Manual List<IObserver> requires you to implement locking, iteration snapshots, and idempotent dispose yourself. Rule of thumb: use event when observers are in the same process and synchronous notification is fine. Use IObservable<T>IObservable<T> gives you a richer contract: OnNext (data), OnError (failure), OnCompleted (stream ends). It also enables composition with Rx.NET operators (Where, Select, Throttle, Buffer, etc.) which turn raw event streams into powerful reactive pipelines. The trade-off: more complex to implement from scratch, but System.Reactive handles that. when you need reactive composition (filtering, throttling, merging streams).

When to use Weak References?

WeakReference<T>A wrapper that holds a reference to an object without preventing garbage collection. The GC can collect the target even if the WeakReference exists. You check if it's still alive with TryGetTarget(). In Observer, using List<WeakReference<IStockObserver>> instead of List<IStockObserver> means observers can be GC'd if no other code holds a reference — preventing memory leaks without explicit Unsubscribe calls. The Subject should periodically prune dead weak references. solves the "forgot to unsubscribe" memory leak — the Subject holds a weak reference to each observer, so the GC can still collect them. Trade-off: you must periodically prune dead references, and the API is more complex. Use weak references in framework/library code where you can't control caller cleanup. In application code, prefer explicit IDisposable subscriptions — they make the intent explicit and are easier to test.

Thread Safety Strategy

Three levels of thread safety in Observer:

  • Level 1 (single-thread): List<T> with .ToList() snapshot. Safe for subscribe/unsubscribe from the same thread as notifications.
  • Level 2 (multi-thread): lock around subscribe/unsubscribe + ToList() snapshot before iterating. The senior code above uses this.
  • Level 3 (high-contention): ImmutableList<T> or CopyOnWrite pattern. No locks during notification — only during subscription changes.

Exception Isolation Policy

When an observer throws, you have three options:

  • Swallow & log: (senior code above) — all observers always receive notifications. Best for independent observers.
  • Stop on first error: Let the exception propagate. Good if observers must all succeed or none should (transactional semantics).
  • Collect all errors: Catch all, then throw AggregateException. Good when you need to know which observers failed without skipping any.

Evolution in .NET

Observer has been in .NET since day one — but the right way to implement it has evolved dramatically. Here's the full timeline.

.NET 1.0–1.1 (2002) — Delegates & Events: The OG Observer

The multicast delegateA delegate that holds references to multiple methods. When invoked, it calls ALL attached methods in order. This is the mechanical heart of .NET events — event += handler adds to the invocation list, event -= removes from it. The delegate handles multicasting automatically; no loop required on your end. was .NET 1.0's built-in Observer. No interfaces, no generics (not invented yet) — just raw delegates with custom EventArgs.

Generics simplified everything. EventHandler<TEventArgs>A built-in generic delegate: void(object? sender, TEventArgs e). Standardized the event pattern across all of .NET. Section 9

When To Use (and When NOT To)

USE Observer when...

Event-driven architecture

Your system reacts to domain eventsDomain Events are meaningful things that happened in your business: OrderPlaced, PaymentReceived, InventoryLow. They represent facts. Observer is the natural dispatch mechanism — the Aggregate (Subject) fires the event, multiple downstream handlers (Observers) react independently without knowing each other. like OrderPlaced, UserRegistered, or StockPriceChanged. New reactions can be added without touching the event source. Classic fit.

UI State Changes (MVVM)

ViewModel properties change, UI controls should update. INotifyPropertyChanged + data binding IS Observer. Don't roll your own — the framework already has it.

Real-time Data Feeds

Stock tickers, sensor readings, live dashboards, telemetry. Multiple consumers need the same data stream with independent processing — IObservable<T> + Rx is the ideal tool.

Audit Logging / Cross-cutting Concerns

Audit trails, metrics, analytics that need to happen on every state change but shouldn't pollute business logic. An audit observer attaches without touching the Subject's business code.

Plugin / Extension Systems

Third-party code hooks into your events without internal access. Expose IObservable<T> streams; plugins subscribe. Core code never changes to accommodate them.

DON'T USE Observer when...

Simple 1-to-1 callback

Only ONE thing ever cares about a change? A simple callback is cleaner. Observer's subscriber list infrastructure is overkill. Use Func<T> or Action<T> instead.

Order of execution matters critically

Observers should be order-agnostic and independent. If Observer B must run after A, use Chain of ResponsibilityPasses a request through a chain of handlers. Unlike Observer (all get notified), CoR typically has one handler process the request. Use CoR when processing order matters and handlers are sequential; use Observer when all observers are independent. or explicit step orchestration instead.

Synchronous request/response

If you need a return value from the “observer” (validation result, computed value), Observer is wrong. It's fire-and-forget by design. Use Strategy or a direct method call.

Distributed / cross-process events

Classic Observer is in-process. Cross-service events need a message broker (RabbitMQ, Azure Service Bus, Kafka). Don't stretch in-process Observer across network boundaries.

When observer count explodes

Thousands of dynamic observers on one high-frequency Subject becomes a bottleneck. Consider topic-based pub/subSystems like Kafka or Azure Service Bus allow observers to subscribe to specific topics rather than a single Subject. Distributes notification load; scales consumers independently. For high-throughput scenarios (>10k events/sec), in-process Observer should give way to proper message infrastructure., batching, or sampling strategies.

Decision Tree: Multiple independent things react to one change? → Observer. Only one thing reacts? → callback/delegate. Need the reaction's return value? → Strategy or direct call. Cross-process? → Message broker. Order critical? → Chain of Responsibility.

Comparisons

Observer gets confused with several similar patterns. Here's how to tell them apart.

Observer vs Mediator

Observer — Direct Notification

Subject notifies observers directly. Subject knows the IObserver abstraction and manages its own subscriber list. Decoupled by abstraction, not by indirection.

observer-direct.cs 
// Subject directly notifies its observers
public void NotifyAll()
{
    foreach (var obs in _observers)
        obs.OnPriceChanged(_price);
}
// Subject owns and manages the observer list

Mediator — Centralized Hub

A MediatorMediator centralizes communication — components don't talk directly, they route through the hub. MediatR is the canonical .NET implementation. Better than Observer when routing logic is complex or you need complete sender/receiver decoupling where even the publisher is unaware of what's downstream. sits between all components. Everything routes through the hub; the publisher doesn't know what's downstream.

mediator-hub.cs 
// Mediator routes to appropriate handlers
await mediator.Publish(new PriceChangedEvent(_price));
// Mediator decides who handles it
// Subject doesn't know what's downstream at all
Rule: Observer when Subject actively manages its listener list. Mediator when you want complete decoupling where even the publisher doesn't know what's listening.

Observer vs Pub/Sub

Observer (in-process)

  • Same process, typically same thread
  • Subject holds direct references to observers
  • Synchronous by default
  • No infrastructure needed
  • Observer list is explicit and inspectable

Pub/Sub (distributed)Publisher/Subscriber via message brokers (RabbitMQ, Kafka, Azure Service Bus). Publishers don't know subscribers exist. Messages go to a broker topic; subscribers consume independently. Fully decoupled: publisher and subscriber can be different services, different languages, even different time zones — subscriber can read messages hours after publishing.

  • Potentially cross-process or cross-service
  • Message broker in between (no direct refs)
  • Asynchronous, with persistence and retry
  • Requires broker infrastructure
  • Publishers completely unaware of subscribers
pubsub-comparison.cs 
// Observer: direct, in-process, synchronous
ticker.Subscribe(myObserver);
ticker.Price = 100m;  // synchronous, same call stack

// Pub/Sub: via broker, async, cross-service
await serviceBus.PublishAsync(new PriceChangedMessage { Price = 100m });
// Message lands in Azure Service Bus topic
// OrderService, AnalyticsService subscribe independently
// They can process at their own pace, even hours later

Events/Delegates vs IObservable<T>

Events/Delegates

  • Language-native, zero dependencies
  • One notification model: OnNext only
  • No built-in error/completion signals
  • Hard to compose (no filter/map/merge)
  • Best for: simple UI events, lifecycle hooks

IObservable<T> / Rx.NET

  • Richer contract: OnNext + OnError + OnCompleted
  • Composable: Where, Select, Throttle, Buffer, Merge
  • Models stream lifecycle, not just one-off events
  • Requires System.Reactive NuGet package
  • Best for: data streams, reactive pipelines, CEP
Quick rule: If you're thinking “single event”, use C# events. If you're thinking “stream of events over time with filtering and transformation”, reach for IObservable<T> + Rx.NET.

SOLID Connections

Observer is one of the most SOLID-aligned patterns in GoF. It practically demonstrates all five principles simultaneously.

SRP — Single Responsibility

Subject has one job: manage state and notify observers. Each Observer has one job: react in its specific way. They never mix. StockTicker doesn't send emails. EmailNotifier doesn't track prices.

srp-example.cs 
// SRP: Subject manages state and notifies -- nothing else
public sealed class OrderTracker
{

    public event EventHandler<OrderStatusChangedArgs>? StatusChanged;

    public void UpdateStatus(OrderStatus newStatus)
    {
        _status = newStatus;
        StatusChanged?.Invoke(this, new(newStatus));  // notify only
    }
}

// SRP: each observer has ONE concern
public sealed class EmailNotifier   : IObserver { /* sends emails */ }
public sealed class InventoryUpdater : IObserver { /* updates stock */ }
public sealed class AuditLogger      : IObserver { /* writes log */ }

Bug Studies

Observer has 5 signature bugs that appear repeatedly in production. Every single one has caused real outages. Learn them here — not in your incident post-mortem.

Bug 1: Memory Leak — The Forgotten Unsubscribe

Scenario: A WPF/MAUI viewmodel subscribes to a long-lived service event on creation. The window closes and the viewmodel should be GC'd — but it never is. Memory grows until OOM crash.

Buggy Code

BuggyViewModel.cs 
public class DashboardViewModel
{
    public DashboardViewModel(StockService service)
    {
        // Subscribes but NEVER unsubscribes
        service.PriceChanged += OnPriceChanged;
        // StockService holds a reference to 'this'
        // 'this' can NEVER be GC'd while service is alive
    }

    private void OnPriceChanged(object? s, decimal price) =>
        CurrentPrice = price;

    public decimal CurrentPrice { get; private set; }
    // No Dispose, no -= handler: memory leak
}

Fixed Code

FixedViewModel.cs 
public sealed class DashboardViewModel : IDisposable
{
    private readonly StockService _service;
    private bool _disposed;

    public DashboardViewModel(StockService service)
    {
        _service = service;
        _service.PriceChanged += OnPriceChanged;
    }

    private void OnPriceChanged(object? s, decimal price) =>
        CurrentPrice = price;

    public decimal CurrentPrice { get; private set; }

    public void Dispose()
    {
        if (!_disposed)

            _service.PriceChanged -= OnPriceChanged; // unsubscribe
            _disposed = true;
        }
    }

Bug 2: Infinite Loop — The Observer That Bites Back

Scenario: Observer A is notified by Subject X. A updates Subject Y. Y notifies Observer B. B updates Subject X. X notifies A again. Stack overflow in seconds.

Buggy Code

BuggyLoop.cs

{

    private readonly TemperatureSensor _fahrenheit;

    {

        f.Subscribe(this);  // which updates F, which notifies this...

Fixed Code

public class TempConverter : IObserver<double>
{

Fixed Code

FixedThreadSafety.cs 
public class StockTicker
{
    private readonly Lock _lock = new();  // C# 13 Lock type
    private readonly List<IObserver<decimal>> _observers = new();

    public void NotifyAll(decimal price)
    {
        List<IObserver<decimal>> snapshot;
        // Lock only to take a snapshot -- release before notifications
        lock (_lock) { snapshot = _observers.ToList(); }

        // Iterate snapshot -- modifications to _observers are safe
        // Don't hold lock during notifications!
        // (observers might subscribe/unsubscribe: deadlock)
        foreach (var obs in snapshot)
            obs.OnNext(price);
    }

    public void Unsubscribe(IObserver<decimal> obs)
    {
        lock (_lock) { _observers.Remove(obs); }  // thread-safe
    }
}

        {
            obs.OnNext(price);  // each observer isolated

        catch (Exception ex)

Pitfalls

Observer is deceptively simple to get wrong. The pattern looks clean on a whiteboard, then you ship it and get memory leaks, deadlocks, and mystery UI crashes at 2am. Here are the 8 most common ways Observer bites you â€” and exactly how to fix each one.

Pitfall 1: Forgetting to Unsubscribe â€” Memory Leaks from Lingering Handlers

Problem: Every event subscription is a reference from the publisher to the subscriber. If the subscriber "goes away" (e.g., a ViewModel is disposed) but never unsubscribes, the publisher's event delegate listC# events are backed by a MulticastDelegate â€” a linked list of method pointers. Each += adds a new entry. The publisher holds a strong reference to each subscriber through this list, preventing GC from collecting them even when the subscriber is logically "gone". keeps a strong reference to it. The subscriber leaks. Forever.

When it bites: Long-lived publishers (singletons, static classes, application-lifetime services) + short-lived subscribers (ViewModels, dialogs, per-request handlers). Classic in WPF/MAUI where a Window subscribes to a service but never unsubscribes on Close.

Bad Code

{
    public OrderViewModel(OrderService svc)
    {
        // Subscribes â€” never unsubscribes!
        svc.OrderPlaced += OnOrderPlaced;
    }

    private void OnOrderPlaced(Order o)
        => LoadOrders(); // called forever

    // No Dispose, no -= anywhere
    // OrderService holds OrderViewModel alive
    // even after the page closes
}
Fix â€” Implement IDisposable
CleanViewModel.cs

{
    private readonly OrderService _svc;

    public OrderViewModel(OrderService svc)
    {
        _svc = svc;
        _svc.OrderPlaced += OnOrderPlaced; // subscribe
    }

    private void OnOrderPlaced(Order o) => LoadOrders();

    public void Dispose()
    {
        _svc.OrderPlaced -= OnOrderPlaced; // always unsubscribe!
    }
}

// Or use IObservable + store the IDisposable subscription:

{

    public BetterViewModel(IObservable<Order> orders)
    {
        _subscription = orders.Subscribe(OnOrderPlaced);

    public void Dispose() => _subscription.Dispose(); // clean!
}

Problem: If an observer's Update() method calls Unsubscribe() on the subject while the subject is iterating its observer list, you get an InvalidOperationException"Collection was modified; enumeration operation may not execute." â€” thrown by foreach when the underlying collection changes during iteration. The fix is to either iterate a snapshot (ToList()), use an immutable list, or collect removals and apply them after the loop.: "Collection was modified; enumeration operation may not execute."

// BETTER: use ImmutableList (no copy needed)

When it bites: Self-unsubscribing observers (one-time handlers), conditional unsubscribes inside Update(), or any time a subscriber decides "I've heard enough."

Q9: How do you make Observer thread-safe?

ThreadSafeSubject.cs
public class ThreadSafeSubject<T>
{
    private readonly object _lock = new();
    private List<IObserver<T>> _observers = new();

    public void Subscribe(IObserver<T> observer)
    {
        lock (_lock) // copy-on-write: atomic list replacement
            _observers = [.._observers, observer];
    }

    public void Unsubscribe(IObserver<T> observer)
    {
        lock (_lock)
        {
            var copy = new List<IObserver<T>>(_observers);
            copy.Remove(observer);
            _observers = copy;
        }
    }

    public void Notify(T value)
    {
        List<IObserver<T>> snapshot;
        lock (_lock) { snapshot = _observers; } // atomic read â€” no copy needed
        // Iterate outside lock â€” observers can safely subscribe/unsubscribe during this
        foreach (var o in snapshot) o.OnNext(value);
    }
}

Maximizes read concurrency â€” notifications never hold the lock. pattern ensures Notify() iterates a stable snapshot without holding the lock, preventing deadlocks if observers subscribe during notification.

Q20: How is Observer related to Event Sourcing?
Advanced

Event Sourcing stores every state change as an immutable event. Projections (read models) are observers of the event stream â€” they subscribe and build their own views by processing events sequentially. The event store is the Subject; projections are Observers. The key difference from basic Observer: ES events are persisted and replayable. Standard Observer notifications are ephemeral â€” missed events are gone. With ES, any projection can be rebuilt from scratch by replaying from event 0, making Observer's "fire-and-forget" limitation irrelevant.

Q21: Distributed Observer across microservices â€” how do you implement it?
Advanced

Distributed Observer is Pub/Sub with a message broker. Subject publishes to a topic (Azure Service Bus, Kafka, RabbitMQ). Subscriber services consume from that topic â€” each service is an independent Observer. The broker provides: durability, fan-out, ordering, and at-least-once delivery. In .NET, MassTransit or NServiceBus abstract the broker: consumers implement IConsumer<TMessage> â€” the Observer interface mapped to a message type. Critical addition over in-process Observer: idempotency â€” network failures can deliver messages twice; design consumers to handle duplicates safely (check-then-act with a processed-event log).

Q22: What are Reactive Streams and how do they extend Observer?
Advanced

{
    string Name { get; }
    Task ReceiveAsync(ChatMessage message, CancellationToken ct);
}

public class ChatRoom(ILogger<ChatRoom> logger)
{
    private readonly ConcurrentDictionary<IChatClient, int> _clients = new();
    public event Action<IChatClient>? ClientDisconnected;

    public IDisposable Join(IChatClient client)
    {
        _clients.TryAdd(client, 0);
        return new Subscription(() =>
        {
            _clients.TryRemove(client, out _);
            logger.LogInformation("{Name} left the room", client.Name);
        });
    }

    // Fire-and-forget â€” PostMessage returns immediately
    public void PostMessage(ChatMessage message)
        => _ = DeliverToAllAsync(message);

    private async Task DeliverToAllAsync(ChatMessage message)
    {
        var deliveryTasks = _clients.Keys.Select(c => DeliverWithRetryAsync(c, message));
        await Task.WhenAll(deliveryTasks);
    }

    private async Task DeliverWithRetryAsync(IChatClient client, ChatMessage message)
    {
        const int maxRetries = 3;
        using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(10));

        for (int attempt = 0; attempt < maxRetries; attempt++)
        {
            try
            {
                await client.ReceiveAsync(message, cts.Token);
                _clients[client] = 0; // reset failure count on success
                return;
            }
            catch (Exception ex) when (attempt < maxRetries - 1)
            {
                var delay = TimeSpan.FromMilliseconds(100 * Math.Pow(2, attempt));
                logger.LogWarning(ex, "{Name}: delivery failed, retry {N} in {Delay}ms",
                    client.Name, attempt + 1, delay.TotalMilliseconds);
                await Task.Delay(delay, cts.Token);
            }
            catch (Exception ex)
            {
                logger.LogError(ex, "{Name}: max retries exceeded â€” disconnecting", client.Name);
                _clients.TryRemove(client, out _);
                ClientDisconnected?.Invoke(client);
            }
        }
    }

    private sealed class Subscription(Action cleanup) : IDisposable
    {
        private int _disposed;
        public void Dispose() { if (Interlocked.Exchange(ref _disposed, 1) == 0) cleanup(); }
    }
}

public sealed record StoredEvent<T>(long Sequence, T Payload, DateTimeOffset OccurredAt);

public class EventStore<T>
{
    private readonly ReaderWriterLockSlim _lock = new();
    private readonly List<StoredEvent<T>> _log = new();
    private readonly List<(long FromSeq, Action<StoredEvent<T>> Handler)> _subscribers = new();
    private long _sequence;

    public long Append(T payload)
    {
        StoredEvent<T> stored;
        List<(long, Action<StoredEvent<T>>)> subs;

        _lock.EnterWriteLock();
        try
        {
            stored = new StoredEvent<T>(++_sequence, payload, DateTimeOffset.UtcNow);
            _log.Add(stored);
            subs = new(_subscribers); // snapshot under write lock
        }
        finally { _lock.ExitWriteLock(); }

        foreach (var (_, handler) in subs) handler(stored);
        return stored.Sequence;
    }

    public IDisposable Subscribe(Action<StoredEvent<T>> handler, long fromSequence = 0)
    {
        _lock.EnterReadLock();
        List<StoredEvent<T>> replay;
        try { replay = _log.Where(e => e.Sequence > fromSequence).ToList(); }
        finally { _lock.ExitReadLock(); }

        // Replay historical events before subscribing to live
        foreach (var evt in replay) handler(evt);

        _lock.EnterWriteLock();
        try { _subscribers.Add((fromSequence, handler)); }
        finally { _lock.ExitWriteLock(); }

        return new Subscription(() =>
        {
            _lock.EnterWriteLock();
            try { _subscribers.RemoveAll(s => s.Handler == handler); }
            finally { _lock.ExitWriteLock(); }
        });
    }

    public Projection<T, TState> CreateProjection<TState>(
        TState initial, Func<TState, T, TState> folder)
        => new(this, initial, folder);

    private sealed class Subscription(Action cleanup) : IDisposable
    {
        private int _disposed;
        public void Dispose() { if (Interlocked.Exchange(ref _disposed, 1) == 0) cleanup(); }
    }
}

public class Projection<T, TState> : IDisposable
{
    private TState _state;
    private readonly IDisposable _subscription;
    private readonly object _stateLock = new();

    public Projection(EventStore<T> store, TState initial, Func<TState, T, TState> folder)
    {
        _state = initial;
        _subscription = store.Subscribe(evt =>
        {
            lock (_stateLock) { _state = folder(_state, evt.Payload); }
        });
    }

    public TState State { get { lock (_stateLock) return _state; } }
    public void Dispose() => _subscription.Dispose();
}

    .Subscribe(/* ... */);

source
    .RetryWhen(errors => errors
        .Zip(Observable.Range(1, 4), (error, attempt) => (error, attempt))
        .SelectMany(tuple =>
        {
            if (tuple.attempt > 3) throw tuple.error; // give up after 3
            var delay = TimeSpan.FromMilliseconds(100 * Math.Pow(2, tuple.attempt));
            return Observable.Timer(delay); // wait before retry
        }))
    .Subscribe(/* ... */);

// â”€â”€â”€ OnErrorResumeNext â€” skip errored sequences â”€â”€â”€
var s1 = source1.Select(v => v);
var s2 = source2.Select(v => v);

    .Subscribe(v => Process(v));

// â”€â”€â”€ Materialize â€” convert errors to notifications (no stream termination) â”€â”€â”€
source
    .Materialize() // wraps each event in Notification<T>
    .Subscribe(notification =>
    {
        if (notification.Kind == NotificationKind.OnError)
            logger.LogError(notification.Exception, "Handled error in stream");
        else if (notification.Kind == NotificationKind.OnNext)
            Process(notification.Value);
        // Stream keeps running â€” errors don't terminate it!
    });
Critical: OnError Terminates the Stream

Unlike exceptions in a loop, OnError is terminal â€” after an error, the subject will never call OnNext again. If you need resilience, use Catch, Retry, or Materialize to prevent termination. Most production Rx pipelines end with a .Catch(ex => Observable.Empty<T>()) or a retry policy so a single network hiccup doesn't kill the entire live feed.

// The dispatcher IS the Subject
public class OrderEventDispatcher
{
    private readonly List<IOrderEventHandler> _handlers = new();

    public IDisposable Subscribe(IOrderEventHandler handler)
    {
        _handlers.Add(handler);
        return new Subscription(() => _handlers.Remove(handler));
    }

    public async Task DispatchAsync(OrderPlacedEvent evt, CancellationToken ct = default)
    {
        // Resilient loop â€” one failing handler doesn't affect others
        var tasks = _handlers.ToList().Select(async h =>
        {
            try { await h.HandleAsync(evt, ct); }
            catch (Exception ex)
            {
                // Log and continue â€” don't let email failure block analytics
                Console.Error.WriteLine($"Handler {h.GetType().Name} failed: {ex.Message}");
            }
        });
        await Task.WhenAll(tasks); // all handlers run in parallel
    }

    private sealed class Subscription(Action cleanup) : IDisposable
    {
        private int _disposed;
        public void Dispose() { if (Interlocked.Exchange(ref _disposed, 1) == 0) cleanup(); }
    }
}

Code Review Checklist

Run every Observer implementation through these 12 checks before it ships. Print this out. Tape it above your monitor. Your future self will thank you at 2am when production isn't leaking memory.

Every subscribe has a matching unsubscribe

Search for += and verify each has a matching -= in a Dispose() or using block. If using IObservable.Subscribe(), the returned IDisposable must be stored and disposed. Red flag: subscribe in constructor with no IDisposable implementation on the class.

Observer list is thread-safe

If subscribe/unsubscribe can happen on any thread (any long-lived subject), the observer list needs protection. Check for: copy-on-write pattern, lock on list mutations, snapshot before iterating, or use of ConcurrentBagConcurrentBag<T> or ImmutableList<T> are thread-safe alternatives to List<T>. For Observer lists, ImmutableList (atomic swap via Interlocked) is usually cleaner than ConcurrentBag because it supports snapshot iteration without locks. / ImmutableList. No lock needed for single-threaded (UI thread) subjects.

Notifications are resilient to observer exceptions

Does a single failing observer stop the rest from being notified? Verify the notification loop has per-observer try-catch. Exceptions should be logged and swallowed at the dispatch level â€” each observer's failure is independent. Exception: validation pipelines where the first failure should stop subsequent observers.

Notifications don't iterate the live list

The notification loop must iterate a snapshot (_observers.ToList() or copy) â€” not the live collection. If an observer calls Unsubscribe() during its Update() call, the live collection changes during iteration: InvalidOperationException or missed observers. Check that Notify() takes a snapshot before the foreach.

Observers do minimal work in Update/OnNext

Notification loops are synchronous by default. An observer doing a 500ms database call blocks every subsequent observer. Verify observers queue work (Channel, Task.Run) rather than doing it inline. Red flag: await db.InsertAsync() or httpClient.PostAsync().Wait() inside an event handler.

No circular observer chains

Observer A's Update() should not trigger a notification that eventually calls A's Update() again. Draw the observer graph â€” it should be a DAG. Red flags: Subject A and Subject B both observing each other, or a handler that calls a method on the subject that triggers re-notification.

Thread marshaling for UI observers

Any observer that touches UI elements (WPF, MAUI, Blazor) must ensure it's on the correct thread. Check for Dispatcher.Invoke, MainThread.BeginInvokeOnMainThread, or ObserveOn(scheduler) for Rx-based feeds. Red flag: Label.Text = value directly in an event handler without thread check.

Observer list is not publicly exposed

The internal observer list should be private. External code should only interact through Subscribe()/Unsubscribe() methods. Exposing the list bypasses validation, locking, and logging in Subscribe(). Red flag: public List<IObserver> Observers { get; }.

Async observers use correct pattern

If observers are async, are they awaited or fire-and-forget intentionally? Verify: (1) async void event handlers have try-catch (exceptions in async void are unobserved), (2) if awaited, understand that the notification is sequential not parallel, (3) for parallel await, use Task.WhenAll with a snapshot. Red flag: .Wait() or .Result inside an observer.

Event payloadThe data passed to observers with each notification. Use immutable records for payloads â€” observers can't accidentally mutate shared state. Include enough context so observers don't need to call back on the subject (avoid pull-model unless necessary). Include a timestamp and correlation ID for distributed tracing. is immutable

Event payloads passed to observers should be immutable â€” use record types. If a payload is mutable and two observers run in parallel, one might mutate it while the other reads it. Red flag: passing a mutable entity directly to all observers â€” one observer's modification affects every subsequent observer.

Static events reviewed carefully

Static events on static classes are the most dangerous form of Observer â€” they live for the entire application lifetime. Any instance subscriber to a static event leaks until the app closes unless it explicitly unsubscribes. If using static events, require IDisposable cleanup on all subscribers and document the lifetime expectation clearly. Prefer DI-managed singleton subjects over static events.

Each observer has unit tests

The dispatcher/subject has tests (verifying all observers notified, resilience to exceptions, thread safety). Each concrete observer has independent tests (verify it handles the event correctly in isolation). Verify unsubscribe tests (no notifications after Dispose). Red flag: 0% test coverage on observer classes because "they're just called by events."
