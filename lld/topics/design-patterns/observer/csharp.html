Prerequisites

Observer relies on a few C# concepts that you'll want to be comfortable with. You don't need to be an expert — just familiar enough that the code examples won't surprise you.

  • Delegates in C# — A delegateA type-safe function pointer in C#. It holds a reference to a method (or multiple methods) and can invoke them later. Think of it as a variable that stores a function. is basically a variable that holds a reference to a method. You can pass methods around like data. If you've used Action<T> or Func<T>, you already know delegates. Observer uses them heavily because the subject needs to "remember" which methods to call when something changes.

  • Events in C# — Events are delegates with guardrails. The event keyword restricts who can raise the event (only the class that owns it) while still letting outsiders subscribe with +=.   • Interfaces — Interfaces define a contract: "any class that implements me must have these methods." Observer depends on this because the subject holds a list of IObserver references, not concrete types. This is what lets you add new observer types without modifying the subject.

  • Generics — GenericsA way to write code that works with any type. List<T> is generic — T gets replaced with a concrete type like int or string when you use it. Avoids code duplication and provides compile-time type safety. let you write type-safe code that works with any data type. IObservable<T> and IObserver<T> are generic interfaces — the T is the type of data being observed (like decimal for stock prices or string for log messages).

Section 4

Core Concept Diagram

Before we draw the class diagram, let's understand each participant in plain English. There are four players in the Observer pattern, and each has a clear job:

1. Subject (ISubject) — This is the thing being watched. It maintains a list of observers and has three core operations: attach (add an observer to the list), detach (remove one), and notify (loop through the list and call each observer's update method). The subject knows the IObserver interface but has no clue what the concrete observers actually do.

2. ConcreteSubject — The actual implementation of the subject. This is where the real state lives — like a StockTicker with a Price property, or a WeatherStation with Temperature and Humidity. When its state changes, it calls Notify() to broadcast the update.

3. Observer (IObserver) — The interface that all observers implement. It typically has a single method: Update(). This is the contract that says "I can handle notifications from a subject."

4. ConcreteObserver — The actual listeners. A PriceDisplay that shows the price on screen. An AlertService that sends an SMS when the price drops below a threshold. A TradingBot that automatically buys or sells. Each one implements Update() differently, but the subject treats them all the same way.

GoF Definition: "Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically."

Code Implementations

C# gives you three distinct ways to implement Observer, each suited to different scenarios. We'll build the same example — a weather station that notifies displays — in all three styles so you can see how the core idea stays the same while the syntax changes dramatically.

This is the textbook GoF approach translated into C#. You define your own IObserver and ISubject interfaces and wire everything manually. It's verbose but shows every moving part clearly — which makes it great for learning.

The WeatherStation is the subject. It keeps a list of observers and loops through them when the temperature changes. PhoneDisplay and WebDashboard are concrete observers — they each implement Update() differently.

// The Observer interface: any class that wants weather updates // must implement this single method. public interface IWeatherObserver { void Update(float temperature, float humidity); } // The Subject: maintains a list of observers and notifies them public class WeatherStation { private readonly List<IWeatherObserver> _observers = new(); private float _temperature; private float _humidity; // Subscribe: add an observer to the notification list public void Attach(IWeatherObserver observer) { _observers.Add(observer); } // Unsubscribe: remove an observer public void Detach(IWeatherObserver observer) { _observers.Remove(observer); } // Broadcast: loop through ALL observers and call Update() private void Notify() { foreach (var observer in _observers) { observer.Update(_temperature, _humidity); } } // When the weather changes, update state and notify everyone public void SetMeasurements(float temp, float humidity) { _temperature = temp; _humidity = humidity; Notify(); // this is where the magic happens } } // Concrete Observer #1: shows weather on a phone screen public class PhoneDisplay : IWeatherObserver { public void Update(float temperature, float humidity) { Console.WriteLine( $"[Phone] Temperature: {temperature}°C, Humidity: {humidity}%"); } } // Concrete Observer #2: shows weather on a web dashboard public class WebDashboard : IWeatherObserver { public void Update(float temperature, float humidity) { Console.WriteLine( $"[Web Dashboard] Current: {temperature}°C / {humidity}% humidity"); } } var station = new WeatherStation(); var phone = new PhoneDisplay(); var dashboard = new WebDashboard(); station.Attach(phone); // phone subscribes station.Attach(dashboard); // dashboard subscribes station.SetMeasurements(25.5f, 65f); // Output: // [Phone] Temperature: 25.5°C, Humidity: 65% // [Web Dashboard] Current: 25.5°C / 65% humidity station.Detach(phone); // phone unsubscribes station.SetMeasurements(30.0f, 70f); // Output: // [Web Dashboard] Current: 30°C / 70% humidity // (phone no longer gets notified)

What just happened: The WeatherStation knows nothing about phones or dashboards. It only knows about IWeatherObserver. When SetMeasurements() fires, it iterates the list and calls Update() on each. Adding a third display — say, a SmartWatchDisplay — means writing one new class and calling station.Attach(smartWatch). The station doesn't change at all.

PriceChanged; private decimal _price; public decimal Price { get => _price; set { var oldPrice = _price; _price = value; // Raise the event = Notify() in classic Observer PriceChanged?.Invoke(this, new PriceChangedEventArgs { Symbol = Symbol, OldPrice = oldPrice, NewPrice = value }); } } public string Symbol { get; init; } = "AAPL"; } public class PriceChangedEventArgs : EventArgs { public string Symbol { get; init; } = ""; public decimal OldPrice { get; init; } public decimal NewPrice { get; init; } } // Subscribing is just += on the event var ticker = new StockTicker { Symbol = "AAPL" }; // Observer 1: log to console ticker.PriceChanged += (sender, args) => Console.WriteLine($"{args.Symbol}: ${args.OldPrice} → ${args.NewPrice}"); // Observer 2: alert on big changes ticker.PriceChanged += (sender, args) => { var change = Math.Abs(args.NewPrice - args.OldPrice); if (change > 5) Console.WriteLine($"⚠ BIG MOVE on {args.Symbol}: ${change} change!"); }; // Observer 3: a named method (easier to unsubscribe later) void LogToFile(object? sender, PriceChangedEventArgs args) { File.AppendAllText("prices.log", $"{DateTime.Now}: {args.Symbol} = ${args.NewPrice}\n"); } ticker.PriceChanged += LogToFile; // Trigger notifications: ticker.Price = 182.50m; // All three observers fire // Unsubscribe the file logger: ticker.PriceChanged -= LogToFile;

Why this is better for most C# code: No custom interfaces, no manual lists, no Notify() method. The event keyword does it all. The ?. null-conditional operator ensures that if nobody is subscribed, nothing happens (no NullReferenceException). And the event keyword enforces encapsulation — only the StockTicker class can raise the event, but anyone can subscribe.

C# events work great, but they have a limitation: they're "fire and forget." You can't filter, transform, combine, or throttle events easily. Enter Reactive Extensions (Rx)A library for composing asynchronous and event-based programs using observable sequences. It extends IObservable<T> with LINQ-style operators like Where, Select, Throttle, Merge, Buffer, etc. and the IObservable<T> / IObserver<T> interfaces. These treat event streams as queryable collections. Think of it as LINQ for events.

With IObservable<T>, you can write things like "give me stock price changes, but only when the price goes above $180, and don't bother me more than once every 2 seconds." Try doing that with event EventHandler<T> — you'll end up writing a lot of plumbing code.

using System.Reactive.Linq; using System.Reactive.Subjects; // Subject<T> is both IObservable<T> AND IObserver<T> // It's the Rx equivalent of our Subject class public class StockPriceStream { // ReplaySubject remembers the last value for late subscribers private readonly Subject<StockPrice> _priceSubject = new(); // Expose as IObservable (read-only to outsiders) public IObservable<StockPrice> Prices => _priceSubject.AsObservable(); public void PublishPrice(string symbol, decimal price) { _priceSubject.OnNext(new StockPrice(symbol, price, DateTime.UtcNow)); } public void Complete() => _priceSubject.OnCompleted(); } public record StockPrice(string Symbol, decimal Price, DateTime Timestamp); var stream = new StockPriceStream(); // Basic subscription — same as += stream.Prices.Subscribe( onNext: price => Console.WriteLine($"{price.Symbol}: ${price.Price}"), onError: ex => Console.WriteLine($"Error: {ex.Message}"), onCompleted: () => Console.WriteLine("Stream closed") ); // POWERFUL: filter + transform with LINQ operators // "Only AAPL prices above $180, at most once per 2 seconds" stream.Prices .Where(p => p.Symbol == "AAPL" && p.Price > 180m) .Throttle(TimeSpan.FromSeconds(2)) .Subscribe(price => Console.WriteLine($"🔔 AAPL alert: ${price.Price}")); // Unsubscription is explicit via IDisposable IDisposable subscription = stream.Prices.Subscribe( price => Console.WriteLine($"Logging: {price}")); // Later: unsubscribe subscription.Dispose(); // same as -= in events

When to reach for Rx: When your observer logic involves filtering, combining multiple streams, throttling, debouncing, buffering, or error handling. For simple "something happened, tell people" scenarios, C# events are simpler. Rx shines when the processing of events is complex, not just the delivery.

Problem Statement

Build a notification system for an e-commerce application. When an order is placed, the system must: send an email confirmation to the customer, update the inventory count, notify the warehouse for fulfillment, and log the event for auditing. How a Junior Thinks

"I know what needs to happen when an order is placed. I'll just call each service directly from the OrderService. It's straightforward, and I can see exactly what happens in one place."

Problems

Tight Coupling

OrderService directly depends on 4 concrete services. It knows about emails, inventory, warehouses, and logging. Adding SMS means adding a 5th dependency. Adding loyalty points means a 6th. The constructor grows forever, and OrderService becomes a god class that knows about everything.

Open-Closed Violation

Every new feature requires modifying OrderService.PlaceOrder(). You're changing existing, tested code to add new behavior. That's the opposite of the Open-Closed PrincipleClasses should be open for extension (add new behavior) but closed for modification (don't change existing code). Observer achieves this by letting you add new observers without touching the subject..

Testing Nightmare

To unit test PlaceOrder(), you need to mock 4 services. Adding SMS? Now it's 5 mocks. The test setup grows linearly with each new feature, and tests break whenever you add or remove a service call.

How a Senior Thinks

"The OrderService should do one thing: place orders. It shouldn't know or care about what happens after the order is placed. I'll use an event-driven approach — the OrderService publishes an OrderPlaced notification, and each downstream concern subscribes independently. Adding SMS next quarter? I just write a new handler class and register it. OrderService never changes."

The senior reaches for MediatRA popular .NET library that implements the Mediator and Notification patterns. For Observer, it provides INotification and INotificationHandler<T> — publish a notification, and all registered handlers fire automatically via DI. — a widely-used .NET library that provides in-process publish/subscribe through dependency injection. Each handler is a separate class with a single responsibility.

Design Decisions

Event-Driven Decoupling

OrderService publishes a notification and moves on. It has zero knowledge of what happens downstream. This means the ordering concern is completely isolated from the email/inventory/warehouse/logging concerns.

MediatR for In-Process Pub/Sub

MediatR provides the observer infrastructure via DI. You don't need to manually register handlers — assembly scanning does it for you. Each handler is resolved from the DI container with its own dependencies properly injected.

Each Handler = Single Responsibility

Each handler does exactly one thing. SendEmailConfirmationHandler sends email. UpdateInventoryHandler updates stock. They're easy to test in isolation — mock one dependency, verify one behavior. Adding SMS next quarter means creating one new file: SendSmsHandler.cs. Nothing else changes.

Solving Tight Coupling

The senior uses an event bus (MediatR). OrderService depends on a single interface (IMediator) instead of 4+ concrete services. When SMS is added, the constructor stays the same. The new handler registers itself through DI. The subject never knows about its observers.

Restoring Open-Closed

With MediatR notifications, adding new behavior means adding a new handler class — you never modify OrderService. The system is open for extension (new handlers) and closed for modification (subject code is frozen). Each handler is a self-contained unit that registers itself through assembly scanning.

Fixing the Testing Nightmare

With the senior approach, testing OrderService requires mocking just one thing: IMediator. Verify that Publish() was called with the right notification. Each handler is tested in isolation with its own mocks. Adding SMS doesn't change any existing tests.

Juniors call downstream services directly (tight coupling, OCP violation, testing pain). Seniors publish events and let handlers subscribe independently via MediatR. Adding new reactions = adding new handler files, never modifying the source.
Section 7

Evolution & History

Observer is one of the oldest design patterns — older than the GoF book that popularized it. Its evolution in the .NET ecosystem mirrors C#'s own growth from a basic OOP language into a modern, reactive, event-driven powerhouse.

Observer-Evolution-Timeline

The "Gang of Four" published Design Patterns: Elements of Reusable Object-Oriented Software, which catalogued 23 patterns including Observer. But the idea was much older — Smalltalk's MVCModel-View-Controller, a UI architecture from the 1970s. The Model is the subject (data), Views are observers (UI). When the Model changes, Views update automatically. MVC was the first mainstream use of Observer. architecture from the late 1970s used Observer as its core mechanism for keeping views in sync with data. The GoF just gave it a formal name and structure.

// The original GoF definition (adapted to C#-like pseudocode) // This is CONCEPTUAL — C# didn't exist yet in 1994 interface ISubject { void Attach(IObserver observer); void Detach(IObserver observer); void Notify(); } interface IObserver { void Update(ISubject subject); }

When C# 1.0 shipped, it baked Observer directly into the language with delegate and the event keyword. This was a deliberate design choice by Anders Hejlsberg (C#'s creator) — events are so fundamental to UI programming that they deserved first-class language support. You no longer needed to write the GoF boilerplate. The compiler handled the observer list (multicast delegate) for you.

Section 10

Comparisons

Observer often gets confused with patterns that also deal with "notifying things." The differences come down to how the notification travels and what kind of coupling exists between sender and receiver. Let's break it down.

Observer vs Mediator

Both decouple objects that communicate, but they do it in opposite ways. Observer uses direct connections from subject to observers. Mediator forces all communication through a central hub. Think of Observer as a radio tower broadcasting to anyone with a receiver, while Mediator is like a call center routing every conversation.

When to pick which: Use Observer when you need simple "fire-and-forget" broadcast with no coordination. Use Mediator when objects need to interact in complex ways that require orchestration (like a chat room where messages need to be routed to specific users, not broadcast to all).

Observer vs Pub/Sub

Pub/Sub is essentially Observer evolved for distributed systems. In classic Observer, the subject holds direct references to its observers — they live in the same process, same memory space. In Pub/Sub, a message brokerA message broker (RabbitMQ, Azure Service Bus, Apache Kafka, Redis Pub/Sub) is infrastructure that sits between publishers and subscribers. Publishers send messages to the broker; the broker delivers them to subscribers. Publisher and subscriber never communicate directly. sits between publishers and subscribers, decoupling them completely.

When to pick which: Observer is perfect for in-process event handling (UI events, domain events within a single service). Pub/Sub is for distributed systems where publisher and subscriber might be different services, different machines, or even different programming languages.

Observer vs Event Sourcing

These two both involve "events," but they solve completely different problems. Observer is about notifying when something changes. Event Sourcing is about storing every change as a permanent record. You can (and often do) combine them: use Event Sourcing to persist the event, then use Observer to notify interested parties about it.

When to pick which: Use Observer alone when you just need notification (email on order placed). Use Event Sourcing when you need a complete audit trail or the ability to reconstruct past state. Combine them when you want both: persist the event (Event Sourcing) and notify subscribers (Observer).

Observer broadcasts directly; Pub/Sub routes through a broker for distributed systems; Mediator orchestrates through a central hub. Observer and Event Sourcing solve different problems (notification vs persistence) and are often combined.

Observer is a poster child for SOLID principles. It scores "Supports" on all five — a rare achievement for any design pattern. That's not a coincidence; it's because Observer was designed around the core idea of decoupling, which is exactly what SOLID optimizes for.

The Incident

Production server memory climbed 2GB over 3 days. A dashboard service subscribed to a real-time price feed (the subject) every time a user opened the dashboard page. Each page load created a new observer and called Subscribe().

But when users navigated away, nobody called Unsubscribe(). The price feed subject held strong references to every observer ever created. The garbage collector couldn't collect them because the subject's observer list kept them alive.

Over 72 hours, thousands of dead observers accumulated in memory. Each one held references to its own dependencies (HTTP clients, database connections, cached data). The server eventually hit its memory limit and started throwing OutOfMemoryException.

Time to Diagnose

4 hours. The memory profiler showed thousands of DashboardObserver instances that were never garbage collected. The team traced the reference chain: PriceFeedSubject._observers list held strong references to every observer ever created. The fix was straightforward once identified, but the diagnosis was slow because memory leaks don't crash immediately — they silently grow until things fall apart.

public class DashboardService { private readonly PriceFeedSubject _priceFeed; public DashboardService(PriceFeedSubject priceFeed) { _priceFeed = priceFeed; } public void OnPageLoad(string userId) { var observer = new DashboardObserver(userId); _priceFeed.Subscribe(observer); // ❌ Subscribes but NEVER unsubscribes // When user navigates away, observer stays in Subject's list // GC can't collect it because Subject holds a strong reference } // No OnPageUnload, no Dispose, no cleanup }

Walking through the buggy code: Every time a user opens the dashboard, OnPageLoad creates a brand-new observer and subscribes it to the price feed. When the user leaves the page, nothing happens. The observer is still in the subject's _observers list. The next visit? Another observer is added. After a few thousand visits, the list has thousands of entries, each holding references to HTTP clients and cached data. Memory grows and grows.

Why the fix works: The subject's Subscribe() method now returns an IDisposable token. When you dispose it, the observer is removed from the subject's list. We store this token and dispose it when the page unloads or when the service is disposed. No dangling references, no memory leak. Lesson Learned

Always return an IDisposable from Subscribe(). This gives callers a clean way to unsubscribe. In .NET, the IObservable<T> interface enforces this pattern by design — Subscribe() returns IDisposable. If you're building your own observer, follow the same convention.
How to Spot This in Your Code

Search for Subscribe( or += (event handler attachment) in your codebase. For each one, look for a matching Unsubscribe, Dispose(), or -=. Customers receiving duplicate emails for every order confirmation. The notification service attached its handler to the OrderPlaced event inside a method that was called on every request. The += operator doesn't check for duplicates — it happily adds the same handler again.

After the first request, the handler fires once. After the second request, it fires twice. After 10 requests, it fires 10 times. Each fire sends an email. Customers received 10 identical order confirmations.

The support team was flooded with complaints about duplicate emails. The ops team initially blamed the email service, but the root cause was in how the event handler was attached.

Time to Diagnose

1 hour. Once the team looked at the GetInvocationList() count on the event delegate, they saw it growing with every request. The += was inside a per-request scope but the event was on a singleton. Classic setup/teardown mismatch.

public class NotificationService { public void Initialize(OrderService orderService) { // ❌ Called on every request (scoped service) // but OrderService is a SINGLETON orderService.OrderPlaced += OnOrderPlaced; // After 10 requests: handler fires 10 times per event! } private void OnOrderPlaced(object? sender, OrderEventArgs e) { SendEmail(e.CustomerEmail, "Order confirmed!"); } }

Walking through the buggy code: NotificationService is scoped (created per request), but OrderService is a singleton. Every request creates a new NotificationService, which attaches its handler via +=. The singleton's event delegate list grows by one handler per request. When the event fires, every attached handler executes — sending duplicate emails.

Why the fix works: The "-= then +=" pattern ensures only one copy of the handler exists. Even better: implement IDisposable to unsubscribe when the scoped service is disposed. Lesson Learned

C#'s += operator does not check for duplicates. If you subscribe the same handler twice, it fires twice. Always use the "-= then +=" pattern, or better yet, use a HashSet<IObserver> to prevent duplicate subscriptions by design.
How to Spot This in Your Code

Search for += in your codebase. For each one, check whether it runs inside a method that's called multiple times (like a constructor of a scoped service). If a scoped service attaches to a singleton's event, you almost certainly have this bug. Also check event.GetInvocationList().Length at runtime — if it grows over time, you have duplicate registrations.

Pitfalls & Anti-Patterns

Mistake: Subscribing to events with += or Subscribe() but never calling -= or Dispose(). The subject holds a strong reference to the observer, keeping it alive in memory long after it should have been garbage collected.

Why This Happens: Subscribing is exciting — you're hooking up functionality! Unsubscribing is boring cleanup that feels optional. But in .NET, event handlers create strong references from the event source to the handler's target object. If the source outlives the handler (a singleton event with scoped subscribers), the subscribers become zombies — dead objects that can't be collected.

Think of it like signing up for a gym membership but never canceling. The gym (subject) keeps your file (reference) forever, even though you haven't visited in years. Your file takes up space in their cabinet (memory), and they can't throw it away because your membership is still "active."

// ❌ BAD: Subscribe without unsubscribe public class DashboardWidget { public DashboardWidget(PriceFeed feed) { feed.PriceChanged += OnPriceChanged; // Subscribes but never unsubscribes } // No IDisposable, no destructor, no cleanup } // ✅ GOOD: Always unsubscribe in Dispose public class DashboardWidget : IDisposable { private readonly PriceFeed _feed; public DashboardWidget(PriceFeed feed) { _feed = feed; _feed.PriceChanged += OnPriceChanged; } public void Dispose() => _feed.PriceChanged -= OnPriceChanged; }

Fix: Implement IDisposable on every observer. Unsubscribe in Dispose(). If using IObservable<T>, store the IDisposable token from Subscribe() and dispose it. Consider WeakReference<T> for observer lists if the subject is long-lived.

Mistake: An observer's Update() method calls Subscribe() or Unsubscribe() on the subject, modifying the observer list while the subject is iterating over it.

Why This Happens: An observer decides during its Update() that it no longer needs notifications and tries to unsubscribe itself. Or it registers a new observer as part of its reaction logic. Both modify the list that the subject is currently iterating, causing InvalidOperationException or skipped/duplicate notifications.

Imagine a teacher calling names from a class roster while a student sneaks in and adds their name to the middle of the list. The teacher loses their place, skips a name, or reads one twice.

// ❌ BAD: Observer modifies the list during notification public class AutoUnsubscribeObserver : IObserver { public void Update(ISubject subject) { ProcessNotification(); subject.Unsubscribe(this); // ❌ Modifies list during foreach! } } // ✅ GOOD: Snapshot the list before iterating public void Notify() { var snapshot = _observers.ToList(); // Copy the list foreach (var obs in snapshot) obs.Update(this); // Safe: iterating a separate copy } // Observers can freely Subscribe/Unsubscribe during Update()

Fix: Always snapshot the observer list before iterating. Use _observers.ToList() or _observers.ToArray() to create a copy, then iterate the copy. Modifications to the original list won't affect the snapshot. Alternatively, collect subscribe/unsubscribe requests during notification and process them after the loop finishes.

Mistake: Stuffing the entire domain object (or the entire subject) into the event args. Every observer receives the full object graph whether it needs it or not.

Why This Happens: It seems convenient — "just pass the whole order, observers can pick what they need." But this creates tight coupling between observers and the subject's internal structure. If the Order model changes (adds a field, renames a property), every observer might need to be updated. It also means observers have access to data they shouldn't — violating the principle of least privilege.

Imagine sending someone your entire filing cabinet when they just asked for one document. It's wasteful, it's a privacy risk, and they have to dig through everything to find what they need.

// ❌ BAD: Passing the entire domain object public class OrderPlacedEvent { public Order Order { get; set; } // Contains customer PII, payment details, etc. // Every observer gets access to sensitive data it doesn't need } // ✅ GOOD: Focused event args with only what observers need public record OrderPlacedEvent( Guid OrderId, string CustomerEmail, // Only what email observer needs decimal TotalAmount, // Only what analytics needs List<Guid> ProductIds // Only what inventory needs );

Fix: Create focused event arg classes that contain only the data observers actually need. Use immutable recordsC# records are immutable by default — once created, their properties can't be changed. This is perfect for event args: the data is set when the event is created and no observer can modify it, preventing accidental side effects. to prevent observers from modifying the data. If different observers need different data, consider multiple specific events rather than one bloated event.

Mistake: Calling all observers synchronously in the subject's thread. If one observer takes 2 seconds (HTTP call, database write), the subject is blocked for 2 seconds before it can continue processing.

Why This Happens: The simplest Observer implementation is a foreach loop that calls Update() on each observer. This is synchronous — the loop waits for each observer to finish before moving to the next. With fast observers (in-memory updates), this is fine. With slow observers (sending emails, calling APIs), it becomes a bottleneck.

Imagine a news anchor who can't read the next story until every viewer has finished reacting to the current one. The broadcast grinds to a halt.

// ❌ BAD: Synchronous notification — each observer blocks the subject public void Notify() { foreach (var obs in _observers) obs.Update(this); // If this takes 2 seconds, subject waits 2 seconds } // 5 observers x 2 seconds each = 10 seconds of blocking! // ✅ GOOD: Async notification — all observers run concurrently public async Task NotifyAsync() { var tasks = _observers.Select(obs => Task.Run(() => obs.Update(this))); await Task.WhenAll(tasks); // All observers run in parallel } // ✅ BEST: Fire-and-forget for non-critical observers public void Notify() { foreach (var obs in _observers) _ = Task.Run(() => obs.Update(this)); // Fire and forget // Subject continues immediately }

Fix: Use Task.WhenAll() for parallel notification when you need to wait for all observers. Use fire-and-forget (Task.Run() without await) for non-critical observers like analytics or logging. For production systems, use a proper background queue (System.Threading.Channels or a message broker) to decouple notification from the subject's thread entirely.

Mistake: A single subject with 15 different events: OrderPlaced, OrderShipped, OrderCancelled, OrderRefunded, CustomerRegistered, CustomerUpdated, InventoryChanged... all on one class.

Why This Happens: It starts small: "OrderService needs an event for order creation." Then someone adds OrderShipped, then OrderCancelled. Then customer events get added because "OrderService already handles customers." Before you know it, one class is the event source for half the application. This is the God classA God class is one that knows too much or does too much. It violates SRP by accumulating unrelated responsibilities. In Observer, a God Subject is one that broadcasts unrelated events, making every observer depend on it even if they only care about one event type. anti-pattern applied to Observer.

Imagine a radio station that broadcasts news, music, sports, cooking shows, and kids' programs on a single frequency. Everyone tunes in and has to filter out the 95% they don't care about.

// ❌ BAD: One subject broadcasts everything public class AppEventHub { public event EventHandler<OrderPlacedEventArgs> OrderPlaced; public event EventHandler<OrderShippedEventArgs> OrderShipped; public event EventHandler<CustomerRegisteredEventArgs> CustomerRegistered; public event EventHandler<InventoryChangedEventArgs> InventoryChanged; // ... 15 more events — single class, single dependency } // ✅ GOOD: Focused subjects per domain concern public class OrderEvents { public event EventHandler<OrderPlacedEventArgs> OrderPlaced; public event EventHandler<OrderShippedEventArgs> OrderShipped; } public class CustomerEvents { public event EventHandler<CustomerRegisteredEventArgs> Registered; } // Or use MediatR INotification for fully decoupled events

Fix: Split your God Subject into focused subjects by domain area. Each subject handles events for one aggregate or bounded context. Alternatively, use an event bus (MediatR INotification) where events are dispatched by type, not by which object raised them. This eliminates the subject class entirely.

Mistake: Declaring an event as a plain Action or EventHandler field instead of using the C# event keyword. This allows external code to invoke the event directly or replace all handlers.

Why This Happens: A developer declares public Action<OrderEventArgs> OrderPlaced; thinking it's equivalent to public event Action<OrderEventArgs> OrderPlaced;. Without the event keyword, the field is a regular delegate. Anyone with access can call OrderPlaced.Invoke() or even set OrderPlaced = null; — wiping out all subscribers.

The event keyword is like a bouncer at a club. It only lets you do two things: add yourself to the list (+=) and remove yourself (-=). Without the bouncer, anyone can walk behind the counter and press the alarm button or tear up the entire guest list.

// ❌ BAD: No event keyword — external code can invoke or wipe handlers public class OrderService { public Action<OrderEventArgs>? OrderPlaced; // Plain delegate field } // External code can do: orderService.OrderPlaced?.Invoke(args); // ❌ Anyone can fire the event orderService.OrderPlaced = null; // ❌ Wipes all handlers! // ✅ GOOD: event keyword restricts to += and -= only public class OrderService { public event Action<OrderEventArgs>? OrderPlaced; // event keyword added public void PlaceOrder(Order order) { // ... process order OrderPlaced?.Invoke(new OrderEventArgs(order.Id)); // Only this class can invoke } } // External code can only: orderService.OrderPlaced += handler; // External code CANNOT: orderService.OrderPlaced?.Invoke(args); // compile error!

Fix: Always use the event keyword when declaring events. The compiler enforces that only the declaring class can invoke the event. External code is restricted to += (subscribe) and -= (unsubscribe). This is a compile-time safety net that costs nothing.

Testing Strategies

Testing Observer has a beautiful property: because subjects and observers are decoupled, you can test each side independently. The subject doesn't know its observers are mocks, and the observers don't need a real subject. Here's the testing hierarchy — start from the top and work down.

The most fundamental test: when the subject's state changes, does it actually call Update() on its observers? Register a mock observerA mock is a test double that records which methods were called on it. Libraries like Moq or NSubstitute create mocks automatically. You can then assert: 'Was Update() called? How many times? With what arguments?', change the subject's state, and verify the mock was called with the correct arguments.

[Fact] public void Subject_NotifiesObserver_WhenStateChanges() { // Arrange var subject = new OrderSubject(); var mockObserver = new Mock<IObserver<OrderPlacedEvent>>(); subject.Subscribe(mockObserver.Object); // Act subject.PlaceOrder(new Order { Id = Guid.NewGuid(), Total = 99.99m }); // Assert — observer's OnNext was called exactly once with correct data mockObserver.Verify( o => o.OnNext(It.Is<OrderPlacedEvent>(e => e.Total == 99.99m)), Times.Once); }

Verify that subscribing actually registers the observer and unsubscribing removes it. After unsubscribing, state changes should not trigger the observer. This catches the memory leak bug (Bug 1) early.

[Fact] public void Observer_DoesNotReceiveNotification_AfterUnsubscribe() { // Arrange var subject = new PriceFeedSubject(); var mockObserver = new Mock<IObserver<PriceUpdate>>(); var subscription = subject.Subscribe(mockObserver.Object); // Act 1: Verify notification works subject.UpdatePrice("AAPL", 150.0m); mockObserver.Verify(o => o.OnNext(It.IsAny<PriceUpdate>()), Times.Once); // Act 2: Unsubscribe and verify silence subscription.Dispose(); // Unsubscribe subject.UpdatePrice("AAPL", 155.0m); // Assert — still only called once (from Act 1), not twice mockObserver.Verify(o => o.OnNext(It.IsAny<PriceUpdate>()), Times.Once); }

Register three observers. Change the subject's state. Assert that all three received the notification. This catches the swallowed exception bug (Bug 4) where one observer failing prevents others from being notified.

[Fact] public void AllObservers_ReceiveNotification_WhenStateChanges() { // Arrange var subject = new OrderSubject(); var observers = Enumerable.Range(0, 3) .Select(_ => new Mock<IObserver<OrderPlacedEvent>>()) .ToList(); foreach (var mock in observers) subject.Subscribe(mock.Object); // Act subject.PlaceOrder(new Order { Id = Guid.NewGuid(), Total = 50m }); // Assert — ALL three observers were called foreach (var mock in observers) mock.Verify(o => o.OnNext(It.IsAny<OrderPlacedEvent>()), Times.Once); }

This is the critical resilience test. Register three observers where the first one throws an exception. Assert that the other two still receive their notification. If this test fails, you have the Bug 4 problem — one misbehaving observer kills the entire chain.

[Fact] public void OtherObservers_StillNotified_WhenOneThrows() { // Arrange var subject = new OrderSubject(); var throwingObserver = new Mock<IObserver<OrderPlacedEvent>>(); throwingObserver .Setup(o => o.OnNext(It.IsAny<OrderPlacedEvent>())) .Throws<SmtpException>(); // Simulate email failure var healthyObserver1 = new Mock<IObserver<OrderPlacedEvent>>(); var healthyObserver2 = new Mock<IObserver<OrderPlacedEvent>>(); subject.Subscribe(throwingObserver.Object); subject.Subscribe(healthyObserver1.Object); subject.Subscribe(healthyObserver2.Object); // Act subject.PlaceOrder(new Order { Id = Guid.NewGuid(), Total = 100m }); // Assert — healthy observers STILL got notified despite the throwing one healthyObserver1.Verify(o => o.OnNext(It.IsAny<OrderPlacedEvent>()), Times.Once); healthyObserver2.Verify(o => o.OnNext(It.IsAny<OrderPlacedEvent>()), Times.Once); }

Good news: Observer is one of the cheapest patterns to use. The overhead is essentially a list iteration plus a method call per observer. Here's what each piece actually costs so you can make informed decisions:

Delegate invocationWhen you invoke a C# event or delegate, the runtime performs a virtual dispatch to each attached handler. For events with 1-10 handlers, this takes roughly 5-15 nanoseconds total. For comparison, a single HTTP call takes 50-500 milliseconds — literally millions of times slower. ~5 ns per subscriber Negligible. Even with 100 observers, you're spending 500ns total — less than a single cache miss.
List iteration for notifications O(n) where n = observers Linear in observer count. With 10-50 observers (typical), this is sub-microsecond. Only a concern if you have 10,000+ observers.
Memory per subscription ~32-64 bytes per observer reference Trivial. 1,000 observers = ~64KB of references. The observers' own state (HTTP clients, caches) is the real cost, not the subscription.

Q2: What's the difference between Push and Pull models in Observer?
Easy
Think First Think about a news alert on your phone. Does the notification include the full article, or just a headline with a "tap to read more" link?

There are two ways the Subject can share data with observers when something changes:

Push model: The Subject sends ALL the relevant data to observers in the notification itself. "Hey, the stock price is now $142.50, volume is 2.3M, change is +3.2%." The observer gets everything it needs without asking follow-up questions. In C#, this is how event EventHandler<StockPriceChangedArgs> works — the EventArgs carries all the data.

Pull model: The Subject just says "Something changed." The observer then calls back to the Subject to ask "Okay, what changed exactly?" and grabs only the data it needs. This is more efficient when different observers care about different parts of the data, but it means the observer needs a reference back to the Subject.

Which is better? It depends. Push is simpler and avoids the observer needing a reference to the Subject. But if the Subject has 20 data fields and an observer only cares about one, Push wastes bandwidth sending 19 unnecessary fields. Pull lets each observer be selective. In practice, most C# code uses Push (EventArgs) because it's cleaner and the overhead of a few extra fields is negligible.

Easy

But here's the crucial detail: C# delegates are multicast by default. That means a single delegate variable can hold references to MULTIPLE methods, not just one.

When you write PriceChanged += OnPriceChanged, you're calling Delegate.Combine() under the hood. This creates a NEW delegate that wraps both the old invocation list and the new method. The += operator creates an entirely new delegate with the combined list. This is why delegate operations are naturally thread-safe for reads — you always see a consistent snapshot.).

When the event fires with PriceChanged?.Invoke(this, args), the runtime walks through the invocation list and calls each method in FIFO order (the order they were added). Memory layout: Each entry in the invocation list stores two things: a Target (the object instance the method belongs to, or null for static methods) and a MethodInfo (which method to call). That Target reference is exactly why event subscriptions can cause memory leaks — the delegate keeps the target object alive.

Q11: How do you prevent memory leaks with Observer?
Medium
Think First If an observer subscribes with += but never unsubscribes with -=, what happens to the observer when it should be garbage collected?

This is the #1 bug with Observer in C#, and it trips up experienced developers too. Here's the problem: when an observer subscribes to an event using +=, the Subject's event delegate holds a strong reference to the observer. Even if all other references to the observer are gone, the garbage collectorThe .NET garbage collector (GC) automatically frees memory by collecting objects that have no references pointing to them. But if a Subject's event delegate holds a reference to an observer, the GC considers that observer "still in use" — even if your code is done with it. The observer stays in memory until the Subject itself is collected. can't collect it because the Subject still "knows about it."

Imagine a web app where every HTTP request creates a new request handler that subscribes to a global event. If those handlers never unsubscribe, you get one leaked object per request. At 1000 requests/second, that's 86 million leaked objects per day.

The three main defenses:

1. IDisposable pattern — The most reliable. Make the observer implement IDisposable and unsubscribe in Dispose(). Use using blocks or DI container lifetime management to guarantee cleanup.

2. WeakReference<T> — The Subject holds weak references to observers instead of strong ones. If the observer has no other references, the GC can collect it. The downside: you have to clean dead references from the list periodically.

3. IObservable<T> subscription tokens — Subscribe() returns an IDisposable that automatically unsubscribes when disposed. This is the cleanest model: using var sub = observable.Subscribe(observer); — when the scope ends, the subscription ends.

Great Answer Bonus "The best defense is structural: make Subscribe() return IDisposable so the caller controls the lifetime. That way, the observer's lifetime is tied to a using block or DI scope — you can't forget to unsubscribe."

Q15: How do you prevent infinite notification loops?
Medium

Infinite loops are the scariest bug in Observer systems because they're hard to predict and can crash your app instantly. They happen when a notification triggers another notification that eventually circles back to the original source. It's like standing between two mirrors — the reflections go on forever.

Here's how it plays out in real code: a PriceService updates the price, which notifies a TaxCalculator, which updates the tax amount, which notifies an InvoiceGenerator, which recalculates the total, which triggers PriceService to re-emit the price... and your stack overflows.

1. Guard flag (re-entrancy lock). The simplest fix. Before notifying, set a boolean flag. If the notification chain circles back, check the flag and bail out. Reset it after notification completes.

2. Depth counter. Instead of a binary flag, count how deep the notification chain is. Allow up to N levels (say 3), then stop. This lets some re-entrancy while preventing infinite loops.

3. Batch/defer updates. Instead of notifying immediately when state changes, queue the notification and process it after the current notification chain finishes. WPF's DispatcherWPF's mechanism for scheduling work on the UI thread. Dispatcher.BeginInvoke queues work to run after the current operation completes. This naturally breaks notification cycles because updates are deferred, not immediate. does this — changes are queued and processed on the next UI tick.

Q16: What's the difference between hot and cold observables in Rx?
Medium

This distinction is crucial when working with Reactive Extensions (Rx)A library for composing asynchronous, event-based programs using observable sequences. It treats event streams like LINQ-queryable collections — you can filter, transform, combine, and throttle them with operators., and getting it wrong leads to subtle bugs.

Cold observable = Netflix. Each subscriber gets their own independent stream that starts from the beginning. The data is produced ON DEMAND, just for that subscriber. If two people subscribe, the work happens twice. Example: reading a file, making an HTTP request, generating a sequence with Observable.Range(1, 10).

Hot observable = live TV. The stream is producing data whether anyone is subscribed or not. When you subscribe, you jump in wherever the stream currently is — you missed everything before you tuned in. All subscribers share the same stream. Example: mouse movements, stock price ticks, sensor readings, a Subject<T>.

Why this matters: If you treat a cold observable as hot, you might think two subscribers share data — but they don't (each triggers a separate HTTP request). The fix is to convert between them: .Publish().RefCount() makes cold observables hot (share one stream among all subscribers), and .Replay() makes hot observables behave cold-ish (replays missed events to late subscribers).

Great Answer Bonus "Cold = data produced per subscriber (like an HTTP call). Hot = data produced regardless (like mouse events). Use .Publish().RefCount() to share a cold observable among multiple subscribers, and .Replay(n) to buffer recent events for late joiners on a hot observable."
Q17: How does IObservable<T> handle backpressure?
Medium
Think First A sensor emits 10,000 readings per second. Your observer can only process 100 per second. What happens to the other 9,900?

BackpressureThe problem that arises when a data producer generates data faster than the consumer can process it. Without backpressure handling, data either piles up in memory (buffer overflow), gets dropped (data loss), or the system crashes (OOM). is what happens when the producer is faster than the consumer. Imagine a firehose pointed at a drinking glass — the glass overflows. In software, the "overflow" is either memory exhaustion (buffering all the excess data) or dropped messages.

The honest answer: IObservable<T> doesn't have built-in backpressure. Unlike Java's Reactive Streams or Project Reactor, .NET's IObservable<T> is a push-only model — the producer pushes data without asking if the consumer is ready. This is by design — it keeps the interface simple. But it means YOU need to handle backpressure explicitly using Rx operators.

Rx strategies for managing fast producers:

1. Throttle / Sample — Drop excess items. .Throttle(TimeSpan.FromMilliseconds(100)) only emits the last item in each 100ms window. Good when you want "latest value" (UI updates, sensor displays).

2. Buffer — Collect items into batches. .Buffer(TimeSpan.FromSeconds(1)) collects all items in a 1-second window and delivers them as a list. Good for batch processing (database writes, analytics).

3. Window — Like Buffer but returns an observable of observables. Each window is itself a stream you can process concurrently. Good for parallel processing.

4. ObserveOn with bounded queue — Move processing to another thread with a bounded buffer. When the buffer fills up, the producer blocks (if synchronous) or items get dropped.

5. Channel<T> bridge — For true backpressure, bridge to a Channel<T> with a bounded capacity and BoundedChannelFullMode.Wait. This physically slows the producer when the consumer can't keep up.

Great Answer Bonus "IObservable<T> is push-only with no built-in backpressure. Use Rx operators to manage fast producers: Throttle for latest-value, Buffer for batching, or bridge to Channel<T> with bounded capacity for true producer-slowing backpressure."

Q19: When should you use Observer vs a message queue (RabbitMQ, Kafka)?
Medium

This is a scale decision, not a "right vs wrong" decision. Observer and message queues solve the same fundamental problem — decoupling producers from consumers — but at vastly different scales and with different guarantees.

Use Observer (in-process) when:

  • Producer and consumers live in the same process (same ASP.NET app, same microservice).
  • You need nanosecond latency — function calls are as fast as it gets.
  • Losing a notification is acceptable — if the process crashes, in-flight events are gone.
  • You have a small number of consumers (under ~100).
  • Example: UI events, domain events within a single bounded context, real-time in-app notifications.

  • Producer and consumers are in different services, potentially on different machines.
  • You need guaranteed delivery — the message persists even if consumers are down.
  • You need replay — new consumers should see historical events (Kafka's killer feature).
  • You need to scale consumers independently — add more Kafka consumers without touching the producer.
  • Example: order placed event consumed by inventory, shipping, analytics, all as separate services.

The gray area: MediatR. MediatR sits in between. It's in-process (like Observer) but provides Pub/Sub decoupling (like a queue). Use MediatR when you want the decoupling of a queue but don't need persistence, replay, or cross-process delivery.

Q22: How would you make Observer thread-safe?
Hard
Think First What happens if Thread A is iterating the observer list to notify, and Thread B adds a new observer at the same time? What about Thread C removing an observer?

Thread safety in Observer is tricky because three things can happen concurrently: subscribing (adding to the list), unsubscribing (removing from the list), and notifying (iterating the list). If two threads modify the list while a third is iterating it, you get InvalidOperationException: Collection was modified — or worse, silent data corruption.

Here are the main strategies, each with different trade-offs:

1. Lock + snapshot iteration — The simplest approach. Lock when modifying the list, and during notification, take a snapshot (copy) of the list before iterating. This way, modifications during notification don't affect the current iteration. The downside: copying the list on every notification has O(n) overhead.

2. ConcurrentBag<T> / ImmutableList<T> — Use a thread-safe collection. ImmutableList<T> is ideal because it creates a new list on every add/remove, so existing iterations see the old version automatically (copy-on-write semantics). Zero locking needed during notification.

3. ReaderWriterLockSlim — Allows multiple threads to read (iterate) concurrently, but only one thread to write (add/remove) at a time. Better throughput than a plain lock when notifications vastly outnumber subscribe/unsubscribe calls.

4. Channel<T> for async notification — Instead of calling observers synchronously, write the notification to a Channel<T> and let each observer read from its own channel asynchronously. This fully decouples the notification from processing and avoids all locking issues — but adds latency.

// Thread-safe Observer using ImmutableList (copy-on-write) using System.Collections.Immutable; public sealed class ThreadSafeSubject<T> : IObservable<T> { // Volatile + Interlocked for lock-free updates private volatile ImmutableList<IObserver<T>> _observers = ImmutableList<IObserver<T>>.Empty; public IDisposable Subscribe(IObserver<T> observer) { // Atomic add — creates a new list, old iterations unaffected ImmutableInterlocked.Update(ref _observers, list => list.Add(observer)); return new Unsubscriber(() => ImmutableInterlocked.Update(ref _observers, list => list.Remove(observer))); } public void Publish(T value) { // Snapshot: grab the reference. Even if _observers changes mid-loop, // we iterate the version we captured — no locks, no exceptions. var snapshot = _observers; foreach (var observer in snapshot) { try { observer.OnNext(value); } catch (Exception ex) { /* log, don't let one observer break others */ } } } private sealed class Unsubscriber(Action unsubscribe) : IDisposable { public void Dispose() => unsubscribe(); } }
Great Answer Bonus "The key insight is snapshot iteration: capture the observer list reference before iterating. With ImmutableList, every add/remove creates a new list — so existing iterations always see a consistent snapshot with zero locking."
Q23: What are the limitations of Observer and when would you choose a different pattern?
Hard
Think First What happens when you have 10,000 observers? What if notification order matters? What if one observer throws an exception — do the rest still get notified?

Observer is powerful, but it has real limits that you need to understand before choosing it:

1. Scalability limits. Observer is in-process. If you have 10,000 observers on a single Subject, every notification iterates through all of them synchronously (unless you add async). At some point, notification latency becomes unacceptable. When you need to scale beyond one machine, switch to a message queue (RabbitMQ, Kafka).

2. Notification ordering. Observers are called in subscription order, but there's no guarantee that Observer A finishes before Observer B starts (especially with async). If ordering matters, you might need Chain of Responsibility (sequential, each handler decides whether to pass along) or a prioritized message queue.

3. Error propagation. If Observer B throws an exception during notification, Observers C through Z never get notified. You must wrap each call in try/catch — but then errors are silently swallowed. For robust error handling, switch to event sourcing where events are persisted first, then processed independently (a failure can be retried).

4. Debugging complexity. With events firing across the codebase, it's hard to trace "who caused this update?" Call stacks become fragmented. A Mediator centralizes communication, making the flow easier to trace.

5. Circular notification. Observer A triggers Subject B, which notifies Observer C, which triggers Subject A again. Infinite loop. The pattern has no built-in protection against this — you need to add cycle detection yourself.

Q29: How do you handle observer ordering guarantees when it matters?
Hard
Think First An order is placed. The validation observer MUST run before the inventory observer (no point deducting stock for an invalid order). And the audit logger MUST run last (it needs to log the final state). How do you enforce this?

Standard Observer gives you no ordering guarantees beyond "subscription order." C# events call handlers in the order they were added with +=, but relying on subscription order is fragile — it depends on initialization order, which can change during refactoring.

When ordering truly matters, you have several approaches:

1. Priority-based notification. Give each observer a priority number. The subject sorts the observer list by priority before notifying. This is explicit and clear — each observer declares its execution order.

2. Pipeline (Chain of Responsibility). Instead of broadcasting to all observers simultaneously, chain them: Observer A runs first, then passes the event to Observer B, which passes to Observer C. Each step can modify, enrich, or even veto the event. This is no longer pure Observer — it's a hybrid with Chain of Responsibility.

3. Phase-based notification. Split notification into phases: validation phase, processing phase, post-processing phase. The subject notifies phase by phase. Observers register for specific phases. This is how ASP.NET Core's middleware pipeline works — each request goes through middleware in a defined order.

4. MediatR Pipeline Behaviors. In .NET, MediatR supports IPipelineBehavior<T> — middleware that wraps around notification handlers. You can add validation, logging, or transaction behaviors that always execute in a defined order around the actual handlers.

    IObservable<T> is Observer with superpowers. Rx.NET wraps it with LINQ-style operators — .Where(), .Select(), .Buffer(), .Throttle(), .CombineLatest() — so you can compose observable streams the same way you compose database queries with LINQ. The result is declarative, readable pipelines that handle complex async data flows in a few lines.

    Think of it this way: LINQ lets you write products.Where(p => p.Price > 10).Select(p => p.Name) to query a collection. Rx lets you write the exact same thing over a stream of events over time. Mouse moves, WebSocket messages, sensor readings, stock ticks — they all become composable streams.

    When to Use Rx vs Plain Events

    Use plain event when you have a simple notification (button clicked, property changed). Use Rx when you need to compose, filter, combine, or throttle streams of data. The crossover point: if you find yourself adding timers, buffers, or complex filtering logic around events, Rx will make it cleaner.

    Strong references in events prevent the garbage collector from cleaning up observers. This is the root cause of the most common Observer bug: memory leaks. Let's understand exactly why this happens and what tools .NET gives you to fight it.

    When you write subject.SomeEvent += observer.Handler, the event's delegate stores a strong reference to observer. Even if you null out every other reference to that observer, the GC sees the event reference and thinks "someone still needs this object." The observer stays in memory for the lifetime of the Subject — which might be the entire application lifetime for singletons.

    WeakEventManager (WPF/MAUI) solves this by holding WeakReference<T> to observers instead of strong references. If the observer has no other references, the GC can collect it, and the next time the event fires, the dead reference is cleaned up automatically.

    // --- Problem: strong reference prevents GC --- public class LongLivedService { public event EventHandler<string>? For most apps this is fine, but for high-frequency financial systems, prefer explicit IDisposable cleanup.

    ChatRoom is now decoupled from User — it has zero knowledge of who subscribes. Adding the ChatLogger required zero changes to ChatRoom or User. But this approach is synchronous, and filtering (like "mute a user") requires logic inside each observer's handler.

    The ChatRoom exposes IObservable<ChatMessage>. Users subscribe with custom Rx pipelines — filtering, throttling, replaying. A user can mute another user without any changes to ChatRoom. Late joiners can get the last N messages automatically with .Replay().

    using System.Reactive.Linq; using System.Reactive.Subjects; public record ChatMessage(string Sender, string Text, DateTimeOffset SentAt); // Subject — exposes IObservable for composable subscriptions public sealed class ChatRoom : IDisposable { private readonly ReplaySubject<ChatMessage> _messages; public ChatRoom(int replayCount = 10) { // ReplaySubject caches last N messages for late joiners _messages = new ReplaySubject<ChatMessage>(replayCount); } public IObservable<ChatMessage> Messages => _messages.AsObservable(); public void SendMessage(string sender, string text) => _messages.OnNext(new ChatMessage(sender, text, DateTimeOffset.UtcNow)); public void Close() { _messages.OnCompleted(); // Signal: room is closed, no more messages } public void Dispose() => _messages.Dispose(); } // User with mute list — filtering happens in the Rx pipeline, NOT in ChatRoom public sealed class User : IDisposable { public string Name { get; } private readonly HashSet<string> _mutedUsers = new(); private IDisposable? _subscription; public User(string name) => Name = name; public void JoinRoom(ChatRoom room) { _subscription = room.Messages .Where(m => m.Sender != Name) // Skip own messages .Where(m => !_mutedUsers.Contains(m.Sender)) // Skip muted users .Subscribe( onNext: m => Console.WriteLine($"[{Name}] {m.Sender}: {m.Text}"), onCompleted: () => Console.WriteLine($"[{Name}] Room closed.")); } public void Mute(string userName) => _mutedUsers.Add(userName); public void Unmute(string userName) => _mutedUsers.Remove(userName); public void LeaveRoom() => _subscription?.Dispose(); public void Dispose() => _subscription?.Dispose(); } // Usage using var room = new ChatRoom(replayCount: 5); room.SendMessage("Alice", "Hey everyone!"); // Sent BEFORE Charlie joins using var alice = new User("Alice"); using var bob = new User("Bob"); using var charlie = new User("Charlie"); alice.JoinRoom(room); bob.JoinRoom(room); room.SendMessage("Alice", "Anyone there?"); // [Bob] Alice: Anyone there? bob.Mute("Alice"); // Bob mutes Alice — zero changes to ChatRoom! room.SendMessage("Alice", "Bob can you hear me?"); // (nothing — Bob muted Alice) charlie.JoinRoom(room); // [Charlie] Alice: Hey everyone! ← ReplaySubject replays cached messages! // [Charlie] Alice: Anyone there? // [Charlie] Alice: Bob can you hear me? room.Close(); // [Alice] Room closed. // [Bob] Room closed. // [Charlie] Room closed.
    Why This Wins

    ReplaySubject caches messages for late joiners — no separate "history" API needed. Muting is just a .Where() filter in the user's pipeline — ChatRoom knows nothing about it. OnCompleted tells all users the room closed — no polling needed. Dispose cleans up subscriptions automatically — no memory leaks.

    Migration Guide

    Refactoring from direct calls to Observer. This guide walks through converting a tightly coupled OrderService that directly calls five downstream services into a clean event-driven design where each service subscribes independently.

    Find the method that calls multiple downstream services. In our case, PlaceOrder() calls _emailService.Send(), _inventoryService.Reserve(), _analyticsService.Track(), etc. That's your future Subject — the point where a single action triggers multiple reactions.

    // BEFORE: PlaceOrder calls 5 services directly public class OrderService( IEmailService email, // dependency 1 IInventoryService inventory, // dependency 2 IAnalyticsService analytics, // dependency 3 ILoyaltyService loyalty, // dependency 4 IAuditService audit) // dependency 5 { public async Task<string> PlaceOrder(OrderRequest request) { var order = CreateOrder(request); // These 5 calls are the "broadcasting point" await email.SendConfirmation(order); await inventory.Reserve(order.Lines); await analytics.TrackPurchase(order); await loyalty.AwardPoints(order.CustomerId, order.Total); await audit.LogOrderPlaced(order); return order.Id; } }

    Risk: Low — This is just analysis. You're reading code, not changing it.

    Create an event class (or record) that carries all the data observers might need. Look at what each of the 5 direct calls needs and include it all in the event. Use INotification from MediatR for automatic handler discovery.

    // Create the event with all data observers need public record OrderPlacedEvent( string OrderId, string CustomerId, string CustomerEmail, List<OrderLine> Lines, decimal Total, DateTimeOffset PlacedAt) : INotification; public record OrderLine(string ProductId, int Quantity, decimal UnitPrice);

    Risk: Low — You're adding new code, not changing existing code.

    Add the event publishing to the OrderService, but keep the direct calls too. This is the "strangle" approach: both paths run in parallel so you can verify the event-driven path works before removing the direct calls.

    public class OrderService( IMediator mediator, // NEW: event publisher IEmailService email, // KEEP for now IInventoryService inventory, // KEEP for now IAnalyticsService analytics, // KEEP for now ILoyaltyService loyalty, // KEEP for now IAuditService audit) // KEEP for now { public async Task<string> PlaceOrder(OrderRequest request) { var order = CreateOrder(request); // OLD: Direct calls (still here for safety) await email.SendConfirmation(order); await inventory.Reserve(order.Lines); await analytics.TrackPurchase(order); await loyalty.AwardPoints(order.CustomerId, order.Total); await audit.LogOrderPlaced(order); // NEW: Also publish event (handlers run in parallel) await mediator.Publish(new OrderPlacedEvent( order.Id, order.CustomerId, order.CustomerEmail, order.Lines, order.Total, DateTimeOffset.UtcNow)); return order.Id; } }

    Risk: Low-Medium — Both paths run, so observers get called twice during migration. Use feature flags or logging to verify the event path works correctly before Step 4.

    One by one, create INotificationHandler<OrderPlacedEvent> for each downstream service. After verifying each handler works, remove the corresponding direct call from OrderService. Eventually, only the event publish remains.

    // Each former direct call becomes an independent handler public sealed class OrderConfirmationEmailHandler(IEmailService email) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await email.SendConfirmation(e.CustomerEmail, e.OrderId, e.Total); } public sealed class InventoryReservationHandler(IInventoryService inventory) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await inventory.Reserve(e.Lines); } // ... same for Analytics, Loyalty, Audit handlers ... // FINAL OrderService — clean, zero downstream dependencies public class OrderService(IMediator mediator) { public async Task<string> PlaceOrder(OrderRequest request) { var order = CreateOrder(request); // Just publish the event — handlers take care of the rest await mediator.Publish(new OrderPlacedEvent( order.Id, order.CustomerId, order.CustomerEmail, order.Lines, order.Total, DateTimeOffset.UtcNow)); return order.Id; } // Constructor went from 6 dependencies to 1 (IMediator) // Adding a 6th reaction = new handler file, zero changes here }

    Risk: Medium — Remove one direct call at a time. Run integration tests after each removal. If a handler fails, the direct call is already gone — make sure handlers have proper error handling and retry logic.

    1 Every += has a matching -= Unsubscribed events hold strong references, causing memory leaks that grow over time service.Event += handler; with no -= in Dispose
    2 Event handlers don't throw unhandled exceptions One observer's exception prevents all subsequent observers from being notified void OnEvent(s, e) { /* no try/catch */ }
    3 Observer list is not modified during notification Adding/removing observers while iterating causes InvalidOperationException or skipped observers foreach (var obs in _observers) without snapshot or lock
    4 Event args are immutable If Observer A mutates shared EventArgs, Observer B sees corrupted data EventArgs with public setters or mutable collections
    5 No circular notification dependencies Observer A triggers Subject B, which notifies Observer C, which triggers Subject A — infinite loop Stack overflow in production, or flag-based _isNotifying guards

    7 Long-running observers use async notification Synchronous notification of a slow observer (HTTP call, DB write) blocks the Subject and all other observers Database calls inside OnNext() without async
    8 Memory profiler shows no subscriber leaks Subscribers accumulate silently — you won't notice until the app runs out of memory in production Growing Gen 2 heap in production without explanation

    Senior Solution: Raise an Event Instead of Direct Calls

    Instead of OrderService directly calling _emailService.Send(), _inventoryService.Reserve(), and _analyticsService.Track(), we raise a single OrderPlaced event. The OrderService's job is done — it just announces "an order was placed" and moves on. The downstream services subscribe to this event and react independently. The result: OrderService goes from 5 constructor dependencies to 1 (the event publisher), and adding a 6th reaction requires zero changes to OrderService.

    // BEFORE: 5 dependencies, 5 direct calls public class OrderService( IEmailService email, IInventoryService inventory, IAnalyticsService analytics, ILoyaltyService loyalty, IAuditService audit) { public async Task PlaceOrder(Order order) { // 5 tightly coupled calls await email.SendConfirmation(order); await inventory.Reserve(order.Lines); await analytics.TrackPurchase(order); await loyalty.AwardPoints(order.CustomerId, order.Total); await audit.LogOrderPlaced(order); } } // AFTER: 1 dependency, 1 event raise public class OrderService(IMediator mediator) { public async Task PlaceOrder(Order order) { await mediator.Publish(new OrderPlacedEvent(order)); // That's it. Each handler reacts independently. } }

    Senior Solution: Each Service Subscribes Independently

    Each downstream service becomes an independent event handler. They subscribe to OrderPlacedEvent through MediatR's INotificationHandler<T> interface. The DI container auto-discovers all handlers — you don't need to register them manually. Each handler has its own dependencies, its own error handling, and its own tests. Adding a new reaction to "order placed" is as simple as creating a new handler class in a new file.

    // Each service is an independent subscriber — lives in its own file public sealed class OrderEmailHandler(IEmailService email) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await email.SendConfirmation(e.CustomerEmail, e.OrderId); } public sealed class InventoryHandler(IInventoryService inventory) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await inventory.Reserve(e.Lines); } public sealed class AnalyticsHandler(IAnalyticsService analytics) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await analytics.TrackPurchase(e.OrderId, e.Total); } // NEW: Adding loyalty points — create file, implement interface, done. // Zero changes to OrderService or any other handler. public sealed class LoyaltyPointsHandler(ILoyaltyService loyalty) : INotificationHandler<OrderPlacedEvent> { public async Task Handle(OrderPlacedEvent e, CancellationToken ct) => await loyalty.AwardPoints(e.CustomerId, e.Total); }
