You should be able to add new behavior to a system without changing the code that already works.

Imagine you've written a perfectly working payment system. It handles credit cards. It's tested, deployed, and customers are happily paying. Then your boss walks over and says: "We need to support PayPal too."

There are two ways to handle this. The scary way: open up your existing payment code, add an if (method == "PayPal") branch, cross your fingers, and hope you didn't accidentally break credit card payments in the process. The smart way: design your system so that "adding PayPal" means writing a new class that plugs into the existing system — without touching a single line of the code that already works.

That second approach is the Open/Closed PrincipleThe second principle in SOLID. Coined by Bertrand Meyer in 1988 and later refined by Robert C. Martin. "Software entities (classes, modules, functions) should be open for extension but closed for modification." In practice: add new behavior by writing NEW code, not by editing OLD code.. It says that software should be open for extension — you can add new behavior — but closed for modification — you don't change what's already working. Those two things sound contradictory, but they're not. The trick is abstractionA simplified representation that hides internal details. In C#, interfaces and abstract classes are abstractions. Your code depends on the abstraction (IPaymentProcessor), not on the specific implementation (StripeProcessor). This way, new implementations can be added without the existing code knowing or caring. — specifically, interfaces and polymorphism.

The laptop didn't need to be "modified" to support them. The USB port (interface) was designed to accept anything that follows the USB spec (contract). The laptop is closed for modification but open for extension via its ports. That's OCP in hardware.

The Principle Defined

Let's get precise about what OCP means structurally. The principle was coined by Bertrand MeyerA French computer scientist who introduced OCP in his 1988 book "Object-Oriented Software Construction." His original formulation was about inheritance — extending classes by subclassing them. Robert C. Martin later refined the principle to emphasize interfaces and polymorphism, which is the modern understanding we use today. in 1988, and later refined by Robert C. Martin as part of SOLID. Here's the formal statement:

"Software entities (classes, modules, functions) should be open for extension, but closed for modification."

— Bertrand Meyer (1988), refined by Robert C. Martin

Let's break this apart, because both halves carry important meaning.

"Open for extension" means you can add new behavior to the system. You can make it do things it couldn't do before. A payment system that only handled credit cards can now handle PayPal. A notification system that only sent emails can now send push notifications. You're extending its capabilities.

"Closed for modification" means you achieve that extension without changing the existing source code. The class that processes payments doesn't get edited. The class that sends notifications doesn't get a new if branch. The existing code is locked — tested, deployed, trusted — and you leave it alone.

The mechanism that makes this possible is abstraction. Instead of your code depending on concrete classes (CreditCardProcessor, EmailSender), it depends on abstractionsIn C#, an abstraction is typically an interface (IPaymentProcessor) or an abstract class. Your code talks to the abstraction, not to the concrete implementation. This means new implementations can be created and plugged in without the existing code knowing they exist. (IPaymentProcessor, INotificationSender). The abstraction defines the contract. Concrete classes implement it. And the core system only ever talks to the abstraction.

Code Implementation

Let's see OCP in action with three complete C# examples. Each one starts with a common real-world problem, shows how interfaces make the system extensible, and proves that adding new behavior requires zero changes to existing code.

The Shape Calculator

This is the textbook OCP example — and for good reason. It's simple enough to understand immediately, but it demonstrates every concept you'll use in real production code.

The problem: you need a class that calculates the total area of a collection of shapes. The obvious (but wrong) approach is a switch statement that checks the type of each shape. That violates OCP because adding a new shape means editing the calculator. The right approach: define an IShape interface and let each shape calculate its own area.

// The contract — the "USB port" for shapes. // Any shape that implements this can plug into our calculator. public interface IShape { // Every shape must know how to calculate its own area. // WHY a method, not a property? Because some area calculations // are expensive (e.g., irregular polygons) and properties should // be cheap to call by convention. double Area(); string Name { get; } } // One concrete shape. Knows ONLY about circles. public class Circle : IShape { public double Radius { get; } public string Name => "Circle"; public Circle(double radius) { if (radius <= 0) throw new ArgumentException("Radius must be positive", nameof(radius)); Radius = radius; } // pi * r^2 — every middle schooler knows this formula. // The calculator doesn't need to know it. The Circle knows it. public double Area() => Math.PI * Radius * Radius; } // Another shape. Same interface, different formula. public class Rectangle : IShape { public double Width { get; } public double Height { get; } public string Name => "Rectangle"; public Rectangle(double width, double height) { if (width <= 0 || height <= 0) throw new ArgumentException("Dimensions must be positive"); Width = width; Height = height; } // width * height — simple multiplication. public double Area() => Width * Height; } // THIS IS THE OCP-COMPLIANT CLASS. // It works with IShape — not Circle, not Rectangle. // Adding new shapes requires ZERO changes here. public class AreaCalculator { // Takes a collection of anything that implements IShape. // Could be 2 shapes or 2,000. Could be circles, rectangles, // or shapes that haven't been invented yet. public double TotalArea(IEnumerable<IShape> shapes) { double total = 0; foreach (var shape in shapes) { // Polymorphism in action: // - If shape is a Circle, calls Circle.Area() // - If shape is a Rectangle, calls Rectangle.Area() // - If shape is a Pentagon (added next week), calls Pentagon.Area() // We never check the type. We never switch/case. total += shape.Area(); } return total; } // Want individual areas with labels? Same approach. public Dictionary<string, double> AreasByShape(IEnumerable<IShape> shapes) => shapes.ToDictionary(s => s.Name, s => s.Area()); } // ✅ ADDED 3 MONTHS LATER — Zero changes to AreaCalculator! // This file is the ONLY thing that was created. // AreaCalculator.cs was NOT edited. IShape.cs was NOT edited. // That's OCP. public class Triangle : IShape { public double Base { get; } public double Height { get; } public string Name => "Triangle"; public Triangle(double @base, double height) { if (@base <= 0 || height <= 0) throw new ArgumentException("Dimensions must be positive"); Base = @base; Height = height; } // 0.5 * base * height public double Area() => 0.5 * Base * Height; }

The proof that OCP works: when Triangle was added, only one new file was created. AreaCalculator.cs didn't change. IShape.cs didn't change. Circle.cs and Rectangle.cs didn't change. The triangle just implemented IShape, and the calculator picked it up automatically. If the calculator had been a switch statement checking types, we would have had to edit it — and risk breaking the area calculations for circles and rectangles.

Building a Notification System That Grows

This is a more realistic, production-style example. Your app starts by sending email notifications. Then marketing wants SMS. Then the mobile team wants push notifications. Then someone asks for Slack integration. With OCP, each new channel is just a new class and one line in DI registration.

// The contract for any notification channel. // Whether it's email, SMS, push, Slack, carrier pigeon — // if it implements this, the system can use it. public interface INotificationChannel { string ChannelName { get; } Task SendAsync(string recipient, string message); bool CanSend(string recipient); // Validates if recipient format is correct } // Channel 1: Email — existed since day one public class EmailChannel(ISmtpClient smtp) : INotificationChannel { public string ChannelName => "Email"; public Task SendAsync(string recipient, string message) => smtp.SendAsync(recipient, "Notification", message); // Simple regex check for email format public bool CanSend(string recipient) => recipient.Contains('@') && recipient.Contains('.'); } // Channel 2: SMS — added 2 months later. ZERO changes to existing code. public class SmsChannel(ITwilioClient twilio) : INotificationChannel { public string ChannelName => "SMS"; public Task SendAsync(string recipient, string message) => twilio.SendSmsAsync(recipient, message); // Phone numbers start with + and have 10-15 digits public bool CanSend(string recipient) => recipient.StartsWith('+') && recipient.Length >= 11; } // Channel 3: Push notifications — added 5 months later. // NotificationService still hasn't been edited. Not once. public class PushChannel(IFirebaseClient firebase) : INotificationChannel { public string ChannelName => "Push"; public Task SendAsync(string recipient, string message) => firebase.SendPushAsync(recipient, "Notification", message); // Device tokens are long alphanumeric strings public bool CanSend(string recipient) => recipient.Length > 20 && recipient.All(c => char.IsLetterOrDigit(c)); } // THE CLOSED CLASS — this is the core that never changes. // It receives ALL registered channels via DI. public class NotificationService(IEnumerable<INotificationChannel> channels, ILogger<NotificationService> logger) { // Send through every channel that can handle this recipient. // Email goes to email addresses. SMS goes to phone numbers. // Push goes to device tokens. The service doesn't know the rules — // each channel decides for itself via CanSend(). public async Task NotifyAsync(string recipient, string message) { foreach (var channel in channels) { if (channel.CanSend(recipient)) { try { await channel.SendAsync(recipient, message); logger.LogInformation("Sent via {Channel} to {Recipient}", channel.ChannelName, recipient); } catch (Exception ex) { logger.LogError(ex, "Failed to send via {Channel}", channel.ChannelName); } } } } } var builder = WebApplication.CreateBuilder(args); // Register all channels. Adding a new one = one new line here. // NotificationService picks them up automatically via IEnumerable<>. builder.Services.AddScoped<INotificationChannel, EmailChannel>(); builder.Services.AddScoped<INotificationChannel, SmsChannel>(); builder.Services.AddScoped<INotificationChannel, PushChannel>(); // builder.Services.AddScoped<INotificationChannel, SlackChannel>(); ← add this next month builder.Services.AddScoped<NotificationService>(); var app = builder.Build(); app.MapPost("/notify", async (NotifyRequest req, NotificationService svc) => await svc.NotifyAsync(req.Recipient, req.Message)); app.Run();

The power of this design: when SlackChannel needs to be added next month, the developer writes one new class and adds one line to Program.cs. They don't touch NotificationService. They don't touch EmailChannel or SmsChannel. No existing tests break. No existing behavior changes. That's the practical payoff of OCP.

Composable Discount Rules

E-commerce sites add new promotions constantly — percentage off, buy-one-get-one, loyalty rewards, holiday specials. If you hard-code these as if/else branches, you'll be editing the order processor every Black Friday. With OCP, each discount rule is its own class.

// Contract for discount rules. // Each rule decides: does it apply to this order? // If so, how much discount? public interface IDiscountRule { string RuleName { get; } bool Applies(Order order); decimal CalculateDiscount(Order order); } // 10% off orders over $100 public class PercentageDiscount : IDiscountRule { public string RuleName => "10% off orders over $100"; public bool Applies(Order order) => order.Total > 100m; public decimal CalculateDiscount(Order order) => order.Total * 0.10m; } // $5 off for customers with 10+ previous orders public class LoyaltyDiscount : IDiscountRule { public string RuleName => "Loyalty: $5 off for returning customers"; public bool Applies(Order order) => order.Customer.OrderCount >= 10; public decimal CalculateDiscount(Order order) => 5m; } // CLOSED for modification. Add new discount rules? // Write a new class. Register it. Done. public class OrderProcessor(IEnumerable<IDiscountRule> rules) { public OrderSummary Process(Order order) { // Apply every rule that matches this order. // The processor doesn't know what the rules are. // It doesn't know about percentages or loyalty programs. // It just asks each rule: "Do you apply? How much off?" var discounts = rules .Where(r => r.Applies(order)) .Select(r => new AppliedDiscount(r.RuleName, r.CalculateDiscount(order))) .ToList(); var totalDiscount = discounts.Sum(d => d.Amount); return new OrderSummary(order.Total, totalDiscount, order.Total - totalDiscount, discounts); } } public record AppliedDiscount(string RuleName, decimal Amount); public record OrderSummary(decimal Subtotal, decimal TotalDiscount, decimal FinalPrice, List<AppliedDiscount> AppliedDiscounts);

Black Friday comes along and marketing wants a "20% off everything" promotion. You write BlackFridayDiscount : IDiscountRule, register it in DI, and the order processor applies it automatically — alongside the existing percentage and loyalty discounts. When Black Friday ends, you remove the DI registration line. OrderProcessor was never modified. Not for adding the promotion, and not for removing it.

 Three examples, one principle. The Shape Calculator shows OCP at its simplest. The Notification System shows it in production ASP.NET Core with DI. The Discount Rules show composable behavior. In every case, the core class (calculator, service, processor) depends on an interface and never changes when new implementations are added. That's OCP.

Why This Hurts

Growing Switch Statement

Every new format adds another 20-40 lines to this method. After 8 formats, the method is 300+ lines long. Nobody wants to touch it. Nobody can review it quickly. It becomes the file everyone is afraid of.

Modification = Risk

Adding HTML export means editing the same method that generates PDFs. If you accidentally delete a brace or change a variable name, PDF export breaks — even though you were only trying to add HTML. Unrelated features share risk.

Testing is All-or-Nothing

You can't test PDF export in isolation — the test file has to instantiate ReportExporter which drags in dependencies for Excel (ClosedXML) and PDF (PDFsharp). Want to mock the Excel library? Too bad — it's hardcoded in the same method.

Using string format with a switch means typos become runtime errors, not compile-time errors. If someone passes "Pdf" instead of "pdf", it hits the default case and throws. With interfaces, the compiler catches mismatches before the code even runs.

The IReportExporter interface includes Format, ContentType, and FileExtension as properties. Evolution & History

OCP wasn't always about interfaces and polymorphism. The principle has evolved significantly over three decades, shaped by changes in language features, architecture styles, and hard lessons from production systems. Understanding this evolution helps you see why the modern approach works and what problems the earlier approaches had.

Bertrand Meyer introduced OCP in his book Object-Oriented Software Construction. His original formulation was about inheritance: you extend a class by creating a subclass that overrides behavior. The base class stays "closed" because you don't edit its source. The subclass is the "extension."

This worked in theory but had a serious problem in practice: the fragile base class problemWhen you inherit from a class and the base class changes its internal behavior, your subclass can break unexpectedly — even if the public API didn't change. This happens because inheritance creates a tight coupling between the base class and all its descendants. It's why modern OCP prefers interfaces over inheritance.. Changing the base class could silently break all subclasses. Inheritance created a tight coupling between parent and child — the opposite of what OCP intended.

// Meyer's approach: extend via inheritance. // Works, but creates tight coupling. public class Shape { public virtual double Area() => 0; } // Extension = new subclass. Base class unchanged. public class Circle : Shape { public double Radius { get; init; } public override double Area() => Math.PI * Radius * Radius; } // But if Shape's constructor changes, ALL subclasses break. // That's the fragile base class problem.

.NET 1.0 brought abstract classes and virtual methods to C#. This gave developers a structured way to define extension points. You'd create an abstract class with some virtual methods, and derived classes would override the behavior they needed to customize.

// Abstract class defines the skeleton. public abstract class ReportGeneratorBase { // Template method — the skeleton is closed. public Report Generate(DateTime from, DateTime to) { var data = FetchData(from, to); // closed var formatted = FormatData(data); // open — subclasses override this return new Report(formatted); } protected abstract string FormatData(DataSet data); // Extension point private DataSet FetchData(DateTime from, DateTime to) { /* ... */ } } // Extension: override the formatting. public class HtmlReportGenerator : ReportGeneratorBase { protected override string FormatData(DataSet data) => $"<table>{data.ToHtmlRows()}</table>"; }

This was better than Meyer's raw inheritance because abstract methods made extension points explicit. But you still had the limitation of single inheritance in C# — a class can only extend one base class.

GenericsA language feature that lets you write code that works with ANY type, specified when the code is used. Instead of writing separate classes for IntList, StringList, UserList, you write List<T> and specify T at usage time. Generics enable type-safe OCP — you can create IProcessor<T> that processes any type without knowing what T is at compile time. were a game-changer for OCP. Generic interfaces like IComparer<T> and IEnumerable<T> meant you could write extension points that were both type-safe and infinitely extensible. The framework itself became OCP-compliant: want to customize how a list sorts? Implement IComparer<T>. No need to modify the list class.

// Generic interface — works with ANY entity type. // Repository pattern: OCP at the data layer. public interface IRepository<T> where T : class { Task<T?> GetByIdAsync(int id); Task<IEnumerable<T>> GetAllAsync(); Task AddAsync(T entity); } // Extension: each entity gets its own repository without // modifying the interface or the generic base class. public class UserRepository : IRepository<User> { /* ... */ } public class ProductRepository : IRepository<Product> { /* ... */ }

This was the inflection point. .NET Core shipped with a built-in dependency injection containerA system that automatically creates and manages the lifecycle of objects your code needs. Instead of 'new PaymentProcessor()', you register IPaymentProcessor and the container creates the right implementation when needed. .NET Core's DI is configured in Program.cs and is used throughout the framework., making OCP trivially easy. The middleware pipeline — app.UseAuthentication(), app.UseRouting(), app.UseAuthorization() — was pure OCP: each middleware is a plugin, the pipeline is closed for modification.

DI meant that registering a new implementation was a single line in Program.cs. No factory patterns. No service locators. No abstract factories. Just builder.Services.AddScoped<IExporter, HtmlExporter>() and the system picks it up.

The latest .NET versions push OCP even further. Source generatorsA compiler feature that generates C# code at compile time based on your existing code. For OCP, this means you can automatically discover and register all implementations of an interface without writing the registration code yourself. The compiler does it for you. can automatically discover all implementations of an interface at compile time and register them — zero manual registration needed. Keyed services let you register multiple implementations and resolve them by name, making the dictionary pattern from our ReportService example a built-in framework feature.

OCP in .NET Core

OCP isn't just a principle you apply to your own code — it's baked into ASP.NET Core's DNA. The framework itself was designed around extension points. Every major subsystem follows OCP: you extend by implementing an interface and registering it, never by modifying framework source code. Let's look at the five most important examples.

The ASP.NET Core middleware pipeline is the purest example of OCP in the .NET ecosystem. The pipeline itself — the infrastructure that passes HTTP requests from one middleware to the next — is completely closed for modification. You never edit it. But the behavior of the pipeline is infinitely extensible: you add middleware by calling app.Use*() methods.

Each middleware is a plugin. It receives a request, does something (authenticate, log, compress, rate-limit), and passes the request to the next middleware. The pipeline doesn't know or care what each middleware does. It just calls them in order.

var app = builder.Build(); // Each line adds a plugin to the pipeline. // The pipeline itself NEVER changes. app.UseHttpsRedirection(); // Plugin: redirect HTTP to HTTPS app.UseAuthentication(); // Plugin: verify JWT tokens app.UseAuthorization(); // Plugin: check permissions app.UseRateLimiter(); // Plugin: throttle requests app.UseResponseCompression(); // Plugin: gzip responses // YOUR custom middleware — plugs right in alongside framework middleware. // Zero changes to the pipeline infrastructure. app.Use(async (context, next) => { var stopwatch = Stopwatch.StartNew(); await next(context); stopwatch.Stop(); context.Response.Headers["X-Response-Time"] = $"{stopwatch.ElapsedMilliseconds}ms"; });

Why is this OCP? Because adding rate limiting didn't require editing the authentication middleware. Adding your custom timing middleware didn't require editing any existing middleware. Each one is independent, plugged into a stable pipeline.

.NET's logging system is built around ILoggerProvider. The logging framework itself — the part that receives log calls from your code — is closed. When To Use / When Not To

OCP sounds like it should apply everywhere — "always be extensible!" But in practice, adding extension points has a cost: more interfaces, more files, more indirection. The question isn't "should I follow OCP?" but rather "will new behavior variants actually be added here?" If yes, OCP saves you from editing tested code. If no, OCP just adds ceremony for no reason. The goal is strategic openness — design extension points where change is likely, not where it's merely possible.

The best heuristic for OCP: wait for the third variant before abstracting. The first time, just write the code. The second time, notice the duplication but resist the urge. The third time — now you have enough data to design the right abstraction. Abstracting on the first variant is gambling; abstracting on the third is informed design. Apply OCP when behavior variants will grow — payment methods, export formats, validation rules, plugin systems. Don't apply it when behavior is fixed, only 2 variants exist, or you're guessing about future needs. The "Rule of Three" is your guide: abstract after the third variant, not the first. Strategic openness beats blanket extensibility.
Section 10

Comparisons

OCP gets confused with other principles and patterns all the time. "Isn't OCP the same as Strategy?" "Isn't it just SRP with a different name?" They're related, but each solves a different problem. Understanding the differences is crucial for applying each one correctly — and for answering interview questions that try to trip you up on these distinctions.

OCP vs SRP

People often mix these up because both deal with managing change. But they focus on different kinds of change. SRP asks "who can force this class to change?" — it's about isolating reasons for change. OCP asks "can I add new behavior without editing existing code?" — it's about isolating the act of change. Here's the important connection: OCP often requires SRP as a prerequisite. If a class does five things, you can't make it extensible for just one of those things without first splitting it up. SRP prepares the battlefield; OCP wins the war.

OCP vs Strategy Pattern

This one confuses people the most. Strategy is one mechanism for achieving OCP, but OCP is much broader than Strategy alone. Think of it this way: OCP is the goal ("add features without editing code"), and Strategy is one tool in the toolbox. You can also achieve OCP through Template Method (override virtual methods), Decorator (wrap and extend), or event-driven architecture (subscribe new handlers). Strategy is the most common approach, especially in DI-heavyDependency Injection heavy: a codebase that relies heavily on injecting interfaces rather than creating concrete objects directly. Common in ASP.NET Core and other modern frameworks. .NET codebases, but it's not the only one.

OCP vs LSP

OCP and LSP are two sides of the same coin. OCP says "extend the system by adding new implementations." LSP says "those new implementations must actually work as drop-in replacements." Without LSP, OCP is dangerous — you could add a new SquarePaymentProcessor that throws exceptions for half the methods, breaking the system even though you "extended" it correctly. LSP is the safety net that ensures your extensions are valid. Think of OCP as the door and LSP as the building code that makes sure the door actually opens properly.

When to pick which approach to OCP: Use composition (Strategy/Decorator) when you have interchangeable algorithms or need runtime swapping — payment processors, formatters, validators. Use inheritance (Template Method) when you have a fixed algorithm skeleton with customizable steps — framework base classes, pipeline stages. Use events (Observer/pub-sub) when you need to add cross-cutting reactions to things that happen — audit logging, notifications, analytics. Most .NET applications use composition as the default because it works naturally with the DI container.

OCP is the goal; Strategy, Template Method, Decorator, and events are mechanisms. OCP differs from SRP (which is about isolating reasons for change, not adding new behavior). Strategy is the most common OCP mechanism in .NET but not the only one. LSP constrains HOW you extend — without it, OCP leads to broken substitutions. Default to composition (interfaces + DI) for most .NET scenarios.

SOLID Connections

OCP doesn't work in isolation — it depends on every other SOLID principle to function properly. Think of SOLID as a machine with five gears: OCP is the output gear (the one that actually produces extensible code), but it can't turn without the other four driving it. Here's how each sibling principle supports — or enables — OCP.

Relationship: SRP prepares classes to be extensible. If a class does five things, you can't make it open for extension on just one axis — you'd need five extension points, and they'd all interfere with each other. SRP splits the class into focused pieces, each of which can then be independently extended. You almost always need to apply SRP before you can apply OCP.

Example: A ReportService that fetches data, formats it, AND exports it. To make the export format extensible (OCP), you first need to extract export into its own class (SRP). Then you create an IReportExporter interface and add CsvExporter, PdfExporter, etc. SRP made the surgery possible.

SRP isolates responsibilities into focused classes. OCP then makes each focused class extensible. Without SRP, OCP is like trying to add an extension to a house that has no walls — there's nothing clean to build onto.

This IS the current principle

OCP is the principle we've been building throughout this page. Software entities should be open for extension (you can add new behavior) but closed for modification (you don't edit existing, tested code). It's the second letter in SOLID and arguably the most impactful one for day-to-day development — because every time a PM says "add support for X," OCP determines whether you're adding one new file or editing twenty existing ones.

Relationship: LSP is OCP's safety net. OCP says "extend by adding new implementations." But what happens if a new implementation doesn't actually work as a drop-in replacement? LSP prevents this. It says every subclass or interface implementation must be usable wherever the base type is expected, without breaking anything. Without LSP, OCP is dangerous — you'd add new implementations that look correct but throw exceptions, return unexpected nulls, or violate preconditions.

Example: You have IPaymentProcessor with CreditCardProcessor and PayPalProcessor. Someone adds CryptoProcessor that throws NotSupportedException for refunds. This violates LSP — and silently breaks every place that calls processor.Refund(). LSP ensures that every new implementation honors the full contract.

OCP gives you permission to extend. LSP sets the rules for HOW you extend. Every new implementation must honor the base contract — no surprises, no exceptions, no silent failures. LSP is the guardrail that prevents OCP extensions from breaking the system.

Relationship: ISP makes extension points clean. If your interface is a 12-method monster, every new implementation has to deal with all 12 methods — even the ones that don't apply. This makes OCP painful because adding a new variant is a huge burden. ISP splits the fat interface into small, focused ones, so each new implementation only needs to handle what's relevant.

Example: IRepository<T> with Create, Read, Update, Delete, BulkInsert, Search, and Export. Adding a new read-only data source means implementing 7 methods when you only need 1. ISP splits this into IReadRepository<T>, IWriteRepository<T>, etc. Now extending is lightweight.

ISP keeps interfaces small and focused, so new implementations are lightweight. Without ISP, every OCP extension means writing boilerplate stubs for methods you don't care about — which also violates LSP (throwing NotSupportedException).

Relationship: DIP is the mechanism that makes OCP physically work. OCP says "depend on abstractions so you can swap implementations." DIP says the exact same thing from a different angle: "high-level modules should not depend on low-level modules; both should depend on abstractions." Without DIP, your code depends on concrete classes — and you literally cannot add new implementations without editing the code that uses them. DIP is OCP's engine room.

Example: OrderService directly creates new CreditCardProcessor(). You want to add PayPal. Without DIP, you'd edit OrderService to add an if/else (violating OCP). With DIP, OrderService depends on IPaymentProcessor, and the DI container injects whatever implementation is registered. Adding PayPal is just a new class + one line of DI registration.

DIP (depend on abstractions, not concretions) is the mechanical requirement that makes OCP possible. Without DIP, there are no extension points. Without OCP, there's no reason to invert dependencies. They are inseparable. OCP is the GOAL. DIP is the MECHANISM. LSP is the SAFETY NET. SRP and ISP keep the interfaces CLEAN enough to extend. All five SOLID principles converge to produce code that is genuinely extensible without modification. Remove any one, and the system has a crack. OCP depends on every other SOLID principle: SRP prepares classes for extension by isolating responsibilities. LSP ensures extensions are valid substitutes. ISP keeps extension points small and manageable. DIP provides the abstraction mechanism that makes extension physically possible. Together, they form a complete system where OCP is the goal and the other four are the enablers.

An e-commerce team added PayPal support. The developer modified the existing PaymentProcessor class with a new if/else branch. Accidentally broke credit card processing because they changed the shared validation logic.

What Went Wrong: The original PaymentProcessor class handled credit card payments with a ProcessPayment method. Section 13

Pitfalls & Anti-Patterns

Even when you know OCP, it's easy to misapply it. These six pitfalls are the most common ways developers get OCP wrong — either by not applying it when they should, or by over-applying it when they shouldn't. Each pitfall includes the mistake, why it's harmful, and the concrete fix.

The Mistake: Using switch or if/else on a type string or enum to dispatch behavior, instead of using polymorphismPolymorphism: the ability to treat different types uniformly through a shared interface. Instead of checking "what type is this?" and branching, you call a method on the interface and each type provides its own behavior..

Why It's Bad: Every time a new type is added, you have to edit the switch statement. This means every new feature requires modifying existing, tested code. If you have 5 switch statements scattered across the codebase that all switch on the same type, you need to update all 5 — and missing one creates a runtime bug with no compile-time warning.

The Fix: Replace the switch with an interface. Each case becomes a class that implements the interface. New types are just new classes — the switch disappears entirely.

// ❌ Switch on type — must edit for every new shape public double CalculateArea(Shape shape) => shape.Type switch { "circle" => Math.PI * shape.Radius * shape.Radius, "rectangle" => shape.Width * shape.Height, // ❌ Adding triangle? Edit THIS method. // ❌ Adding pentagon? Edit THIS method again. _ => throw new NotSupportedException(shape.Type) }; // ✅ Each shape knows how to calculate its own area public interface IShape { double CalculateArea(); } public class Circle(double radius) : IShape { public double CalculateArea() => Math.PI * radius * radius; } public class Rectangle(double width, double height) : IShape { public double CalculateArea() => width * height; } // Adding Triangle? Just create a new file. Zero edits to existing code. public class Triangle(double baseLen, double height) : IShape { public double CalculateArea() => 0.5 * baseLen * height; }

The Mistake: Creating extension points for behavior that will never vary. Wrapping everything in interfaces "just in case" when there's no evidence that multiple implementations will ever be needed.

Why It's Bad: Every interface adds a layer of indirection. If IMathHelper has exactly one implementation and will never have another, the interface just makes the code harder to navigate. You pay the complexity cost of OCP without getting any of its benefits. This is YAGNIYAGNI: "You Aren't Gonna Need It." A principle that says don't build features (or abstractions) until you actually need them. Predicting future requirements is unreliable — wait for evidence. in action.

The Fix: Wait for the second or third implementation before extracting an interface. When you only have one implementation, use a concrete class. Extracting an interface later takes 30 seconds in any modern IDE — you're not locked in.

// ❌ YAGNI: interface for something that will NEVER have a second implementation public interface IMathHelper { double Add(double a, double b); double Multiply(double a, double b); double Sqrt(double x); } public class MathHelper : IMathHelper // the ONLY implementation, forever { public double Add(double a, double b) => a + b; public double Multiply(double a, double b) => a * b; public double Sqrt(double x) => Math.Sqrt(x); } // DI registration for... what exactly? services.AddSingleton<IMathHelper, MathHelper>(); // Will there ever be an AlternativeMathHelper? No. This interface is pure ceremony. // ✅ Concrete class — extract interface WHEN you need a second implementation public class MathHelper { public double Add(double a, double b) => a + b; public double Multiply(double a, double b) => a * b; public double Sqrt(double x) => Math.Sqrt(x); } // No interface needed. If someday you need an ApproximateMathHelper, // extracting an interface takes 30 seconds in any IDE. // "Extract Interface" is a safe, mechanical refactoring.

The Mistake: Creating an interface that exposes implementation details, making true extension impossible. The interface looks extensible but is secretly coupled to a specific technology or approach.

Why It's Bad: If your IRepository interface has a method IQueryable<T> GetQueryable(), any new implementation must support LINQ-to-SQL queries. A Redis cache, a file system, or an API endpoint can't implement this meaningfully. The "abstraction" forces all implementations to be SQL databases.

The Fix: Design interfaces around what the consumer needs, not how the implementation works. Instead of exposing IQueryable, expose specific query methods like GetById, GetByFilter, etc.

// ❌ Leaky: forces all implementations to support IQueryable (SQL) public interface IRepository<T> { IQueryable<T> GetQueryable(); // ❌ Only SQL databases support this void ExecuteRawSql(string sql); // ❌ Literally requires SQL! DbContext GetContext(); // ❌ Exposes EF Core internals } // ✅ Clean: describes WHAT, not HOW public interface IRepository<T> where T : class { Task<T?> GetByIdAsync(int id); Task<IReadOnlyList<T>> GetAllAsync(); Task<IReadOnlyList<T>> FindAsync(Expression<Func<T, bool>> predicate); Task AddAsync(T entity); Task UpdateAsync(T entity); Task DeleteAsync(int id); } // Now ANY storage can implement this: // SqlRepository, RedisRepository, InMemoryRepository, ApiRepository...

The Mistake: One giant interface with 15+ methods instead of focused, specific ones. This makes every new implementation a massive undertaking, discouraging extension.

Why It's Bad: If your IDataService has Read, Write, Delete, Export, Import, Search, Cache, Validate, and Transform methods, adding a read-only data source means implementing stubs for 8 methods you don't need. This violates ISPInterface Segregation Principle: clients should not be forced to depend on methods they don't use. Split fat interfaces into smaller, role-specific ones. and makes OCP impractical — nobody wants to implement a 15-method interface just to add one new behavior.

The Fix: Split the god interface into small, focused interfaces. Each implementation only needs to handle its relevant methods.

// ❌ God interface — 12 methods, every new impl must handle ALL of them public interface IDataService { Task<T> Read(int id); Task Write(T entity); Task Delete(int id); Task<byte[]> Export(string format); Task Import(byte[] data); Task<List<T>> Search(string query); Task Cache(T entity); Task Validate(T entity); Task<T> Transform(T entity); Task Backup(); Task Restore(); Task Sync(); // Adding a read-only API source? Must implement ALL 12 methods. // Most will throw NotSupportedException. That's LSP violation too. } // ✅ Small, focused interfaces — implement only what you need public interface IReadable<T> { Task<T?> GetByIdAsync(int id); Task<List<T>> SearchAsync(string query); } public interface IWritable<T> { Task CreateAsync(T entity); Task UpdateAsync(T entity); Task DeleteAsync(int id); } public interface IExportable<T> { Task<byte[]> ExportAsync(string format); } // Read-only API source? Just implement IReadable. Done. public class ExternalApiSource : IReadable<Product> { public Task<Product?> GetByIdAsync(int id) => _api.FetchProduct(id); public Task<List<Product>> SearchAsync(string q) => _api.Search(q); // No stubs, no NotSupportedException, no wasted code }

The Mistake: Writing a beautiful new implementation class that follows OCP perfectly... but forgetting to register it in the DI containerDependency Injection container: the runtime system that creates and wires objects. In .NET, configured in Program.cs with services.AddScoped, AddSingleton, etc. If a class isn't registered, the container can't inject it.. The code compiles, the tests pass (because tests mock the interface), but the new implementation never actually runs in production.

Why It's Bad: This is the most frustrating OCP pitfall because everything looks correct. You wrote the class, it implements the interface, the code is clean. But the DI container doesn't know about it, so the old implementation keeps running. It can take days to notice that the new behavior isn't actually active.

The Fix: Use assembly scanning to auto-register all implementations, or write integration tests that verify the DI container has all expected implementations.

// ❌ Manual registration — easy to forget new implementations services.AddScoped<IPaymentProcessor, CreditCardProcessor>(); services.AddScoped<IPaymentProcessor, PayPalProcessor>(); // ❌ Forgot: services.AddScoped<IPaymentProcessor, ApplePayProcessor>(); // ApplePayProcessor exists in the codebase but never runs! // ✅ Assembly scanning — new implementations are auto-registered services.Scan(scan => scan .FromAssemblyOf<IPaymentProcessor>() .AddClasses(c => c.AssignableTo<IPaymentProcessor>()) .AsImplementedInterfaces() .WithScopedLifetime()); // ✅ OR: Integration test that catches missing registrations [Fact] public void AllPaymentProcessors_AreRegistered() { var provider = new ServiceCollection() .AddPaymentServices() // your registration method .BuildServiceProvider(); var processors = provider.GetServices<IPaymentProcessor>(); Assert.Contains(processors, p => p is CreditCardProcessor); Assert.Contains(processors, p => p is PayPalProcessor); Assert.Contains(processors, p => p is ApplePayProcessor); }

The Mistake: Adding new methods to an existing interface when new behavior is needed, instead of creating a new interface that extends the original. This breaks every existing implementation because they all need the new method.

Why It's Bad: If you add RefundAsync to IPaymentProcessor, every existing processor — CreditCardProcessor, PayPalProcessor, CryptoProcessor — fails to compile until you add the method to each one. This is the opposite of OCP: instead of adding new behavior without touching existing code, you're touching every existing implementation.

The Fix: Create a new interface (like IRefundable) that only the relevant implementations adopt. Or use C# default interface methods if you need backward compatibility.

// ❌ Adding Refund to existing interface — breaks ALL implementations public interface IPaymentProcessor { PaymentResult Process(PaymentRequest request); Task<RefundResult> RefundAsync(string transactionId); // ❌ NEW METHOD } // ❌ CryptoProcessor can't refund — forced to throw public class CryptoProcessor : IPaymentProcessor { public PaymentResult Process(PaymentRequest r) => /* ... */; public Task<RefundResult> RefundAsync(string txId) => throw new NotSupportedException("Crypto can't refund"); // ❌ LSP violation too! } // ✅ New interface — only adopted by processors that actually support refunds public interface IPaymentProcessor { PaymentResult Process(PaymentRequest request); // untouched! } public interface IRefundable { Task<RefundResult> RefundAsync(string transactionId); } // Only credit card and PayPal support refunds public class CreditCardProcessor : IPaymentProcessor, IRefundable { public PaymentResult Process(PaymentRequest r) => /* ... */; public Task<RefundResult> RefundAsync(string txId) => /* ... */; } // Crypto doesn't support refund — and that's fine. No forced stubs. public class CryptoProcessor : IPaymentProcessor { public PaymentResult Process(PaymentRequest r) => /* ... */; // No RefundAsync — CryptoProcessor is honest about its capabilities } // Usage: check if the processor supports refunds if (processor is IRefundable refundable) await refundable.RefundAsync(transactionId); The six OCP pitfalls: (1) Switch statements — replace with polymorphism. (2) Premature abstraction — wait for the third variant. (3) Leaky abstractions — design interfaces around behavior, not implementation. (4) God interfaces — split into focused roles. (5) Missing DI registration — use assembly scanning or integration tests. (6) Modifying interfaces — create new ones instead. The common thread: OCP isn't just "add interfaces." It requires thoughtful design of what those interfaces expose and how implementations are wired.

Testing Strategies

OCP-compliant code is a dream to test. Why? Because each implementation is a standalone class with minimal dependencies. You test the interface contract once, and every new implementation gets its own isolated test suite. The ultimate proof of OCP: you add a new implementation, and zero existing tests need to change. If adding a new feature requires editing old tests, you probably have an OCP violation. Here are the four strategies that keep OCP code well-tested.

Write tests against the interface, not the concrete class. This gives you a test suite that validates the contract once — and every new implementation can be plugged in to verify it satisfies the same contract. This is the most powerful OCP testing technique: the test suite grows with implementations but never needs editing.

// ✅ Test the contract — any IShape must satisfy these public abstract class ShapeContractTests { protected abstract IShape CreateShape(); // subclass provides the impl [Fact] public void CalculateArea_ReturnsPositiveNumber() { var shape = CreateShape(); var area = shape.CalculateArea(); Assert.True(area > 0, "Area must be positive"); } [Fact] public void CalculateArea_IsIdempotent() { var shape = CreateShape(); var first = shape.CalculateArea(); var second = shape.CalculateArea(); Assert.Equal(first, second); // same input = same output } } // Each implementation plugs into the contract tests public class CircleContractTests : ShapeContractTests { protected override IShape CreateShape() => new Circle(5.0); } public class RectangleContractTests : ShapeContractTests { protected override IShape CreateShape() => new Rectangle(4.0, 6.0); } // ✅ Adding Pentagon? Create PentagonContractTests : ShapeContractTests // Zero changes to existing test code. All contract tests run automatically.

Beyond contract tests, each implementation gets its own unit tests that verify implementation-specific behavior. CreditCardProcessor has tests for CVV validation, expiry checks, and card network routing. PayPalProcessor has tests for email validation and PayPal API error handling. These are independent — a bug in PayPal tests never affects credit card tests.

public class CreditCardProcessorTests { private readonly CreditCardProcessor _sut = new(new MockGateway()); [Fact] public void Process_ExpiredCard_ReturnsDeclined() { var request = new PaymentRequest { CardNumber = "4111111111111111", ExpiryDate = new DateTime(2020, 1, 1), // expired Amount = 99.99m }; var result = _sut.Process(request); Assert.False(result.IsSuccess); Assert.Equal("Card expired", result.ErrorMessage); } // Implementation-specific tests: CVV, card network, 3D Secure... // These tests are ONLY about CreditCardProcessor // Adding PayPalProcessor never touches this file }

This catches Pitfall #5 (forgetting to register). Write an integration test that builds the DI container and verifies all expected implementations are registered. If someone adds a new processor class but forgets the DI registration, this test fails immediately.

public class DependencyRegistrationTests { [Fact] public void AllPaymentProcessors_AreRegistered() { // Build the actual DI container var services = new ServiceCollection(); services.AddPaymentServices(); // your registration method var provider = services.BuildServiceProvider(); var processors = provider.GetServices<IPaymentProcessor>().ToList(); // Verify every known processor is registered Assert.Contains(processors, p => p is CreditCardProcessor); Assert.Contains(processors, p => p is PayPalProcessor); Assert.Contains(processors, p => p is ApplePayProcessor); // Bonus: verify no unexpected processors sneaked in Assert.Equal(3, processors.Count); } [Fact] public void AllProcessors_CanBeResolved() { // Verify the DI container can actually create instances var provider = BuildTestProvider(); var processors = provider.GetServices<IPaymentProcessor>(); foreach (var processor in processors) { // If this throws, the processor has unregistered dependencies Assert.NotNull(processor); } } }

The ultimate proof that your code follows OCP: create a new implementation, register it, and verify that zero existing tests need to change. If every existing test still passes after adding a new feature, your design is truly open for extension and closed for modification.

// This isn't a real test — it's a PROCESS. // When you add a new IPaymentProcessor implementation: // // Step 1: Create StripeProcessor.cs (new file) // Step 2: Create StripeProcessorTests.cs (new file) // Step 3: Register in DI (one line in Program.cs) // Step 4: Run ALL existing tests // // If Step 4 passes with ZERO changes to existing tests: // ✅ Your code follows OCP // // If Step 4 requires editing existing tests: // ❌ You have an OCP violation somewhere // // Common failures: // - Tests hardcode processor count: Assert.Equal(3, count) ← must update // - Tests switch on type: if (p is CreditCard) ← must add new branch // - Tests depend on ordering: processors[0] ← fragile After adding a new implementation, ask: "Did I edit any existing file other than DI registration?" Four OCP testing strategies: (1) Contract tests through the abstraction — one test suite validates ALL implementations. (2) Isolated unit tests per implementation — independent, focused, no cross-contamination. (3) DI registration tests — catch missing registrations before production. (4) The OCP litmus test: new feature = zero edits to existing tests. If your existing tests break when you add a feature, you have an OCP violation.

Performance Considerations

"Doesn't OCP mean more interfaces, more virtual dispatch, more DI overhead?" This is the most common performance concern about OCP — and it's almost always misplaced. OCP does have a runtime cost: virtual dispatchVirtual dispatch: when the runtime looks up which implementation to call through an interface or virtual method. This adds ~1.5-2.5 nanoseconds per call compared to a direct method call, because the CPU must follow a pointer to the method table. through interfaces is slightly slower than direct method calls. But "slightly slower" means nanoseconds — and your real bottlenecks are in database queries, network calls, and serialization that take milliseconds.

Cost Source Overhead Context
Virtual dispatch (interface call) ~2ns per call Direct call is ~0.5ns. Difference: 1.5ns.
Interface boxing for value types ~20ns per box Only affects structs implementing interfaces. Use generics to avoid.
DI container resolution (scoped) ~45ns per resolve Cached after first resolve per scope. Trivial.
Additional class instances ~24 bytes per object Object header (16B) + minimum fields (8B). GC handles this easily.
Database query (for comparison) 5,000,000ns (5ms) 2,500,000x more expensive than virtual dispatch
HTTP API call (for comparison) 100,000,000ns (100ms) 50,000,000x more expensive than virtual dispatch
The maintainability gains from OCP far outweigh the nanosecond cost of virtual dispatch. You would need to make 500,000 interface calls to equal the cost of ONE database query. The only time OCP overhead matters is in tight inner loops processing millions of items per second — game engines, audio processing, HFT. For web APIs, background jobs, and business logic, it's completely irrelevant.

Let's see this in actual benchmark numbers. The code below compares a direct method call, an interface call, and a DI-resolved interface call — the three levels of OCP overhead.

How to Explain OCP in an Interview

OCP comes up in virtually every SOLID interview. The key is sounding like you've applied it in real code, not like you've memorized a Wikipedia definition. Interviewers want to hear: a clear definition, a concrete example, and practical judgment about when to apply it. Here's how to nail all three.

Drop these naturally: "open for extension, closed for modification", "interface + new implementation", "DI container wires it up", "zero changes to existing code." Interviewers listen for these — they signal you understand OCP beyond the textbook.

Opening (define it): "OCP says software should be open for extension but closed for modification. That means when you need new behavior — a new payment method, a new export format, a new validation rule — you add it through new code, not by editing existing code."

Core (explain the mechanism): "The way you achieve this is through abstractions — usually interfaces in C#. Instead of a giant switch statement that checks payment type, you define an IPaymentProcessor interface. Each payment method is its own class. When you need a new one, you create a new class, register it in the DI container, and you're done. The existing processors are never touched."

Example (make it real): "In our e-commerce system, adding Apple Pay support was literally one new class that implemented IPaymentProcessor and one line in Program.cs for DI registration. We didn't edit the credit card or PayPal code at all. That's OCP in action — the feature was deployed in one PR with zero risk to existing functionality."

When to apply (show judgment): "I apply OCP when behavior varies and will grow — payment methods, notification channels, export formats. But I don't prematurely abstract. If there's only one implementation and no evidence of a second, I keep it concrete. The Rule of Three helps: I abstract after the third variant, not the first."

Close (nail the benefit): "The real benefit of OCP is confidence. When I add a new feature, I know it can't break existing ones — because I never touch existing code. New feature, new file, zero regression risk. That's what makes OCP worth the upfront design cost."

Interviewers love to push back after your OCP answer. Here are the most common follow-ups:

  • "How do you decide what to make extensible?" — "I look at what changes most frequently. Payment methods, business rules, and export formats are classic extension points. Math utilities and configuration parsers are not. I watch for the pattern: if the third variant appears, that's my signal to abstract."
  • "Doesn't OCP create too many classes?" — "More files, yes, but each is small, focused, and independently testable. I'd rather have 10 files I can understand in isolation than 1 file with 500 lines and 8 responsibilities. IDE navigation makes file count irrelevant."
  • "What's the relationship between OCP and DIP?" — "They're inseparable. DIP says depend on abstractions — that's the mechanism. OCP says extend through those abstractions — that's the goal. Without DIP, you'd depend on concrete classes and couldn't add new implementations without editing callers. DIP is OCP's engine."
  • "Can you over-apply OCP?" — "Absolutely. Wrapping everything in interfaces 'just in case' is YAGNI. An interface with one implementation that will never have a second is pure overhead. The cost of OCP is indirection — and indirection should always pay for itself with actual extensibility."
 Nail the OCP interview with five beats: (1) Define it — open for extension, closed for modification. (2) Explain the mechanism — interfaces + DI. (3) Give a concrete example — payment processors, notification channels. (4) Show judgment — Rule of Three, don't prematurely abstract. (5) Close with the benefit — new features can't break existing ones. Prepare for follow-ups about deciding extension points, class count, and the relationship between OCP and DIP.

Easy Questions

Q1: What is the Open/Closed Principle?
Easy
Think First Imagine a sealed box that does its job perfectly. Now imagine it has open slots where you can plug in new functionality. The box never changes — but what it can do keeps growing.

The strong answer: OCP says your code should be open for extension (you can add new behavior) but closed for modification (you don't change existing, tested code to do it). Bertrand Meyer coined it in 1988, and it remains one of the most practically useful SOLID principles.

Think of it like a smartphone. When Apple released the iPhone, they didn't ship a phone that could only make calls and then rewrite the phone's OS every time someone wanted a new feature. Instead, they built an extension mechanism — the App Store. You add new capabilities (apps) without ever touching the phone's core operating system. The core is closed for modification; the app ecosystem is open for extension.

In C#, the most common way to achieve this is through interfacesA contract that defines what a class can do (method signatures) without specifying how. Any class that implements the interface promises to provide those methods. This lets you swap implementations without changing calling code. and dependency injectionInstead of a class creating its own dependencies (new PaypalGateway()), the dependencies are passed in from outside (via the constructor). This lets you swap implementations without touching the class that uses them.. You define an interface like IPaymentProcessor, and existing code works with that interface. Adding PayPal support means creating a new class — PayPalProcessor : IPaymentProcessor — not editing anything that already works.

Why interviewers love this: OCP is the principle that separates junior code (giant switch statements that grow with every feature) from senior code (pluggable architectures that scale). Show them you understand the difference.

Q2: How does OCP relate to interfaces in C#?
Easy
Think First An interface is like a contract posted on a wall. Anyone who signs the contract can do the job. The person hiring never has to change — they just accept anyone who signed.

The strong answer: Interfaces are the mechanism that makes OCP work in C#. When a class depends on an interface instead of a concrete class, it becomes automatically open for extension.

Here's why: let's say your PaymentService depends on IPaymentProcessor instead of StripeProcessor. The day your business adds PayPal, you create PayPalProcessor : IPaymentProcessor and register it in DI. The PaymentService never changes — it doesn't even know PayPal exists. It just calls processor.ChargeAsync() and the right implementation runs.

// The interface IS the extension point public interface IPaymentProcessor { string Provider { get; } Task<ChargeResult> ChargeAsync(decimal amount, string currency); } // Existing: works fine, tested, deployed public class StripeProcessor : IPaymentProcessor { public string Provider => "Stripe"; public async Task<ChargeResult> ChargeAsync(decimal amount, string currency) => await _stripeClient.Charges.CreateAsync(amount, currency); } // NEW: added without touching StripeProcessor or PaymentService public class PayPalProcessor : IPaymentProcessor { public string Provider => "PayPal"; public async Task<ChargeResult> ChargeAsync(decimal amount, string currency) => await _paypalClient.CreatePaymentAsync(amount, currency); } // PaymentService never changes — it depends on the interface public class PaymentService(IEnumerable<IPaymentProcessor> processors) { public async Task<ChargeResult> ChargeAsync(string provider, decimal amount) { var processor = processors.First(p => p.Provider == provider); return await processor.ChargeAsync(amount, "USD"); } }

Key insight: The interface is the "seam" where new code plugs in. Without it, you'd need a switch statement that grows every time — which is the exact anti-pattern OCP prevents.

Q3: What's a real-world example of OCP that a non-programmer would understand?
Easy
Think First Think about USB ports, electrical outlets, or LEGO bricks. The port doesn't change, but what you plug in is unlimited.

The strong answer: The best example is a USB port. Your laptop has USB-C ports that were designed and manufactured years ago. Those ports are closed — nobody's going back to the factory to modify them. But they're open for extension — you can plug in a mouse, a keyboard, an external monitor, a flash drive, or devices that haven't been invented yet.

The USB specification is the "interface." Any device that follows the USB spec can plug in and work. The laptop doesn't care what the device is. It doesn't need a firmware update to support a new brand of mouse. That's OCP in physical form.

Other great examples: Electrical outlets (the socket never changes, but you can plug in anything from a toaster to a phone charger), LEGO bricks (the stud pattern is fixed, but what you build is unlimited), and browser extensions (Chrome's core doesn't change when you install an ad blocker).

Q4: What happens when you violate OCP?
Easy
Think First Every time a new requirement arrives, you open the same file, add another case, re-test everything, and pray you didn't break something.

The strong answer: Violating OCP creates a cascade of pain that gets worse with every new feature. Here's the chain of consequences:

  1. Modification risk: Adding a new payment method means editing PaymentProcessor.cs. Every edit to working code risks introducing bugs in the credit card flow that's been running fine for 2 years.
  2. Testing burden: You changed the file, so now you must re-test everything in that class — all existing payment types — not just the new one. The test suite grows quadratically.
  3. Merge conflicts: Two developers adding two different payment methods simultaneously? Both touch the same switch statement. Merge conflict guaranteed.
  4. Deployment coupling: You wanted to deploy PayPal support. But the file also contains Apple Pay changes from another team that aren't ready. Now either both ship or neither ships.
  5. Knowledge bottleneck: That one file becomes "the file everyone's afraid to touch." New developers avoid it. Senior developers guard it. The entire team slows down.

Real impact: I've seen teams where adding a single report format (Excel export) took 3 weeks because of OCP violations — not because the feature was complex, but because the existing ReportGenerator class was a 2,000-line switch statement where every change needed full regression testing across 15 existing formats.

Q5: What's the difference between OCP and just "not touching old code"?
Easy
Think First "Don't touch old code" could mean "copy-paste a whole file and edit the copy." That's not OCP — that's fear-driven duplication. OCP is about designing for extension, not just avoiding edits.

The strong answer: There's a crucial difference. "Not touching old code" is a behavior. OCP is a design property. You can avoid touching old code by copy-pasting the entire class and editing the copy — but now you have two nearly identical classes to maintain, which is far worse than editing one.

OCP says the system should be designed so that new behavior plugs in through well-defined extension points (interfaces, events, configuration). The old code isn't just "not touched" — it doesn't need to be touched because the architecture anticipated that new variants would arrive.

Think of it this way: a locked door with no keyhole means "nobody gets in" (not touching old code). A locked door with a keyhole means "authorized people get in without breaking anything" (OCP). Both prevent unauthorized changes, but only one provides a path for legitimate extension.

The litmus test: If adding a new feature requires you to copy-paste, you're avoiding modification but violating OCP. If adding a new feature means writing a new class that implements an existing interface, you're following OCP.

Q6: Can a simple if/else ever be OK, or does it always violate OCP?
Easy
Think First Not every if/else is an OCP violation. The question is whether that branching is likely to grow. A boolean check is fine. A switch that gains a new case every sprint is not.

The strong answer: A simple if/else is absolutely fine when the condition is stable and binary. Here's the rule: if the number of branches is fixed and unlikely to grow, if/else is fine. If new branches arrive with new business requirements, extract an abstraction.

OK if/else (not an OCP violation):

  • if (user.IsActive) — boolean, only ever true/false
  • if (order.Total > 100) — threshold check, not a growing list of types
  • if (string.IsNullOrEmpty(name)) — validation, not extensible behavior

Problematic if/else (OCP violation):

  • if (type == "pdf") ... else if (type == "excel") ... else if (type == "csv") ... — new export formats will keep arriving
  • if (country == "US") ... else if (country == "UK") ... — you'll add more countries

The heuristic: Count the branches. If there are 2 and the domain is stable, keep it simple. If there are 3+ and growing, extract. If there are 2 but the business says "we're adding more," extract now.

Q7: How does OCP apply to configuration files vs code?
Easy
Think First Configuration is behavior change without code change. Isn't that what OCP wants? Moving hard-coded values to config files is one of the simplest forms of OCP.

The strong answer: Configuration files are one of the simplest and most overlooked ways to achieve OCP. When you move a hard-coded value into appsettings.json, you're making the code closed for modification (the C# file doesn't change) but open for extension (the behavior changes via config).

Consider a retry policy. The non-OCP way: maxRetries = 3 hard-coded. To change it, you edit code, recompile, redeploy. The OCP way: maxRetries comes from config. To change it, you update a JSON file or environment variable. The code is never touched.

// BEFORE: Hard-coded, must edit code to change public class RetryHandler { private const int MaxRetries = 3; // Closed for extension! private const int DelayMs = 1000; // Change requires recompile } // AFTER: Config-driven, open for extension via settings public class RetryHandler(IOptions<RetryOptions> opts) { public async Task<T> ExecuteAsync<T>(Func<Task<T>> action) { for (int i = 0; i <= opts.Value.MaxRetries; i++) { try { return await action(); } catch when (i < opts.Value.MaxRetries) { await Task.Delay(opts.Value.DelayMs * (i + 1)); } } throw new InvalidOperationException("Unreachable"); } } // appsettings.json — change behavior without touching code // { "Retry": { "MaxRetries": 5, "DelayMs": 2000 } }

Key insight: OCP isn't just about interfaces and polymorphism. It's about designing systems where behavior can change without editing source code. Config, feature flags, environment variables, and even database-driven rules all count as extension mechanisms.

Q8: What's the simplest way to make code open for extension in C#?
Easy
Think First You don't always need a full interface hierarchy. Sometimes a Func<T> parameter is enough. Start with the lightest mechanism that works.

The strong answer: The simplest OCP mechanism in C# is accepting a delegate or Func<T> parameter. It requires zero new types, zero interfaces, and zero additional files. You just let the caller pass in behavior.

// Simplest OCP: accept a Func<T> parameter public class Sorter { // "How to compare" is open for extension public List<T> Sort<T>(List<T> items, Func<T, T, int> compare) { // ... sorting logic using compare ... return items.OrderBy(x => x, Comparer<T>.Create( (a, b) => compare(a, b))).ToList(); } } // Extend with any comparison — no interface, no new class var byName = sorter.Sort(users, (a, b) => string.Compare(a.Name, b.Name)); var byAge = sorter.Sort(users, (a, b) => a.Age.CompareTo(b.Age)); var byScore = sorter.Sort(users, (a, b) => b.Score.CompareTo(a.Score)); // descending!

Progression from simplest to most structured:

  1. Func/Action parameter — zero ceremony, good for 1-method behaviors
  2. Events — good for "notify me when X happens" extension points
  3. Interface — good when the extension point has multiple methods or needs a name
  4. Abstract base class — good when extensions share significant common logic
  5. Plugin system — good when extensions come from separate assemblies

Rule of thumb: Start with the lightest mechanism. Upgrade to a heavier one only when you need the extra structure. A Func<Order, decimal> is perfectly fine for a discount strategy — you don't need IDiscountStrategy until you have 5 implementations with 3 methods each.

Q9: Does adding a method to an existing class violate OCP?
Easy
Think First OCP says "closed for modification." But does adding a new method count as "modification"? It depends on whether existing behavior is at risk.

The strong answer: It depends on the kind of addition. OCP cares about protecting existing behavior from unintended changes. Adding a brand-new, independent method to a class doesn't break any existing method — so most people consider it acceptable.

Not a violation: Adding ExportToPdf() to a ReportService that already has ExportToCsv(), when the two methods are completely independent. You didn't touch ExportToCsv(), so its behavior is safe.

Is a violation: Adding a method that requires changing an interface that other classes implement. If you add Validate() to IPaymentProcessor, now every existing implementation must be edited to add that method. You've modified closed code.

The nuanced answer: Strictly speaking, adding a method to a class is a modification of that class's source file. In a strict OCP interpretation, you should extend via subclass or decorator instead. But in practice, most teams follow a pragmatic OCP — if you're adding genuinely new behavior that doesn't affect existing methods, it's fine. The spirit of OCP is about protecting working behavior, not about never opening a file.

C# 8+ tip: If you need to add a method to an interface without breaking implementors, use a default interface methodA feature introduced in C# 8 that lets you add methods with a default body to an interface. Existing implementors don't break because they inherit the default. Only override if you need custom behavior.: interface IProcessor { void New() => DefaultImpl(); }. Existing implementors don't break.

Q10: How do enums relate to OCP violations?
Easy
Think First Every time you add a new value to an enum, you need to find every switch statement that uses it and add a new case. That's the textbook definition of "modification required to extend."

The strong answer: Enums are one of the most common OCP violation magnets in C# codebases. The problem isn't the enum itself — it's the switch statements that inevitably follow.

Here's the pattern: you define enum NotificationType { Email, Sms }. Somewhere a switch dispatches on it. Then the business adds Push. You edit the enum (modification #1), then hunt down every switch that uses it (modification #2, #3, #4...). Miss one? Silent bug at runtime.

// The enum itself public enum NotificationType { Email, Sms, Push /* new! */ } // Every switch must be found and updated — OCP violation public void Send(NotificationType type, string message) { switch (type) { case NotificationType.Email: SendEmail(message); break; case NotificationType.Sms: SendSms(message); break; // Forgot to add Push? No compile error, just a silent bug! } } // OCP-compliant alternative: interface + DI public interface INotificationSender { string Channel { get; } Task SendAsync(string message); } // Adding Push = new class, zero edits to existing code public class PushSender : INotificationSender { public string Channel => "Push"; public Task SendAsync(string msg) => _pushService.NotifyAsync(msg); }

When enums are fine: If the enum represents a truly fixed, closed set that won't grow — like enum DayOfWeek (there will never be an 8th day) or enum Direction { North, South, East, West }. These are stable domains where OCP doesn't apply.

The red flag: If your enum has grown by more than one value in the past year, it's probably an OCP violation waiting to happen. Replace the enum + switch pattern with an interface + DI registration.

Medium Questions

Q11: How do you decide WHERE to put extension points?
Medium
Think First Not every piece of code needs to be extensible. The skill is knowing WHERE change is likely and WHERE it isn't.

The strong answer: This is the hardest part of OCP — and the part most people get wrong. The rule of thumb is: put extension points where you have evidence of variation, not where you imagine it might happen someday.

Here's the decision framework I use:

  1. Has it changed before? If the same switch statement or if-else chain has been modified in the last 3 PRs, that's strong evidence. Extract an interface there.
  2. Is the business telling you more variants are coming? "We'll add Stripe next quarter" is evidence. "Maybe someday we'll support Bitcoin" is speculation.
  3. Is it a known variation point in the domain? Payment methods, notification channels, export formats — these are areas where variation is inherent to the business domain.
  4. Are there only 2 variants and the domain is stable? A boolean flag is fine. Don't create IIsActiveStrategy for true/false.

The anti-pattern: Creating interfaces for everything "just in case." If you have IUserNameFormatter, IUserEmailFormatter, and IUserPhoneFormatter but they've never changed in 3 years, you've added complexity for no benefit. That's YAGNIYou Aren't Gonna Need It — a principle that says don't build something until you actually need it. Speculative interfaces add complexity without delivering value. Wait for evidence before abstracting. violation disguised as OCP compliance.

Q12: Compare OCP via inheritance vs composition. Which is better?
Medium
Think First Meyer said "extend via subclass." Martin said "extend via interface." Both achieve OCP — but one leads to deep inheritance hierarchies and the other leads to flexible, swappable components.

The strong answer: There are actually two versions of OCP, and understanding the difference shows deep knowledge.

Meyer's OCP (1988): Bertrand Meyer's original version says: the base class is "closed" (you don't edit it), and you "open" it by creating subclasses that override or extend behavior. This is inheritance-based OCP. You extend via class GoldCustomer : Customer.

Polymorphic OCP (1990s): Robert C. Martin reinterpreted OCP for polymorphismThe ability for different classes to be used interchangeably through a shared interface. A method that takes IShape can receive Circle, Rectangle, or Triangle — it doesn't care which. This is how modern OCP works.. Instead of extending via subclass, you extend via implementing an interface. The consumer depends on an abstraction (IPaymentProcessor), and new implementations can be plugged in without touching the consumer.

Modern answer: Prefer polymorphic OCP (interfaces + composition) in almost all cases. Meyer's inheritance-based approach leads to the fragile base class problemWhen you change a base class, every subclass might break in unexpected ways. The more levels of inheritance you have, the worse this gets. A change in Customer might break GoldCustomer, SilverCustomer, and BronzeCustomer all at once. — deep inheritance trees where a change in the base class ripples through dozens of subclasses. Interfaces keep implementations independent.

Q13: How does .NET Core's middleware pipeline demonstrate OCP?
Medium
Think First ASP.NET Core processes every HTTP request through a pipeline of middleware. Each middleware is independent. You add new ones without touching existing ones. That's OCP built into the framework itself.

The strong answer: ASP.NET Core's middleware pipeline is probably the most visible example of OCP in the entire .NET ecosystem. Here's why it's brilliant.

Each middleware is a separate class. The pipeline is configured in Program.cs by calling app.UseXxx(). When you add app.UseRateLimiting(), you're adding a new middleware to the pipeline. You don't edit UseAuthentication() or UseRouting(). You don't even recompile them. The existing middleware doesn't know or care that rate limiting now exists.

var app = builder.Build(); // Each .UseXxx() adds a middleware — OCP in action app.UseHttpsRedirection(); // Existing: never touched app.UseAuthentication(); // Existing: never touched app.UseAuthorization(); // Existing: never touched // NEW: Added without modifying ANY existing middleware app.UseRateLimiting(); app.UseResponseCaching(); app.UseCustomTelemetry(); // Your own middleware! app.MapControllers(); app.Run(); // Writing a custom middleware = adding behavior without modifying anything public class TelemetryMiddleware(RequestDelegate next) { public async Task InvokeAsync(HttpContext context) { var sw = Stopwatch.StartNew(); await next(context); // Call the next middleware sw.Stop(); // Log timing — existing pipeline untouched context.Response.Headers["X-Response-Time"] = $"{sw.ElapsedMilliseconds}ms"; } }

Why this matters in interviews: It shows you can connect abstract principles to real framework design. The middleware pipeline is OCP, the DI container is OCP (register new services without changing consumers), and the options pattern is OCP (add new configuration sections without editing existing ones). The whole framework is built on it.

Q14: How do you apply OCP in ASP.NET Core middleware?
Medium
Think First Middleware is a pipeline of independent steps. Each step does one thing. Adding a new step means writing a new class and one line in Program.cs — nothing existing changes.

The strong answer: ASP.NET Core middleware is OCP incarnate. The pipeline is a chain of RequestDelegateA delegate that represents a function which takes an HttpContext and returns a Task. Each middleware calls the next RequestDelegate in the chain, forming a pipeline. This is the "interface" that makes middleware pluggable. functions. Each middleware decides whether to pass the request to the next one. You extend the pipeline by writing a new middleware class — you never edit existing middleware.

Here's how to build a custom middleware that follows OCP perfectly. Notice how the existing pipeline (authentication, authorization, routing) is completely untouched:

// Step 1: Write the middleware class (new code, nothing edited) public class RequestTimingMiddleware(RequestDelegate next, ILogger<RequestTimingMiddleware> logger) { public async Task InvokeAsync(HttpContext context) { var stopwatch = Stopwatch.StartNew(); // Add a header BEFORE the response starts context.Response.OnStarting(() => { context.Response.Headers["X-Response-Time"] = $"{stopwatch.ElapsedMilliseconds}ms"; return Task.CompletedTask; }); await next(context); // Pass to the next middleware stopwatch.Stop(); logger.LogInformation("Request {Method} {Path} took {Ms}ms", context.Request.Method, context.Request.Path, stopwatch.ElapsedMilliseconds); } } // Step 2: Register it — one line in Program.cs app.UseHttpsRedirection(); app.UseAuthentication(); // Existing — untouched app.UseAuthorization(); // Existing — untouched app.UseMiddleware<RequestTimingMiddleware>(); // NEW! app.MapControllers();

Why this is superior OCP: Each middleware is its own class, in its own file, with its own tests. Adding rate limiting doesn't risk breaking authentication. Adding telemetry doesn't risk breaking authorization. The pipeline's "interface" is just RequestDelegate — call next() to pass through, or don't call it to short-circuit.

Q15: How does the Strategy pattern implement OCP?
Medium
Think First Strategy says "define a family of algorithms, make them interchangeable." The context class is closed — it never changes. New algorithms plug in as new classes. That IS OCP.

The strong answer: The Strategy patternA behavioral design pattern that defines a family of algorithms, puts each one in its own class, and makes them interchangeable. The class that uses the algorithm (the "context") doesn't know or care which specific strategy it's running. is essentially OCP wrapped in a design pattern name. The pattern has three parts: an interface (the strategy), concrete implementations (the algorithms), and a context class that uses the interface. The context is closed — it never changes. New algorithms are extensions — new classes that implement the interface.

// The strategy interface — the extension point public interface IShippingStrategy { string Name { get; } decimal CalculateCost(Order order); } // Context class — NEVER changes, regardless of how many strategies exist public class ShippingCostCalculator(IEnumerable<IShippingStrategy> strategies) { public decimal GetCost(string method, Order order) { var strategy = strategies.First(s => s.Name.Equals(method, StringComparison.OrdinalIgnoreCase)); return strategy.CalculateCost(order); } } // Extension: new file, new class, zero edits to anything above public class SameDayExpressStrategy : IShippingStrategy { public string Name => "SameDay"; public decimal CalculateCost(Order order) => order.Weight * 2.5m + 15.00m; // Premium pricing }

Key connection: Every time you see a Strategy pattern, you're looking at OCP. Every time you see an OCP violation (growing switch), the Strategy pattern is the fix. They're two sides of the same coin.

Q16: What role does Dependency Injection play in achieving OCP?
Medium
Think First An interface creates the extension point. But who decides which implementation to use? That's DI's job. DI is the "wiring" that connects extensions to the system without the system knowing which extension it got.

The strong answer: Dependency InjectionA technique where a class receives its dependencies from the outside (typically via constructor parameters) rather than creating them internally. In ASP.NET Core, the built-in DI container resolves and injects dependencies automatically based on registrations in Program.cs. is the delivery mechanism that makes OCP practical at scale. Without DI, even if you have perfect interfaces, something has to decide which implementation to create. That "something" becomes an OCP violation — a factory or switch that grows with each new implementation. DI eliminates that last switch.

The chain:

  1. Interface creates the extension point (IPaymentProcessor)
  2. Implementations are the extensions (StripeProcessor, PayPalProcessor)
  3. DI wires them together without any class knowing about the others
// Program.cs — the ONLY place that knows about concrete types builder.Services.AddTransient<IPaymentProcessor, StripeProcessor>(); builder.Services.AddTransient<IPaymentProcessor, PayPalProcessor>(); // Add a new payment method: ONE line here, ONE new class file builder.Services.AddTransient<IPaymentProcessor, ApplePayProcessor>(); // PaymentService — depends only on the interface // This class is 100% OCP-compliant: adding ApplePay required zero edits here public class PaymentService(IEnumerable<IPaymentProcessor> processors) { public Task<Result> ChargeAsync(string provider, decimal amount) { var proc = processors.First(p => p.Provider == provider); return proc.ChargeAsync(amount); } }

Without DI, OCP breaks at the last mile: You'd need a factory like new StripeProcessor() or new PayPalProcessor() somewhere, which means that code must be edited every time. DI moves all the "which concrete type?" decisions to one place (Program.cs) and keeps all other code working purely with interfaces.

Advanced move: Use assembly scanning (services.Scan() with Scrutor) to auto-register all implementations of an interface. Then even Program.cs doesn't change when you add a new implementation — it discovers them automatically.

Q17: How do you unit test OCP-compliant code vs OCP-violating code?
Medium
Think First OCP-compliant code is easier to test because each implementation is independent. OCP-violating code forces you to test every branch in a single monster class.

The strong answer: The testing difference is dramatic. OCP-compliant code is isolated by design — each implementation lives in its own class with its own test file. OCP-violating code is entangled — all behaviors share one class, so adding a test for one branch risks affecting another.

The practical difference:

  • OCP violation: Adding Slack means editing NotificationService.cs AND editing NotificationServiceTests.cs. All existing tests must still pass with the new mock added to the setup.
  • OCP compliant: Adding Slack means creating SlackChannel.cs AND SlackChannelTests.cs. Existing test files are never opened. Existing tests don't even know Slack exists.

Speed impact: OCP-compliant tests run faster because each test file only loads the mocks it needs. The violation version loads all mocks for every test, even when testing only email.

Q18: Can you apply OCP to database schemas?
Medium
Think First Database schemas feel rigid — you can't just "implement an interface" in SQL. But there are patterns that let you extend a schema without modifying existing tables.

The strong answer: Yes, and it's more important than most people realize. Schema changes are some of the riskiest modifications in production systems because they require migrationsScripts that change database schema (add tables, alter columns, etc.) in a controlled, versioned way. Tools like Entity Framework Migrations or Flyway generate and apply these scripts. Each migration is a permanent, ordered change to the database structure. that are hard to roll back and affect every row in a table.

OCP-friendly schema patterns:

  1. EAV (Entity-Attribute-Value): Store custom fields as rows instead of columns. Adding a new attribute means inserting a row, not altering the table. Used by Magento, WordPress for custom fields.
  2. JSON columns: Store extensible data in a JSONB column. Add new fields by just writing new JSON keys — no schema migration needed. PostgreSQL and SQL Server both support indexed JSON queries.
  3. Separate extension tables: Instead of adding columns to Users, create UserPreferences, UserAnalytics, etc. Each new feature gets its own table. The core Users table stays untouched.
  4. Polymorphic associations with a type table: Instead of NotificationType as an enum column (requires ALTER TABLE for new types), use a notification_types lookup table. New types = new rows, not schema changes.
-- BEFORE (OCP violation): Adding a notification type = ALTER TABLE ALTER TABLE notifications ADD COLUMN push_token VARCHAR(255); -- Scary! -- AFTER (OCP compliant): Extension table pattern -- Core table — never modified CREATE TABLE notifications ( id BIGINT PRIMARY KEY, user_id BIGINT NOT NULL, channel_id INT NOT NULL, -- FK to channels table message TEXT NOT NULL, sent_at TIMESTAMPTZ ); -- Extension: new channel = new row, not a schema change CREATE TABLE notification_channels ( id INT PRIMARY KEY, name VARCHAR(50) NOT NULL, -- 'email', 'sms', 'push', 'slack' config JSONB NOT NULL -- Channel-specific settings ); -- Adding Slack support = INSERT, not ALTER TABLE INSERT INTO notification_channels (id, name, config) VALUES (4, 'slack', '{"webhook_url": "...", "default_channel": "#alerts"}');

Trade-off: These patterns add query complexity (JOINs, JSON parsing) in exchange for schema stability. Use them where the domain is genuinely extensible (notification types, custom fields, plugin data). Don't use them for stable columns like created_at — that's overengineering.

Q19: How does OCP apply to API versioning?
Medium
Think First API versioning is OCP at the HTTP level. Version 1 is "closed" — existing clients rely on it. Version 2 is "open for extension" — new behavior without breaking V1 consumers.

The strong answer: API versioning is one of the most visible applications of OCP, because breaking a public API breaks every client that depends on it. The principle is the same: existing behavior (V1) must be preserved while new behavior (V2) extends the system.

Three versioning strategies (all follow OCP):

  1. URL path versioning: /api/v1/orders and /api/v2/orders coexist. V1 controller is closed — never touched. V2 is a new controller with new behavior.
  2. Header versioning: Api-Version: 2 header routes to different implementations. Same principle, less visible in URLs.
  3. Query parameter: /api/orders?version=2. Simple but pollutes the URL.
// V1 controller — CLOSED for modification, deployed, clients depend on it [ApiController] [Route("api/v1/orders")] public class OrdersV1Controller(IOrderServiceV1 svc) : ControllerBase { [HttpGet("{id}")] public IActionResult Get(int id) => Ok(svc.GetOrder(id)); // Returns: { "id": 1, "total": 99.99 } } // V2 controller — NEW file, extends the API without touching V1 [ApiController] [Route("api/v2/orders")] public class OrdersV2Controller(IOrderServiceV2 svc) : ControllerBase { [HttpGet("{id}")] public IActionResult Get(int id) => Ok(svc.GetOrderDetailed(id)); // Returns: { "id": 1, "total": 99.99, "items": [...], "tracking": {...} } } // V1 clients keep working. Q21: How do you balance OCP with YAGNI?
Hard
Think First OCP says "prepare for extension." YAGNI says "don't build what you don't need." They sound contradictory — but both are right, at different times.

The strong answer: This is the question that separates mid-level engineers from senior engineers. Both principles are correct, and the tension between them is real. The resolution is timing.

YAGNI wins at the start: When building a feature for the first time, keep it simple. One concrete class, no interfaces, no abstraction layers. Don't create INotificationChannel when you only have email. That interface adds a layer of indirection that serves nobody today.

OCP wins on the second variant: The moment the business says "we also need SMS notifications," that's your signal. Now you have evidence of variation. Extract the interface, refactor email into one implementation, add SMS as a second. Both principles are satisfied: you didn't pre-build the abstraction (YAGNI), and you can now add push notifications without modifying email or SMS (OCP).

The "Rule of Two": The first time you write something, write it directly. The second time you see the same pattern of variation, extract the abstraction. This way you never pre-build interfaces you won't need, and you always have the abstraction when you do need it.

Q22: Can you achieve OCP without interfaces?
Hard
Think First Interfaces are the most common mechanism, but C# has other tools: events, delegates, Func<T>, and even extension methods. Each one can create extension points.

The strong answer: Absolutely. Interfaces are the most common mechanism, but C# gives us several other ways to achieve OCP. Here are four:

1. Events and delegates: A class raises an event; any number of subscribers can handle it. Adding a new subscriber requires zero changes to the publisher.

OrderCompleted; public async Task ProcessAsync(Order order) { // ... process the order ... 2. Func<T> delegates: Pass behavior as a parameter. The consuming code is open for extension because you can pass any function that matches the signature.

3. Extension methods: Add new behavior to existing types without modifying them. The original class is literally sealed, yet you've extended what it can do.

4. Decorator pattern with generics: Wrap existing behavior without interfaces, using generic delegates to compose pipelines.

Key insight: OCP is a principle, not a pattern. Interfaces are just one way to achieve it. The real question is: "Can I add new behavior without changing existing code?" If yes, OCP is satisfied — regardless of the mechanism.

Q23: How would you refactor a 500-line switch statement to follow OCP?
Hard
Think First A giant switch is the most common OCP violation. Each case block is a candidate for its own class. The switch becomes a dictionary lookup or DI resolution.

The strong answer: Here's the step-by-step playbook I use. It's safe, incremental, and can be done across multiple PRs.

Step 1: Write characterization tests. Before touching the switch, write tests that cover every existing case. These are your safety net — they catch regressions during refactoring.

Step 2: Extract the interface. Look at what each case block does. They all do the same kind of thing (generate a report, process a payment, apply a discount) but in different ways. That's your interface: the "what" is the interface method, the "how" is each implementation.

Step 3: Create one class per case. Each case becomes its own class that implements the interface. Start with the simplest case, verify the tests still pass, then move to the next.

Step 4: Replace the switch with a lookup. Register all implementations in DI and inject them as IEnumerable<IHandler>. The switch becomes a dictionary or LINQ lookup by some discriminator (like a string key or enum value).

// The switch is gone — replaced by a DI-powered lookup public class ReportService(IEnumerable<IReportExporter> exporters) { public byte[] Generate(string format, ReportData data) { // Dictionary lookup instead of switch var exporter = exporters.FirstOrDefault( e => e.Format.Equals(format, StringComparison.OrdinalIgnoreCase)) ?? throw new NotSupportedException($"Format '{format}' not registered"); return exporter.Export(data); } }

The key point for interviews: This refactoring doesn't have to happen in one giant PR. Extract one case at a time. Keep the switch as a fallback for cases you haven't migrated yet. The switch shrinks with each PR until it disappears.

Q24: How do you apply OCP to a plugin architecture in .NET?
Hard
Think First A plugin architecture is OCP taken to its extreme: the host application is deployed and sealed, and new behavior arrives as separate DLLs loaded at runtime. The host literally cannot be modified — it just discovers and runs plugins.

The strong answer: A plugin architecture is the purest form of OCP. The host application defines interfaces (contracts), ships, and is sealed. Third-party developers write plugins that implement those interfaces in separate assemblies. The host discovers and loads them at runtime without recompilation.

// 1. Shared contract (published as a NuGet package) public interface IPlugin { string Name { get; } void Configure(IServiceCollection services); void Execute(PluginContext context); } // 2. Host application — discovers & loads plugins at startup public class PluginLoader { public IReadOnlyList<IPlugin> LoadPlugins(string pluginsDir) { var plugins = new List<IPlugin>(); foreach (var dll in Directory.GetFiles(pluginsDir, "*.dll")) { // Load assembly in isolated context (prevents version conflicts) var context = new AssemblyLoadContext( Path.GetFileNameWithoutExtension(dll), isCollectible: true); var assembly = context.LoadFromAssemblyPath( Path.GetFullPath(dll)); // Find all types that implement IPlugin var pluginTypes = assembly.GetTypes() .Where(t => typeof(IPlugin).IsAssignableFrom(t) && !t.IsAbstract); foreach (var type in pluginTypes) { if (Activator.CreateInstance(type) is IPlugin plugin) plugins.Add(plugin); } } return plugins; } } // 3. A third-party plugin (separate project, separate DLL) public class CustomReportPlugin : IPlugin { public string Name => "Custom Reports"; public void Configure(IServiceCollection services) => services.AddTransient<IReportGenerator, CustomReportGenerator>(); public void Execute(PluginContext ctx) => ctx.Logger.LogInformation("Custom Report plugin loaded!"); }

Key design decisions:

  • AssemblyLoadContext: Load each plugin in an isolated context so they can have different dependency versions without conflicts.
  • Shared contract via NuGet: The IPlugin interface lives in a shared package that both the host and plugins reference. This is the "USB spec" of the architecture.
  • Hot reload: With isCollectible: true, you can unload and reload plugins without restarting the host. True runtime extension.
Q25: What's the relationship between OCP and the Template Method pattern?
Hard
Think First Template Method defines a skeleton algorithm in a base class. Subclasses override specific steps without changing the skeleton. The skeleton is "closed," the steps are "open." That's OCP via inheritance.

The strong answer: Template MethodA behavioral design pattern where a base class defines the structure (skeleton) of an algorithm, with some steps left as abstract or virtual methods. Subclasses override those steps to provide specific behavior, without changing the algorithm's overall structure. is OCP achieved through inheritance rather than interfaces. The base class defines the algorithm's skeleton and marks specific steps as abstract or virtual. Those steps are the extension points. The skeleton is closed (sealed in the base class), while the steps are open (overridden in subclasses).

// The skeleton is CLOSED — this algorithm structure never changes public abstract class DataImporter { // Template method — sealed, nobody can change the order public async Task ImportAsync(string source) { var raw = await ReadDataAsync(source); // Step 1 — open var validated = Validate(raw); // Step 2 — open var transformed = Transform(validated); // Step 3 — open await SaveAsync(transformed); // Step 4 — open await NotifyAsync(transformed.Count); // Step 5 — has default } protected abstract Task<RawData> ReadDataAsync(string source); protected abstract ValidatedData Validate(RawData data); protected abstract TransformedData Transform(ValidatedData data); protected abstract Task SaveAsync(TransformedData data); // Virtual — has a default but can be overridden protected virtual Task NotifyAsync(int count) => Task.CompletedTask; // Default: no notification } // EXTENSION: new importer, skeleton untouched public class CsvImporter : DataImporter { protected override async Task<RawData> ReadDataAsync(string source) => await CsvParser.ParseAsync(source); protected override ValidatedData Validate(RawData data) => CsvValidator.Validate(data); protected override TransformedData Transform(ValidatedData data) => CsvTransformer.Transform(data); protected override Task SaveAsync(TransformedData data) => _repository.BulkInsertAsync(data); } // Another EXTENSION: different format, same skeleton public class JsonImporter : DataImporter { protected override async Task<RawData> ReadDataAsync(string source) => await JsonDocument.ParseAsync(File.OpenRead(source)); // ... each step customized for JSON protected override Task NotifyAsync(int count) => _slack.PostAsync($"Imported {count} JSON records"); }

OCP comparison — Strategy vs Template Method:

  • Strategy (interface-based OCP): The whole algorithm is swapped. The context doesn't know the internals.
  • Template Method (inheritance-based OCP): The algorithm structure is fixed. Only specific steps are swapped. The base class controls the flow.

When to use which: Use Template Method when the sequence of steps is important and shouldn't change (import: read, validate, transform, save). Use Strategy when the entire algorithm is replaceable (shipping cost: flat rate vs weight-based vs zone-based).

Q26: How does OCP apply at the architecture level (not just class level)?
Hard
Think First At the class level, OCP is about interfaces and polymorphism. At the architecture level, OCP is about service boundaries, message contracts, and system design that lets you add new services without modifying existing ones.

The strong answer: OCP scales far beyond individual classes. At the architecture level, the same principle applies: existing services should not need modification when new capabilities are added to the system. Here's how it manifests at each level:

1. Microservices: Each service is a "closed module." Adding a new recommendation engine means deploying a new service, not editing the order service. Services communicate through defined contracts (APIs, message schemas) — the architectural equivalent of interfaces.

2. Event-driven architecture: An order service publishes OrderPlaced events. It doesn't know or care who listens. Adding email notifications, inventory updates, or analytics tracking means adding new subscribers — the publisher is closed for modification.

3. API Gateway: The gateway routes requests to backend services. Adding a new service means adding a route configuration, not editing existing services. The gateway is the extension point.

4. Feature flags: New behavior is deployed but hidden behind a flag. "Extending" the system means flipping a flag, not deploying new code. The feature flag system itself is the OCP mechanism.

// Architectural OCP: Event-driven // OrderService publishes — never changes when new subscribers appear public class OrderService(IEventBus bus) { public async Task PlaceOrderAsync(Order order) { await _repository.SaveAsync(order); // Publish once. Who listens? OrderService doesn't care. await bus.PublishAsync(new OrderPlacedEvent(order)); } } // Subscriber 1: existed since day 1 — untouched public class EmailNotifier : IHandle<OrderPlacedEvent> { public Task HandleAsync(OrderPlacedEvent e) => _email.SendOrderConfirmation(e.Order); } // Subscriber 2: added 6 months later — zero changes to OrderService public class InventoryUpdater : IHandle<OrderPlacedEvent> { public Task HandleAsync(OrderPlacedEvent e) => _inventory.ReserveItems(e.Order.Items); } // Subscriber 3: added by a different TEAM — still zero changes public class FraudDetector : IHandle<OrderPlacedEvent> { public Task HandleAsync(OrderPlacedEvent e) => _fraudEngine.AnalyzeAsync(e.Order); }

The senior insight: Architecture-level OCP is more important than class-level OCP because the cost of modifying a deployed service is far higher than modifying a class. It involves deployment pipelines, testing across service boundaries, and coordinating between teams. Designing for architectural OCP from the start prevents the kind of cross-team coordination nightmares that slow organizations down.

Q27: Compare OCP in an event-driven system vs a request-response system.
Hard
Think First In request-response, the caller must know who to call. Adding a new handler means updating the caller or a router. In event-driven, the publisher doesn't know subscribers at all — extension is truly invisible.

The strong answer: Both architectures can follow OCP, but event-driven systems achieve it more naturally because the publisher-subscriber relationship is inherently decoupled.

Summary:

  • Request-response: OCP is possible (via interfaces, DI, mediator pattern) but the caller must be updated to route to new handlers. The caller is a soft OCP violation point.
  • Event-driven: OCP is automatic. The publisher fires an event and walks away. New subscribers appear without the publisher knowing. Extension is completely invisible to existing code.

You send a command/query, and MediatR routes it to the correct handler via DI. Adding a new handler for a new command requires zero changes to existing code. or a similar mediator in request-response systems to get event-like decoupling. The controller sends a command, the mediator routes it, and new handlers can be added without touching controllers.

Q28: How do you handle the tension between OCP and keeping things simple?
Hard
Think First OCP adds layers of indirection — interfaces, DI registrations, separate files. That complexity is justified when there's real variation. It's overengineering when there isn't. The skill is knowing the difference.

The strong answer: This is the question that reveals truly senior engineers. The answer is: OCP is a tool, not a religion. Apply it where the cost of future modification is high, and skip it where the cost of indirection exceeds the cost of direct editing.

When OCP complexity is justified:

  • Public APIs and libraries: Users can't modify your code. Extension points are essential.
  • Core business rules that vary by customer/region: Discount strategies, tax rules, validation rules that differ per market.
  • Integration points: Payment gateways, notification channels, export formats — these grow by nature.
  • Team boundaries: When different teams own different implementations, OCP prevents deployment coupling.

When OCP complexity is NOT justified:

  • Internal utilities with one implementation: Don't create IDateTimeProvider if you're never going to swap it (unless for testing).
  • Startup/prototype code: Move fast, validate the idea, then refactor with OCP once you know what varies.
  • Stable, simple logic: A helper method that calculates sales tax for one country doesn't need ITaxStrategy until you expand internationally.
  • One-off scripts and tools: A migration script that runs once doesn't need extensibility.

The decision framework:

  1. How likely is change? High certainty = apply OCP now. Low certainty = keep it simple.
  2. How costly is change? Public API = very costly, apply OCP. Internal method = cheap to edit, maybe skip OCP.
  3. How many people are affected? 50 consumers = apply OCP. 1 consumer (yourself) = keep it direct.

The mantra: "Simple until proven otherwise." Start concrete. Extract when you have evidence. The best code is the simplest code that handles today's requirements AND can be refactored to handle tomorrow's when tomorrow actually arrives.

Q30: How would you migrate a legacy codebase with 100+ switch statements toward OCP?
Hard
Think First You can't refactor 100 switches in one sprint. You need a triage strategy: which switches hurt the most? Start there. Leave stable ones alone. This is a marathon, not a sprint.

The strong answer: This is a real-world scenario I've seen in multiple legacy codebases. The key insight is: you don't refactor all 100 switches. You triage, prioritize, and migrate incrementally.

Phase 1: Triage (Week 1)

Categorize every switch statement into three buckets:

  1. Hot switches (refactor first): Switches that have been modified in the last 3 months, cause merge conflicts, or have TODOs saying "add new case for X." These are actively hurting the team.
  2. Warm switches (refactor later): Switches with 5+ cases that are growing slowly. They'll become hot eventually but aren't urgent.
  3. Cold switches (leave alone): Switches over stable, fixed domains (DayOfWeek, HttpMethod). They don't grow. Refactoring them adds complexity with zero benefit.

Phase 2: Strangler Fig Pattern (Ongoing)

For each hot switch, apply the Strangler FigA migration pattern named after a vine that slowly wraps around and replaces a tree. Instead of rewriting a module all at once, you build the new implementation alongside the old one, routing traffic incrementally. Eventually the old code is empty and can be deleted. approach:

  1. Write characterization tests for the existing switch (capture current behavior).
  2. Extract the interface from the common shape of the cases.
  3. Migrate one case at a time into its own class implementing the interface.
  4. Keep the switch as a fallback for unmigrated cases. It shrinks with each PR.
  5. Delete the switch when all cases are migrated.
// Phase 2: Hybrid approach — switch shrinks over time public class ReportGenerator { private readonly Dictionary<string, IReportExporter> _newExporters; public ReportGenerator(IEnumerable<IReportExporter> exporters) { _newExporters = exporters.ToDictionary(e => e.Format); } public byte[] Generate(string format, ReportData data) { // Try new OCP-compliant path first if (_newExporters.TryGetValue(format, out var exporter)) return exporter.Export(data); // Fallback to legacy switch for unmigrated formats // This switch SHRINKS with each PR until it's gone return format switch { "html" => LegacyHtmlExport(data), // TODO: migrate sprint 12 "rtf" => LegacyRtfExport(data), // TODO: migrate sprint 13 _ => throw new NotSupportedException(format) }; } }

Add a code review rule: no new switch statements that dispatch behavior by type. New features must use the interface + DI pattern from day one. This stops the bleeding while Phase 2 heals existing wounds.

Realistic timeline for 100 switches:

  • ~20 are "cold" (leave forever) = 0 effort
  • ~50 are "warm" (migrate when they're next touched) = organic, low-effort
  • ~30 are "hot" (actively refactor) = ~2 switches per sprint = ~15 sprints

The honest answer for interviewers: "I wouldn't try to fix them all at once.     Deep Dive — OCP Through the Ages

    OCP has evolved significantly since Bertrand Meyer coined it in 1988. Understanding the history isn't academic trivia — it explains why modern C# uses interfaces instead of inheritance, and reveals a deeper computer science problem that OCP only half-solves.

    Topic 1: Meyer's OCP vs Polymorphic OCP

    Bertrand Meyer's original 1988 formulation was simple: a module is "closed" when it's published for other modules to use, and "open" when it can be extended. His mechanism for extension? Inheritance. You extend a class by creating a subclass that overrides or adds behavior. The base class (the "closed" part) never changes.

    This worked well in Eiffel (Meyer's language), but by the early 1990s, the problems of deep inheritance were becoming clear. Robert C. Martin reinterpreted OCP for the real world: instead of extending via subclass, you extend by implementing an interface. The consumer depends on an abstraction, and new implementations plug in without touching the consumer.

    // Meyer's approach: extend via inheritance public class Logger { public virtual void Log(string message) { Console.WriteLine($"[LOG] {message}"); } } // Extension: override in a subclass public class FileLogger : Logger { public override void Log(string message) { File.AppendAllText("app.log", $"[LOG] {message}\n"); } } // Problem: what if you need BOTH console AND file? // You'd need multiple inheritance or complex chaining. // What if FileLogger also needs to add timestamps? // Now you have FileLogger > TimestampLogger > Logger. // 3 levels deep and it's already getting fragile. // Modern approach: extend via interface implementation public interface ILogger { void Log(string message); } public class ConsoleLogger : ILogger { public void Log(string message) => Console.WriteLine($"[LOG] {message}"); } public class FileLogger : ILogger { public void Log(string message) => File.AppendAllText("app.log", $"[LOG] {message}\n"); } // Need both? Easy — no inheritance chain needed public class CompositeLogger(IEnumerable<ILogger> loggers) : ILogger { public void Log(string message) { foreach (var logger in loggers) logger.Log(message); } } // Each implementation is independent — no fragile base class // New loggers never touch existing ones

    The modern consensus is clear: prefer polymorphic OCP (interfaces + composition) over Meyer's OCP (inheritance). Inheritance creates coupling between parent and child classes. Interfaces keep implementations independent.

    Topic 2: The Expression Problem

    Here's a deeper question that OCP raises: you can easily add new types that implement an interface (just create a new class). But what about adding new operations to all existing types? That's the Expression ProblemA classic computer science challenge: how do you add both new data types AND new operations without modifying existing code? OCP (with interfaces) makes adding new types easy, but adding a new method to the interface forces changes to all existing implementations. — one of the most famous problems in programming language theory.

    OCP solves half the problem. With interfaces, adding a new HexagonShape that implements IShape is easy. But if you want to add a new operation — say Draw() — to the IShape interface, you must modify every existing implementation (Circle, Rectangle, Triangle). That's a modification, not an extension.

    // OCP makes this trivial: // Add Pentagon WITHOUT touching Circle or Rectangle public class Pentagon : IShape { public double Area() => /* formula */; public double Perimeter() => /* formula */; } // Zero existing code changed. OCP wins! // OCP struggles here: // Adding Draw() to IShape means editing the interface public interface IShape { double Area(); double Perimeter(); void Draw(Canvas canvas); // NEW — but now Circle, Rectangle, // Triangle all must implement it! } // That's modification, not extension. OCP violation. // The Visitor pattern attempts to solve this: public interface IShapeVisitor { void Visit(Circle c); void Visit(Rectangle r); void Visit(Triangle t); } public class DrawVisitor : IShapeVisitor { public void Visit(Circle c) => /* draw circle */; public void Visit(Rectangle r) => /* draw rectangle */; public void Visit(Triangle t) => /* draw triangle */; } // Now you can add new operations (new visitors) // without changing shapes. BUT adding a new shape // forces changes to every visitor. Trade-off!

    The Expression Problem has no perfect solution in any mainstream language. OCP with interfaces makes adding types easy but operations hard. The Visitor pattern flips it — new operations are easy but new types are hard. In practice, you choose based on which direction of change is more common in your domain. For most business applications, new types (new payment methods, new report formats) are more common, so interface-based OCP is the right choice.

    Topic 3: OCP in Functional C#

    You don't need classes and interfaces to follow OCP. Modern C# supports a functional style where Func<T>, delegates, and higher-order functionsFunctions that take other functions as parameters or return functions as results. In C#, this is done with Func<T> and Action<T>. They let you parameterize behavior without creating classes. create extension points that are just as powerful — and often simpler.

    // OCP via function composition — no classes needed! public class Pipeline<T> { private readonly List<Func<T, T>> _steps = []; // Extension point: add steps without modifying existing ones public Pipeline<T> AddStep(Func<T, T> step) { _steps.Add(step); return this; } public T Execute(T input) => _steps.Aggregate(input, (current, step) => step(current)); } // Usage: build pipelines by composing functions var orderPipeline = new Pipeline<Order>() .AddStep(order => ApplyDiscount(order)) // existing .AddStep(order => CalculateTax(order)) // existing .AddStep(order => ApplyLoyaltyPoints(order)) // NEW! no edits .AddStep(order => RoundTotal(order)); // NEW! no edits var processed = orderPipeline.Execute(myOrder); // OCP via delegates — even simpler public class EventProcessor { // Extension point: subscribers can be added anytime public event Func<OrderEvent, Task>? OnOrderPlaced; public event Func<OrderEvent, Task>? OnOrderShipped; public async Task ProcessAsync(OrderEvent evt) { if (OnOrderPlaced != null) await OnOrderPlaced.Invoke(evt); } } // Extend by subscribing — zero changes to EventProcessor processor.OnOrderPlaced += async evt => await emailService.SendConfirmation(evt); processor.OnOrderPlaced += async evt => await analytics.Track("order.placed", evt); processor.OnOrderPlaced += async evt => // NEW handler await slackNotifier.PostToChannel(evt);

    Functional OCP is especially powerful for transformation pipelines, event handlers, and validation chains. It's lighter than the full interface + class + DI approach and works well for in-process extension where you don't need the full power of polymorphism.

    OCP has two historical versions: Meyer's (inheritance-based, 1988) and Martin's (interface-based, 1990s). Modern C# uses the latter. OCP solves half the Expression Problem (new types are easy; new operations are hard). Functional OCP with Func<T> and delegates provides a lightweight alternative when full interface hierarchies are overkill.

    Mini-Project: Extensible Tax Calculator

    Let's build a tax calculator that handles different tax rules by country. We'll start with the naive approach (giant switch), evolve to an interface-based design, and finally build a production-ready plugin architecture. This is the same evolution your real-world code will follow — and by the end, you'll have a template for any extensible system.

    You start simple. One class, one method, a switch on the country code. It works for 3 countries. Then the business expands to 15 countries. Every new country means editing this file, re-testing all existing countries, and praying the India tax changes didn't break the UK calculation.

    Migration Guide

    You've inherited a codebase with 500-line switch statements and if-else chains that grow with every feature. You can't rewrite everything at once. Here's how to refactor toward OCP safely and incrementally — one step per pull request.

    Find the switch statement or if-else chain that changes every time a new requirement arrives. This is where OCP is most needed and most valuable.

    # Find switch statements in your codebase grep -rn "switch\s*(" --include="*.cs" src/ | head -20 # Find if-else chains with 4+ branches (likely OCP candidates) grep -rn "else if" --include="*.cs" src/ | wc -l # Check git history: which files change with every new feature? git log --oneline --name-only | grep ".cs" | sort | uniq -c | sort -rn | head -10

    Risk level: Low. You're just reading code and git history. No changes to production.

    Look at what each case block does. They all do the same kind of thing (apply a discount, export a format, calculate tax) but in different ways. That "kind of thing" becomes your interface method.

    // BEFORE: the switch tells you the interface shape switch (format) { case "pdf": return RenderPdf(data); // returns byte[] case "excel": return RenderExcel(data); // returns byte[] case "csv": return RenderCsv(data); // returns byte[] } // AFTER: extract the common shape into an interface public interface IReportExporter { string Format { get; } byte[] Export(ReportData data); } // Keep the switch temporarily — it becomes a factory // for the migration period. Delete it in Step 4.

    Risk level: Low-Medium. You're adding a new interface file. No existing behavior changes. The switch still works as before.

    Extract each case block into its own class. Start with the simplest, least risky case. Verify tests pass. Move to the next.

    // Extracted from the switch case "pdf" block public class PdfExporter : IReportExporter { public string Format => "pdf"; public byte[] Export(ReportData data) { using var doc = new PdfDocument(); // ... exact same logic from the old switch case ... return doc.Save(); } } // Extracted from the switch case "excel" block public class ExcelExporter : IReportExporter { public string Format => "excel"; public byte[] Export(ReportData data) { using var workbook = new XLWorkbook(); // ... exact same logic from the old switch case ... using var stream = new MemoryStream(); workbook.SaveAs(stream); return stream.ToArray(); } }

    Risk level: Medium. You're moving code between files. Run all existing tests after each extraction to catch regressions.

    Register all implementations in DI. Update the consumer to inject IEnumerable<IReportExporter>. Delete the switch statement entirely. The DI container is now the "registry" of available implementations.

    // Program.cs — register all exporters builder.Services.AddSingleton<IReportExporter, PdfExporter>(); builder.Services.AddSingleton<IReportExporter, ExcelExporter>(); builder.Services.AddSingleton<IReportExporter, CsvExporter>(); // The consumer: switch is GONE public class ReportService(IEnumerable<IReportExporter> exporters) { public byte[] GenerateReport(string format, ReportData data) { var exporter = exporters.FirstOrDefault( e => e.Format.Equals(format, StringComparison.OrdinalIgnoreCase)) ?? throw new NotSupportedException($"Unknown format: {format}"); return exporter.Export(data); } } // From now on, adding a new format = new class + 1 DI line // ZERO changes to ReportService Risk level: Medium-High. This is the step where you delete the old switch. Test thoroughly! Run integration tests, not just unit tests. Deploy to staging before production. Keep the old code in a branch for a week as a safety net.     Migrate to OCP in 4 safe steps: (1) find the switch/if-else that grows with features, (2) extract an interface from the common behavior, (3) create one class per case, (4) wire through DI and delete the switch. One PR per step. Each step is independently deployable and testable.

    OCP doesn't exist in isolation — it works hand-in-hand with other SOLID principles and design patterns. Here's where to go next to build a complete understanding of extensible software design.

    Solution 1: Extract an Interface

    The first step is to identify the variation. In the report exporter switch statement, every case block does the same kind of thing — export data to a specific format — but each format has different logic. That "same kind of thing" becomes your interface.

    Instead of one method with a switch, you define IReportExporter with an Export() method. Each format becomes a class that implements this interface. The consumer doesn't care which format it's working with — it just calls Export().

    IReportExporter.cs
    // The interface IS the extension point
public interface IReportExporter
{
    string Format { get; }          // "pdf", "excel", "csv"
    byte[] Export(ReportData data);  // The actual export logic
}

// Each format: its own class, its own file, its own tests
// Adding a new format means creating a new class
// NOT editing the switch statement

    Why this works: The interface creates a "seam" in your code — a boundary where new behavior can plug in. The consumer depends on IReportExporter, not on any specific exporter. That dependency on abstraction is what makes it "open for extension."

    Solution 2: One Class Per Format

    Each case block from the old switch becomes its own class. Each class is small, focused, and independently testable. If the PDF rendering library changes, only PdfExporter changes — ExcelExporter and CsvExporter are completely unaffected.

    Exporters.cs
    public class PdfExporter : IReportExporter

    public string Format => "pdf";
    public byte[] Export(ReportData data)
    {
        using var doc = new PdfDocument();
        var page = doc.AddPage();
        // ... PDF rendering logic ...
        return doc.Save();
    }

public class ExcelExporter : IReportExporter
{
    public string Format => "excel";
    public byte[] Export(ReportData data)
    {
        using var workbook = new XLWorkbook();
        var sheet = workbook.AddWorksheet("Report");
        // ... Excel generation logic ...
        using var ms = new MemoryStream();
        workbook.SaveAs(ms);
        return ms.ToArray();
    }
}

public class CsvExporter : IReportExporter
{
    public string Format => "csv";
    public byte[] Export(ReportData data)
    {
        var sb = new StringBuilder();
        sb.AppendLine(string.Join(",", data.Headers));
        foreach (var row in data.Rows)
            sb.AppendLine(string.Join(",", row));
        return Encoding.UTF8.GetBytes(sb.ToString());
    }
}

    Solution 3: DI Wiring

    The final piece: register all implementations in the DI containerDependency Injection container — the system in ASP.NET Core that creates objects and provides them to classes that need them. Instead of classes creating their own dependencies, the container creates and injects them. and let the consumer receive them automatically. No factory methods, no service locators — just clean constructor injection.

    Program.cs
    var builder = WebApplication.CreateBuilder(args);

builder.Services.AddTransient<IReportExporter, PdfExporter>();
builder.Services.AddTransient<IReportExporter, ExcelExporter>();

// The consumer: receives ALL registered exporters
builder.Services.AddScoped<ReportService>();

var app = builder.Build();
app.Run();

// ReportService receives IEnumerable<IReportExporter>
// It doesn't know or care which exporters exist
// Adding a new format = new class + 1 DI line above
// ZERO changes to ReportService

    The magic: When you inject IEnumerable<IReportExporter>, .NET's built-in DI gives you ALL registered implementations. Adding a new format is literally one new class file and one new AddTransient line. The ReportService is permanently closed for modification.
