The Principle Defined

Let's get precise. The Liskov Substitution Principle was first described by Barbara LiskovA pioneering computer scientist who delivered her seminal keynote "Data Abstraction and Hierarchy" at the OOPSLA conference in 1987. She formalized the principle with Jeannette Wing in 1994. Liskov won the Turing Award in 2008 for her contributions to programming language design and software engineering. in a 1987 keynote, then formally defined with Jeannette Wing in 1994. Here's the official statement:

"If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering any of the desirable properties of the program."

— Barbara Liskov & Jeannette Wing, 1994

That's pretty dense, so let's unpack it in plain English. The principle boils down to three rules that every subtype must follow. Think of them as the "behavioral contract" — the promises that a base type makes to anyone who uses it. Every subtype must keep those promises.

The Three Rules of LSP

Rule 1: Preconditions can't be strengthened. A preconditionA condition that must be true BEFORE a method runs. It defines what inputs the method accepts. For example, a Deposit(amount) method might require that amount > 0. That's a precondition. The caller is responsible for meeting preconditions before calling the method. is a requirement on the input — what the method demands before it will run. If the base type says "I accept any positive number," the subtype can't suddenly say "I only accept even positive numbers." The subtype can be MORE relaxed (accept zero too), but it can't be MORE strict. Why? Because the caller already wrote code that passes any positive number. If the subtype rejects half of those inputs, the caller's code breaks.

Rule 2: Postconditions can't be weakened. A postconditionA condition that must be true AFTER a method runs. It defines what the method promises to deliver. For example, a GetUser() method might guarantee that it never returns null — it either returns a valid User or throws an exception. That's a postcondition. The caller relies on postconditions to write their code. is a promise about the output — what the method guarantees after it runs. If the base type promises "I always return a non-null result," the subtype can't return null. The subtype can make STRONGER promises (return a specific subtype instead of the base return type), but it can't make WEAKER ones. Why? Because the caller wrote code that assumes the result is never null. If the subtype breaks that promise, the caller gets a NullReferenceException.

Rule 3: Invariants must be preserved. An invariantA condition that is ALWAYS true for an object, throughout its lifetime. For a Rectangle, the invariant is: width and height are independent — changing one doesn't affect the other. For a BankAccount, the invariant might be: balance is always >= 0 (no overdraft). Invariants define the fundamental "rules of reality" for the object. is something that must ALWAYS be true about the object — during its entire lifetime. For a Rectangle, the invariant is that width and height are independent. You can set one without affecting the other. If a Square inherits from Rectangle and breaks this invariant (setting width also sets height), it violates LSP. Why? Because any code that relies on width and height being independent will produce wrong results.

LSP isn't about what a subtype IS — it's about how a subtype BEHAVES. A Square IS-A Rectangle mathematically. Every geometry textbook says so. But in OOP, a Square that changes both dimensions when you set one BEHAVES differently than a Rectangle. And behavior is what code depends on. When you write rect.Width = 4; rect.Height = 5;, you're counting on the behavior that those are independent operations. If they're not, your code breaks — even though the types are "correct."

Code Implementation

Let's see LSP in action with three complete C# examples. Each one starts with a violation — code that compiles but produces wrong results — then shows the fix. The pattern is always the same: replace broken inheritance with independent implementations of a shared interface.

The Rectangle/Square Problem

This is the most famous LSP violation in all of object-oriented programming. It's so well-known that interviewers use it as a litmus test. Let's start with the broken version, understand exactly why it fails, and then fix it properly.

The idea seems perfectly logical: a square is a special kind of rectangle where width equals height. So it should inherit from Rectangle, right? Let's see what happens.

// A rectangle with independently settable width and height. // The INVARIANT: setting width does NOT affect height (and vice versa). // This invariant is critical — callers DEPEND on it. public class Rectangle { public virtual int Width { get; set; } public virtual int Height { get; set; } // Area is width times height. Simple and correct // — as long as width and height are truly independent. public int Area => Width * Height; } // BAD: Square inherits from Rectangle. // "A square IS a rectangle" — mathematically correct, OOP disaster. public class Square : Rectangle { // Override Width setter to also change Height. // This keeps the "square" constraint (w == h), // but BREAKS Rectangle's invariant (independent dimensions). public override int Width { get => base.Width; set { base.Width = value; base.Height = value; } } // Override Height setter to also change Width. // Same problem — the caller thinks they're setting only height. public override int Height { get => base.Height; set { base.Width = value; base.Height = value; } } } // This test PASSES for Rectangle, FAILS for Square: void AssertAreaIsCorrect(Rectangle rect) { rect.Width = 4; rect.Height = 5; Debug.Assert(rect.Area == 20); // FAILS for Square! Area is 25. } // The RIGHT approach: don't use inheritance. // Rectangle and Square implement the same INTERFACE independently. public interface IShape { double Area(); double Perimeter(); string Name { get; } } // Rectangle implements IShape with independent width and height. // No virtual setters. No risk of overrides breaking the contract. public class Rectangle : IShape { public double Width { get; } public double Height { get; } public string Name => "Rectangle"; public Rectangle(double width, double height) { if (width <= 0 || height <= 0) throw new ArgumentException("Dimensions must be positive."); Width = width; Height = height; } public double Area() => Width * Height; public double Perimeter() => 2 * (Width + Height); } // Square implements IShape with a single side length. // It does NOT inherit from Rectangle. No shared state. No surprises. public class Square : IShape { public double Side { get; } public string Name => "Square"; public Square(double side) { if (side <= 0) throw new ArgumentException("Side must be positive."); Side = side; } // Area is side * side — correct by definition. // No broken invariants because there's nothing to break. public double Area() => Side * Side; public double Perimeter() => 4 * Side; } // Now ANY IShape works correctly with this function. // No surprises, no special cases, no silent wrong answers. void PrintShapeInfo(IShape shape) { Console.WriteLine($"{shape.Name}: area = {shape.Area():F1}"); } PrintShapeInfo(new Rectangle(4, 5)); // Rectangle: area = 20.0 PrintShapeInfo(new Square(5)); // Square: area = 25.0 PrintShapeInfo(new Circle(3)); // Circle: area = 28.3 // The function doesn't know or care what shape it gets. // Every shape calculates its own area correctly. // That's LSP: substitution without surprises.

The key insight: making Rectangle's properties virtual was the mistake. It created an extension point that allowed subclasses to break the invariant. In the fixed version, both Rectangle and Square implement IShape independently, use constructor-only initialization (immutable), and each calculates its own area correctly. There's no inheritance relationship to abuse.

The Payment Processor That Explodes

This is a more realistic example from production code. Imagine a payment system with an IPaymentProcessor interface. Everything works fine with credit cards and PayPal. Then someone adds a cryptocurrency processor that throws NotSupportedException for refunds. Surprise! The calling code assumed every processor supports refunds.

// This interface promises too much. // It says every processor can process AND refund. // But what if some payment methods can't be refunded? public interface IPaymentProcessor { Task<PaymentResult> ProcessAsync(decimal amount); Task<RefundResult> RefundAsync(string transactionId, decimal amount); } // Crypto can process payments, but CANNOT refund. // Blockchain transactions are irreversible by design. public class CryptoProcessor : IPaymentProcessor { public async Task<PaymentResult> ProcessAsync(decimal amount) { // This works fine — crypto CAN accept payments. var txHash = await _blockchain.SendAsync(amount); return new PaymentResult(true, txHash); } public Task<RefundResult> RefundAsync(string transactionId, decimal amount) { // LSP VIOLATION: the interface promises Refund works. // Throwing an exception is a SURPRISE — the caller // didn't expect this. They assumed refund always works. throw new NotSupportedException( "Crypto transactions cannot be refunded."); } } // Split into two interfaces — each makes only the promises // it can keep. This is called Interface Segregation (ISP), // and it directly supports LSP. // Every processor can process a payment. public interface IPaymentProcessor { Task<PaymentResult> ProcessAsync(decimal amount); } // Only processors that support refunds implement this. public interface IRefundableProcessor : IPaymentProcessor { Task<RefundResult> RefundAsync(string transactionId, decimal amount); } // Credit card supports both — implements IRefundableProcessor. public class CreditCardProcessor : IRefundableProcessor { public async Task<PaymentResult> ProcessAsync(decimal amount) => await _gateway.ChargeAsync(amount); public async Task<RefundResult> RefundAsync(string txId, decimal amount) => await _gateway.RefundAsync(txId, amount); } // Crypto supports only processing — implements IPaymentProcessor. // It NEVER promises to refund, so it can NEVER break that promise. public class CryptoProcessor : IPaymentProcessor { public async Task<PaymentResult> ProcessAsync(decimal amount) => await _blockchain.SendAsync(amount); // No RefundAsync method. No exception. No surprise. } // Usage: code that needs refunds explicitly asks for IRefundableProcessor. // Code that only needs to process accepts IPaymentProcessor. // The type system enforces LSP at compile time.

The principle at work: instead of one fat interface that makes promises not every implementation can keep, we split it into a base interface (what everyone can do) and an extended interface (what some can do additionally). Now the compiler prevents LSP violations — you literally can't call RefundAsync on a CryptoProcessor because the type system won't let you. The violation was caught at compile time, not at 3 AM in production.

IReadOnlyList vs IList

This one is straight from the .NET framework itself — a lesson Microsoft learned about LSP. The IList<T> interface has an Add() method. But what happens when someone passes a ReadOnlyCollection<T> as an IList<T>?

// IList<T> promises you can Add items. void AddDefaults(IList<string> items) { items.Add("default-1"); items.Add("default-2"); // This works fine with List<string>. } // But ReadOnlyCollection also implements IList<T>! var readOnly = new ReadOnlyCollection<string>(new[] { "a", "b" }); IList<string> list = readOnly; // compiles fine AddDefaults(list); // BOOM! NotSupportedException at runtime. // The caller trusted that IList<T>.Add() works. // ReadOnlyCollection broke that trust. // Use IReadOnlyList when you only need to READ. // This way, ReadOnlyCollection is a perfect substitute. void PrintAll(IReadOnlyList<string> items) { foreach (var item in items) Console.WriteLine(item); // No Add(), no Remove(), no surprise exceptions. } // Use IList only when you genuinely NEED to modify. void AddDefaults(IList<string> items) { items.Add("default-1"); items.Add("default-2"); } // Now the compiler protects you: var readOnly = new ReadOnlyCollection<string>(new[] { "a", "b" }); PrintAll(readOnly); // Works perfectly — LSP compliant. // AddDefaults(readOnly); // Won't compile — readOnly is IReadOnlyList, // not IList. The compiler PREVENTS the violation. var mutable = new List<string> { "a", "b" }; PrintAll(mutable); // Also works — List implements IReadOnlyList too. AddDefaults(mutable); // Works — List implements IList too.

This example shows a real-world LSP lesson from the .NET BCL. The solution: use the most restrictive interface that meets your needs. If you only read from a collection, accept IReadOnlyList<T>. If you need to modify it, accept IList<T>. The type system then prevents callers from passing in implementations that can't honor the contract.

 Three examples, one pattern. The Rectangle/Square problem shows the classic invariant violation. The Payment Processor shows a postcondition violation (throwing unexpected exceptions). Junior vs Senior Implementation

Build a bird simulation. The zoo wants every bird to fly and make a sound. Later, they want to add penguins, ostriches, and kiwis. How you design the hierarchy determines whether "adding a penguin" takes 5 minutes or causes a cascade of runtime exceptions.

How a Junior Thinks

"All birds fly and make sounds. I'll create a Bird base class with Fly() and MakeSound(). If a bird can't fly, I'll override Fly() to throw an exception. Simple!"

// The "all birds fly" assumption — sounds reasonable at first. public abstract class Bird { public string Name { get; } protected Bird(string name) => Name = name; // Every bird makes a sound — this is fine. public abstract string MakeSound(); // Every bird can fly — THIS is the problem. // What about penguins? Ostriches? Kiwis? Emus? public virtual void Fly() { Console.WriteLine($"{Name} is flying!"); } } // Eagle — works perfectly. Eagles fly. public class Eagle : Bird { public Eagle() : base("Eagle") { } public override string MakeSound() => "Screech!"; // Fly() inherited — works fine. } // Penguin — LSP violation! Penguins can't fly. public class Penguin : Bird { public Penguin() : base("Penguin") { } public override string MakeSound() => "Squawk!"; // The junior "fix": override Fly() to throw. // This COMPILES, but any code calling bird.Fly() will crash. public override void Fly() { throw new NotSupportedException("Penguins can't fly!"); } } // Ostrich — same problem. public class Ostrich : Bird { public Ostrich() : base("Ostrich") { } public override string MakeSound() => "Boom boom!"; public override void Fly() { throw new NotSupportedException("Ostriches can't fly!"); } }

Why This Hurts

Silent Runtime Bombs

Code that calls bird.Fly() works for Eagles but explodes for Penguins. The compiler says everything is fine — you only discover the crash when a Penguin reaches the flight simulation at runtime. In production, that could be a customer-facing error.

Defensive Programming Everywhere

To avoid crashes, every caller must now check: if (bird is not Penguin && bird is not Ostrich) before calling Fly(). That's exactly what LSP warns against — the caller shouldn't need to know which specific subtype it has. If you have to check the type, your hierarchy is broken.

Every New Flightless Bird = More Checks

When Kiwis and Emus are added, every if (bird is not Penguin) check needs to be updated. Miss one? Runtime crash. This gets worse over time — the list of exceptions grows, and each caller might have a different set of checks.

When you see a method that throws NotSupportedException, that's almost always an LSP violation in disguise. The subtype is pretending to support an operation it actually can't perform. The interface or base class promised that the method works, and the subtype is breaking that promise.

How a Senior Thinks

"Not all birds fly. That's not an edge case — it's a fundamental fact about birds. I should NOT put Fly() in the base Bird interface. Instead, I'll separate the capabilities: all birds make sounds, but only SOME birds fly. The type system should enforce this — if you need a flying bird, ask for an IFlyingBird."

// The base contract: every bird has a name and makes a sound. // Notice: NO Fly() method. Design Decisions

By splitting IBird (all birds) from IFlyingBird (birds that fly), we ensure that no subtype ever has to throw NotSupportedException. If a class implements IFlyingBird, it genuinely can fly. If it only implements IBird, nobody expects it to fly. The type system does the enforcement — not runtime checks. The junior asked: "Is a Penguin a Bird?" (Yes.) The senior asked: "Does a Penguin BEHAVE LIKE a FlyingBird?" (No.) The IS-A test is about taxonomy. The BEHAVES-LIKE test is about contracts. LSP cares about behavior, not taxonomy. Always ask "behaves like?" before using inheritance. In the junior version, you discover the Penguin can't fly when it crashes at runtime. In the senior version, you discover it at compile time — the Penguin can't be passed to RunFlightShow() because it doesn't implement IFlyingBird. Compile-time errors are infinitely cheaper than production crashes.

Bottom Line

The junior puts Fly() in the base class and throws NotSupportedException for penguins. The senior splits the interface: IBird for all birds, IFlyingBird for birds that can actually fly. The key difference: the senior uses the type system to prevent violations at compile time, not runtime exceptions to handle them after the fact.

LSP has a fascinating origin story. It started as an academic concept in the late 1980s, got formalized in the 1990s, was popularized by SOLID in the 2000s, and is now baked into .NET's type system through features like generic varianceC# generics support covariance (out T — reading is safe) and contravariance (in T — writing is safe). Barbara Liskov delivered her keynote "Data Abstraction and Hierarchy" at the OOPSLAObject-Oriented Programming, Systems, Languages, and Applications — the premier academic conference for object-oriented programming. First held in 1986, it became the birthplace of many foundational OOP concepts. It was merged into SPLASH (Systems, Programming, Languages, and Applications: Software for Humanity) in 2010. conference in 1987. She described a key property that subtypes should have: any program that works with a supertype should continue to work when a subtype is used instead. She wasn't just talking about method signatures matching — she meant the behavior should be compatible.

At this point, the idea was informal. There were no precise rules. Liskov was observing a problem she saw in real Eiffel and Smalltalk programs: inheritance hierarchies that looked correct but produced wrong results. The Rectangle/Square problem was already being discussed in academic circles.

Seven years later, Liskov and Jeannette Wing published the formal definition. They introduced the precise concepts of behavioral subtypingThe formal theory behind LSP. It says that subtype substitutability must be defined by behavior (preconditions, postconditions, invariants), not just by type signatures. A type S is a behavioral subtype of T if every provable property of T also holds for S. This is much stronger than structural subtyping (just matching method signatures). — the three rules we covered in Section 4: preconditions can't be strengthened, postconditions can't be weakened, and invariants must be preserved.

This was the mathematical foundation. It gave programmers a precise vocabulary for discussing substitutability. Before this, the best you could say was "it should work." After this, you could say exactly why it didn't — "the subtype strengthened the precondition" or "the subtype weakened the postcondition."

Robert C. Martin (Uncle Bob) included LSP as the "L" in SOLID in his 2003 book Agile Software Development: Principles, Patterns, and Practices. He popularized the Rectangle/Square example that we've been using throughout this page. He also connected LSP to practical software design — showing that LSP violations lead to fragile hierarchies, defensive if/else checks, and unintended coupling.

Martin's key contribution was making LSP accessible to working programmers. The formal 1994 paper was rigorous but academic. Martin translated it into practical advice: "if you have to check what type a subclass is before using it, you've got an LSP violation."

When .NET Core shipped with built-in dependency injectionInstead of a class creating its own dependencies (new EmailSender()), the dependencies are "injected" from outside, usually through the constructor. This makes it trivial to swap implementations. ASP.NET Core has DI built into its core architecture., LSP became critical for everyday development. Every service registered in DI makes an LSP promise: "this implementation will honor the interface contract." If it doesn't — if your custom ILogger silently drops messages, or your Stream subclass doesn't support Read() — the whole system misbehaves.

.NET Core also pushed generic constraints (where T : IComparable<T>) which are essentially LSP enforcement — you can only plug in types that satisfy the behavioral requirements.

C# 8 and later added language features that directly support LSP. Default interface methods let you add new behavior to an interface without breaking existing implementations — a direct LSP concern (adding a method to an interface used to force all implementations to change). Covariant return types (C# 9) let overrides return a more specific type, which strengthens the postcondition — always safe under LSP. Pattern matching makes it easy to work with different subtypes without violating LSP through type-checking.

Comparisons

LSP gets mixed up with other SOLID principles all the time — especially OCP and ISP. They're related, but each solves a different problem. LSP is about behavior: "does the subtype keep its promises?" OCP is about extension: "can I add new behavior without editing old code?" ISP is about scope: "is the interface small enough to implement correctly?" Understanding these distinctions matters in interviews and in real design decisions. Let's break each comparison down.

LSP vs OCP

LSP says: "Subtypes must behave correctly when substituted for their base types." OCP says: "Extend behavior by adding new code, not by modifying existing code." They sound like different concerns, but they're tightly coupled. Here's why: OCP tells you to extend your system by adding new implementations of an interface. But what guarantees those new implementations actually WORK as drop-in replacements? That's LSP. Without LSP, every OCP extension is a gamble — you add a new implementation, plug it in, and cross your fingers that it doesn't break the callers.

Think of it this way: OCP is the permission to extend. LSP is the quality control that ensures extensions are safe. OCP opens the door to new implementations; LSP makes sure nothing dangerous walks through that door.

OCP says "add new implementations freely." LSP says "but those implementations must honor the contract." Without LSP, OCP is dangerous — you'd add new classes that compile fine but silently break callers at runtime. They work as a pair: OCP provides the extensibility, LSP provides the trust.

LSP vs ISP

LSP says: "Don't break the promises your base type made." ISP says: "Don't make promises you can't keep." They attack the same problem from opposite directions. ISP is preventive — it keeps interfaces small so that every implementor can realistically fulfill every method. LSP is detective — it catches the cases where an implementor failed to fulfill a method properly.

Here's the classic example: imagine an IReadableWritable interface with both Read() and Write() methods. A read-only file system can implement Read() perfectly but has to throw NotSupportedException for Write(). That's an LSP violation — the subtype can't fulfill the contract. But the root cause is an ISP violation — the interface is too fat. Split it into IReadable and IWritable, and the read-only file system only implements IReadable. No LSP violation because no broken promise.

ISP keeps interfaces small enough that every implementor can genuinely fulfill every method. This eliminates the most common source of LSP violations: methods that throw NotSupportedException because the implementor can't do what the interface demands. Fix the interface design (ISP) and you prevent the behavioral violation (LSP).

LSP vs Design by Contract

Design by ContractDesign by Contract (DbC): A methodology invented by Bertrand Meyer where every method has explicit preconditions (what must be true before calling), postconditions (what will be true after calling), and invariants (what is always true about the object). Think of it as a legal contract between caller and callee. (DbC) is the formal, mathematical version of what LSP describes informally. Bertrand Meyer invented DbC in the 1980s with his language Eiffel, and Barbara Liskov independently formalized the substitution principle around the same time. They arrived at the same core insight from different angles.

DbC uses three explicit mechanisms: preconditions (what must be true before you call a method), postconditions (what will be true after the method returns), and invariants (what is always true about the object's state). LSP applies these to subtyping with three rules:

DbC Concept LSP Rule What It Means
Preconditions Can't be stricter in subtype If the base accepts any positive number, the subtype can't reject numbers below 10. It must accept at LEAST what the base accepts.
Postconditions Can't be weaker in subtype If the base promises a non-null return, the subtype can't return null. It must deliver at LEAST what the base promises.
Invariants Must be preserved in subtype If the base guarantees that Width and Height are independent, the subtype (e.g., Square) can't secretly couple them.
DbC gives you the formal language — preconditions, postconditions, invariants. LSP gives you the rule for how those contracts change (or don't) in subtypes. In practice, you rarely write formal contracts in C#, but the mental model is invaluable: every time you override a method, ask "Am I making the preconditions stricter? Am I making the postconditions weaker? Am I breaking an invariant?" Three key comparisons: (1) LSP vs OCP — LSP is the safety net that ensures OCP extensions actually work. OCP gives permission to extend; LSP ensures quality. (2) LSP vs ISP — ISP prevents LSP violations by keeping interfaces small enough to implement correctly. Fix the interface (ISP) and you prevent the broken promise (LSP). (3) LSP vs Design by Contract — LSP is DbC applied to subtyping. Subtypes can't tighten preconditions, can't weaken postconditions, and must preserve invariants.

SOLID Connections

LSP doesn't live in isolation — it's the bridge that connects the rest of SOLID together. SRP and ISP keep interfaces clean. OCP uses polymorphism to extend. DIP inverts dependencies so you depend on abstractions. But NONE of that works if substitutions break behavior. LSP is the principle that makes polymorphism trustworthy, and without trust in polymorphism, the entire SOLID framework collapses. Here's how each principle connects to LSP.

Relationship: SRP supports LSP by keeping classes focused. A God classGod class: a class that tries to do everything — data access, business logic, formatting, validation, logging. It has so many responsibilities that it's nearly impossible to extend or substitute correctly. Splitting it into focused classes is the SRP cure. with 10 responsibilities is nearly impossible to substitute correctly — a subclass would need to honor contracts across all 10 dimensions simultaneously. But a focused class with one responsibility has a narrow, well-defined contract that's straightforward to honor in a subclass.

Example: An OrderService that validates, prices, ships, and emails is a substitution nightmare. Subclassing it means you must get validation, pricing, shipping, AND emailing all correct. But if you split it into OrderValidator, PricingEngine, ShippingService, and EmailSender (SRP), each one has a small contract that's easy for a subtype to honor (LSP).

The fewer responsibilities a class has, the fewer promises its subtypes need to keep. SRP shrinks the contract surface area, making correct substitution practical rather than heroic.

Relationship: OCP depends on LSP. OCP says "extend by adding new implementations." But what if a new implementation doesn't actually work as a drop-in replacement? That's exactly the disaster LSP prevents. Without LSP, OCP is like adding new plug-in modules to a spaceship without testing whether they'll blow up in space — the extension mechanism exists, but there's no guarantee the extension is safe.

Example: You follow OCP by defining IPaymentProcessor and adding CryptoProcessor. But CryptoProcessor.Refund() throws NotSupportedException. Every caller that processes refunds through IPaymentProcessor now explodes. OCP gave you the extension point. LSP violations made it a trap. If LSP had been honored, the hierarchy would've been redesigned so that CryptoProcessor only implements the methods it can actually fulfill.

OCP provides the ability to extend. LSP guarantees extensions are safe. Without LSP, every OCP extension is a ticking time bomb — it compiles, it looks right, but it blows up when callers encounter unexpected behavior.

This IS the current principle

LSP is the principle we've been exploring throughout this entire page. Subtypes must be substitutable for their base types without altering program correctness. Every subclass, every interface implementation, every generic type parameter must honor the contract of its base — no exceptions, no surprises, no silent failures. It's the guarantee that makes polymorphism trustworthy.

Relationship: ISP supports LSP by keeping interfaces small and focused. A fat interface with 12 methods forces every implementor to deal with all 12 — even if 8 of them don't apply. Those 8 methods become NotSupportedException traps, which are LSP violations. ISP prevents this by splitting the fat interface into small, focused ones so each implementor only promises what it can deliver.

Example: IBird with Fly(), Swim(), Walk(), Peck(), Migrate(). A penguin can swim, walk, and peck — but not fly or migrate. With the fat interface, Penguin must throw for Fly() and Migrate() — two LSP violations. ISP splits it into IFlyingBird, ISwimmingBird, IWalkingBird. Penguin implements only ISwimmingBird and IWalkingBird. No broken promises.

Every fat interface method that an implementor can't fulfill is an LSP violation waiting to happen. ISP eliminates these by ensuring implementors only sign up for contracts they can honor. Fewer methods per interface = fewer chances to break a promise.

Relationship: DIP supports LSP by making substitution the normal way code operates. DIP says "depend on abstractions, not concretions." When high-level code depends on ILogger instead of ConsoleLogger, substitution is the DEFAULT behavior — every time the DI container resolves ILogger, it's performing a substitution. LSP ensures that substitution is safe.

Example: OrderService depends on ICacheProvider (DIP). The DI container injects RedisCache in production and InMemoryCache in tests. LSP guarantees that both caches honor the same contract — same timeout behavior, same null handling, same error semantics. Without LSP, the test environment would behave differently from production, and bugs would only appear in prod.

DIP makes code depend on abstractions. LSP makes those abstractions trustworthy. Together, they enable you to swap implementations freely — through DI, through generics, through factory methods — without fear. Remove either one, and the substitution mechanism becomes unreliable. SRP keeps classes focused, making their contracts easy to honor. ISP keeps interfaces small, preventing impossible promises. OCP provides extension points through polymorphism. DIP makes code depend on abstractions. But ALL of this relies on one thing: substitutions must be safe. That's LSP. Without it, SRP classes can't be subclassed reliably, ISP interfaces can still be violated, OCP extensions blow up callers, and DIP abstractions are untrustworthy. LSP is the principle that ties everything together. LSP connects to every SOLID principle: SRP makes contracts small enough to honor correctly. OCP relies on LSP to ensure new implementations are safe. ISP prevents the fat-interface trap that causes LSP violations. DIP creates the substitution mechanism that LSP guards. Remove LSP, and polymorphism — the core mechanism of SOLID — becomes unreliable.

Bug Case Studies

LSP violations don't cause compiler errors — they cause runtime surprises. The code compiles fine, the tests pass (because tests usually only test one implementation), and the bug hides until production, when someone substitutes a different implementation and everything breaks. These six real-world-style bugs show exactly how LSP violations manifest and how to fix them.

A UI layout engine calculated container area as width x height. Rectangles worked perfectly. When Square (a Rectangle subclass) was introduced, setting Width=100 and Height=50 produced area=2500 instead of the expected 5000. Two hundred UI components rendered at half their intended size.

What Went Wrong: Square extended Rectangle because "a square IS a rectangle" — mathematically true, but behaviorally false in code. Section 13

Pitfalls & Anti-Patterns

Even when you know LSP, these six traps catch developers over and over. Each pitfall is a specific mistake, the reason it's dangerous, and the concrete fix. If the bug studies (Section 12) show what HAPPENS when LSP is violated, this section shows the HABITS that cause violations in the first place.

The Mistake: Overriding a method from the base type and throwing NotImplementedException or NotSupportedException because the subtype "doesn't need" that method.

Why It's Bad: Throwing in an override is the loudest possible LSP violation. It tells callers "I implement this interface" while simultaneously saying "but I can't actually do what the interface promises." The compiler sees a valid implementation. The runtime sees a bomb. Every caller that uses the base type polymorphically will eventually hit the exception.

The Fix: If a subtype can't implement a method, it shouldn't be forced to. Move the method to a more specific interface. The subtype only implements interfaces whose methods it can genuinely fulfill.

// ❌ Forced to implement Save() but can't actually save public interface IDataStore { void Save(string key, object value); object Load(string key); } public class ReadOnlyStore : IDataStore { public object Load(string key) => _data[key]; // ✅ Fine public void Save(string key, object value) => throw new NotSupportedException("Read-only!"); // ❌ LSP violation } // ✅ Split into read and write interfaces public interface IReadableStore { object Load(string key); } public interface IWritableStore : IReadableStore { void Save(string key, object value); } public class ReadOnlyStore : IReadableStore // ✅ only promises what it can do { public object Load(string key) => _data[key]; // No Save() method — no broken promise } public class FileStore : IWritableStore // ✅ promises both { public object Load(string key) => File.ReadAllText(key); public void Save(string key, object value) => File.WriteAllText(key, value.ToString()); }

The Mistake: Making a subtype's method accept LESS input than the base type's method. The base type accepts any integer; the subtype rejects negatives. Callers that worked with the base type now fail with the subtype.

Why It's Bad: If the base type's Deposit(int amount) accepts any positive or negative integer (treating negatives as corrections), and the subtype rejects negatives, every caller that previously relied on negative-amount corrections now gets a runtime error. The subtype made the precondition stricter than what callers were promised.

The Fix: Subtype preconditions must be the same or WEAKER (more permissive) than the base. If you need stricter validation, do it at the caller level or create a new method with an explicit contract.

// ❌ Subtype adds validation that base didn't have public class BankAccount { public virtual void Deposit(int amount) { Balance += amount; // accepts any int — positive or negative } } public class SavingsAccount : BankAccount { public override void Deposit(int amount) { if (amount < 0) // ❌ Strengthened precondition! throw new ArgumentException("Amount must be positive"); Balance += amount; } } // Caller code that worked with BankAccount breaks with SavingsAccount: account.Deposit(-50); // ❌ Was a valid correction, now throws // ✅ Subtype accepts everything base accepts public class SavingsAccount : BankAccount { public override void Deposit(int amount) { // ✅ Accepts same input as base (any int) // Additional behavior: logs negative deposits for auditing if (amount < 0) _auditLog.LogCorrection(amount); Balance += amount; } } // Or: if you genuinely need strict validation, make it explicit public class StrictSavingsAccount : BankAccount { // New method with explicit strict contract — callers opt-in public void DepositPositive(int amount) { if (amount <= 0) throw new ArgumentException("Must be positive"); Deposit(amount); } }

The Mistake: Making a subtype's method deliver LESS than what the base type promised. The base guarantees a non-null return value; the subtype returns null. The base promises a sorted result; the subtype returns unsorted.

Why It's Bad: Every caller that uses the base type trusts the postcondition. If the base says "this method always returns a non-null string," callers don't check for null. When a subtype returns null, every caller gets a NullReferenceException. The subtype weakened the guarantee that callers depended on.

The Fix: Subtype postconditions must be the same or STRONGER (more guaranteed) than the base. If you can't maintain the guarantee, don't extend that base type.

// ❌ Subtype weakens the non-null postcondition public class User { public virtual string GetDisplayName() { return _name; // guaranteed non-null (set in constructor) } } public class AnonymousUser : User { public override string GetDisplayName() { return null; // ❌ Weakened postcondition! } } // Caller trusts base contract: string greeting = $"Hello, {user.GetDisplayName().ToUpper()}!"; // 💥 NullReferenceException when user is AnonymousUser // ✅ Subtype maintains non-null postcondition public class AnonymousUser : User { public override string GetDisplayName() { return "Guest"; // ✅ Non-null, meaningful default } } // Caller code works perfectly: string greeting = $"Hello, {user.GetDisplayName().ToUpper()}!"; // Output: "Hello, GUEST!" — no crash, no surprise

The Mistake: A subtype changes the relationship between properties that the base type guarantees. The classic: Rectangle guarantees Width and Height are independent. Square couples them. Area calculations go wrong.

Why It's Bad: An invariantInvariant: a condition that is ALWAYS true about an object's state. For example, a Rectangle's invariant might be "Width and Height can be set independently." An invariant is a promise that callers rely on for correctness. is a promise about the object's state that callers depend on for correctness. When a subtype breaks an invariant, every calculation, assertion, and optimization that relied on it produces wrong results — often silently, without any exception.

The Fix: Identify the invariants of the base type before subclassing. If the subtype can't maintain all of them, don't use inheritance. Use composition or a separate type behind a shared interface instead.

// ❌ Square breaks Rectangle's independence invariant public class Square : Rectangle { public override int Width { set { base.Width = value; base.Height = value; } // ❌ couples both } public override int Height { set { base.Height = value; base.Width = value; } // ❌ couples both } } // Caller relies on independence: Rectangle r = new Square(); r.Width = 10; r.Height = 5; // Expected: Area = 50. Actual: Area = 25 (Width was silently changed to 5) // ✅ Separate types, shared interface — no invariant coupling public interface IShape { int Area { get; } } public class Rectangle : IShape { public int Width { get; set; } public int Height { get; set; } public int Area => Width * Height; // ✅ independent dimensions } public class Square : IShape { public int Side { get; set; } public int Area => Side * Side; // ✅ one dimension, no coupling confusion }

The Mistake: Writing if (shape is Circle) or switch (animal.GetType()) to handle different subtypes, instead of using virtual methods or interfaces.

Why It's Bad: If you need to check the concrete type before calling a method, the hierarchy isn't doing its job. Polymorphism means "one interface, many behaviors" — you call shape.Draw() and each shape draws itself. Type-checking defeats the purpose. Every new subtype requires editing every type-check location, and missing one creates a runtime bug.

The Fix: Move the behavior into the subtype via virtual methods. The caller uses the base type and lets polymorphism dispatch to the right implementation. If different subtypes need fundamentally different treatment, they probably shouldn't share a base type.

// ❌ Type-checking — must edit for every new shape public void Render(IShape shape) { if (shape is Circle c) DrawCircle(c.Center, c.Radius); else if (shape is Rectangle r) DrawRect(r.X, r.Y, r.Width, r.Height); else if (shape is Triangle t) DrawTriangle(t.P1, t.P2, t.P3); // ❌ New Pentagon? Must add another else-if here // ❌ Forgot to add it? Pentagon is silently not rendered! else throw new NotSupportedException(shape.GetType().Name); } // ✅ Each shape renders itself — polymorphism handles dispatch public interface IShape { void Draw(ICanvas canvas); } public class Circle : IShape { public void Draw(ICanvas canvas) => canvas.DrawCircle(Center, Radius); } public class Rectangle : IShape { public void Draw(ICanvas canvas) => canvas.DrawRect(X, Y, Width, Height); } // Renderer — no type checks needed public void Render(IShape shape) { shape.Draw(_canvas); // ✅ Works for ANY shape, present or future } // Adding Pentagon? The Mistake: Building inheritance chains 5+ levels deep: Animal > Mammal > Carnivore > Feline > DomesticCat > PersianCat. Each level adds constraints, and by level 5, the accumulated contract is so complex that any new subclass is almost guaranteed to violate some invariant from an ancestor.

Why It's Bad: Every level in the hierarchy adds preconditions, postconditions, and invariants. A class at level 5 must honor the contracts of ALL five ancestors. The probability of accidentally violating an inherited invariant increases exponentially with depth. Deep hierarchies also make the code brutally hard to understand — you have to read five classes to understand one.

The Fix: Favor composition over inheritanceComposition over inheritance: instead of making class B extend class A (is-a), make class B CONTAIN an instance of class A (has-a). This avoids the tight coupling of inheritance while still reusing A's behavior. The Go language doesn't even have inheritance — it uses composition exclusively.. Keep inheritance to 1-2 levels maximum. Use interfaces for polymorphism and composition for code reuse. If you need behavior from multiple sources, inject them as dependencies rather than building a tower of base classes.

// ❌ 5-level hierarchy — each level adds constraints public class Animal { public virtual void Eat() { ... } } public class Mammal : Animal { public virtual void Nurse() { ... } } public class Carnivore : Mammal { public virtual void Hunt() { ... } } public class Feline : Carnivore { public virtual void Purr() { ... } } public class DomesticCat : Feline { public virtual void UseLitterBox() { ... } } // ❌ PersianCat must honor 5 levels of contracts — good luck! // ✅ Flat: interfaces for polymorphism, composition for behavior public interface IAnimal { string Species { get; } } public interface IFeedable { void Feed(Food food); } public interface IHuntable { Prey Hunt(); } public class Cat : IAnimal, IFeedable { private readonly IFeedingBehavior _feeding; private readonly IMovementBehavior _movement; public Cat(IFeedingBehavior feeding, IMovementBehavior movement) { _feeding = feeding; _movement = movement; } public string Species => "Cat"; public void Feed(Food food) => _feeding.Feed(food); // Each behavior is injected, swappable, and independently testable } Six LSP pitfalls to avoid: (1) Throwing NotImplementedException in overrides — split the interface instead. (2) Strengthening preconditions — subtypes must accept everything the base accepts. (3) Weakening postconditions — subtypes must deliver everything the base promises. (4) Breaking invariants — if you can't maintain the base's guarantees, don't inherit. (5) Type-checking instead of polymorphism — if you check types before calling methods, the hierarchy is broken. (6) Deep inheritance hierarchies — favor composition to avoid accumulating impossible-to-honor contracts.

Testing Strategies

LSP violations hide until production because most test suites only test one implementation at a time. You test InMemoryCache — it passes. You test RedisCache — it passes. But they pass for DIFFERENT reasons, and the behavioral differences between them are the LSP violations that crash production. The fix is to test the contract, not the implementation. Run the same tests against every implementation, and any behavioral difference surfaces immediately.

The most powerful LSP testing technique. Write your test suite against the interface, not any specific implementation. Then run that exact same test suite against every implementation. If any implementation fails a test that others pass, you've found an LSP violation. This is the gold standard because it directly tests the definition of LSP: "can I substitute any implementation without changing behavior?"

// ✅ Contract tests — one suite, every implementation public abstract class CacheProviderContractTests { protected abstract ICacheProvider CreateProvider(); // subclass provides the impl [Fact] public void Set_ThenGet_ReturnsSameValue() { var cache = CreateProvider(); cache.Set("key", "hello", TimeSpan.FromMinutes(5)); var result = cache.Get<string>("key"); Assert.Equal("hello", result); // MUST be true for ALL implementations } [Fact] public void Get_NonexistentKey_ReturnsNull() { var cache = CreateProvider(); var result = cache.Get<string>("missing"); Assert.Null(result); // MUST be true for ALL implementations } [Fact] public void Set_Overwrite_ReturnsNewValue() { var cache = CreateProvider(); cache.Set("key", "old", TimeSpan.FromMinutes(5)); cache.Set("key", "new", TimeSpan.FromMinutes(5)); Assert.Equal("new", cache.Get<string>("key")); } } // ✅ Each implementation plugs into the SAME tests public class InMemoryCacheTests : CacheProviderContractTests { protected override ICacheProvider CreateProvider() => new InMemoryCache(); } public class RedisCacheTests : CacheProviderContractTests { protected override ICacheProvider CreateProvider() => new RedisCache(_testConnection); } // Adding MemcachedCache? Section 17

Interview Q&As

Thirty questions you'll face in real interviews — 10 Easy, 10 Medium, 10 Hard — arranged from warmup to expert. For each one, pause at the "Think First" prompt and form your own answer before reading ours. That's how the learning sticks — not by memorizing answers, but by practising the thinking.

Q1: What is LSP in simple terms?
Easy
Think First Imagine a function that works perfectly with a base type. What should happen when you hand it a subtype instead? What should NEVER happen?

The strong answer: LSP says that anywhere your code expects a base type, you should be able to hand it any subtype and the program still works correctly. No surprises, no crashes, no wrong results.

Think of it like an electrical outlet. The outlet (your code) expects a "plug" (base type). Any device with a compatible plug — lamp, phone charger, toaster — should work. If you plug in a device and it blows a fuse, that device violates the outlet's "contract." LSP is the same thing for code: subtypes must honour every promise the base type makes.

The why behind this is polymorphism itself. The whole point of having base types and subtypes is so you can write code once and reuse it with many implementations. If a subtype misbehaves, polymorphism breaks — you're forced to add if (obj is SpecialCase) checks everywhere, which defeats the purpose of the hierarchy in the first place.

One-liner for interviews: "If a function works with the base type, every subtype should drop in without changing the function's behavior."

Q2: Explain the Rectangle/Square problem.
Easy
Think First In math class, every square IS a rectangle. So it seems natural to write Square : Rectangle. What goes wrong?

The strong answer: In geometry, a square is a rectangle — that's true. But in code, a Rectangle promises that width and height are independent. You can set width to 5 and height to 10 and get an area of 50. That's the contract.

A Square breaks this contract. When you set its width to 5, it also sets height to 5 (because a square must have equal sides). Now code that sets width=5, height=10 and expects area=50 gets area=100 instead. No exception is thrown — the code runs fine. But the result is wrong. That's the most dangerous kind of bug: a silent behavioral change.

The fix: Don't make Square extend Rectangle. Instead, create an IShape interface with Area(), and have both Rectangle and Square implement it independently. Each shape owns its own geometry, no surprises.

The deeper lesson: Inheritance models behavior, not real-world taxonomy. A square behaves differently from a rectangle in code (setting one dimension changes the other), so it shouldn't inherit from it — even though it "is a" rectangle in math.

Q3: What are preconditions and postconditions?
Easy
Think First When you call a method, there's an unwritten deal: "I'll give you valid input, and you'll give me a valid result." What are the formal names for each side of that deal?

The strong answer: Every method has an invisible contract with its callers. That contract has two sides:

Preconditions are what the method requires before it runs. Think of it as the entry ticket. "Give me a non-null string" or "pass a positive number." If you break the precondition, the method isn't obligated to work correctly.

Postconditions are what the method guarantees after it runs. Think of it as the receipt. "I'll return a non-empty list" or "the file will be saved to disk." If the method breaks its postcondition, it's the method's fault, not the caller's.

LSP puts strict rules on these:

  • Preconditions can't be strengthened. If the base type accepts any string, the subtype can't suddenly demand "only non-empty strings." That's tightening the entry requirement, and callers who were sending empty strings will now break.
  • Postconditions can't be weakened. If the base type guarantees a non-null return, the subtype can't return null. Callers trust the guarantee and don't check for null — they'll crash.

Real example: A base class BankAccount.Withdraw(amount) accepts any positive amount. A subtype PremiumAccount that restricts withdrawals to amounts over $100 strengthens the precondition — LSP violation. But a PremiumAccount that also returns loyalty points alongside the withdrawal strengthens the postcondition — that's fine, because it delivers more than promised.

Q4: How does LSP relate to polymorphism?
Easy
Think First Polymorphism lets you write Process(IShape shape) and it works for Circle, Rectangle, Triangle. What assumption does the caller make about every implementation?

The strong answer: Polymorphism is the mechanism; LSP is the guarantee that makes the mechanism work safely.

When you write void Render(IShape shape), polymorphism lets the runtime call the correct Draw() method for whichever shape was passed. But polymorphism alone doesn't promise that every implementation behaves correctly. You could have a shape whose Draw() method deletes files instead of drawing — polymorphism would happily dispatch to it.

LSP is the rule that says "every implementation must honor the behavioral contract of the interface." Polymorphism gives you type substitution. LSP gives you safe type substitution. Without LSP, polymorphism is a loaded gun — you never know if the next implementation will surprise you.

Practical consequence: If your codebase has if (shape is Circle) type checks before calling methods, that means the caller doesn't trust that all subtypes behave correctly. That's a symptom of LSP violations — and a sign that polymorphism isn't working as intended.

Q5: How would you fix an LSP violation in existing code?
Medium
Think First You've found a subtype that throws NotSupportedException in an override. You can't delete the class — production depends on it. What's your step-by-step plan?

The strong answer: Fixing LSP violations is a refactoring exercise. Here's the decision tree I use:

Step-by-step:

  1. Write characterization tests — capture what the current code actually does (including the violation). These are your safety net.
  2. Identify the violated contract — which method is misbehaving? Is it throwing? Returning unexpected values? Ignoring calls?
  3. Decide the fix strategy — if the subtype can support the method, fix the implementation. If it genuinely can't, the interface is too broad. Split it into smaller, focused interfaces (this is where LSP meets ISP).
  4. Update callers — change method signatures from the broad interface to the focused one. The compiler will catch any callers that assumed too much.
Q6: Compare LSP violations via inheritance vs interface. Which is more dangerous?
Medium
Think First Both inheritance and interfaces can have subtypes that misbehave. But one of them makes violations much harder to detect. Which one, and why?

The strong answer: Inheritance-based violations are more dangerous because they involve implicit contracts. With an interface, the contract is explicit — you see the method signatures and can reason about them. With inheritance, the base class carries hidden behavioral expectations that aren't spelled out in the method signatures.

Why inheritance is worse: When Square extends Rectangle, the implicit contract is "width and height are independent." This isn't written anywhere — it's baked into the base class's behavior. You only discover the violation when a test fails or a user reports wrong calculations.

With interfaces, the violation is usually more visible. A ReadOnlyRepository that throws NotSupportedException from Add() is obvious — you see the exception in testing. Still a violation, but much easier to spot and fix (split the interface).

Bottom line: This is why modern C# favors composition and interfaces over deep inheritance. Interfaces keep violations visible and fixable. Inheritance hides them.

Q7: How does .NET's Stream class handle the seekable/non-seekable problem?
Medium
Think First Stream has a Seek() method, but NetworkStream can't seek. How does the BCL handle this without violating LSP?

The strong answer: The .NET Stream class uses a pattern I call "capability flags" — boolean properties like CanRead, CanWrite, and CanSeek that tell callers what operations are available before they try to use them.

When you receive a Stream, the contract says: "Check CanSeek before calling Seek(). If it's false, Seek() will throw NotSupportedException." This makes the exception part of the documented contract, not a surprise. The caller knows to check the flag first.

// .NET's Stream uses capability flags to manage LSP Stream stream = GetStream(); // could be any stream type // WRONG: assume all streams can seek stream.Seek(0, SeekOrigin.Begin); // throws on NetworkStream! // RIGHT: check capability first if (stream.CanSeek) { stream.Seek(0, SeekOrigin.Begin); } else { // Buffer the stream, or handle non-seekable case var buffer = new MemoryStream(); await stream.CopyToAsync(buffer); buffer.Position = 0; }

Is this a true LSP violation? Technically, yes — a NetworkStream can't do everything a FileStream can. The ideal solution would be separate interfaces: IReadableStream, IWritableStream, ISeekableStream. But the Stream class was designed in .NET 1.0 (2002) before the community fully understood ISP. The capability-flag pattern is a pragmatic compromise — not perfect, but workable and well-documented.

Lesson for interviews: Real-world code isn't always textbook-perfect. The Stream example shows that acknowledging a limitation with clear documentation and capability flags is better than pretending the limitation doesn't exist.

Q8: Can LSP be violated without throwing exceptions?
Hard
Think First Most developers think LSP violations = NotSupportedException. But what about a subtype that silently returns the wrong value? Or one that has unexpected side effects?

The strong answer: Absolutely. In fact, silent LSP violations are far more dangerous than exceptions. An exception at least tells you something went wrong. A silent violation produces wrong results that might go unnoticed for weeks.

Here are three ways to violate LSP without throwing anything:

1. Returning wrong values. The base class's GetBalance() returns the actual balance. A subtype returns the balance minus pending fees. Callers expecting the true balance now make wrong decisions.

2. Silent side effects. The base class's Save() writes to the database. A subtype also sends a notification email on every save. Code that calls Save() ten times in a loop just sent ten emails — without knowing it.

3. Breaking invariants. A SortedList promises items are always in order. A subtype's Add() method appends without sorting. The list looks fine but iteration returns items out of order.

How to catch silent violations: Contract tests. Write a test suite against the interface, then run it against every implementation. If the base class test says "Save() produces exactly 1 database row and no side effects," the subtype's email-sending behavior will fail that test.

Q9: How do covariance and contravariance in C# relate to LSP?
Hard
Think First C# has out T (covariance) and in T (contravariance) keywords on generic type parameters. How are these related to "safe substitution"?

The strong answer: Covariance and contravariance are the compiler's way of enforcing LSP at the type level for generics. Here's the plain-English version:

Covariance (out T) — "I only produce T values, never consume them." Because you're only reading, it's safe to substitute a more specific type. You can use IEnumerable<Dog> wherever IEnumerable<Animal> is expected, because reading a Dog as an Animal is perfectly safe.

Contravariance (in T) — "I only consume T values, never produce them." Because you're only writing, it's safe to substitute a less specific type. You can use IComparer<Animal> wherever IComparer<Dog> is expected, because a comparer that handles any Animal can certainly handle Dogs.

Invariance (no keyword) — "I both read and write T." No substitution is safe. IList<Dog> is NOT an IList<Animal>, because if it were, you could add a Cat to a list of Dogs. That's a type-safety (and LSP) violation.

// IEnumerable<out T> is covariant — SAFE for LSP // "I only produce T values" IEnumerable<Dog> dogs = new List<Dog> { new Dog("Rex"), new Dog("Buddy") }; // This works because IEnumerable only READS (out T) // Reading a Dog as an Animal is safe IEnumerable<Animal> animals = dogs; // compiles! foreach (Animal a in animals) { Console.WriteLine(a.Name); // works perfectly } // WHY it's safe: you can't ADD to IEnumerable // So no risk of putting a Cat into the Dog list // IComparer<in T> is contravariant — SAFE for LSP // "I only consume T values" IComparer<Animal> animalComparer = new AnimalByWeightComparer(); // This works because IComparer only CONSUMES (in T) // A comparer that handles ANY animal can handle Dogs IComparer<Dog> dogComparer = animalComparer; // compiles! var dogs = new List<Dog> { new("Rex"), new("Buddy") }; dogs.Sort(dogComparer); // works — it compares Dogs as Animals // WHY it's safe: comparing Animals by weight works for Dogs // because Dogs ARE Animals. The comparer never creates Animals, // it only reads their Weight property. // IList<T> is invariant — NO substitution allowed // "I both read AND write T" IList<Dog> dogs = new List<Dog>(); // This does NOT compile — and for good reason! // IList<Animal> animals = dogs; // ERROR! // WHY it's unsafe: // If it compiled, you could do this: // animals.Add(new Cat("Whiskers")); // Adding Cat to Dog list! // Dog firstDog = dogs[0]; // CRASH: it's a Cat! // The compiler prevents this LSP violation at compile time. // That's the WHOLE POINT of variance annotations.

The connection to LSP: Variance keywords are the compiler doing the LSP check for you. When you mark a type parameter as out T, you're saying "this type is safe for substituting more specific types." The compiler then proves it by checking that T only appears in output positions. If you try to use T in an input position, the compiler rejects it — because that would allow a violation.

Q10: Design a payment system that handles gateways with different capabilities without violating LSP.
Hard
Think First Stripe supports refunds, PayPal supports partial refunds, and CryptoGateway doesn't support refunds at all. If you put Refund() on IPaymentGateway, CryptoGateway must throw. How do you design this cleanly?

The strong answer: The key insight is that "payment gateway" is actually a bundle of capabilities, not a single thing. Different gateways support different subsets of operations. Forcing them all into one fat interface is what creates the LSP problem. The fix: split by capability.

// Step 1: Split by capability — each interface is a focused contract public interface IChargeable { Task<ChargeResult> ChargeAsync(Money amount, PaymentMethod method); } public interface IRefundable { Task<RefundResult> RefundAsync(string transactionId); } public interface IPartialRefundable : IRefundable { Task<RefundResult> PartialRefundAsync(string transactionId, Money amount); } public interface IRecurringBillable { Task<SubscriptionResult> SubscribeAsync(Plan plan, PaymentMethod method); Task CancelSubscriptionAsync(string subscriptionId); } // Step 2: Each gateway implements ONLY what it supports public class StripeGateway : IChargeable, IRefundable, IPartialRefundable, IRecurringBillable { // Stripe supports everything — implements all four } public class PayPalGateway : IChargeable, IRefundable, IPartialRefundable { // PayPal: charges, full refunds, partial refunds — no recurring } public class CryptoGateway : IChargeable { // Crypto: charge only — blockchain transactions are irreversible } // Step 3: Callers depend on the capability they NEED public class RefundService(IRefundable gateway) { // CryptoGateway can NEVER be passed here // The compiler enforces LSP at compile time! public async Task ProcessRefund(string txnId) => await gateway.RefundAsync(txnId); } public class CheckoutService(IChargeable gateway) { // Any gateway works here — all of them can charge public async Task<ChargeResult> Checkout(Order order) => await gateway.ChargeAsync(order.Total, order.PaymentMethod); }

Why this design is LSP-clean: No implementation ever needs to throw NotSupportedException. Each class only promises what it can deliver. The compiler prevents you from passing a charge-only gateway to code that expects refunds. Zero runtime surprises.

Q11: What's the simplest way to explain LSP to a non-programmer?
Easy
Think First Forget code entirely. How would you explain "substitutability" to someone who has never written a line of code?

The strong answer: Think about a TV remote. You buy a universal remote, and the box says "works with any TV." That's a promise. If you point it at a Samsung, a Sony, or an LG, it should turn the TV on, change channels, and adjust volume — the basic operations work on all of them.

Now imagine someone sells a "TV" that only displays static and can't change channels. It's technically a screen, but when you point the universal remote at it and press "Channel Up," nothing happens. The remote didn't break. The "TV" just lied about what it was. That's an LSP violation.

LSP says: if something claims to be a type of [thing], it must actually behave like that thing in every way that matters. A sedan that can't steer. A pen that can't write. A bank account that silently loses your money. All violations — they claim to be something they can't fully be.

One-liner: "If it says it's an X, it must do everything an X does — no faking."

Q12: Why is "is-a" not enough for good inheritance?
Easy
Think First Every OOP textbook says "use inheritance when there's an is-a relationship." A square IS a rectangle. An ostrich IS a bird. So why do these examples cause problems?

The strong answer: The "is-a" test only checks taxonomy — it asks "is this thing a kind of that thing?" But LSP requires behavioral compatibility — "does this thing act like that thing in every context the code cares about?"

A square is a rectangle in geometry class. But in code, a Rectangle promises that you can set width and height independently. A Square can't keep that promise. The "is-a" relationship is real in the domain, but the behavioral contract doesn't hold in the code.

Same with birds. An Ostrich is a bird. But if your Bird base class promises a Fly() method that returns an altitude, Ostrich violates that contract. The taxonomy is fine; the behavior isn't.

The better test: Instead of asking "is A a kind of B?" ask "can A do everything B promises to do, without surprises?" That's the behavioral substitutability test, and it's what LSP actually requires.

Rule of thumb: "Is-a" gets you in the door. "Behaves-as" gets you past the bouncer. Both must pass.

Q13: What happens when you violate LSP in a DI container?
Easy
Think First A DI container resolves interfaces to concrete implementations at runtime. What happens when the concrete class doesn't behave like the interface promises?

The strong answer: Dependency injectionA technique where objects receive their dependencies from the outside (usually a container) instead of creating them internally. This makes code more testable and loosely coupled. In .NET, the built-in DI container is Microsoft.Extensions.DependencyInjection. makes LSP violations invisible at compile time and explosive at runtime. Here's why.

When you register services.AddScoped<IRepository, CachingRepository>(), every class that takes IRepository in its constructor gets the caching version. If CachingRepository has an LSP violation — say, its Delete() method silently does nothing because "you shouldn't delete cached data" — then every service in your entire app that calls Delete() is now broken. And nobody knows it, because there's no exception.

The damage is amplified because DI containers wire up dependencies everywhere. A single bad registration pollutes dozens of classes. You change one line in Startup.cs and suddenly orders aren't being deleted, users can't be removed, and audit logs keep growing forever.

// Registration looks innocent services.AddScoped<IRepository, CachingRepository>(); // CachingRepository has a silent LSP violation: public class CachingRepository : IRepository { public void Delete(int id) { // "Cached data shouldn't be deleted" // So this method does... NOTHING. // No exception. No log. Just silence. } } // Now EVERY service that depends on IRepository // has a broken Delete() — and the compiler can't warn you.

Prevention: Write contract tests against the interface, then run them against every registered implementation. If IRepository has a contract test that says "after Delete(id), GetById(id) returns null," then CachingRepository fails immediately during CI, not in production.

Q14: How do you know if a method override is LSP-safe?
Easy
Think First You're reviewing a PR that overrides a base class method. What checklist would you run through to verify the override doesn't break LSP?

The strong answer: Run through this five-point checklist. If any point fails, the override is unsafe:

  1. Does it accept everything the base accepts? The override must not reject inputs that the base method handles. If the base accepts null, the override must accept null. Strengthening preconditionsThe requirements that must be true BEFORE a method executes. Like entry requirements for a concert: if the original says "anyone with a ticket," the subtype can't add "must also have VIP pass." is a violation.
  2. Does it return compatible results? If the base guarantees a non-null return, the override must also return non-null. If the base returns a value in range 0-100, the override can't return -1. Weakening postconditionsThe guarantees that must be true AFTER a method executes. Like a restaurant promise: if the menu says "served within 30 minutes," the kitchen can't take 2 hours. is a violation.
  3. Does it preserve the same side effects? If the base's Save() only writes to the database, the override shouldn't also send emails. No unexpected side effects.
  4. Does it throw only expected exceptions? If the base throws ArgumentException for bad input, the override shouldn't throw InvalidOperationException for the same input. New exception types surprise callers.
  5. Does it maintain class invariants? If the base class guarantees that Balance >= 0 after any operation, the override must keep that guarantee.

Quick mnemonic — PRISE: Preconditions same or weaker, Return values compatible, Invariants preserved, Side effects unchanged, Exceptions expected. If all five pass, the override is LSP-safe.

Q15: What's the difference between LSP and duck typing?
Easy
Think First Duck typing says "if it walks like a duck and quacks like a duck, it's a duck." LSP says subtypes must be substitutable. They sound similar — what's the actual difference?

The strong answer: They're related but operate at different levels:

Duck typingA concept from dynamic languages (Python, Ruby, JavaScript) where the type of an object is determined by what methods it has, not by what class it inherits from. If an object has a quack() method, you can treat it as a duck — no formal interface needed. asks: "Does this object have the right method signatures?" It checks structural compatibility — the shape of the API. In Python, if an object has .read() and .close(), you can use it as a file-like object regardless of its actual class.

LSP asks: "Does this object have the right behavior?" It checks behavioral compatibility — not just that the methods exist, but that they do the right thing. An object might have .read() and .close(), but if .read() returns garbage or .close() silently does nothing, it violates LSP even though it passes the duck test.

The key insight: Duck typing is necessary but not sufficient for LSP. Having the right shape gets you past the compiler (or the dynamic dispatch). Having the right behavior gets you past LSP. You need both for safe substitution.

Analogy: A counterfeit $20 bill passes the "looks like a $20" duck test. But it fails the behavioral test at the bank — it doesn't actually hold value. LSP is the bank's verification; duck typing is the visual check.

Q16: Is throwing NotSupportedException always an LSP violation?
Easy
Think First .NET's ReadOnlyCollection throws NotSupportedException from Add(). The Stream class throws it from Seek() in non-seekable streams. Are these violations? Or is there a nuance?

The strong answer: It depends on whether the exception is part of the documented contract or a surprise.

NOT a violation — when the contract says "might throw." The Stream class documents that Seek() throws NotSupportedException when CanSeek is false. That's part of the contract. Callers know to check CanSeek first. The exception isn't a surprise — it's an expected outcome of calling a method you were warned not to call.

YES a violation — when the contract implies the method works. If IRepository has an Add() method with no capability flag and no documentation about exceptions, then an implementation that throws NotSupportedException from Add() is a surprise. Callers had no reason to expect failure. That's a violation.

The deciding question: "Did the caller have a reasonable way to know this would throw, before calling it?" If yes (capability flags, documentation, method naming), it's a legitimate part of the contract. If no, it's a violation.

Best practice: If you find yourself wanting to throw NotSupportedException, that's a design smell. It usually means the interface is too broad. Split it so each implementation can honestly fulfill its entire interface — no exceptions needed.

Q20: How does interface segregation prevent LSP violations?
Medium
Think First ISP says "don't force classes to implement methods they don't use." LSP says "subtypes must honor all base type promises." How do these two work together?

The strong answer: ISPInterface Segregation Principle — the fourth SOLID principle. It says: no client should be forced to depend on methods it doesn't use. Instead of one fat interface, create several small, focused ones. is LSP's best friend. Here's the chain of reasoning:

Fat interfaces cause LSP violations. When an interface has 10 methods but a particular implementation only supports 7 of them, the implementation must either throw NotSupportedException for the other 3 (obvious violation) or silently do nothing (sneaky violation). Either way, any caller that uses those 3 methods gets broken behavior.

ISP prevents this by splitting the fat interface. Instead of one 10-method interface, you create three focused interfaces: one for reading (4 methods), one for writing (3 methods), and one for admin (3 methods). Now each implementation only implements the interfaces it can genuinely fulfill. No fake methods. No exceptions. No violations.

The pattern:

  1. You discover an LSP violation (a class throwing NotSupportedException).
  2. You ask "why can't this class support this method?" Answer: the interface is too broad for this class.
  3. You apply ISP: split the interface so each piece is small enough that every implementation can genuinely honor it.
  4. Result: LSP violations disappear because no class is forced to fake methods it can't do.

Bottom line: ISP makes interfaces small enough that LSP becomes easy to satisfy. They're two sides of the same coin — ISP shapes the interfaces, LSP ensures the implementations are honest.

Q21: How do you apply LSP when designing an API for external consumers?
Medium
Think First Internal code can be refactored together. But external APIs have consumers you can't control. How does this change the LSP equation?

The strong answer: When you publish a public API — a NuGet package, a REST endpoint, an SDK — the stakes for LSP go through the roof. Why? Because you can't update the callers. Thousands of applications depend on your contract. If a new version changes behavior, those apps break silently in production. You'll never even know.

API versioning IS LSP at scale. When you release v2 of your API, it must be substitutable for v1 in every scenario that v1 supported. New endpoints are fine (that's adding capability). But changing what existing endpoints return — that's a postcondition change and an LSP violation against every v1 consumer.

Practical rules for public APIs:

  • Never remove fields from responses. Consumers parse them. Removing a field is weakening a postcondition.
  • Never add required fields to requests. Existing callers don't send them. That's strengthening a precondition.
  • Never change the meaning of a field. If status: "active" used to mean "account exists," don't change it to mean "account exists AND verified." That's an invariant change.
  • New optional fields are safe. Adding an optional request field with a sensible default keeps backward compatibility. Adding a new response field that old clients ignore is also fine.

The test: Before releasing any API change, ask: "If a consumer upgrades from v1 to v2 without changing their code, does everything still work exactly as before?" If yes, you're LSP-compliant. If no, you need a new major version.

Q22: What's the relationship between LSP and the Template Method pattern?
Medium
Think First Template Method defines the skeleton of an algorithm in a base class and lets subclasses fill in the steps. How does this pattern naturally enforce (or violate) LSP?

The strong answer: The Template Method patternA behavioral design pattern where a base class defines the overall algorithm structure (the "template") with some steps left as abstract methods. Subclasses override those abstract steps to provide specific behavior, but the overall algorithm structure remains unchanged. is one of the best patterns for enforcing LSP — when used correctly. Here's why:

The base class controls the contract. The template method itself (the public method that runs the algorithm) is non-virtual. Subclasses can't change the overall flow. They can only fill in the specific steps that the base class declares as abstract or virtual. This limits the surface area where violations can happen.

// The base class CONTROLS the contract public abstract class DataExporter { // Template method — sealed (not overridable) // This IS the contract: validate, transform, write, log public void Export(DataSet data) { Validate(data); // subclass fills in var transformed = Transform(data); // subclass fills in Write(transformed); // subclass fills in LogCompletion(data.Name); // base class handles } protected abstract void Validate(DataSet data); protected abstract DataSet Transform(DataSet data); protected abstract void Write(DataSet data); private void LogCompletion(string name) => Console.WriteLine($"Export of {name} complete"); } // CSV and JSON exporters fill in the steps // but CAN'T change the order or skip steps // The algorithm structure IS the contract, and it's locked down

Where it can still go wrong: If a subclass's Validate() silently accepts invalid data (weakening precondition checking), or if Transform() returns data in the wrong format (breaking postconditions), you still get LSP violations. The pattern limits structural violations (can't change the algorithm order) but can't prevent behavioral violations within individual steps.

The lesson: Template Method narrows the LSP surface area from "the entire class" to "individual hook methods." That makes violations easier to find and fix. Combine it with contract tests on each hook, and you get strong LSP guarantees.

Q23: How do you refactor a deep inheritance hierarchy to satisfy LSP?
Medium
Think First You inherit a codebase with Vehicle → MotorVehicle → Car → ElectricCar, four levels deep. Several classes throw NotSupportedException. How do you flatten this without breaking production?

The strong answer: Deep hierarchies are LSP minefields because every level adds implicit contracts that lower levels might violate. The fix is to flatten the hierarchy by extracting interfaces and using composition.

The step-by-step refactoring plan:

  1. Identify the violations. Search for NotSupportedException, NotImplementedException, empty overrides, and if (this is SpecificType) checks. Each is a symptom.
  2. Extract capability interfaces. For each group of related methods, create a focused interface. IFuelPowered for gas operations, IElectricPowered for charging, ITrunkEquipped for trunk access.
  3. Make each class implement only what it supports. ElectricCar implements IDriveable and IElectricPowered but NOT IFuelPowered.
  4. Update callers. Change method parameters from the fat base type to the specific capability interface. void Refuel(IFuelPowered vehicle) instead of void Refuel(Vehicle vehicle).
  5. Delete the deep hierarchy. Once all callers use interfaces, the intermediate classes (MotorVehicle) become unnecessary.

Safety tip: Do this in small PRs. Extract one interface at a time, update its callers, run tests, ship. Don't try to flatten the entire hierarchy in one PR — that's a recipe for merge conflicts and missed edge cases.

Q25: How does LSP apply to async/await patterns in C#?
Hard
Think First An interface returns Task<Result>. One implementation makes a real HTTP call (takes 200ms). Another returns a cached value synchronously wrapped in Task.FromResult (takes 0ms). Is the timing difference an LSP concern?

The strong answer: Async methods introduce a whole new dimension to LSP: timing and cancellation contracts. Here are the three ways async code can violate LSP:

1. Ignoring CancellationToken. If the base interface accepts a CancellationToken, every implementation must honor it. An implementation that ignores the token and always runs to completion violates the cancellation contract. Callers that set timeouts will hang forever.

2. Blocking the calling thread. If a method returns Task, callers expect it to be non-blocking. An implementation that calls .Result or .Wait() internally can deadlock the caller. The method looks async but behaves synchronously — that's a behavioral mismatch.

3. Different exception wrapping. Async methods wrap exceptions in AggregateException or surface them when the task is awaited. An implementation that throws synchronously (before returning the Task) behaves differently from one that returns a faulted task. Callers with try { await ... } won't catch the synchronous throw the same way.

public interface IDataFetcher { // Contract: respects cancellation, returns within token's timeout Task<Data> FetchAsync(string key, CancellationToken ct); } // GOOD: honors cancellation public class ApiFetcher : IDataFetcher { public async Task<Data> FetchAsync(string key, CancellationToken ct) { // HttpClient automatically respects the token var response = await _http.GetAsync($"/data/{key}", ct); return await response.Content.ReadFromJsonAsync<Data>(ct); } } // BAD: ignores cancellation — LSP violation! public class SlowDbFetcher : IDataFetcher { public async Task<Data> FetchAsync(string key, CancellationToken ct) { // NEVER passes ct to the database call // If caller cancels, this keeps running for 30 seconds var result = await _db.QueryAsync($"SELECT * FROM data WHERE key = '{key}'"); return result; } } // WRONG: synchronous implementation pretending to be async public class CachedFetcher : IDataFetcher { public Task<Data> FetchAsync(string key, CancellationToken ct) { // This is fine — Task.FromResult doesn't block if (_cache.TryGetValue(key, out var data)) return Task.FromResult(data); // BUT: Don't do this inside an async method: // var result = _slowService.GetData(key); // BLOCKS the thread! // return Task.FromResult(result); // DO: use async all the way down return FetchFromSourceAsync(key, ct); } } // RIGHT: truly async, respects cancellation public class CachedFetcher : IDataFetcher { public async Task<Data> FetchAsync(string key, CancellationToken ct) { ct.ThrowIfCancellationRequested(); if (_cache.TryGetValue(key, out var data)) return data; data = await _source.FetchAsync(key, ct); _cache[key] = data; return data; } }

The timing question: Is returning instantly from cache vs 200ms from HTTP an LSP violation? No. The contract is "returns data for this key." Faster is better, and the postcondition (correct data returned) is the same. Performance differences are generally not LSP violations unless the interface explicitly specifies timing constraints (which is rare).

Rule of thumb for async LSP: Honor cancellation tokens. Don't block the thread. Throw exceptions in the same way (via faulted Task, not synchronous throw). If you do these three things, your async implementations are LSP-safe.

Q26: What is the "history constraint" in behavioral subtyping and when does it matter?
Hard
Think First Most developers know about preconditions and postconditions. But Liskov and Wing's original paper mentions a THIRD rule — the "history constraint." What is it, and why do most tutorials skip it?

The strong answer: The history constraintThe rule in Liskov and Wing's formal behavioral subtyping theory that says: a subtype cannot allow state changes that the base type would not permit. The entire "history" of state transitions must remain valid. This goes beyond individual method contracts — it covers sequences of operations over time. (also called the "history rule") says: a subtype cannot allow state transitions that the base type would forbid. It's not about what a single method does — it's about the sequence of states an object can go through over its lifetime.

Plain English: If the base type says "once an order is shipped, it can never go back to pending," then no subtype can add a method that moves a shipped order back to pending. The allowed history of state changes is part of the contract.

Why it matters: Preconditions and postconditions check individual method calls. The history constraint checks the object's lifecycle. You could have a subtype where every method individually passes pre/postcondition checks, but the sequence of states it allows is different from the base type. That's a history constraint violation.

// Base type: immutable after creation public class ImmutableConfig { public string DbConnection { get; } public int Timeout { get; } public ImmutableConfig(string conn, int timeout) { DbConnection = conn; Timeout = timeout; } // No setters. Once created, it NEVER changes. // History constraint: the state after construction // is the state FOREVER. } // Subtype: adds mutability — HISTORY CONSTRAINT VIOLATION! public class MutableConfig : ImmutableConfig { public MutableConfig(string conn, int timeout) : base(conn, timeout) { } // Adds a way to change state after creation public new string DbConnection { get; set; } // mutable! // The base type promises "state never changes after construction." // This subtype breaks that promise by allowing mutation. // Code that caches the config value (trusting immutability) // will now use stale values. }

Another example: A Stack has a history constraint: "the next Pop() returns the last Push()ed item" (LIFO order). A subtype that internally sorts items violates this — Pop() returns items in a different order. Each individual Push() and Pop() works fine. But the sequence of states (the history) is wrong.

Why tutorials skip it: The history constraint is harder to test automatically. Preconditions and postconditions can be checked with single method calls. The history constraint requires testing sequences of operations — "do A, then B, then C, then verify the state." It's the most subtle of the three LSP rules, but in systems with complex state machines (orders, workflows, transactions), it's the one that causes the most production bugs.

Q28: Compare LSP in OOP vs protocol-based dispatch (like Clojure/Go interfaces).
Hard
Think First In Go, a type satisfies an interface just by having the right methods — no implements keyword. In Clojure, protocols dispatch on the first argument's type. How does LSP manifest differently in these paradigms?

The strong answer: LSP is a universal principle, but how it manifests depends on how the language handles substitution. Here's the comparison:

C# / Java (Nominal typing): A class explicitly declares what it implements (class Dog : IAnimal). The compiler checks structural compliance at compile time. The strong answer: There's a clear dividing line. The compiler can enforce structural compatibility (method signatures, types, accessibility). It cannot enforce behavioral compatibility (does the method do the right thing). Here's the full breakdown:

What the C# compiler CAN enforce (compile time):

  • Method signatures. If an interface requires Task<int> Calculate(), every implementation must return Task<int>. Can't return void or string.

  • Generic constraints. where T : IComparable<T> ensures structural compliance. Won't compile without the interface.
  • Nullable reference types. string? vs string makes null-return violations visible. If the interface says string GetName() (non-nullable), an implementation that returns null gets a compiler warning.

What ONLY tests can enforce (runtime):

  • Precondition rules. "This method accepts any positive number" — the compiler can't verify that an override doesn't add if (n > 100) throw.
  • Postcondition rules. "This method always returns a sorted list" — the compiler can't verify sorting order.
  • Invariant preservation. "Balance is always >= 0" — requires property-based testing or contracts.
  • Side effect rules. "Save() only writes to DB, nothing else" — the compiler can't detect email-sending side effects.
  • History constraints. "Once frozen, the object can't be modified" — requires state-sequence testing.

    Deep Dive — The Theory Behind LSP

    LSP isn't just a "good practice" — it's rooted in formal computer science from Barbara Liskov and Jeannette Wing's 1994 paper. Understanding the theory helps you reason about edge cases that rules of thumb don't cover.

    Topic 1: Behavioral Subtyping — The Formal Definition

    The formal statement from Liskov and Wing goes like this: if S is a subtype of T, then for all programs P defined in terms of T, the behavior of P is unchanged when objects of type S are substituted for objects of type T.

    That sounds academic, but in practice it means: write a test using the base type. Run it. Now swap in a subtype. If the test still passes with identical behavior, the subtype satisfies LSP. If the test fails — or passes but produces different results — the subtype violates it.

    The word behavioral is doing heavy lifting here. It's not enough for the subtype to have the same method signatures (that's just structural subtyping). The subtype must exhibit the same observable behavior: same outputs for same inputs, same side effects, same exceptions (or fewer). The signatures are the easy part. The behavior is what trips people up.

    // Structural subtyping: matches the method signatures ✓ // Behavioral subtyping: matches the BEHAVIOR ✗ public class BaseLogger { public virtual void Log(string message) { File.AppendAllText("app.log", $"{DateTime.UtcNow}: {message}\n"); } } // Structurally valid — same signature // Behaviorally INVALID — different behavior! public class NullLogger : BaseLogger { public override void Log(string message) { // Does nothing. Callers expect a log entry to appear. // If a monitoring system reads the log file, it will // think no events happened. Silent data loss. } } // The fix: make NullLogger a separate implementation of ILogger // where the INTERFACE contract explicitly allows no-op logging public interface ILogger { /// <summary> /// Logs a message. Implementations MAY choose different destinations /// (file, console, /dev/null). Caller should not assume where /// the message goes — only that the call succeeds without throwing. /// </summary> void Log(string message); } // Now NullLogger is LSP-compliant because the interface // doesn't promise WHERE the log goes — only that it succeeds public class NullLogger : ILogger { public void Log(string message) { } // valid per contract } The key insight: The contract defines what's allowed. If the contract says "logs to a file," a no-op logger violates LSP. If the contract says "accepts a log message" (without specifying destination), a no-op logger is fine. The contract determines the boundary — not the implementation.

    Topic 2: Covariance and Contravariance Deep Dive

    C# uses two keywords on generic type parameters to control how types relate to each other in a hierarchy: out (covariant) and in (contravariant). These aren't just syntax — they're the compiler's way of proving that substitution is safe.

    Why does this matter? Because without variance, generic types are completely rigid. IList<Dog> has no relationship to IList<Animal> even though Dog inherits from Animal. Variance restores the inheritance relationship when it's safe to do so.

    The rule is simple once you see the pattern:

    • Producer (out) — only outputs T. Safe to use a more derived type. IEnumerable<out T> lets you treat IEnumerable<Dog> as IEnumerable<Animal> because you're only reading, never inserting.
    • Consumer (in) — only inputs T. Safe to use a less derived type. IComparer<in T> lets you use an IComparer<Animal> where IComparer<Dog> is needed because a comparer for all animals certainly works for dogs.
    • Both (no keyword) — reads and writes T. No substitution is safe. IList<T> is invariant because allowing substitution could insert the wrong type.
    // WHY IEnumerable<out T> is covariant: // The "out" keyword means T only appears in OUTPUT positions // (return types, out parameters). The compiler VERIFIES this. public interface IEnumerable<out T> { IEnumerator<T> GetEnumerator(); // T is output (return) // void Add(T item); // COMPILER ERROR! T in input position } // WHY IList<T> is invariant: // T appears in BOTH input and output positions public interface IList<T> { T this[int index] { get; set; } // T is both input AND output void Add(T item); // T is input bool Contains(T item); // T is input } // If IList were covariant, you could do this disaster: // IList<Animal> animals = new List<Dog>(); // hypothetical // animals.Add(new Cat()); // Inserting Cat into Dog list! // Dog d = ((List<Dog>)animals)[0]; // BOOM: ClassCastException // WHY IReadOnlyList<out T> IS covariant: // By removing all write methods, T only appears as output public interface IReadOnlyList<out T> { T this[int index] { get; } // T is output only int Count { get; } } // So this works perfectly: IReadOnlyList<Dog> dogs = new List<Dog> { new("Rex") }; IReadOnlyList<Animal> animals = dogs; // Safe! Read-only.

    The takeaway for LSP: When designing your own generic interfaces, ask yourself: "Does T only flow out?" If yes, mark it out. "Does T only flow in?" Mark it in. If T flows both ways, leave it invariant. The compiler then enforces LSP-safe substitution automatically — you get compile-time safety instead of runtime surprises.

    Topic 3: The History Constraint

    This is the least-known rule of LSP, and the one that catches even experienced developers off guard. The history constraintThe fourth and least-known LSP rule: a subtype must not allow state transitions that the base type doesn't allow. If the base type's state only grows (append-only), the subtype can't introduce deletion. If the base is immutable, the subtype can't add mutability. says: a subtype must not introduce new state modifications that the base type doesn't allow.

    In plain English: if the base type is immutable (its state never changes after creation), a subtype that adds a mutable field violates LSP. Why? Because callers who work with the base type assume immutability. They might cache the object, share it across threads, or use it as a dictionary key — all safe operations with immutable objects. A mutable subtype breaks these assumptions silently.

    // The base type is immutable — state set once, never changes public record Point(double X, double Y); // LSP VIOLATION: subtype adds mutability public class MutablePoint : Point { public MutablePoint(double x, double y) : base(x, y) { } // This setter lets callers change the point AFTER creation // But code working with Point assumes it's immutable! public new double X { get; set; } public new double Y { get; set; } } // Why this is dangerous: var cache = new Dictionary<Point, string>(); Point p = new MutablePoint(3, 4); cache[p] = "my point"; // Caller thinks Point is immutable, so this should always work: Console.WriteLine(cache[p]); // "my point" — works // But the subtype allows mutation: ((MutablePoint)p).X = 99; // hash code changes! Console.WriteLine(cache[p]); // KeyNotFoundException! // The point is still in the dictionary, but with the OLD hash. // It's corrupted. No exception told us. Silent data loss. // THE FIX: use sealed records public sealed record Point(double X, double Y); // "sealed" prevents subclassing entirely // No subtype can break the immutability guarantee

    Where this matters most: value objects, records, and anything used as a dictionary key or in a HashSet. If the base type is immutable, use sealed to prevent subtypes from introducing mutability. C# records are a great tool here — they're immutable by default, and sealing them closes the door on history-constraint violations.

    If your base type is immutable, make it sealed. If you can't seal it (because you need subclasses), document the immutability contract prominently and write contract tests that verify no subtype introduces mutability. Three deep-dive topics: (1) Behavioral subtyping means matching behavior, not just signatures — contracts define the boundary. (2) Covariance (out) and contravariance (in) are the compiler's LSP enforcement for generics — they prove substitution is safe at compile time. (3) The history constraint prevents subtypes from introducing state changes the base doesn't allow — seal immutable types to prevent this.

    You've inherited a codebase with NotSupportedException scattered across overrides, empty method bodies, and type-checking if (obj is SpecialType) everywhere. Here's how to find and fix LSP violations safely and incrementally — one pull request at a time.

    Before fixing anything, find out how widespread the problem is. Search your codebase for these red flags:

    # Red flag 1: NotImplementedException / NotSupportedException grep -rn "throw new NotImplementedException\|throw new NotSupportedException" --include="*.cs" src/ # Red flag 2: Empty method overrides (silent no-ops) grep -rn "override.*{[\s]*}" --include="*.cs" src/ # Red flag 3: Type-checking patterns grep -rn "is \w\+)" --include="*.cs" src/ grep -rn "as \w\+)" --include="*.cs" src/ # Red flag 4: Catch-all exception swallowing around polymorphic calls grep -rn "catch.*NotSupportedException" --include="*.cs" src/

    Risk: Low. You're only reading code and searching. No production changes.

    Look at the methods that some subtypes throw from. Group them by capability. Each group becomes an interface. The base class might have 10 methods — but they cluster into 3-4 capability groups.

    // The fat base class has Fly(), Swim(), Walk(), Eat(), Sleep() // Fly+Swim+Walk = capability groups. Eat+Sleep = shared by all. // Extract capability interfaces: public interface IAnimal { string Name { get; } void Eat(); void Sleep(); } public interface IFlyable { void Fly(); } public interface ISwimmable { void Swim(); } public interface IWalkable { void Walk(); } // Keep the old base class for now — it can implement all interfaces // to maintain backward compatibility during migration

    Risk: Low-Medium. You're adding new interfaces. Existing code doesn't change yet.

    This is the core migration step. Each subclass drops the base class and implements only the interfaces matching its real capabilities. One class at a time, one PR at a time.

    // BEFORE: Penguin extends Animal (has Fly() that throws) public class Penguin : Animal { public override void Fly() => throw new NotSupportedException(); // LSP violation! } // AFTER: Penguin implements only its real capabilities public class Penguin : IAnimal, ISwimmable, IWalkable { public string Name => "Penguin"; public void Eat() => Console.WriteLine("Penguin eats fish"); public void Sleep() => Console.WriteLine("Penguin naps"); public void Swim() => Console.WriteLine("Penguin swims!"); public void Walk() => Console.WriteLine("Penguin waddles!"); // No Fly() — no violation possible }

    Risk: Medium. You're changing class hierarchies. Run all tests after each class migration. If a caller was catching NotSupportedException from Penguin.Fly(), that code needs updating too (see Step 4).

    Change method signatures from the broad base type to the focused interfaces. This is where the compiler becomes your best friend — it will flag every caller that was assuming capabilities that don't exist.

    // BEFORE: takes Animal, hopes for the best void RunFlyingShow(Animal animal) { try { animal.Fly(); } catch (NotSupportedException) { /* swallow */ } } // AFTER: takes IFlyable, compiler guarantees it can fly void RunFlyingShow(IFlyable flyer) { flyer.Fly(); // always works, no try-catch needed } // If someone tries to pass a Penguin (which is ISwimmable, not IFlyable): // RunFlyingShow(penguin); // COMPILER ERROR! // The violation is caught at COMPILE TIME, not runtime. Risk: Medium-High. Changing method signatures may break compilation across the codebase. But that's the point — every compile error is a caller that was relying on an LSP violation. Fix each one, and your codebase becomes more robust. Deploy to staging first.     8 Generic constraints match actual behavioral requirements Over-constraining excludes valid types; under-constraining allows invalid ones where T : class when where T : ISerializable is what's actually needed
    9 Inheritance depth ≤ 3 levels Deep hierarchies make it harder to verify that all subtypes honor all contracts at every level Class hierarchy 4+ levels deep with overrides at each level
    10 Override methods don't add surprising side effects If base's Save() writes to DB, subtype shouldn't also send emails. Callers don't expect it. Override does what base does plus something callers don't know about
    Search for throw new NotImplementedException() and throw new NotSupportedException() in your codebase — each one is a potential LSP violation. Also search for if (x is patterns — type-checking around polymorphic calls is a strong signal that the hierarchy doesn't honor LSP. During code review, the two most important LSP checks are: (1) does any override throw NotSupportedException or do nothing? (2) are callers using is/as type checks before calling methods? If either answer is yes, the hierarchy needs splitting.

    Related Topics & What to Study Next

    LSP doesn't live alone — it works hand-in-hand with other SOLID principles and design patterns. Understanding how they reinforce each other gives you a complete toolkit for building robust, substitutable hierarchies.

    Separate Flying from Being a Bird

    The problem: Bird base class has a Fly() method, but Penguin can't fly. The inheritance hierarchy assumes all birds fly — a factual error baked into the class structure.

    The fix is to separate the capability (flying) from the taxonomy (being a bird). Create an IFlyingBird interface that extends IBird. Only birds that actually fly implement IFlyingBird. Penguin stays as IBird without any flying contract.

    IFlyingBird.cs
    // Base: all birds can do these
public interface IBird
{
    string Name { get; }
    void Eat();
}

// Extension: only for birds that ACTUALLY fly
public interface IFlyingBird : IBird
{
    void Fly();
    double MaxAltitude { get; }
}

// Eagle implements both — it really can fly
public class Eagle : IFlyingBird
{
    public string Name => "Eagle";
    public void Eat() => Console.WriteLine("Eagle hunts");
    public void Fly() => Console.WriteLine("Eagle soars!");
    public double MaxAltitude => 3000;
}

// Penguin: just IBird — no Fly() contract to violate
public class Penguin : IBird
{
    public string Name => "Penguin";
    public void Eat() => Console.WriteLine("Penguin eats fish");
    // No Fly(). No throw. No violation.
}

    Result: Code that needs flying birds takes IFlyingBird. Penguin can never be passed to it. The compiler enforces LSP automatically.

    Composition Over Inheritance

    Instead of putting Fly() in the class hierarchy at all, make flight a composable behavior. Any animal can have a flight capability, rather than being a flying animal. This is more flexible than inheritance because capabilities can be mixed and matched independently.

    FlightCapability.cs
    // Capability as a composable component
public interface IFlightCapability
{
    void Fly();
    double MaxAltitude { get; }
}

public class StandardFlight(double maxAlt) : IFlightCapability
{
    public void Fly() => Console.WriteLine("Flying!");
    public double MaxAltitude => maxAlt;
}

// Animals compose capabilities instead of inheriting them
public class Eagle : IAnimal
{
    public string Name => "Eagle";

    // Eagle HAS flight (composition), doesn't INHERIT it
    public IFlightCapability? Flight { get; } = new StandardFlight(3000);
}

public class Penguin : IAnimal
{
    public string Name => "Penguin";

    // Penguin has no flight capability — null, not throwing
    public IFlightCapability? Flight => null;
}

// Callers check for capability, never assume it
if (animal.Flight is { } flight)
    flight.Fly();

    Why composition wins: Adding a new capability (climbing, burrowing) means creating a new component class — no changes to existing animals. Each animal composes only the capabilities it actually has. No inheritance chain, no LSP risk.

    The Behavioral Test

    The most practical way to catch LSP violations is to write contract tests — a test suite defined once against the interface, then run against every implementation. If a subtype fails any contract test, it violates LSP.

    BirdContractTests.cs
    // Contract test: defined ONCE, run against ALL IFlyingBird impls
public abstract class IFlyingBirdContractTests
{
    protected abstract IFlyingBird CreateBird();

    [Fact]
    public void Fly_DoesNotThrow()
    {
        var bird = CreateBird();
        var ex = Record.Exception(() => bird.Fly());
        Assert.Null(ex); // must not throw
    }

    [Fact]
    public void MaxAltitude_IsPositive()
    {
        var bird = CreateBird();
        Assert.True(bird.MaxAltitude > 0,
            "Flying birds must have positive max altitude");
    }
}

// Every flying bird runs the SAME tests
public class EagleTests : IFlyingBirdContractTests
{
    protected override IFlyingBird CreateBird() => new Eagle();
}

public class SparrowTests : IFlyingBirdContractTests
{
    protected override IFlyingBird CreateBird() => new Sparrow();
}

// If Penguin were IFlyingBird (it shouldn't be!), it would FAIL:
// Fly_DoesNotThrow → FAILS (throws NotSupportedException)
// The contract test catches the violation automatically!

    Key insight: Contract tests are the automated enforcement of LSP. Every new implementation just needs one line: protected override IFlyingBird CreateBird() => new MyBird();. All contract tests run automatically. Violations are caught before code review.
