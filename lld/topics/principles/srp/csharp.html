Section 1

TL;DR

Most misunderstood SOLID principle: SRP does NOT mean "a class should do one thing." It means a class should have one reason to change — one actor, one stakeholder whose requirements drive modifications.

What: Every class should be responsible to exactly one actorA group of users or stakeholders who want the same kind of changes. For example, the accounting department is one actor — they all want the same financial reports. The HR department is a different actor. — one group of people whose requirements drive changes to that class.

Why: When a class serves multiple actors, a change requested by one actor can break functionality relied on by another. Separate responsibilities = isolated change = fewer bugs.

Modern .NET: ASP.NET Core's entire architecture embodies SRP — middlewareA component in the ASP.NET Core request pipeline that handles one cross-cutting concern (auth, logging, CORS, etc.). Each middleware is a separate class with a single responsibility. handles cross-cutting concerns, controllers handle routing, services handle business logic, repositories handle data access.

Quick Code:

public class OrderService
{
    public decimal CalculateTotal(Order order) => /* business logic */;
    public void SaveToDatabase(Order order) => /* SQL queries */;
}

// ✓ Each class has ONE reason to change
public class OrderCalculator           // Changes when: business rules change
{
    public decimal CalculateTotal(Order order) => /* business logic */;
}

public class OrderRepository           // Changes when: database schema changes
{
    public void Save(Order order) => /* SQL queries */;
}

Real-World Analogies

The Hospital Department Analogy

A hospital doesn't have one "Doctor of Everything" who performs surgery, reads X-rays, mixes prescriptions, and handles billing. Instead, each department has one responsibility — and when medical regulations change, only the affected department needs to update its procedures. The billing department doesn't care about new surgical techniques, and the radiology department doesn't care about insurance policy changes.

Core Principle & Visualization

Robert C. Martin originally stated SRP as "A class should have one, and only one, reason to change." In his 2017 book Clean Architecture, he refined it to: "A module should be responsible to one, and only one, actorAn actor is a group of users or stakeholders who want the system to change in the same way. The Key Insight

SRP isn't about limiting what a class does — it's about limiting who can cause it to change. A class that calculates payroll AND generates reports serves two actors (Accounting and HR). When Accounting changes their calculation rules, the report format shouldn't break.

Violation: The God Controller

This controller does everything — validation, business logic, data access, email sending, and PDF generation. Six different stakeholders could force changes to this one file.

OrderController.cs — ✗ SRP Violation 
// ✗ 6 responsibilities in ONE controller
[ApiController]
[Route("api/[controller]")]
public class OrderController(AppDbContext db) : ControllerBase
{
    // ✗ Responsibility 1: Input validation
    private bool ValidateOrder(OrderDto dto)
    {
        if (string.IsNullOrEmpty(dto.CustomerId)) return false;
        if (dto.Items.Count == 0) return false;
        if (dto.Items.Any(i => i.Quantity <= 0)) return false;
        return true;
    }

    // ✗ Responsibility 2: Price calculation (business logic)
    private decimal CalculateTotal(List<OrderItemDto> items)
    {
        decimal total = 0;
        foreach (var item in items)
        {
            var product = db.Products.Find(item.ProductId);
            total += product!.Price * item.Quantity;
        }
        if (total > 100) total *= 0.9m; // 10% discount
        return total;
    }

    // ✗ Responsibility 3: Persistence
    private Order SaveOrder(OrderDto dto, decimal total)
    {
        var order = new Order
        {
            CustomerId = dto.CustomerId,
            Total = total,
            CreatedAt = DateTime.UtcNow
        };
        db.Orders.Add(order);
        db.SaveChanges();
        return order;
    }

    // ✗ Responsibility 4: Email notification
    private void SendConfirmation(Order order)
    {
        var smtp = new SmtpClient("smtp.company.com");

            Body = $"Your total: {order.Total:C}"

        smtp.Send(msg);
    }

    private byte[] GenerateInvoice(Order order)
    {
        // 50 lines of PDF generation code...
        return Array.Empty<byte>();

        var order = SaveOrder(dto, total);

Proper SRP: One Class Per Responsibility

Each class now has exactly one actor who can force it to change. The controller just orchestrates — it delegates to specialists.

SRP-Refactored.cs 
// ── Responsibility 1: Validation (changes when: input rules change) ──
public class OrderValidator : AbstractValidator<OrderDto>
{
    public OrderValidator()
    {
        RuleFor(x => x.CustomerId).NotEmpty();
        RuleFor(x => x.Items).NotEmpty();
        RuleForEach(x => x.Items)
            .Must(i => i.Quantity > 0)
            .WithMessage("Quantity must be positive");
    }
}

// ── Responsibility 2: Pricing (changes when: business rules change) ──
public class OrderPricingService(IProductRepository products)
{
    public decimal CalculateTotal(IReadOnlyList<OrderItemDto> items)
    {
        decimal total = items.Sum(i =>
            products.GetPrice(i.ProductId) * i.Quantity);

        return total > 100 ? total * 0.9m : total;
    }
}

// ── Responsibility 3: Persistence (changes when: schema changes) ──
public class OrderRepository(AppDbContext db) : IOrderRepository
{
    public Order? GetById(int id) => db.Orders.Find(id);
    public void Save(Order order)
    {
        db.Orders.Add(order);
        db.SaveChanges();
    }
}

// ── Responsibility 4: Notification (changes when: email provider changes) ──
public class OrderNotificationService(IEmailSender sender) : IOrderNotifier
{
    public async Task SendConfirmation(Order order) =>
        await sender.SendAsync(order.CustomerEmail,
            $"Order {order.Id} Confirmed",
            $"Your total: {order.Total:C}");
}

// ── Controller: ONLY orchestration (changes when: API contract changes) ──
[ApiController, Route("api/orders")]
public class OrderController(
    OrderPricingService pricing,
    IOrderRepository repo,
    IOrderNotifier notifier) : ControllerBase
{
    [HttpPost]
    public async Task<IActionResult> Create(OrderDto dto)
    {
        var total = pricing.CalculateTotal(dto.Items);
        var order = new Order(dto.CustomerId, total);
        repo.Save(order);
        await notifier.SendConfirmation(order);
        return CreatedAtAction(nameof(Get), new { id = order.Id }, order);
    }

    [HttpGet("{id}")]
    public IActionResult Get(int id) => Ok(repo.GetById(id));
}

How a Junior Thinks

"It's all related to user registration, so it belongs in one class. I'll keep it organized with regions."

UserService.cs — Junior 
public class UserService
{
    private readonly string _connString;
    private readonly SmtpClient _smtp;

    public UserService(string connString)
    {
        _connString = connString;
        _smtp = new SmtpClient("smtp.company.com"); // ✗ hard-coded
    }

    #region Validation
    public bool ValidateEmail(string email) =>
        email.Contains('@') && email.Contains('.');

    public bool ValidatePassword(string pw) =>
        pw.Length >= 8;
    #endregion

    #region Security
    public string HashPassword(string pw) =>
        Convert.ToBase64String(
            System.Security.Cryptography.SHA256.HashData(
                System.Text.Encoding.UTF8.GetBytes(pw))); // ✗ SHA256 for passwords
    #endregion

    #region Database
    public void SaveUser(string email, string hash)
    {
        using var conn = new SqlConnection(_connString);
        conn.Open();
        using var cmd = conn.CreateCommand();
        cmd.CommandText = $"INSERT INTO Users VALUES ('{email}', '{hash}')"; // ✗ SQL injection!
        cmd.ExecuteNonQuery();
        Console.WriteLine($"Saved user {email}"); // ✗ console logging
    }
    #endregion

    #region Email
    public void SendWelcomeEmail(string email)
    {
        var msg = new MailMessage("no-reply@co.com", email)
        { Subject = "Welcome!", Body = "Thanks for signing up!" };
        _smtp.Send(msg); // ✗ synchronous, blocks thread
    }
    #endregion

    public void Register(string email, string password)
    {
        if (!ValidateEmail(email)) throw new Exception("Bad email");
        if (!ValidatePassword(password)) throw new Exception("Bad pw");
        var hash = HashPassword(password);
        SaveUser(email, hash);
        SendWelcomeEmail(email);
    }
}

Problems

5+ Responsibilities

Validation, hashing, persistence, email, and logging all live in one class. Any of these changing forces a recompile and retest of the entire service.

Untestable

Can't test password hashing without a database connection. Can't test registration without an SMTP server. Everything is coupled together.

Security Vulnerabilities

SHA256 for password hashing (should be bcrypt/Argon2), SQL injection via string interpolation, hard-coded SMTP — a security team's nightmare.

How a Senior Thinks

"Registration is a workflow, not a class. Each step is a separate responsibility — validation, hashing, persistence, notification. I'll create focused classes and wire them through DI."

RegistrationService.cs 
// Responsibility: Registration ORCHESTRATION only
// Changes when: registration workflow steps change
// Does NOT implement any step — just coordinates
public sealed class RegistrationService(
    IValidator<RegisterCommand> validator,
    IPasswordHasher hasher,
    IUserRepository users,
    IWelcomeNotifier notifier,
    ILogger<RegistrationService> logger)
{
    public async Task<Result<User>> RegisterAsync(RegisterCommand cmd)
    {
        var validation = validator.Validate(cmd);
        if (!validation.IsValid)
            return Result<User>.Fail(validation.Errors.First().ErrorMessage);

        if (await users.ExistsAsync(cmd.Email))
            return Result<User>.Fail("Email already registered");

        var user = new User
        {
            Email = cmd.Email,
            Name = cmd.Name,
            PasswordHash = hasher.Hash(cmd.Password),
            CreatedAt = DateTime.UtcNow
        };

        await users.AddAsync(user);
        await notifier.NotifyAsync(user);

        logger.LogInformation("User {Email} registered", user.Email);
        return Result<User>.Ok(user);
    }
}

Design Decisions

Interface Per Responsibility

Each interface represents exactly one capability (IPasswordHasher, IUserRepository, IWelcomeNotifier). Swap implementations without changing the orchestrator.

Orchestrator ≠ Implementor

RegistrationService doesn't hash, store, or email — it coordinates classes that do. If registration gains a step (like fraud check), only the orchestrator changes.

Testable in Isolation

Test hashing without a database. Test persistence without an email server. Mock any dependency independently. SRP = testability.

SRP in .NET Core

The .NET Core framework itself is a masterclass in SRP. Here's where the framework practices what we preach — each component has exactly one job.

Middleware Pipeline

Each middlewareA delegate in ASP.NET Core's request pipeline. Each middleware handles one concern (auth, CORS, compression, etc.) and either handles the request or passes it to the next middleware via next(). handles exactly one cross-cutting concern. Authentication doesn't know about CORS. Rate limiting doesn't know about compression. Each middleware has one reason to change — when its specific concern's requirements change.

MiddlewareSRP.cs 
// Each middleware = one responsibility
// Changes to auth rules don't affect CORS or compression

public class RequestTimingMiddleware(RequestDelegate next, ILogger<RequestTimingMiddleware> logger)
{
    public async Task InvokeAsync(HttpContext ctx)
    {
        var sw = Stopwatch.StartNew();
        await next(ctx);                     // Pass to next middleware
        sw.Stop();
        logger.LogInformation("{Method} {Path} completed in {Ms}ms",
            ctx.Request.Method, ctx.Request.Path, sw.ElapsedMilliseconds);
    }
}

// Registration — one line per concern:
app.UseMiddleware<RequestTimingMiddleware>();  // Timing only
app.UseAuthentication();                       // Auth only
app.UseAuthorization();                        // Authz only

Bug 1: Pricing Change Broke Email Templates

The Incident

2023, .NET 7 e-commerce platform. The pricing team updated discount calculation to return amounts in cents (integer) instead of dollars (decimal) for precision. After deployment, all order confirmation emails started showing "$0.45" instead of "$45.00" — the email template code was in the same service class and called a shared GetTotal() helper that now returned cents, but the email formatter still treated it as dollars.

Imagine you and your roommate share a single kitchen knife. You decide to sharpen it for cutting vegetables, but your roommate picks it up to spread butter and accidentally cuts through the bread. That's what happened here: two teams shared a single helper method, and when one team "sharpened" it for their needs, it cut the other team's work to pieces.

The pricing team had a good reason for the change. Working with integers (cents) instead of decimals (dollars) avoids floating-point rounding errors. A $45.99 item times 3 can give you $137.96999... in decimal math, but 4599 cents times 3 is always exactly 13797 cents. So they updated GetTotal() to multiply by 100 and return cents. Clean, correct, well-tested.

The problem? That same GetTotal() method was also called by BuildEmailBody(), written by the email team six months earlier. The email method used C#'s :C currency formatter, which assumed the number was in dollars. So when GetTotal() returned 4500 (cents), the email happily formatted it as "$4,500.00" instead of "$45.00". Every customer got a confirmation email showing 100x the actual price.

The truly painful part: it took 3 hours to diagnose because OrderService.cs was 800 lines long. The pricing team's PR only touched CalculateDiscountCents() and GetTotal(). No one realized that BuildEmailBody() also depended on GetTotal() because it was buried 400 lines below, invisible in the PR diff.

Time to Diagnose

3 hours — the connection between pricing logic and email formatting wasn't obvious because they were in the same 800-line OrderService.

OrderService.cs — Buggy 
public class OrderService
{
    // ✗ Shared helper used by BOTH pricing and email
    private decimal GetTotal(Order order) =>
        order.Items.Sum(i => i.Price * i.Qty) * 100; // Was dollars, pricing team changed to CENTS

    public int CalculateDiscountCents(Order order)
    {
        // Pricing team changed to cents for precision
        return (int)(GetTotal(order) * 0.1m);
    }

    public string BuildEmailBody(Order order)
    {
        // ✗ This broke — GetTotal now returns cents, but email still treats as dollars
        return $"Your total: {GetTotal(order):C}"; // Shows "$45.00" as "$4500.00"
    }
}

Walking through the buggy code: Look at GetTotal() on line 4. It used to just sum up prices. The pricing team added * 100 at the end to convert to cents. That's the only line that changed. But scroll down to BuildEmailBody() — it also calls GetTotal() and passes the result straight into C#'s currency formatter (:C). The formatter doesn't know the number is in cents. It just adds a dollar sign and decimal point. So 4500 cents becomes "$4,500.00" instead of "$45.00". Two different teams, two different assumptions about the same shared method.

SRP-Fix.cs 
// ✓ Pricing and email are separate classes — can't accidentally break each other
public class OrderPricingService
{
    public decimal CalculateDiscount(Order order) =>
        Math.Round(order.Total * 0.1m, 2);
}

public class OrderEmailService(IEmailSender sender)
{
    private static string FormatAmount(decimal amount) =>
        amount.ToString("C2", CultureInfo.InvariantCulture);

    public async Task SendConfirmation(Order order) =>
        await sender.SendAsync(order.Email,
            "Order Confirmed",
            $"Your total: {FormatAmount(order.Total)}");
}

Why the fix works: Now the pricing service and the email service are completely separate classes. They don't share any private methods. The pricing team can change how discounts are calculated — switching to cents, changing rounding rules, whatever they need. The email service has its own FormatAmount() method that reads order.Total directly from the Order model (which always stores dollars). Neither team can accidentally break the other's code, because they don't touch the same files.

How to Spot This in Your Code

Look for private helper methods that are called by multiple public methods serving different purposes. If you see a private method used by both "business logic" methods and "presentation/email" methods, that's invisible coupling waiting to break. The fix: each concern gets its own class with its own private helpers.

Lesson Learned

Shared private methods between responsibilities are invisible coupling. When pricing and email live in separate classes, a pricing change cannot break email formatting.

Bug 2: Logging Caused a Deadlock

The Incident

Friday 5:47 PM. Production API stopped responding. A service class that handled both order processing AND audit logging held an open database transactionA unit of work that groups multiple SQL operations. While a transaction is open, it holds locks on affected rows. If the code does slow work (HTTP calls, file I/O) inside the transaction, those locks are held for the entire duration — blocking other requests. while synchronously writing to the audit log table. Under load, the long-held row locks caused a cascade: concurrent requests queued up waiting for locks, exhausting the connection pool within seconds.

Think of a database transaction like borrowing a library study room. While you have the room, nobody else can use it. Now imagine you borrowed the room to write an essay (your main task), but you also decided to update the room's visitor logbook while still inside (your audit log). The logbook update takes extra time, and during that time, 10 other students are standing in line waiting for the room. Eventually they all give up and leave. That's a deadlock under load.

Here's what happened step by step. The OrderProcessor class opened a database transaction, updated the order status (locking the Orders row), then — while still holding those locks — also wrote an audit log entry to the AuditLogs table. Both writes used the same database context and the same transaction. Under normal traffic, this worked fine. Each request finished in about 50 milliseconds.

But on Friday at 5:47 PM, a flash sale drove traffic 20x higher than usual. Suddenly dozens of requests tried to process orders at the same time. Each one opened a transaction, locked an Orders row, and then tried to write to AuditLogs. But AuditLogs was already locked by other transactions doing the same thing. Requests started queuing. The connection poolA fixed-size pool of database connections (default: 100 in .NET). When all connections are busy, new requests wait. If they wait too long, the pool is "exhausted" and requests fail with a timeout error. Long-running transactions are the #1 cause. (capped at 100 connections by default) filled up within seconds. New requests got "connection pool exhausted" errors. The API returned 500 errors for every request.

The worst part: it was invisible in testing. Local development hits the database sequentially (one request at a time), so the locks never overlapped. Only under concurrent production load did the deadlock appear. The team spent 4 hours staring at logs before they realized the audit writes inside the transaction were the bottleneck.

Time to Diagnose

4 hours — the connection pool exhaustion only appeared under concurrent load. Local testing with sequential requests never triggered it.

OrderProcessor.cs — Deadlock 
public class OrderProcessor(AppDbContext db)
{
    public async Task ProcessOrder(Order order)
    {
        using var tx = await db.Database.BeginTransactionAsync();
        try
        {
            order.Status = OrderStatus.Processing;
            db.Orders.Update(order);       // ✗ Locks Orders row

            // ✗ SRP violation: audit logging in the same transaction
            db.AuditLogs.Add(new AuditLog   // ✗ Extra write inside open transaction
            {
                Action = "OrderProcessed",
                EntityId = order.Id,
                Timestamp = DateTime.UtcNow
            });

            await db.SaveChangesAsync();   // ✗ Transaction holds locks during both writes
            await tx.CommitAsync();        // ✗ Under load: locks held too long → connection pool exhaustion
        }
        catch { await tx.RollbackAsync(); throw; }
    }
}

Walking through the buggy code: Line 5 opens a transaction — from this point, every database write is wrapped in a "hold everything" lock. Line 8 updates the order status and locks that row. Then, instead of committing right away, line 11 also writes an audit log entry. Now the transaction holds locks on TWO tables. SaveChangesAsync() on line 18 pushes both writes to the database, but the transaction doesn't release its locks until CommitAsync() on line 19 completes. Under heavy load, that gap between "acquire lock" and "release lock" overlaps with other requests doing the same thing, and everyone ends up waiting for everyone else.

SeparatedConcerns.cs 
// ✓ Order processing and audit logging are separate classes with separate DbContexts
public class OrderProcessor(AppDbContext db)
{
    public async Task ProcessOrder(Order order)
    {
        order.Status = OrderStatus.Processing;
        db.Orders.Update(order);
        await db.SaveChangesAsync(); // Only locks Orders — no deadlock risk
    }
}

public class AuditLogger(AuditDbContext auditDb) : IAuditLogger
{
    public async Task LogAsync(string action, int entityId)
    {
        auditDb.AuditLogs.Add(new AuditLog
        {
            Action = action, EntityId = entityId, Timestamp = DateTime.UtcNow
        });
        await auditDb.SaveChangesAsync(); // Separate context, separate connection
    }
}

Why the fix works: Each class now has its own DbContext, which means its own database connection and its own transaction scope. The OrderProcessor locks the Orders row, writes, and immediately releases. The AuditLogger writes to AuditLogs on a completely separate connection. Even under 20x traffic, the order locks are held for milliseconds instead of the entire combined operation. The connection pool never fills up because each individual operation is fast.

How to Spot This in Your Code

Search for methods that call BeginTransactionAsync() and then do multiple unrelated writes inside that transaction. If the writes serve different purposes (business data + audit trail, order status + notification queue), they should use separate DbContext instances. Also watch for any SaveChangesAsync() call that's bundling work for different stakeholders into a single round-trip.

Lesson Learned

Mixing business logic with logging in the same class (and same transaction) creates invisible concurrency hazards. Separate responsibilities = separate transactions = no deadlocks.

Bug 3: Validation Change Broke the Happy Path

The Incident

2024, .NET 8 SaaS app. A junior developer added a new validation rule to the UserService.Register() method. The new rule used throw for invalid input — but the method also had a try/catch around the database save that swallowed exceptions. Valid users started getting "registration failed" errors because the new validation exception was caught by the persistence error handler.

Imagine a restaurant where the waiter takes your order AND cooks the food. The kitchen has a rule: "if anything goes wrong with the stove, throw away the dish and tell the customer it's a kitchen fire." Now the waiter adds a new rule: "reject orders without a drink." But when a customer orders food without a drink, the waiter throws an error — which gets caught by the kitchen's "stove fire" handler. The customer hears "sorry, kitchen fire" when the real problem was just a missing drink order. That's exactly what happened here.

The original Register() method was written by a senior developer who wrapped the database save in a try/catch to handle transient database errors gracefully. The catch block returned Result.Fail("Registration failed: DB error"). This worked perfectly for months — when the database was down or a unique constraint failed, users got a clean error message.

Then a junior developer was assigned to add email format validation and a minimum password length rule. They added two throw new ArgumentException() checks at the top of the same method. Their code was correct in isolation — if the email is empty, throw. If the password is too short, throw. But they didn't realize those throws would be caught by the catch (Exception ex) block below, which was designed for database errors. The catch block ate the validation exception and returned "DB error" to the user.

The bug was especially sneaky because it only affected users who submitted invalid data. Users with valid data registered fine. The team didn't notice until a customer emailed support saying "I keep getting a database error but I just want to sign up." The support team assumed the database was down and escalated to infrastructure — who found nothing wrong. Only after an hour of digging did they realize it was a validation exception being misreported as a database error.

Time to Diagnose

1 hour — but only after a customer reported they couldn't sign up. The catch block logged "DB error" even though it was a validation error.

UserService.cs — Buggy

{

    {

        if (string.IsNullOrEmpty(dto.Email))

        if (dto.Password.Length < 8)

Walking through the buggy code: Lines 6-9 are the new validation rules — they throw ArgumentException when input is bad. Lines 12-21 are the original try/catch wrapper around the database save. When line 7 throws ArgumentException("Email required"), execution jumps straight to line 18's catch block. The catch block has no idea it caught a validation error — it always returns "DB error". The original developer wrote the catch for DbUpdateException, but used the broad Exception base class. The junior developer had no way to know this trap existed.

{

    {
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
        RuleFor(x => x.Password).MinimumLength(8);
    }
}

            return Result.Fail(validation.Errors.First().ErrorMessage);

        await repo.AddAsync(new User { Email = dto.Email });
        return Result.Ok();
    }
}

Why the fix works: Validation is now a completely separate class (RegisterValidator) that uses FluentValidation. It returns validation results — it never throws exceptions. The repository (UserRepository) only deals with database operations, so only database exceptions can occur there. The orchestrator (RegistrationService) calls validation first: if invalid, it returns the validation error immediately and never reaches the database. There's no way for a validation failure to be misreported as a database error because the two concerns run in different classes at different times.

How to Spot This in Your Code

Look for methods that have both throw statements and try/catch blocks in the same method body. If the throw is for one concern (validation) and the catch is for another (persistence), that's a cross-contamination bug waiting to happen. Also watch for catch (Exception) with no type filtering — it catches everything indiscriminately.

Lesson Learned

When validation and persistence are in the same method, error handling becomes tangled. Separate them: validate first, persist second, each in its own class with its own error handling strategy.

Bug 4: Cache Invalidation Broke Reports

The Incident

2023, .NET 7 analytics platform. A DashboardService handled both data fetching and caching. The performance team changed the cache TTL from 1 hour to 24 hours and modified the LINQ query to use a faster projection — but they didn't realize the cache key included a date component that was now stale. Dashboard showed yesterday's numbers all day because the 24-hour cache was populated before midnight.

Think of caching like a sticky note on your fridge that says "we have milk." If you check the note once an hour, you'll know pretty quickly when the milk runs out. But if someone changes the rule to "check the note once a day," and the milk runs out in the afternoon, you'll think you have milk until tomorrow morning. That's what happened with this dashboard — the "sticky note" (cache) said "here are today's sales numbers," but "today" was actually yesterday.

The DashboardService was responsible for two completely different things: fetching sales data from the database (a data access concern) and caching that data in memory (a performance concern). The performance team was asked to improve response times. Their change was simple: bump the cache duration from 1 hour to 24 hours, and optimize the LINQ query with a faster projection. Both changes were correct in isolation.

But here's the trap. The cache key was just "dashboard-{userId}" — it didn't include a date. When the cache was set to 1 hour, this didn't matter much because the data refreshed 24 times a day. But at 24 hours, a user who viewed their dashboard at 11:30 PM would get that snapshot cached. The next morning at 9 AM, they'd still see last night's numbers. Sales that happened overnight were invisible. The dashboard looked frozen.

The bug went undetected for a full business day. The data looked real — it was actual sales numbers, just outdated. Nobody noticed until a user reported "my sales counter hasn't moved all day." The team checked the database: new sales were there. They checked the API: it returned old data. Only then did they trace it to the cache TTL change, hidden in a seemingly innocent performance PR.

Time to Diagnose

1 day — because the data looked correct (it was real data, just outdated). Only after a user reported "today's sales haven't moved" did the team investigate the cache.

        sp.GetRequiredService<DashboardRepository>(),

Why the fix works: The repository (DashboardRepository) only knows about SQL — it has zero awareness of caching. The cache decorator (CachedDashboardRepository) only knows about caching — it has zero awareness of SQL. Now the performance team can change cache TTL, cache keys, or even swap from in-memory to Redis without ever touching the data access code. And the data team can optimize the LINQ query without worrying about cache invalidation. Each concern lives in its own class, reviewed by the team that owns it.

How to Spot This in Your Code

Look for classes that inject both a data access dependency (DbContext, HttpClient) AND a caching dependency (IMemoryCache, IDistributedCache). That's two concerns in one class. The Decorator pattern lets you layer caching on top of data access without mixing them. Bonus: you can turn caching on/off just by changing DI registration — no code changes.

Use the Decorator patternA structural design pattern that wraps an object to add behavior without modifying it. Bug 5: Authorization Check Leaked Into Business Logic

The Incident

2 AM. Production. A multi-tenant SaaS app had authorization checks scattered throughout a ProjectService. When a new feature was added, the developer forgot to add the tenant check. Users from Tenant A could see Tenant B's projects. Data breach reported, incident response triggered.

Imagine an apartment building where every door has its own separate lock, and it's each tenant's responsibility to lock their own door. If someone installs a new door and forgets to add a lock, anyone in the building can walk right in. That's what happened here: every method in ProjectService was responsible for adding its own tenant check, and the new method simply forgot.

This was a multi-tenant SaaS application. "Multi-tenant" means multiple companies (tenants) share the same database, but each company should only see its own data. The existing GetAll() and GetById() methods both had a .Where(p => p.TenantId == currentUser.TenantId) filter. This worked, but the security check was duplicated in every method — it was the developer's job to remember to add it each time.

A developer was assigned to build a new project search feature. They wrote a perfectly functional Search() method with a clean LINQ query: db.Projects.Where(p => p.Name.Contains(query)). It passed all unit tests (which used a single-tenant test database). It passed code review (the reviewer focused on the search logic, not the missing filter). It shipped to production at 6 PM on a Friday.

At 2 AM, an automated security scanner flagged an anomaly: a user from Company A's search results included project names belonging to Company B. The incident response team was woken up. It took only 30 minutes to find the missing filter — but the real damage was the 2-week audit that followed. The legal team needed proof that no sensitive data was actually leaked across tenants. Every search query made between deployment and the fix had to be reviewed. The company had to notify affected customers under their data processing agreement.

Bug 6: File Upload Exhausted Server Memory

The Incident

2024, .NET 8 document management system. A DocumentService handled upload, virus scanning, thumbnail generation, and storage — all in one method. When a user uploaded a 500MB file, the method loaded it into memory for virus scanning, then kept it in memory while generating a thumbnail, then kept it in memory while uploading to blob storage. Peak memory: 1.5GB for one request. Three concurrent uploads crashed the server with OutOfMemoryException.

Think of moving into a new apartment. The wrong way: carry ALL your furniture to the front door, then carry ALL of it to the elevator, then carry ALL of it to your apartment — keeping everything in your hands the whole time. The right way: move each piece separately. Couch to apartment, done. Table to apartment, done. Each trip, you're only carrying one thing. That's the difference between buffering the entire file in memory versus streaming it through separate steps.

The DocumentService had four distinct jobs packed into a single method: scan for viruses, generate a thumbnail preview, upload to cloud storage, and save metadata to the database. On line 6, it read the entire file into a byte[] array. A 500MB file means 500MB of RAM consumed. So far, that's bad but survivable.

But here's where it gets catastrophic. The virus scanner held a reference to the original bytes array. The thumbnail generator created a second copy of the file data while processing it. The blob upload created a third copy when wrapping it in BinaryData. Peak memory for a single 500MB upload: approximately 1.5GB (500MB times 3 copies). The .NET garbage collector couldn't reclaim any of these copies because they were all referenced within the same method scope — the method hadn't finished yet, so nothing was eligible for garbage collection.

With the server's default 2GB memory limit for an ASP.NET Core process, just three concurrent 500MB uploads (3 x 1.5GB = 4.5GB) exceeded the memory limit instantly. The server threw OutOfMemoryException, crashed the process, and every in-flight request — not just uploads — failed. The monitoring dashboard showed a cliff: healthy, healthy, healthy, dead.

Time to Diagnose

2 hours — memory profiling showed the entire file buffered 3 times because each "responsibility" within the method held a reference.

DocumentService.cs — Memory Hog 
public class DocumentService(AppDbContext db, BlobServiceClient blob)
{
    public async Task Upload(IFormFile file)
    {
        // ✗ Load entire file into memory
        var bytes = new byte[file.Length];
        await file.OpenReadStream().ReadAsync(bytes);

        // ✗ Responsibility 1: Virus scan (holds bytes in memory)
        if (ContainsVirus(bytes))
            throw new InvalidOperationException("Virus detected");

        // ✗ Responsibility 3: Upload to blob (STILL holds bytes)
        var container = blob.GetBlobContainerClient("docs");
        await container.UploadBlobAsync(file.FileName, new BinaryData(bytes));

        // ✗ Responsibility 4: Save metadata
        db.Documents.Add(new Document { Name = file.FileName, Size = file.Length });
        await db.SaveChangesAsync();
        // Peak memory: file.Length × 3 (bytes + thumbnail + upload buffer)
    }
}

Walking through the buggy code: Line 6 allocates a byte[] the size of the entire file — for a 500MB file, that's 500MB of RAM. Line 7 reads the file into that array. Now lines 10-11 pass the full array to the virus scanner, which processes it (keeping the reference alive). Line 14 calls GenerateThumbnail(bytes), which internally creates more buffers for image processing. Line 18 wraps it in BinaryData(bytes) — yet another copy. All these variables (bytes, thumbnail, the blob upload buffer) are in scope simultaneously. The garbage collector cannot free any of them until the entire method completes. That's why three concurrent uploads total 4.5GB and crash the process.

// ✓ Each step is a separate class that streams data — no full-file buffering
public class VirusScanner : IVirusScanner
{
    public async Task<bool> IsSafeAsync(Stream fileStream)
    {
        // Scans in chunks — doesn't load entire file
        // Returns true if clean, false if virus detected
        // Resets stream position after scanning
    }
}

public class ThumbnailGenerator : IThumbnailGenerator
{
    public async Task<byte[]> GenerateAsync(Stream fileStream)
    {
        // Reads only first N bytes for thumbnail
        // Small memory footprint regardless of file size
    }
}

public class BlobUploader(BlobServiceClient blob) : IBlobUploader
{
    public async Task<string> UploadAsync(string name, Stream fileStream)
    {

// Orchestrator: passes streams, never buffers the full file
public class DocumentUploadService(
    IVirusScanner scanner,
    IThumbnailGenerator thumbs,
    IBlobUploader uploader,
    IDocumentRepository repo)
{
    public async Task UploadAsync(IFormFile file)
    {
        await using var stream = file.OpenReadStream();
        if (!await scanner.IsSafeAsync(stream)) throw new Exception("Virus!");
        stream.Position = 0;
        var thumb = await thumbs.GenerateAsync(stream);
        stream.Position = 0;
        var url = await uploader.UploadAsync(file.FileName, stream);
        await repo.AddAsync(new Document(file.FileName, file.Length, url, thumb));
    }
}

Why the fix works: Each class now works with a Stream instead of a byte[]. The virus scanner reads the stream in small chunks (64KB at a time), never holding the entire file. The thumbnail generator reads only the first few kilobytes (enough to decode image headers). The blob uploader streams directly to Azure — the data flows from disk to network without ever fully loading into RAM. The orchestrator resets the stream position between steps so each class can read from the beginning. Peak memory for a 500MB upload: roughly 1MB (a 64KB scan buffer + a thumbnail + stream overhead). That's 1,500 times less memory than the monolithic version.

How to Spot This in Your Code

Look for methods where a byte[] or large object is created early and then passed through multiple processing steps in the same method. If you see new byte[file.Length] followed by multiple operations that all reference the same array, that's a memory multiplication bug. The fix: separate each processing step into its own class that accepts a Stream, so each step can choose its own memory strategy (chunked reading, streaming, partial reads).

Lesson Learned

SRP doesn't just improve code organization — it improves resource management. When each step is a separate class, each can optimize memory independently (streaming vs buffering). A monolithic method holds everything in scope.

Section 13

Pitfalls & Anti-Patterns

1. "One Method Per Class" Extremism

Mistake: Interpreting SRP as "each class should have exactly one public method."

Why This Happens: Developers hear "single responsibility" and think "single method." It's a natural misunderstanding — the word "single" seems to point at quantity. But Robert Martin's actual principle is about actors (stakeholders), not method count. A class that calculates subtotals, applies discounts, and computes tax is serving one actor: the pricing team. All three methods change together when pricing rules change. Splitting them into three separate classes doesn't improve anything — it just scatters related logic across files, making the codebase harder to navigate and understand.

TooGranular.cs
// ✗ Absurd — these all serve the same actor and operate on the same data
public class OrderTotalCalculator { public decimal Calculate(Order o) => /* ... */; }
public class OrderDiscountApplier { public decimal Apply(Order o) => /* ... */; }
public class OrderTaxCalculator { public decimal Calculate(Order o) => /* ... */; }
public class OrderFinalizer { public Order Finalize(Order o) => /* ... */; }

// ✓ These all serve the same actor (pricing team) — keep them together
public class OrderPricingService
{
    public decimal CalculateSubtotal(Order o) => /* ... */;
    public decimal ApplyDiscount(Order o, string code) => /* ... */;
    public decimal CalculateTax(Order o) => /* ... */;
    public Order Finalize(Order o) => /* ... */;
}

Why Bad: Creates a "class explosion" — dozens of trivial classes that are harder to navigate than one cohesive class. SRP is about actors, not method count.

Fix: Group methods by actor. If all methods serve the same stakeholder and operate on the same data, they belong together. The test: "If the pricing rules change, which methods need to change?" If all four methods change together, they belong in one class.

2. #region as a Substitute for SRP

Mistake: Using C# #regionA C# preprocessor directive that collapses a block of code in the IDE. It's purely cosmetic — the compiler ignores it. If you need regions to navigate a file, the file is too large. blocks to "organize" responsibilities within a single class instead of actually separating them.

Why This Happens: Regions feel like a "quick fix" for messy code. You've got a 500-line class and it's hard to navigate, so you collapse the validation stuff into a region and the database stuff into another region. Now your IDE shows a clean outline — problem solved? Not really. It's like organizing a cluttered room by stuffing things into drawers. The room looks clean, but the mess is still there. The class still has multiple reasons to change, and regions are purely cosmetic — the compiler ignores them entirely.

RegionAbuse.cs

{

    // 50 lines of validation code
    #endregion

    #region Database Access   // ← This should be its own class
    // 100 lines of SQL code
    #endregion

    #region Email             // ← This should be its own class
    // 30 lines of email code
    #endregion

// If you need #regions to navigate a class, it's too big.

Why Bad: Regions hide the problem. The class still has multiple reasons to change. Regions are a code smell that screams "extract these into separate classes."

Fix: Each region becomes its own class. If your IDE needs regions to make a class readable, the class is violating SRP. A good rule of thumb: the moment you type #region, stop and ask "should this be a separate file instead?"

3. God Controller (Fat Controller)

Mistake: Putting business logic, validation, data access, and notification in controller action methodsMethods in an ASP.NET Core controller that handle HTTP requests. They should only translate HTTP → domain calls → HTTP responses. Business logic belongs in services, not controllers..

Why This Happens: Controllers are the first place you write code in an ASP.NET tutorial. "Hello World" always goes in the controller. So when deadlines are tight, it feels natural to keep adding logic there — "I'll refactor later." But "later" never comes. The controller grows from 10 lines to 100 to 500. Now it validates input, calculates prices, queries the database, sends emails, and returns responses. Every team on the project ends up editing the same controller file, causing merge conflicts and making deployments risky.

Why Bad: Controllers should be thin — they map HTTP requests to application commands. They shouldn't contain business logic.

4. Mixing Business Logic with Data Access

Mistake: A "service" class that contains both LINQ queries AND business rule calculations.

Why This Happens: When you're building a feature, the fastest path from "I need data" to "I need to process data" is a single class. You write a LINQ query, get the results, and immediately run calculations on them. It feels efficient — everything in one place, no extra files, no interfaces. But the problem shows up when you try to write unit tests: you can't test your pricing math without spinning up a database (or an in-memory database), because the business logic is tangled with the data access code.

5. Premature Splitting (YAGNI Violation)

Mistake: Creating IOrderValidator, IOrderPricer, IOrderSaver, IOrderNotifier on day one of an MVP when you have 3 users and one developer.

Why This Happens: You read Clean Architecture, watched a YouTube video about SOLID, and got excited. Now every feature starts with 5 interfaces, 5 implementations, and 5 DI registrations. The problem: SRP is a response to pain, not a prediction of pain. On day one of an MVP with one developer and three users, there's only one "actor" — you. Nobody is requesting conflicting changes. Splitting everything upfront adds complexity without any payoff. It's like buying a fire extinguisher for every room in a house you haven't built yet.

PrematureSplit.cs
// ✗ Day 1 MVP with 3 users — this is over-engineering
public interface IOrderValidator { Result Validate(OrderDto dto); }
public interface IOrderPricer { decimal Calculate(OrderDto dto); }
public interface IOrderSaver { Task Save(Order order); }
public interface IOrderNotifier { Task Notify(Order order); }
public interface IOrderAuditor { Task Audit(Order order); }
// ... 5 interfaces, 5 implementations, 5 DI registrations
// for a feature that has ONE developer and changes once a month

Why Bad: Adds navigation overhead, increases file count, and complicates debugging — all for hypothetical future changes that may never happen.

Fix: Start with a simple class. Extract when you feel the pain — when merge conflicts happen, tests become hard to write, or different teams need to change the same file. SRP is reactive, not proactive. Split because you have two teams stepping on each other, not because you might someday have two teams.

6. Helper/Utility Classes (Responsibility Dumping Grounds)

Mistake: Creating a Helpers or Utilities class that accumulates unrelated static methods.

Why This Happens: Every codebase has that one file. It starts innocently: someone needs a date formatter and doesn't know where to put it, so they create Helpers.cs. Then another developer adds a tax calculator. Then another adds an email validator. Six months later, Helpers is 2000 lines with 40 unrelated methods. It's the junk drawer of your codebase — everyone dumps things there because the name is so vague that anything fits. The real problem: "Helpers" is not a responsibility. It's an admission that you didn't know where the code belonged.

HelperHell.cs

Fix: Ask "who would change this method?" and move it to a class that serves that actor. If it's truly generic (string formatting), use extension methods in a StringExtensions class.

7. Static Methods Hiding Dependencies

Mistake: Using static methods for responsibilities that should be injectable (email, logging, file I/O).

Why This Happens: Static methods are convenient — you call EmailHelper.Send() from anywhere without wiring up DI. It feels faster than creating an interface, an implementation, and a DI registration. But convenience has a price: when you look at a class's constructor, you see its visible dependencies. Static calls are invisible dependencies. A class might look like it only depends on IOrderRepository, but deep inside a method it calls FileHelper.WriteLog() and CacheManager.Get(). Now the class secretly depends on the file system and a cache — you just can't see it from the outside.

HiddenDependencies.cs

        FileHelper.WriteLog($"Order {order.Id} placed");      // ✗ Hidden dependency!

    }

    IOrderRepository repo,
    IEmailSender email,
    ILogger<OrderService> logger)
{

    {

        await email.SendAsync(order.CustomerEmail, "Confirmed"); // ✓ Visible, mockable
        logger.LogInformation("Order {Id} placed", order.Id);    // ✓ Visible, mockable
    }

Why Bad: Static methods can't be mocked, can't be swapped via DI, and hide the dependency graph. The class appears to have one responsibility but secretly depends on file system, network, or database through static calls. In unit tests, the static calls will actually send emails or write files — there's no way to prevent it.

Fix: Wrap side effects in injectable interfaces. IEmailSender, IFileStorage, IClock — make dependencies explicit. If a class's constructor has too many parameters after this, that's SRP telling you the class does too much.

8. Confusing "Reason to Change" with "Reason to Edit"

Mistake: Thinking "if I need to fix a typo, that's a reason to change, so even comments violate SRP."

Why This Happens: The phrase "one reason to change" is vague enough to be taken too literally. If you interpret "reason to change" as "any reason I might edit this file," then formatting code, fixing typos, updating comments, and refactoring variable names are all "reasons to change." Taken to its extreme, even a class with one method would violate SRP because you might change the method's implementation AND its documentation. This interpretation leads to analysis paralysis — you can't write a class without worrying about SRP.

ReasonToChange.cs

// ✗ "I optimized a LINQ query" — same actor, same responsibility

Fix: Remember the refined definition: responsible to one actor. Cosmetic edits don't count. Only changes driven by different business stakeholders matter. The test: "Which team requested this change?" If the answer is "no team — I just cleaned up the code," it's not an SRP concern.

Mistake: Adding ILogger to every service and sprinkling log statements throughout business logic.

Why This Happens: Logging feels like a "good practice" that belongs everywhere. In tutorials, every class injects ILogger and logs every step. But when you step back and look at the result, your business logic is 30% actual logic and 70% log statements.         logger.LogInformation("Order {Id} saved successfully", order.Id);

// ✓ Use a Decorator or MediatR pipeline behavior for logging
public class LoggingBehavior<TReq, TRes>(ILogger<LoggingBehavior<TReq, TRes>> logger)
    : IPipelineBehavior<TReq, TRes> where TReq : IRequest<TRes>
{
    public async Task<TRes> Handle(TReq request, RequestHandlerDelegate<TRes> next, CancellationToken ct)
    {
        logger.LogInformation("Handling {Request}", typeof(TReq).Name);
        var result = await next();
        logger.LogInformation("Handled {Request}", typeof(TReq).Name);
        return result;
    }
}

Why Bad: Logging is a cross-cutting concernA concern that spans multiple modules (logging, auth, caching, transactions). Cross-cutting concerns should be handled in a single place (middleware, decorators, pipeline behaviors) — not scattered across every class.. Scattering it through business logic classes gives them a second reason to change (when log format/level changes).

Fix: Use middleware, decorators, or MediatR IPipelineBehavior for cross-cutting log concerns (request/response, timing, correlation IDs). Business-meaningful logs ("order placed", "payment failed") are fine inside services — they're part of the service's responsibility, not a separate concern.

10. "Manager" and "Handler" God Objects

Mistake: Naming a class OrderManager, UserHandler, or DataProcessor — vague names that attract every responsibility.

Why This Happens: When you're not sure what a class does (or it does too many things), you reach for a generic name. "Manager" can mean anything — create, update, delete, validate, notify, cache, and report. Because the name is vague, every developer feels comfortable adding their code to it. It's like a box labeled "stuff" — everyone puts things in, nobody takes things out. Over time, OrderManager becomes a 1000-line God class with 15 dependencies, and nobody knows what it's "managing" anymore. Specific names like OrderPlacer or ShippingCalculator set a boundary — if your method doesn't place orders, it doesn't go in OrderPlacer.

Interview Q&As

Easy

Q1: What does the Single Responsibility Principle say?
Easy
Think First Is it "do one thing" or something more specific?

Think of a chef in a restaurant. If the chef cooks the food, takes orders, washes dishes, and does the accounting, any one of those jobs changing would force the chef to adapt. If the health inspector changes kitchen rules, the chef has to change. If the accounting rules change, the chef also has to change. Too many different bosses pulling in different directions.

SRP states that a class should have one, and only one, reason to change. Robert Martin later refined this to: "A module should be responsible to one, and only one, actor." An "actor" is a group of stakeholders who want the same kind of changes. In our restaurant analogy, the chef should only cook — that way, only kitchen-related changes affect them.

It's commonly misquoted as "a class should do one thing" — but that's not the actual principle. A class can have multiple methods as long as they all serve the same actor. A TaxCalculator class can have CalculateSalesTax(), CalculateVAT(), and CalculateGST() — three methods, but one actor (the Finance team).

Great Answer Bonus "The refinement from 'reason to change' to 'actor' in Clean Architecture (2017) was important because 'reason to change' was too vague — it could mean anything. 'Actor' gives a concrete test: who would request this change?"
Q2: What's the difference between SRP and "do one thing"?
Easy
Think First Can a class with 10 methods follow SRP?

Imagine a toolbox. "Do one thing" would mean each toolbox should hold exactly one tool. SRP says something different: each toolbox should belong to one person (or team). A carpenter's toolbox can have a hammer, saw, measuring tape, and sandpaper — four tools, but they all serve the same craftsperson. That's fine. What's NOT fine is if the carpenter, electrician, and plumber all share one toolbox — now changing the electrical tools might bury the plumbing tools.

"Do one thing" is a function-level guideline from Clean Code — it applies to individual methods. SRP operates at the class level and is about who drives changes, not what the class does. A PayrollCalculator with methods for base pay, overtime, bonuses, and deductions has 4 methods but ONE actor (Accounting). All four methods change when payroll rules change.

Conversely, a class with one public method can violate SRP if that method internally mixes responsibilities serving different actors. A single ProcessOrder() method that validates, charges payment, sends email, and updates inventory serves four different teams.

Great Answer Bonus "Think of cohesion: if all methods operate on the same data for the same stakeholder, they're cohesive. 10 cohesive methods > 10 separate single-method classes."
Q3: Give a real .NET example where SRP is built into the framework.
Easy
Think First Where in ASP.NET Core do you see one-class-per-concern?

The middleware pipeline is a perfect example. Think of it like an airport security line — each checkpoint handles one thing. One person checks your boarding pass, another scans your bag, another does the metal detector. In ASP.NET Core, each middleware handles one cross-cutting concern: UseAuthentication() handles auth, UseCors() handles CORS, UseRateLimiting() handles rate limits. They're separate classes in separate files, each with one reason to change.

Another example: IOptions<T> separates configuration binding from consumption. Your service receives a typed options object — it doesn't know if it came from appsettings.json, environment variables, or Azure Key Vault. The configuration source is one class's responsibility; using the configuration is another's.

Great Answer Bonus "Even the DI container itself follows SRP — registration (what services exist) is separate from resolution (how to create them). You register in Program.cs, resolve via constructor injection."
Q4: How does SRP relate to testability?
Easy
Think First Why are God classes hard to test?

Think about testing a Swiss Army knife versus testing a single screwdriver. To test the Swiss Army knife, you need to set up scenarios for the blade, the can opener, the corkscrew, the scissors, and the file. To test the screwdriver, you just need a screw. SRP gives you screwdrivers — classes so focused that testing them is trivial.

SRP makes classes testable by limiting dependencies. A class with one responsibility typically has 1-2 dependencies, which are easy to mock. A class with 5 responsibilities needs 5+ dependencies — setting up tests becomes painful, and failures are hard to diagnose. When a test with 6 mocks fails, which dependency caused the failure? You have to investigate all 6.

The "mock count" smell: if you need 4+ mocks in a test, the class under test probably violates SRP. In a well-designed SRP system, most test files are shorter than the production code they test.

Great Answer Bonus "I've seen test files that are 3x longer than the production code — all because the class had so many dependencies to mock. After splitting by SRP, each test file was 20 lines."
Q5: What does "cohesion" mean and how does it relate to SRP?
Easy
Think First What makes methods "belong together" in a class?

Imagine a band where everyone plays the same genre — that's high cohesion. A jazz quartet where all four musicians play jazz, improvise together, and respond to each other's cues. Now imagine a "band" with a jazz pianist, a heavy metal drummer, a country singer, and a DJ. They share a stage, but they're not really playing together. That's low cohesion — technically one group, but practically four separate acts.

In code, cohesion measures how closely a class's methods relate to each other. High cohesion means all methods work on the same data toward the same purpose. Low cohesion means methods do unrelated things and share nothing — they're just roommates who happen to live in the same file.

SRP is the OOP principle that formalizes the idea of high cohesion. When a class follows SRP, its methods are naturally cohesive — they all serve the same actor and typically share the same instance fields. You can even measure this: the LCOM4 metric counts how many disconnected "groups" of methods exist in a class. LCOM4 = 1 means fully cohesive. LCOM4 = 3 means the class can be split into 3 separate classes.

Great Answer Bonus "Larry Constantine defined cohesion in the 1970s. SRP is essentially the OOP restatement of 'maximize cohesion' — Robert Martin added the 'actor' dimension to make it testable."
Q6: What's wrong with a class named "OrderManager"?
Easy
Think First What does the name "Manager" tell you about the class's responsibility?

Imagine a job title: "Office Manager." What do they do? Everything. They order supplies, schedule meetings, handle HR issues, fix the printer, and organize the holiday party. The title is so vague that any task can be dumped on them. That's exactly what happens with OrderManager in code.

The name "Manager" is too vague — it attracts every order-related responsibility: creation, cancellation, pricing, notification, reporting. Six months after creation, it's 800 lines long and every developer on the team has committed to it. It's a naming anti-pattern that signals an SRP violation waiting to happen (or already happening).

Better names describe the specific responsibility: OrderPlacer, OrderPricingService, OrderCanceller. These names set boundaries. If your code doesn't place orders, it doesn't belong in OrderPlacer. If you can't name the one thing the class does, it probably does too many things.

Great Answer Bonus "Same applies to 'Handler', 'Processor', 'Helper', 'Utility' — all magnets for unrelated code. A good class name should tell you both what it does AND what it doesn't do."
Q7: How does Dependency Injection enable SRP?
Easy
Think First What would SRP look like WITHOUT DI?

Imagine you're building with LEGO bricks, but every time you want to use a special piece, you have to go to the store, buy it, bring it home, and assemble it yourself. That's what coding without DI is like. Now imagine a magical bin that automatically provides exactly the right piece when you need it. That's DI — and it makes separating responsibilities painless.

Without DI, splitting a class into 5 focused classes means the consumer must manually create and wire all 5. You'd write new OrderService(new OrderRepository(new AppDbContext()), new EmailSender(new SmtpClient()), ...). That's painful enough that developers resist splitting — they'd rather keep one big class than manage 5 constructors with nested dependencies.

With .NET's DI container, you register each class once in Program.cs, and the container automatically injects them wherever they're needed. The "cost" of SRP drops to near-zero. Adding a new focused class is just one line of DI registration. That's why SRP and DI go hand in hand — DI removes the practical friction that makes SRP feel expensive.

Great Answer Bonus "DI is the reason SRP became practical. Before DI containers, SOLID principles were mostly academic. .NET Core's built-in container made SRP the default architecture."

Q8: How would you refactor a God class in an existing codebase?
Medium
Think First Can you refactor a 2000-line class in one PR?

You never refactor a God class in one big bang. Think of it like renovating a house while people are living in it — you do one room at a time, keeping everything functional between changes.

Step 1: Identify actors — who are the different stakeholders that cause changes to this class? Group methods by actor. Look at the constructor parameters as clues: DbContext = data access actor, IEmailSender = notification actor, IMemoryCache = performance actor.

Step 2: Extract one responsibility at a time. Start with the most painful (most merge conflicts, hardest to test). Create an interface + implementation. Inject the new class into the original. This is one PR — small, reviewable, reversible.

Step 3: Keep the original class as a facade that delegates to the extracted classes. This maintains backward compatibility: existing callers don't need to change. New code uses the focused classes directly.

Step 4: Over multiple PRs, migrate callers from the facade to the focused classes. Eventually delete the facade.

Great Answer Bonus "The Strangler Fig pattern works at the class level too. You don't rewrite — you gradually extract and redirect. Each PR is small, reviewable, and backward-compatible."
Q9: Is a controller with 10 endpoints violating SRP?
Medium

But if the controller has 10 endpoints AND each one contains 50 lines of business logic, validation, and data access — that's a violation. The fix isn't splitting into 10 controllers; it's extracting the logic into services.

Great Answer Bonus "Minimal APIs take the opposite approach — each endpoint is a separate delegate. Both are valid. The SRP question isn't 'how many endpoints?' but 'does the controller contain business logic?'"
Q10: How does SRP apply to EF Core's DbContext?
Medium
Think First DbContext has DbSets, configuration, and change tracking. Is that one responsibility or three?

PipelineBehavior.cs
// One behavior for ALL commands — validation concern separated from business logic
public class ValidationBehavior<TReq, TRes>(IEnumerable<IValidator<TReq>> validators)
    : IPipelineBehavior<TReq, TRes> where TReq : IRequest<TRes>
{
    public async Task<TRes> Handle(TReq request, RequestHandlerDelegate<TRes> next, CancellationToken ct)
    {
        var failures = validators
            .Select(v => v.Validate(request))
            .SelectMany(r => r.Errors)
            .Where(f => f is not null)
            .ToList();

        if (failures.Count > 0)
            throw new ValidationException(failures);

        return await next();
    }
}

Q20: How does SRP apply to domain events and eventual consistency?
Hard
Think First When an order is placed, should the order service send the email, update inventory, AND log the audit?

No. The order service raises a domain eventA notification that something important happened in the domain (e.g., OrderPlaced, PaymentReceived). Domain events decouple the trigger from the reactions — the order service doesn't know who listens. (OrderPlacedEvent). Separate handlers react independently: SendConfirmationHandler, UpdateInventoryHandler, AuditLogHandler. Each handler has one responsibility and can fail independently.

This is SRP at the workflow level — the order service doesn't know (or care) what happens after the order is placed. New reactions can be added without modifying the order service.

Great Answer Bonus "This is also the Open/Closed Principle in action — open for extension (add new handlers) without modifying the order service. SRP + OCP work together naturally."
Q21: Can SRP conflict with DRY? How do you resolve it?
Hard
Think First If two SRP-separated classes need the same calculation, do you duplicate it?

Yes, SRP and DRY can conflict. If PayCalculator and HourReporter both need to calculate regular hours, extracting that into a shared utility couples them — a change to the shared code affects both actors.

Resolution: Ask whether the duplication is coincidental or essential. If both actors calculate hours the same way today but might diverge tomorrow (Accounting counts overtime differently than HR), keep them separate. If they'll always be identical, extract to a shared value object.

Great Answer Bonus "Martin himself says: 'Beware of false duplication.' Two pieces of code that look the same but serve different actors aren't really duplicated — they'll evolve independently. Premature DRY creates the exact coupling SRP tries to prevent."
Q22: How would you design a multi-tenant system with SRP?
Hard

Q23: How does SRP relate to Clean Architecture's dependency rule?
Hard
Think First Why do inner layers define interfaces that outer layers implement?

Clean Architecture's dependency rule says inner layers (domain, application) define interfaces; outer layers (infrastructure, UI) implement them. This is SRP applied to layers:

  • Domain layer: Business rules only — changes when business rules change
  • Application layer: Use cases only — changes when workflows change
  • Infrastructure layer: External integrations — changes when providers change
  • Presentation layer: UI/API — changes when display requirements change

Each layer has one type of actor driving changes. The dependency rule ensures inner layers don't know about outer layers — SRP prevents framework details from contaminating business logic.

Great Answer Bonus "This is why domain classes don't reference EF Core, Dapper, or HttpClient. Those are infrastructure concerns. The domain defines IOrderRepository; infrastructure provides EfOrderRepository."
Q24: How do you balance SRP with performance in hot paths?
Hard
Think First What if the extra indirection from SRP hurts performance in a tight loop?

In 99% of cases, SRP overhead is unmeasurable (nanoseconds vs millisecond-level I/O). But in genuine hot paths (game loops, high-frequency trading, tight serialization loops), the extra virtual dispatch can matter.

Approach: Start with SRP. Profile. If (and only if) profiling shows the indirection is a bottleneck, optimize the hot path — but keep the SRP-compliant version as the "readable" implementation and mark the optimized version clearly.

Great Answer Bonus "In practice, I've never seen SRP class splitting be the bottleneck. It's always N+1 queries, unindexed columns, or synchronous I/O. Profile first, optimize second."

Q26: Can you explain the "Axis of Change" concept in SRP?
Hard
Think First How do you identify the different "directions" a class can change?

Each actor represents an "axis of changeA direction along which requirements evolve independently. The CFO changes pay rules, the COO changes reports. These are orthogonal axes — a change along one axis should not require changes along the other." — a direction along which requirements evolve. An Employee class with pay calculation and report generation has two axes: business rules axis (CFO) and presentation axis (COO). These axes are orthogonal — changes along one shouldn't affect the other.

When you identify multiple axes, split the class so each resulting class has only one axis. Changes become linear (one class, one direction) instead of cross-cutting (one change, multiple affected areas).

Great Answer Bonus "Think of it geometrically: a class with two axes of change is a 2D surface — any point on it can change. A class with one axis is a line — changes are predictable and localized. SRP reduces dimensionality."

LCOM4: Measuring Cohesion

LCOM (Lack of Cohesion of Methods)A metric that measures how related a class's methods are. LCOM4 builds a graph: each method is a node, and methods are connected if they share instance fields. The number of connected components = LCOM4. An LCOM4 of 1 means the class is cohesive (good). Higher means it can likely be split. measures how well a class's methods relate to each other. LCOM4 builds a graph of methods connected by shared fields. If the graph has multiple disconnected components, the class can be split.

LCOM4Example.cs
// LCOM4 = 2 (two disconnected groups → can be split into 2 classes)
public class OrderService
{
    private readonly IOrderRepo _repo;          // Group A
    private readonly IEmailSender _email;       // Group B

    // Group A: uses _repo
    public Order GetOrder(int id) => _repo.Get(id);
    public void SaveOrder(Order o) => _repo.Save(o);

    // Group B: uses _email (DISCONNECTED from Group A!)
    public void SendReceipt(Order o) => _email.Send(o);
    public void SendReminder(Order o) => _email.Send(o);
}

        var lines = order.Items.Select(i => new InvoiceLine(
            i.ProductName, i.Quantity, i.Price, i.Quantity * i.Price
        )).ToList();

        var subtotal = lines.Sum(l => l.Total);
        var tax = taxCalc.Calculate(subtotal);

        return new Invoice
        {
            Number = GenerateNumber(),
            Lines = lines,
            Subtotal = subtotal, Tax = tax, Total = subtotal + tax,
            CreatedAt = DateTime.UtcNow
        };
    }

    private static string GenerateNumber() =>
        $"INV-{DateTime.UtcNow:yyyyMMdd}-{Guid.NewGuid().ToString("N")[..8]}";
}

Section 24

Related Topics

Open/Closed Principle

When classes follow SRP, extending behavior means adding new classes (open for extension) without modifying existing ones (closed for modification).

Interface Segregation

SRP's sibling for interfaces. Split fat interfaces into focused ones — clients should only depend on methods they actually use.

Strategy Pattern

The pattern that implements SRP for varying algorithms. Extract each strategy into its own class — swap without modifying the context.

Decorator Pattern

Add responsibilities (caching, logging, retry) without modifying existing classes. Each decorator has one responsibility.

Observer Pattern

Decouple event producers from consumers. The publisher doesn't know about email, inventory, or analytics — each observer has its own responsibility.

Clean Architecture

SRP applied at the layer level. Domain, Application, Infrastructure, and Presentation each have one type of actor driving changes.

Senior Solution: Extract Responsibilities

Create one class per responsibility. Each class has one reason to change. The original class becomes a thin orchestrator that delegates to specialists via constructor injection.

// Each class = one reason to change
public interface IPasswordHasher { string Hash(string pw); }
public interface IUserRepository { Task AddAsync(User user); }
public interface IWelcomeNotifier { Task NotifyAsync(User user); }

// Orchestrator: only coordinates, doesn't implement
public class RegistrationService(
    IPasswordHasher hasher,
    IUserRepository users,
    IWelcomeNotifier notifier)
{
    public async Task RegisterAsync(string email, string pw)
    {
        var user = new User(email, hasher.Hash(pw));
        await users.AddAsync(user);
        await notifier.NotifyAsync(user);
    }
}

Senior Solution: Interface-Based Testing

When each responsibility is behind an interface, you can mock any dependency. Test the hasher without a database. Test the repository without an email server. Test the orchestrator with all mocks.

[Fact]
public async Task Register_HashesPassword_AndSavesUser()
{
    // Arrange — mock each responsibility independently
    var hasher = Substitute.For<IPasswordHasher>();
    hasher.Hash("secret").Returns("hashed_secret");

    var repo = Substitute.For<IUserRepository>();
    var notifier = Substitute.For<IWelcomeNotifier>();

    var sut = new RegistrationService(hasher, repo, notifier);

    // Act
    await sut.RegisterAsync("test@co.com", "secret");

    // Assert — each mock verifiable in isolation
    hasher.Received(1).Hash("secret");
    await repo.Received(1).AddAsync(Arg.Is<User>(u =>
        u.Email == "test@co.com" && u.PasswordHash == "hashed_secret"));

}
